# Структуры данных

Структура данных — это способ организации и хранения данных так, чтобы над ними можно было эффективно выполнять нужные операции. Правильный выбор DS решает большую часть прикладных задач по производительности.

> Здесь мы будем говорить больше про теоретическую составляющую и общие концепции, больше кода будет в следующих страницах.

---

## Интерфейс vs Реализация

Когда мы говорим "структура данных", у неё есть две стороны:

<b class="text-on-semi-light">Интерфейс (API)</b> — какие операции доступны. Это "что мы можем делать". Например:

- `insert(x)` — вставить элемент
- `delete(x)` — удалить элемент
- `lookup(x)` — найти элемент
- `iterate()` — пройтись по всем
- `rangeQuery(a, b)` — найти все элементы в диапазоне
- `merge(A, B)` — объединить две структуры

→ это "снаружи": как разработчик будет использовать структуру.

<b class="text-on-semi-light">Реализация (implementation)</b> — как эти операции устроены внутри.
Например:

- `insert` в массиве = дописать в конец (`O(1)`)
- `insert` в отсортированном массиве = найти позицию и сдвинуть элементы (`O(n)`)
- `lookup` в хеш-таблице ≈ `O(1)`
- `lookup` в связном списке = `O(n)`

→ это "под капотом": какие механизмы, указатели, хеши или балансировки деревьев.

## Массивы

<b class="text-on-semi-light">Массив (Array)</b> — это структура данных, которая используется для хранения множества элементов (значений), обычно одного типа, в одном объекте. Массивы позволяют хранить и упорядочивать данные, чтобы с ними можно было удобно работать: получать, изменять, удалять элементы, а также выполнять другие операции, такие как сортировка и фильтрация.

**Основные характеристики массива:**

1. <b class="text-on-semi-light">Порядок:</b> Элементы массива сохраняют порядок, в котором они были добавлены. То есть можно обращаться к элементам по их позиции (индексу).

2. <b class="text-on-semi-light">Индекс:</b> Каждый элемент массива имеет уникальный индекс. Индексы обычно начинаются с 0, т.е. первый элемент массива имеет индекс 0, второй — индекс 1, и так далее.

3. <b class="text-on-semi-light">Тип данных:</b> В зависимости от языка программирования, массивы могут содержать элементы одного типа (например, только числа или строки), хотя в некоторых языках, таких как JavaScript, элементы массива могут быть разных типов.

4. <b class="text-on-semi-light">Размер:</b> В большинстве языков массивы имеют фиксированный размер (например, C, C++), но в некоторых языках (например, в JavaScript, Python) размер массива может изменяться динамически.

Массив в JavaScript — это объект, содержащий набор значений. Вы можете работать с ним как с обычным объектом, но с добавлением возможности индексации элементов. Более подробно работу с массивами в JS мы разберем в <NuxtLink to="/content/javascript">7. JavaScript</NuxtLink>

```js
let arr = [1, 2, 3, 4, 5];
console.log(arr[0]); // 1 — доступ к первому элементу массива
arr[2] = 10; // изменение третьего элемента
console.log(arr); // [1, 2, 10, 4, 5]
```

В Python массивы называются списки (list). Они тоже могут содержать элементы разных типов и динамически изменять размер.

```py
arr = [1, 2, 3, 4, 5]
print(arr[0])  # 1 — доступ к первому элементу массива
arr[2] = 10  # изменение третьего элемента
print(arr)  # [1, 2, 10, 4, 5]
```

В C массивы — это наборы данных фиксированного размера, где все элементы должны быть одного типа.

```c
int arr[5] = {1, 2, 3, 4, 5};
printf("%d", arr[0]); // 1 — доступ к первому элементу массива
arr[2] = 10;  // изменение третьего элемента
```

### Статический массив

Статический массив: фиксированный размер, более быстрый доступ к данным, но ограничен в гибкости.

Адрес элемента `i`: `base + i * sizeof(element)`.

```
address(i) = base + i * sizeof(element)
```

**Операции и сложность:**

- Доступ по индексу `A[i]` — `O(1)`.
- Перебор — `O(n)`.
- Вставка/удаление в середине — `O(n)` (сдвиг элементов).

**Преимущества:**

- Отличная локальность (cache-friendly).
- Низкие накладные расходы.

**Недостатки:**

- Фиксированный размер.
- Дорогие вставки/удаления в середине.

**Пример (C):**

```c
#include <stdio.h>

int main() {
    int arr[5]; // статический массив из 5 элементов
    arr[0] = 42; // доступ по индексу O(1)
    arr[1] = 13;

    for (int i = 0; i < 5; i++) {
        printf("%d ", arr[i]); // перебор O(n)
    }
}
```

**Пример (JavaScript — динамический, но похожий синтаксис):**

```js
// В JS массивы динамические, статического нет
const arr = new Array(5); // выделена емкость 5, но массив можно расширять
arr[0] = 42;
arr[1] = 13;

console.log(arr[0]); // 42
```

### Динамический массив

Динамический массив: гибкий размер, требует больше времени для управления памятью и может быть медленнее, но позволяет работать с изменяемыми объемами данных.

**Идея.** Храним `data[]` и два числа:

- `size` — сколько элементов реально хранится
- `capacity` — сколько памяти выделено под массив

При переполнении выделяется новый блок памяти (обычно в ×2 раза больше), и все элементы копируются.

**Операции и сложность:**

- `append(x)` — амортизированно `O(1)`
- `pop()` (с конца) — `O(1)`
- Доступ по индексу — `O(1)`
- Вставка/удаление в середине — `O(n)` (сдвиг элементов)

**Амортизированный анализ (doubling):**

- Начинаем с `capacity = 1`.
- При заполнении массива: capacity удваивается → 1, 2, 4, 8, ...
- Каждое удвоение требует скопировать все элементы.
- Всего перемещений до `n` элементов < `2n`.
- Следовательно: суммарная стоимость `O(n)`, средняя стоимость `append` — `O(1)`.

**Почему именно ×2?**

- Умножение на константу >1 даёт логарифмическое количество дорогих операций ресайза.
- В Java `ArrayList` использует коэффициент ≈1.5, чтобы экономить память.

**Псевдокод `append`:**

```text
function append(arr, x):
  if arr.size < arr.capacity:
    arr.data[arr.size] = x
    arr.size += 1
  else:
    new_capacity = max(1, arr.capacity * 2)
    new_data = allocate(new_capacity)
    copy(arr.data -> new_data)
    new_data[arr.size] = x
    arr.data = new_data
    arr.capacity = new_capacity
    arr.size += 1
```

**Пример (JavaScript):**

```js
// В JS массивы динамические изначально
const a = [];
for (let i = 0; i < 1000; i++) {
  a.push(i); // амортизированно O(1)
}
```

**Практические замечания:**

Если заранее знаешь размер, вызывай `reserve(n)` (C++) или задавай `length = n` (JS) → меньше копирований.

Для больших объектов часто выгоднее хранить ссылки/указатели, чтобы удешевить копирование при ресайзе. Но это ухудшает локальность (cache-friendliness).

**Кэш-френдлиность (Cache friendliness):**

**Суть.** CPU читает память блоками (cache lines, обычно 64 байта). Последовательный доступ к массиву загружает соседние элементы в кеш сразу.

**Последствия:**

- Проход по массиву быстрее, чем по связному списку с такими же асимптотиками, т.к. pointer-chasing вынуждает читать много разных блоков.
- Алгоритмы, оптимизированные для локальности (т.е. читают данные в порядке расположения в памяти), часто быстрее на практике даже при одинаковой асимптотике.

**Пример:** summation через `for i from 0 to n` vs `for each node in linked list` — массив обычно быстрее в несколько раз.

### Многомерные массивы

**Многомерный массив** — это массив, элементы которого сами являются массивами. Наиболее распространённый случай — **двумерные массивы**, которые можно рассматривать как матрицы или таблицы. По сути — это структуры для хранения данных в виде сетки (grid), куба или N-мерного пространства.

**Идея.**

- 1D массив: доступ по одному индексу `A[i]`.
- 2D массив: доступ по паре индексов `A[i][j]`.
- 3D массив: доступ по тройке `A[i][j][k]`.
- И так далее для N-мерного случая.

Под капотом многомерные массивы обычно сводятся к **одномерному блоку памяти** с вычислением смещения (offset) по формуле.

**Формула адресации (row-major order, как в C/C++):**

```
address(A[i][j]) = base + (i * cols + j) * sizeof(element)
```

- `i` — номер строки
- `j` — номер столбца
- `cols` — количество столбцов
- `base` — адрес первого элемента

В **column-major order** (как в Fortran, MATLAB):

```
address(A[i][j]) = base + (j * rows + i) * sizeof(element)
```

**Операции и сложность**

- Доступ `A[i][j]` — `O(1)`.
- Перебор всех элементов — `O(n * m)` для матрицы `n × m`.
- Вставка/удаление строки или столбца — `O(n * m)` (дорогая операция, нужно перераспределять память).

**Пример (JavaScript):**

```js
// двумерный массив (матрица 3x3)
const matrix = [
  [1, 2, 3],
  [4, 5, 6],
  [7, 8, 9],
];

console.log(matrix[1][2]); // 6

// проход по всем элементам
for (let i = 0; i < matrix.length; i++) {
  for (let j = 0; j < matrix[i].length; j++) {
    console.log(matrix[i][j]);
  }
}
```

**Пример (визуализация):**

```
Массив 2x3:

[ [1, 2, 3],
  [4, 5, 6] ]

A[0][2] = 3
A[1][0] = 4
```

**Практические замечания:**

1. **Локальность доступа.** Если обходить массив построчно (в row-major памяти), доступ будет кэш-дружелюбным. Если постолбцово — хуже.
2. **Разреженные матрицы.** Если большинство элементов = 0, то многомерный массив неэффективен по памяти → используют специальные структуры (CSR, CSC, COO).
3. **Dynamically jagged arrays.** В некоторых языках (Java, JS, Python) «многомерный массив» — это массив массивов, и длины вложенных могут отличаться. Это даёт гибкость, но может ломать предположения об эффективности.

**Применение:**

- Хранение изображений (пиксели как матрица `height × width`).
- Таблицы и матрицы в линейной алгебре.
- Тензоры (N-мерные массивы) в ML/AI.
- Игровые поля (например, шахматы, клеточные автоматы).

**Заметки:**

- При работе с **большими матрицами** важно учитывать layout (row-major vs column-major) и порядок доступа.
- В высокопроизводительных вычислениях (HPC, ML) используют **блочные структуры**, чтобы уменьшить cache misses.
- Для GPU/тензорных библиотек (NumPy, PyTorch) ключевой аспект — **stride**: смещение между соседними элементами по измерениям.
- Иногда выгодно хранить матрицу в **разреженном формате**, если ненулевых значений мало (<10%).

## Связные списки

<b class="text-on-semi-light">Связный список</b> — структура, где каждый элемент хранит ссылку(и) на соседей. Полезны для быстрых вставок/удалений при известной позиции.

Каждый элемент хранит:

- **Данные:** информация, которую хранит узел (например, число, строка, объект).
- **Ссылка (или указатель):** ссылка на следующий узел в списке (для однонаправленного связного списка) или на следующий и предыдущий узлы (для двусвязного связного списка).

JavaScript не имеет встроенной структуры для связных списков, поэтому их обычно реализуют вручную. Вот пример односвязного списка:

```js
class Node {
  constructor(data) {
    this.data = data; // Данные
    this.next = null; // Ссылка на следующий узел
  }
}

class LinkedList {
  constructor() {
    this.head = null; // Начало списка
  }

  // Добавление в конец списка
  append(data) {
    const newNode = new Node(data);
    if (this.head === null) {
      this.head = newNode;
    } else {
      let current = this.head;
      while (current.next !== null) {
        current = current.next;
      }
      current.next = newNode;
    }
  }

  // Вывод списка
  printList() {
    let current = this.head;
    while (current !== null) {
      console.log(current.data);
      current = current.next;
    }
  }
}

// Пример использования
let list = new LinkedList();
list.append(1);
list.append(2);
list.append(3);
list.printList(); // Выведет: 1 2 3
```

В Python можно реализовать связный список с помощью классов:

```py
class Node:
    def __init__(self, data):
        self.data = data  # Данные
        self.next = None  # Ссылка на следующий узел

class LinkedList:
    def __init__(self):
        self.head = None  # Начало списка

    # Добавление в конец списка
    def append(self, data):
        new_node = Node(data)
        if self.head is None:
            self.head = new_node
        else:
            current = self.head
            while current.next:
                current = current.next
            current.next = new_node

    # Вывод списка
    def print_list(self):
        current = self.head
        while current:
            print(current.data)
            current = current.next

# Пример использования
ll = LinkedList()
ll.append(1)
ll.append(2)
ll.append(3)
ll.print_list()  # Выведет: 1 2 3
```

В C++ можно использовать указатели для реализации связных списков:

```c
#include <iostream>
using namespace std;

class Node {
public:
    int data;
    Node* next;

    Node(int val) {
        data = val;
        next = nullptr;
    }
};

class LinkedList {
public:
    Node* head;

    LinkedList() {
        head = nullptr;
    }

    // Добавление в конец списка
    void append(int data) {
        Node* newNode = new Node(data);
        if (head == nullptr) {
            head = newNode;
        } else {
            Node* temp = head;
            while (temp->next != nullptr) {
                temp = temp->next;
            }
            temp->next = newNode;
        }
    }

    // Вывод списка
    void printList() {
        Node* temp = head;
        while (temp != nullptr) {
            cout << temp->data << " ";
            temp = temp->next;
        }
        cout << endl;
    }
};

// Пример использования
int main() {
    LinkedList list;
    list.append(1);
    list.append(2);
    list.append(3);
    list.printList();  // Выведет: 1 2 3
    return 0;
}
```

### Односвязный список (Singly Linked List)

Односвязный список — это последовательность элементов (узлов), где каждый узел хранит:

- **value** — данные узла
- **next** — указатель (ссылку) на следующий узел

Узел **не хранит ссылку на предыдущий**, поэтому можно двигаться только вперёд. Это делает операции добавления и удаления быстрыми в начале списка, но поиск по индексу — медленным.

**Структура узла:** `(value, next)`

#### Операции и сложность:

- `insert_after(node, x)` → `O(1)`
- `delete_after(node)` → `O(1)`
- `find(value)` → `O(n)`
- `append` (без `tail`) → `O(n)`, с `tail` → `O(1)`
- `iterate` → `O(n)`

#### Псевдокод (JS):

```js
class Node {
  constructor(val) {
    this.val = val;
    this.next = null;
  }
}

class SinglyLinkedList {
  constructor() {
    this.head = null;
    this.tail = null;
  }

  append(val) {
    const node = new Node(val);
    if (!this.head) {
      this.head = this.tail = node;
    } else {
      this.tail.next = node;
      this.tail = node;
    }
  }

  find(val) {
    let curr = this.head;
    while (curr) {
      if (curr.val === val) return curr;
      curr = curr.next;
    }
    return null;
  }
}
```

#### Плюсы:

- Простая реализация.
- Быстрая вставка/удаление при известной позиции (`O(1)`).
- Хорош для реализаций **стеков**, **очередей** и простых динамических коллекций.

#### Минусы:

- Плохая локальность данных (каждый узел в памяти может находиться далеко → хуже для CPU cache).
- Нет быстрого доступа по индексу (`O(n)`).
- Дополнительная память на указатели.

#### Заметки:

- **Cache-unfriendly**: в отличие от массивов, данные «разбросаны» в памяти.
- **GC pressure (в JVM/JS/Python)**: каждый узел — отдельный объект, что создаёт нагрузку на сборщик мусора.
- **Хорош в теориях, плох на практике**: в реальных продакшен-системах чаще используют массивы/ArrayList из-за кэш-френдлиности.
- **Persistent linked lists**: в функциональных языках (Clojure, Haskell) часто используют неизменяемые списки, где при добавлении создаётся новый «головной» узел, а старая версия остаётся доступной.

### Двусвязный список (Doubly Linked List)

В двусвязном списке каждый элемент (или узел) содержит три поля:

- Данные (или значение)

- Ссылка на следующий элемент

- Ссылка на предыдущий элемент

Каждый узел ссылается как на следующий, так и на предыдущий узел, что позволяет двигаться в обе стороны (вперед и назад). Это упрощает удаление узлов и вставку в середине списка, но требует больше памяти для хранения дополнительных ссылок.

Структура: `(prev, value, next)`.

#### Операции и сложность:

- `insert_after(node, x)` — `O(1)`.
- `insert_before(node, x)` — `O(1)`.
- `delete(node)` — `O(1)`.
- `find(value)` — `O(n)`.
- `append/prepend` — `O(1)`.

#### Псевдокод (JS):

```js
class Node {
  constructor(val) {
    this.val = val;
    this.next = null;
    this.prev = null;
  }
}

class DoublyLinkedList {
  constructor() {
    this.head = null;
    this.tail = null;
  }

  append(val) {
    const node = new Node(val);
    if (!this.head) {
      this.head = this.tail = node;
    } else {
      this.tail.next = node;
      node.prev = this.tail;
      this.tail = node;
    }
  }

  // insert_before, delete, iterate — стандартно
}
```

#### Особенности:

- Удобная навигация в обе стороны.
- Требует больше памяти (два указателя на узел).
- Более сложная реализация (надо аккуратно обновлять оба указателя).

#### Применение:

- Реализация `Deque`, `LRU Cache` (через связку с `HashMap`).

#### Заметки:

- Всегда думать о корректности обновления ссылок.
- При удалении важно правильно обнулить связи.

---

### Циклический список (Circular Linked List)

В циклическом списке последний элемент указывает не на `null`, а на первый элемент списка. _Это может быть как односвязный, так и двусвязный список._

Структура (односвязный вариант): `(value, next)` где `tail.next = head`.

#### Операции и сложность:

- `traverse` (обход) — идём по кругу, важно остановиться (например, после `n` шагов).
- `insert_after(node, x)` — `O(1)`.
- `delete_after(node)` — `O(1)`.
- `find(value)` — `O(n)`.
- `append/prepend` — `O(1)` при хранении tail.

#### Псевдокод (JS):

```js
class Node {
  constructor(val) {
    this.val = val;
    this.next = null;
  }
}

class CircularLinkedList {
  constructor() {
    this.head = null;
  }

  append(val) {
    const node = new Node(val);
    if (!this.head) {
      this.head = node;
      node.next = node; // указывает на себя
    } else {
      let tail = this.head;
      while (tail.next !== this.head) tail = tail.next;
      tail.next = node;
      node.next = this.head;
    }
  }
}
```

#### Особенности:

- Нет «null»-окончания, можно ходить бесконечно.
- Удобно для циклических буферов, задач на «карусель» (round-robin), например: `Round-Robin Scheduler`.

#### Применение:

- Реализация **круговых очередей**, **игровых структур** (например, модель игроков по кругу).

#### Заметки:

- Нужно аккуратно избегать бесконечных циклов.
- Часто используется в ОС (распределение процессорного времени).

---

### Skip List

Skip List — это вероятностная структура данных для хранения отсортированных элементов с быстрыми операциями поиска, вставки и удаления. Это практичная альтернатива сбалансированным деревьям (AVL, Red-Black): проще в реализации, хороша для многопоточных реализаций и реально используется в продакшне (например, Redis `sorted set`).

**Ключевая идея.**

- Элементы организованы в несколько уровней. Уровень 0 — обычный отсортированный связный список. Каждый более высокий уровень содержит «пропуски» через несколько элементов нижнего уровня, образуя "быстрые дорожки".
- Каждый узел имеет случайную «высоту» (число уровней), обычно генерируемую серией подбрасываний монетки (probability `p`, часто `0.5`).
- Поиск начинается с верхнего уровня и идёт вправо, пока следующий узел меньше искомого, затем спускается на уровень ниже и продолжается.

**Структура узла:**

```
Node {
  value
  next[0..level-1] // ссылки для каждого уровня
}
```

---

#### Операции и сложность (ожидаемая)

- `search(x)` — `O(log n)` в среднем
- `insert(x)` — `O(log n)` в среднем
- `delete(x)` — `O(log n)` в среднем
- Память — `O(n)` в среднем (дополнительные указатели увеличивают константу)

> В худшем случае (крайне маловероятном) сложность может стать `O(n)`, но вероятность такого исхода экспоненциально мала.

---

#### Псевдокод (JS):

```js
class Node {
  constructor(val, level) {
    this.val = val;
    this.next = new Array(level).fill(null);
  }
}

class SkipList {
  constructor(maxLevel = 32, p = 0.5) {
    this.maxLevel = maxLevel; // верхний возможный уровень
    this.p = p; // вероятность повышения уровня
    this.head = new Node(null, maxLevel);
    this.level = 1; // текущий максимальный уровень
  }

  // Генерация уровня узла (случайно по p)
  randomLevel() {
    let lvl = 1;
    while (Math.random() < this.p && lvl < this.maxLevel) {
      lvl += 1;
    }
    return lvl;
  }

  // Поиск: возвращает node, предок на каждом уровне в update
  _findPath(val) {
    const update = new Array(this.maxLevel).fill(null);
    let cur = this.head;
    for (let i = this.level - 1; i >= 0; i--) {
      while (cur.next[i] && cur.next[i].val < val) cur = cur.next[i];
      update[i] = cur;
    }
    return update;
  }
}
```

---

#### Почему probabilistic (вероятностный) подход работает?

- Каждый узел получает высоту, которая распределена геометрически: `P(height >= k) = p^(k-1)`.
- Ожидаемая длина пути при поиске порядка `O(log_{1/p} n)` — логарифм по основанию `1/p`.
- Случайность устраняет необходимость в сложных балансировках (поворотах), которые есть в AVL/Red-Black деревьях.

---

#### Детали реализации и optimizations

- **Выбор p**: чаще используют `p = 0.5` — тогда ожидаемое число уровней ≈ `log2 n`. Можно выбирать `p = 0.25` или другие в зависимости от памяти/скорости trade-off.
- **maxLevel**: задаётся как `ceil(log_{1/p}(MAX_N))` или просто фиксируется (например, 32).
- **Duplicate keys**: можно хранить счётчики или разрешать мульти-ключи через linked-list на уровне 0.
- **Memory layout**: для performance-sensitive реализаций целесообразно хранить массивы `next` как плотные блоки и избегать частых аллокаций.
- **Concurrent skip lists**: структура выгодна для concurrent реализаций (меньше сложной синхронизации, чем в деревьях). Java `ConcurrentSkipListMap` — пример.

---

#### Преимущества:

- Простая реализация по сравнению с self-balancing деревьями.
- Хорошая ожидаемая производительность `O(log n)`.
- Удобство для concurrent реализаций.
- Реально используется в продакшне (Redis, Java ConcurrentSkipListMap).

#### Ограничения:

- Ожидаемая, а не детерминированная гарантия времени.
- Дополнительная память на указатели (множество ссылок на уровне).

---

#### Применения:

- In-memory sorted sets / indexes (Redis).
- Ordered maps/sets в многопоточных окружениях.
- Альтернатива деревьям при необходимости простоты реализации.

---

#### Заметки:

- Можно строить слои детерминированно (например, каждый 2-й элемент на верхнем уровне), но это сложнее при динамических вставках.
- Детерминированные skip-like структуры встречаются в теории, но в практике чаще используют probabilistic variant.

## Стек и очереди

### Стек (Stack, LIFO)

<b class="text-on-semi-light">Стек</b> — это абстрактная структура данных, работающая по принципу LIFO. Все операции происходят только на вершине стека, что делает его одной из самых простых и в то же время фундаментальных структур.

**Основные характеристики:**

1. **Ограниченный доступ:** доступен только последний добавленный элемент (вершина).
2. **Упорядоченность:** элементы сохраняют порядок вставки, но доступны не напрямую, а через последовательные удаления.
3. **Абстракция:** стек можно реализовать на массиве или на связном списке, но интерфейс остаётся одинаковым.

**Операции и их сложность:**

- `push(x)` — добавить элемент наверх стека — `O(1)`.
- `pop()` — удалить элемент с вершины — `O(1)`.
- `peek()` — получить значение вершины без удаления — `O(1)`.
- `isEmpty()` — проверить, пуст ли стек — `O(1)`.
- `size()` — узнать количество элементов — `O(1)`.

**Реализация на массиве:**

- Быстрая и простая.
- Возможна перераспределение памяти (динамический массив).

**Реализация на связном списке:**

- Гибкий размер.
- Более высокие накладные расходы на память (указатели).

**Пример (JavaScript):**

```js
const stack = [];
stack.push(1);
stack.push(2);
console.log(stack.pop()); // 2
console.log(stack[stack.length - 1]); // peek -> 1
```

**Применение стека:**

- **Системный стек вызовов:** поддержка рекурсии и вложенных функций.
- **Алгоритмы разбора:** проверка правильности скобок, работа компиляторов.
- **Алгоритмы обхода графов:** DFS (обход в глубину).
- **История действий:** Undo/Redo в редакторах, браузерах.
- **Обратная польская запись (RPN):** вычисление выражений без скобок.

**Недостатки:**

- Нет произвольного доступа (нельзя получить `k`-й элемент напрямую).
- Не подходит для сценариев, где требуется выборка из середины.

---

### Очередь (Queue, FIFO)

<b class="text-on-semi-light">Очередь</b> — структура данных, работающая по принципу `FIFO`. Элементы добавляются в конец и извлекаются из начала.

**Основные характеристики:**

1. **Упорядоченность:** Элементы обслуживаются строго в порядке поступления.
2. **Два конца:** Добавление идёт в хвост, удаление — из головы.
3. **Линейная структура:** Нет случайного доступа, только последовательная обработка.

**Операции и сложность:**

- `enqueue(x)` — добавить элемент в конец — `O(1)`.
- `dequeue()` — удалить элемент из начала — `O(1)` (в массиве может быть `O(n)`, если не использовать кольцевую буферизацию).
- `peek()` — посмотреть первый элемент — `O(1)`.
- `isEmpty()` — проверка на пустоту — `O(1)`.

**Пример (JavaScript):**

```js
const queue = [];
queue.push(1);
queue.push(2);
console.log(queue.shift()); // 1
console.log(queue[0]); // peek -> 2
```

**Применение:**

- Планирование задач (task scheduling, job queues).
- Очереди сообщений (message queues, event loops).
- BFS в графах (поиск в ширину).
- Буферизация потоков данных (например, IO-операции, сетевые пакеты).

**Заметки:**

- При реализации на массиве важно избегать `O(n)` сдвигов → лучше использовать **кольцевую очередь** (circular buffer) или связный список.
- В многопоточной среде применяются lock-free структуры (например, **Michael-Scott queue**).
- Для высоконагруженных систем важна кэш-локальность: связные реализации менее эффективны, чем массивные.
- В реальных системах часто используется очередь с приоритетом вместо обычной FIFO (например, планировщики ОС).

---

### Дек (Deque, Double-Ended Queue)

<b class="text-on-semi-light">Дек</b> — это структура данных, представляющая собой двустороннюю очередь, где добавление и удаление элементов возможно как с начала, так и с конца. Название происходит от английского _**double-ended queue**_.

**Основные характеристики:**

1. **Гибкость:** Поддерживает операции и в голове, и в хвосте.
2. **Объединяет стек и очередь:** Может работать как LIFO, так и FIFO.
3. **Линейная структура:** Нет произвольного доступа по индексу (в отличие от массива).

**Операции и сложность:**

- `push_front(x)` — добавить элемент в начало — `O(1)`.
- `push_back(x)` — добавить элемент в конец — `O(1)`.
- `pop_front()` — удалить элемент из начала — `O(1)`.
- `pop_back()` — удалить элемент с конца — `O(1)`.
- `peek_front()` — посмотреть первый элемент — `O(1)`.
- `peek_back()` — посмотреть последний элемент — `O(1)`.

**Пример (JavaScript — реализация на массиве):**

```js
const deque = [];
deque.push(1); // push_back
deque.unshift(2); // push_front
console.log(deque.pop()); // pop_back -> 1
console.log(deque.shift()); // pop_front -> 2
```

**Применение:**

- Реализация **двусторонних буферов**.
- Слайдинг-алгоритмы (например, поиск max/min на подотрезке).
- Хранение состояний для undo/redo (можно двигаться в обе стороны).
- Дек используется в **алгоритме BFS** с приоритетом (0-1 BFS).

**Заметки:**

- В JS `Array` неэффективен для `shift/unshift` (O(n)), поэтому для серьёзных задач нужен связный список или специализированная структура.
- В C++ `std::deque` — контейнер с амортизированным `O(1)` на операции по краям.
- В многопоточных системах возможны lock-free реализации deques (например, work-stealing queues в пулах потоков).
- Дек часто используется как основа для более сложных структур: **Deque + HashMap** = LRU Cache.

---

### Очередь с приоритетами (Priority Queue)

Очередь с приоритетами — это структура данных, в которой каждому элементу назначается **приоритет**, и элементы извлекаются **не по порядку добавления**, а по наивысшему (или наименьшему) приоритету.

**Основные характеристики:**

1. **Упорядоченность по приоритету:** порядок определяется не временем вставки, а значением приоритета.
2. **Гибкость:** можно реализовать как _max-priority queue_ (сначала наибольшие приоритеты), так и _min-priority queue_ (сначала наименьшие).
3. **Не обязательно FIFO:** в отличие от обычной очереди, здесь доступ к «самому важному» элементу.

**Операции и сложность:**

- `insert(x, priority)` — вставка элемента — `O(log n)` (через кучу).
- `extract_max()` / `extract_min()` — достать элемент с наибольшим/наименьшим приоритетом — `O(log n)`.
- `peek()` — посмотреть элемент с наивысшим приоритетом — `O(1)`.
- `increase_key(x, newPriority)` — увеличить приоритет элемента — `O(log n)`.

**Пример (JavaScript — реализация на куче):**

```js
class PriorityQueue {
  constructor() {
    this.heap = [];
  }

  parent(i) {
    return Math.floor((i - 1) / 2);
  }
  left(i) {
    return 2 * i + 1;
  }
  right(i) {
    return 2 * i + 2;
  }

  swap(i, j) {
    [this.heap[i], this.heap[j]] = [this.heap[j], this.heap[i]];
  }

  push(value, priority) {
    this.heap.push({ value, priority });
    this.bubbleUp(this.heap.length - 1);
  }

  bubbleUp(i) {
    while (
      i > 0 &&
      this.heap[this.parent(i)].priority < this.heap[i].priority
    ) {
      this.swap(i, this.parent(i));
      i = this.parent(i);
    }
  }

  pop() {
    if (this.heap.length === 0) return null;
    const top = this.heap[0];
    const end = this.heap.pop();
    if (this.heap.length > 0) {
      this.heap[0] = end;
      this.bubbleDown(0);
    }
    return top;
  }

  bubbleDown(i) {
    let largest = i;
    const left = this.left(i),
      right = this.right(i);

    if (
      left < this.heap.length &&
      this.heap[left].priority > this.heap[largest].priority
    ) {
      largest = left;
    }
    if (
      right < this.heap.length &&
      this.heap[right].priority > this.heap[largest].priority
    ) {
      largest = right;
    }
    if (largest !== i) {
      this.swap(i, largest);
      this.bubbleDown(largest);
    }
  }
}

// Пример использования
const pq = new PriorityQueue();
pq.push("low", 1);
pq.push("medium", 5);
pq.push("high", 10);

console.log(pq.pop()); // { value: "high", priority: 10 }
console.log(pq.pop()); // { value: "medium", priority: 5 }
```

**Применение:**

- Планирование задач (scheduler): выполнение в порядке важности.
- Алгоритмы на графах: **Dijkstra, A\*** используют min-priority queue.
- Системы обработки событий (например, симуляции).
- Реализация **job queues** с приоритетом (очередь задач).

**Заметки:**

- Чаще всего очередь с приоритетами реализуется через **бинарную кучу (heap)**.
- В C++ есть `std::priority_queue` (по умолчанию max-heap).
- В Python — модуль `heapq` (min-heap, для max нужно приоритет хранить со знаком `-`).
- Существуют более сложные реализации (фибоначчиевы кучи, pairing heap), которые дают лучшую асимптотику в некоторых задачах.
- Можно строить специализированные версии: **bounded priority queue** (ограниченное количество приоритетов), **multiqueue** (параллельные очереди для масштабируемости).

### Кольцевая очередь (Circular Queue)

Кольцевая очередь — это разновидность обычной очереди, в которой конец структуры соединён с началом, образуя кольцо. Такая организация позволяет эффективно использовать фиксированный массив без затрат на сдвиги элементов.

**Основные характеристики:**

1. **Фиксированный размер:** Реализуется поверх массива с ограниченной ёмкостью.
2. **Кольцевое поведение:** Когда хвост достигает конца массива, он переходит в начало.
3. **Эффективность:** Операции добавления и удаления выполняются за `O(1)`.

**Операции и сложность:**

- `enqueue(x)` — добавить элемент в конец — `O(1)`.
- `dequeue()` — удалить элемент из начала — `O(1)`.
- `peek_front()` — посмотреть первый элемент — `O(1)`.
- `peek_rear()` — посмотреть последний элемент — `O(1)`.
- `isEmpty()` — проверка на пустоту — `O(1)`.
- `isFull()` — проверка на заполненность — `O(1)`.

**Пример (JavaScript):**

```js
class CircularQueue {
  constructor(capacity) {
    this.queue = new Array(capacity);
    this.capacity = capacity;
    this.front = 0;
    this.rear = -1;
    this.size = 0;
  }

  enqueue(x) {
    if (this.isFull()) throw new Error("Queue is full");
    this.rear = (this.rear + 1) % this.capacity;
    this.queue[this.rear] = x;
    this.size++;
  }

  dequeue() {
    if (this.isEmpty()) throw new Error("Queue is empty");
    const val = this.queue[this.front];
    this.front = (this.front + 1) % this.capacity;
    this.size--;
    return val;
  }

  peek_front() {
    return this.isEmpty() ? null : this.queue[this.front];
  }

  peek_rear() {
    return this.isEmpty() ? null : this.queue[this.rear];
  }

  isEmpty() {
    return this.size === 0;
  }

  isFull() {
    return this.size === this.capacity;
  }
}
```

**Применение:**

- Буферизация потоков данных (например, в сетевых драйверах).
- Реализация **циклических буферов** в системах реального времени.
- Очереди заданий в ограниченной памяти.
- Системы обработки событий (например, клавиатурные буферы).

**Заметки:**

- Отличие от обычной очереди: отсутствует проблема «ложного переполнения», так как пространство переиспользуется.
- Широко используется в embedded-разработке, ОС и драйверах.
- В многопоточной среде может использоваться как lock-free структура при правильной синхронизации.

## Хеш-структуры

### Хеш-таблица (Hash Table, Hash Map)

**Хеш-таблица** — структура данных, которая хранит пары «ключ–значение» и обеспечивает быстрый доступ к данным с помощью _хеш-функции_. Основная идея — преобразовать ключ в индекс массива (bucket), где лежит элемент.

**Основные характеристики:**

1. **Ассоциативное хранение:** доступ к элементам по ключу, а не по индексу.
2. **Хеш-функция:** преобразует ключ в целое число (индекс).
3. **Коллизии:** разные ключи могут давать один и тот же индекс, решаются методами _chaining_ или _open addressing_.
4. **Load factor:** отношение числа элементов к размеру таблицы, влияет на скорость работы и необходимость ресайза.

**Операции и их сложность:**

- `insert(key, value)` — вставить или обновить элемент — `O(1)` амортизированно.
- `get(key)` — получить значение по ключу — `O(1)` амортизированно.
- `delete(key)` — удалить элемент по ключу — `O(1)` амортизированно.
- `contains(key)` — проверить наличие ключа — `O(1)` амортизированно.
- При плохом хешировании или переполнении бакетов операции могут деградировать до `O(n)`.

**Пример (JavaScript):**

```js
const map = new Map();
map.set("apple", 1);
map.set("banana", 2);
console.log(map.get("apple")); // 1
console.log(map.has("banana")); // true
map.delete("apple");
```

**Применение:**

- Реализация словарей и кэшей.
- Индексация в базах данных.
- Подсчёт частоты слов или элементов.
- Символьные таблицы в компиляторах.

**Недостатки:**

- Нет упорядоченности ключей.
- При плохой хеш-функции возможны большие цепочки коллизий.
- Дополнительные накладные расходы на память.

### Множество (Set) и Мультимножество (Multiset)

**Set** — структура, хранящая только уникальные элементы.
**Multiset** — допускает дубликаты элементов.

**Основные характеристики:**

1. **Уникальность (Set):** каждый элемент встречается максимум один раз.
2. **Счётчик (Multiset):** хранит частоту появления каждого элемента.
3. **Быстрый доступ:** большинство операций работают за `O(1)` амортизированно.

**Операции и их сложность:**

- `insert(x)` — добавить элемент — `O(1)`.
- `delete(x)` — удалить элемент — `O(1)`.
- `contains(x)` — проверить наличие — `O(1)`.
- `count(x)` (только для Multiset) — узнать количество копий — `O(1)`.
- Итерация по множеству — `O(n)`.

**Пример (JavaScript — Set):**

```js
const set = new Set();
set.add(1);
set.add(2);
set.add(1); // дубликат не добавится
console.log(set.has(2)); // true
set.delete(1);
```

**Применение:**

- Удаление дубликатов (например, из массива).
- Проверка принадлежности элемента множеству.
- Подсчёт частот (multiset = hash map с числами).
- Реализация множест­венных операций: пересечение, объединение, разность.

**Недостатки:**

- Нет прямого доступа к k-му элементу.
- В `Set` нельзя хранить дубликаты (нужно `Multiset`).
- Зависимость от качества хеш-функции.

## Деревья (Trees)

**Деревья** — иерархические структуры данных, где каждый узел хранит значение и ссылки на дочерние узлы. Корень дерева — это начальная точка, у каждого узла может быть 0 или более детей. Используются для поиска, хранения и организации данных.

**Основные обходы деревьев:**

1. **Preorder (NLR):** сначала посещаем текущий узел (N), затем левое поддерево (L), потом правое (R).
   - Пример: используется для копирования дерева или сериализации.

2. **Inorder (LNR):** сначала левое поддерево (L), потом узел (N), затем правое (R).
   - Пример: в бинарном дереве поиска (BST) даёт отсортированную последовательность.

3. **Postorder (LRN):** сначала левое поддерево (L), затем правое (R), и в конце узел (N).
   - Пример: используется для удаления дерева или вычисления выражений.

4. **Level-order (BFS):** обход по уровням сверху вниз, слева направо (через очередь).
   - Пример: применяется для поиска кратчайшего пути в невзвешенных графах.

**Сложность обходов:**

- Любой обход дерева требует `O(n)`, где `n` — количество узлов.

**Применение деревьев:**

- Хранение иерархий (файловая система, организационные структуры).
- Реализация структур поиска (BST, Heap, Trie).
- Разбор арифметических выражений.
- Построение индексных структур в базах данных.

### Binary Tree (Бинарное дерево)

<b class="text-on-semi-light">Бинарное дерево</b> — это структура, в которой каждый узел имеет не более двух дочерних элементов: `left` (левый) и `right` (правый). Оно является фундаментальной основой для многих специализированных деревьев (BST, AVL, Red-Black).

**Основные характеристики:**

1. **Ограниченность детей:** максимум два потомка у каждого узла.
2. **Иерархия:** чёткая структура от корня к листьям.
3. **Универсальность:** бинарное дерево может быть неупорядоченным (просто контейнером) или упорядоченным (например, BST).

**Классификация бинарных деревьев:**

- **Полное бинарное дерево (Complete Binary Tree):** все уровни, кроме последнего, полностью заполнены, а последний уровень заполняется слева направо.
- **Полное (Full) бинарное дерево:** каждый узел имеет либо 0, либо 2 потомка.
- **Совершенное (Perfect) бинарное дерево:** все внутренние узлы имеют 2 потомка, и все листья находятся на одном уровне.
- **Сбалансированное бинарное дерево:** высота левого и правого поддеревьев различается не более чем на 1 (пример — AVL).

**Операции и их сложность:**

- **Обход (traversal):**
  - Preorder (NLR), Inorder (LNR), Postorder (LRN), Level-order.
  - Время — `O(n)`, где `n` — количество узлов.

- **Поиск элемента:** зависит от типа дерева:
  - В произвольном бинарном дереве — `O(n)`.
  - В бинарном дереве поиска (BST) — `O(log n)` в среднем.

- **Вставка:**
  - В простом бинарном дереве — `O(1)` при известной позиции.
  - В BST — `O(log n)` в среднем.

- **Удаление:** аналогично вставке — зависит от упорядоченности и балансировки.

**Реализация на практике (JavaScript):**

```js
class TreeNode {
  constructor(value) {
    this.value = value;
    this.left = null;
    this.right = null;
  }
}

// Пример обхода Inorder
function inorder(node) {
  if (!node) return;
  inorder(node.left);
  console.log(node.value);
  inorder(node.right);
}

const root = new TreeNode(1);
root.left = new TreeNode(2);
root.right = new TreeNode(3);
root.left.left = new TreeNode(4);

inorder(root); // 4, 2, 1, 3
```

**Применение бинарных деревьев:**

- **Хранение иерархий:** структура папок, орг. схемы.
- **Парсинг выражений:** арифметические и логические выражения.
- **Алгоритмы:** основа для бинарного поиска (BST), кучи, сбалансированных деревьев.
- **Обход графов:** бинарные деревья используются для DFS-подобных задач.

**Недостатки:**

- В несбалансированном виде может вырождаться в список (высота = `n`).
- Низкая эффективность поиска/вставки в худшем случае.

### Binary Search Tree (BST, Бинарное дерево поиска)

**Binary Search Tree (BST)** — это специализированный вид бинарного дерева, где для каждого узла выполняется свойство порядка:

- Все значения в левом поддереве **меньше** значения узла.
- Все значения в правом поддереве **больше** значения узла.

Это свойство делает BST особенно полезным для эффективного поиска, вставки и удаления элементов.

**Основные характеристики:**

1. **Упорядоченность:** структура хранит данные в отсортированном виде.
2. **Рекурсивная структура:** каждое поддерево также является BST.
3. **Неуникальность:** может хранить как уникальные ключи, так и дубликаты (зависит от реализации).

**Операции и их сложность:**

- `search(x)` — поиск элемента — `O(log n)` в среднем, `O(n)` в худшем.
- `insert(x)` — вставка нового узла — `O(log n)` / `O(n)`.
- `delete(x)` — удаление узла — `O(log n)` / `O(n)`.
- `min()` — минимальный элемент (самый левый узел) — `O(h)`, где `h` — высота.
- `max()` — максимальный элемент (самый правый узел) — `O(h)`.
- `traverse()` (обход in-order) — выдаёт отсортированную последовательность за `O(n)`.

**Пример (JavaScript — реализация вставки и поиска):**

```js
class Node {
  constructor(value) {
    this.value = value;
    this.left = null;
    this.right = null;
  }
}

class BST {
  constructor() {
    this.root = null;
  }

  insert(value) {
    const newNode = new Node(value);
    if (!this.root) {
      this.root = newNode;
      return;
    }
    let current = this.root;
    while (true) {
      if (value < current.value) {
        if (!current.left) {
          current.left = newNode;
          return;
        }
        current = current.left;
      } else {
        if (!current.right) {
          current.right = newNode;
          return;
        }
        current = current.right;
      }
    }
  }

  search(value) {
    let current = this.root;
    while (current) {
      if (value === current.value) return true;
      current = value < current.value ? current.left : current.right;
    }
    return false;
  }
}

const tree = new BST();
tree.insert(8);
tree.insert(3);
tree.insert(10);
console.log(tree.search(10)); // true
console.log(tree.search(5)); // false
```

**Применение BST:**

- Быстрый поиск, вставка и удаление данных.
- Реализация множеств (`Set`) и ассоциативных массивов (`Map`).
- Построение словарей и индексов.
- Сортировка через обход in-order.
- Используется в базах данных и компиляторах.

**Недостатки:**

- В худшем случае (если вставлять элементы в отсортированном порядке) дерево вырождается в список → операции становятся `O(n)`.
- Решение: применять самобалансирующиеся деревья (AVL, Red-Black Tree).

### AVL Tree

**AVL-дерево** — это самобалансирующееся бинарное дерево поиска, в котором для любого узла разница высот его левого и правого поддерева (баланс-фактор) не превышает 1. Названо в честь авторов — Адельсона-Вельского и Ландиса.

**Основные характеристики:**

1. **Самобалансировка:** после каждой вставки или удаления выполняются повороты для восстановления свойства баланса.
2. **Баланс-фактор:** `balance = height(left) - height(right)` ∈ { -1, 0, 1 }.
3. **Гарантированная высота:** `O(log n)` для `n` элементов, что обеспечивает эффективный поиск.

**Операции и сложность:**

- `search(x)` — поиск — `O(log n)`.
- `insert(x)` — вставка с возможной балансировкой — `O(log n)`.
- `delete(x)` — удаление с возможной балансировкой — `O(log n)`.

Балансировка требует максимум одного или двух поворотов (левый, правый или двойные).

**Типы поворотов:**

1. **Правый поворот (Right Rotation, LL-case)** — применяется, если вставка произошла в левом поддереве левого ребёнка.
2. **Левый поворот (Left Rotation, RR-case)** — если вставка произошла в правом поддереве правого ребёнка.
3. **Лево-правый поворот (Left-Right, LR-case)** — комбинация: сначала левый поворот на левом поддереве, затем правый.
4. **Право-левый поворот (Right-Left, RL-case)** — сначала правый поворот на правом поддереве, затем левый.

**Пример (псевдокод вставки):**

```text
function insert(node, key):
  if node == null:
    return new Node(key)

  if key < node.key:
    node.left = insert(node.left, key)
  else if key > node.key:
    node.right = insert(node.right, key)
  else:
    return node  // дубликаты не допускаются

  updateHeight(node)
  balance = height(node.left) - height(node.right)

  if balance > 1 and key < node.left.key:
    return rotateRight(node)   // LL-case
  if balance < -1 and key > node.right.key:
    return rotateLeft(node)    // RR-case
  if balance > 1 and key > node.left.key:
    node.left = rotateLeft(node.left)
    return rotateRight(node)   // LR-case
  if balance < -1 and key < node.right.key:
    node.right = rotateRight(node.right)
    return rotateLeft(node)    // RL-case

  return node
```

**Преимущества:**

- Высокая предсказуемость времени выполнения (`O(log n)` всегда).
- Отличный выбор для систем, где **важен быстрый поиск**.

**Недостатки:**

- Вставка и удаление сложнее, чем в простом BST (из-за поворотов).
- Требует хранения высоты каждого узла → больше памяти.

**Применение AVL-деревьев:**

- **Базы данных и индексы:** когда критичен быстрый поиск.
- **Кэш и словари:** гарантированно быстрые операции.
- **Системы реального времени:** где важно предсказуемое время доступа.

### Red-Black Tree

**Красно-черное дерево** — это самобалансирующееся бинарное дерево поиска, в котором каждый узел имеет дополнительный атрибут цвета (красный или чёрный). Балансировка достигается за счёт правил окраски и поворотов, что гарантирует высоту дерева `O(log n)`.

**Основные характеристики:**

1. **Окрашивание узлов:** каждый узел либо красный, либо чёрный.
2. **Свойства дерева:**
   - Корень и листья `null` всегда чёрные.
   - У красного узла не может быть красных детей.
   - Все пути от корня до `null`-узлов содержат одинаковое количество чёрных узлов (черная высота).

3. **Гарантированная высота:** не более чем `2 * log(n+1)`.

**Операции и сложность:**

- `search(x)` — поиск — `O(log n)`.
- `insert(x)` — вставка с перекрашиванием и поворотами — `O(log n)`.
- `delete(x)` — удаление с балансировкой — `O(log n)`.

**Балансировка:**

При вставке или удалении может нарушиться одно из свойств. В таких случаях выполняются:

- **Перекраска (recoloring):** изменение цвета узлов.
- **Повороты (rotations):** левый или правый, чтобы восстановить баланс.

**Типы поворотов:**

1. **Левый поворот (Left Rotation):** поднимает правого ребёнка.
2. **Правый поворот (Right Rotation):** поднимает левого ребёнка.
3. В комбинации с перекраской восстанавливают свойства дерева.

**Пример (псевдокод вставки):**

```text
function insert(tree, key):
  node = BSTInsert(tree, key)
  color(node) = RED

  while node != root and color(parent(node)) == RED:
    if parent(node) is left child:
      uncle = parent(parent(node)).right
      if color(uncle) == RED:
        color(parent(node)) = BLACK
        color(uncle) = BLACK
        color(parent(parent(node))) = RED
        node = parent(parent(node))
      else:
        if node == parent(node).right:
          node = parent(node)
          rotateLeft(node)
        color(parent(node)) = BLACK
        color(parent(parent(node))) = RED
        rotateRight(parent(parent(node)))
    else:
      ... // симметричный случай

  color(root) = BLACK
```

**Преимущества:**

- Высокая эффективность операций (`O(log n)` всегда).
- Проще реализации, чем AVL, при вставке/удалении.
- Широко используется в стандартных библиотеках.

**Недостатки:**

- Балансировка слабее, чем у AVL → поиск может быть немного медленнее.
- Более сложная логика удаления.

**Применение красно-черных деревьев:**

- **Стандартные структуры:** Java `TreeMap`, C++ `std::map`, `std::set`.
- **Операционные системы:** планировщики, кэш.
- **Базы данных и индексы:** где нужны гарантии `O(log n)` при вставке и удалении.

---

### Splay Tree

**Splay-дерево** — это самобалансирующееся бинарное дерево поиска, в котором при каждом обращении к элементу (поиск, вставка, удаление) выполняется операция **splay (подтягивание)** — доступный узел поднимается в корень с помощью серии вращений. Это обеспечивает быструю работу с часто используемыми элементами.

**Основные характеристики:**

1. **Подтягивание (Splaying):** после каждой операции затронутый элемент становится корнем.
2. **Адаптивность:** часто используемые элементы оказываются ближе к корню, ускоряя повторные обращения.
3. **Амортизированная эффективность:** несмотря на то, что отдельная операция может занимать `O(n)`, амортизированная сложность — `O(log n)`.

**Операции и сложность:**

- `search(x)` — поиск + подтягивание элемента в корень — `O(log n)` амортизированно.
- `insert(x)` — вставка и подтягивание нового узла в корень — `O(log n)` амортизированно.
- `delete(x)` — удаление и подтягивание ближайшего узла к корню — `O(log n)` амортизированно.

**Типы вращений (Splaying Cases):**

1. **Zig (одинарный поворот):** когда узел — ребёнок корня.
2. **Zig-Zig (двойной однотипный поворот):** когда узел и его родитель оба являются левыми или правыми детьми.
3. **Zig-Zag (двойной разнотипный поворот):** когда узел — левый ребёнок правого родителя или наоборот.

**Пример (псевдокод splay):**

```text
function splay(node, key):
  if node == null or node.key == key:
    return node

  if key < node.key:
    if node.left == null:
      return node
    if key < node.left.key:  // Zig-Zig
      node.left.left = splay(node.left.left, key)
      node = rotateRight(node)
    else if key > node.left.key:  // Zig-Zag
      node.left.right = splay(node.left.right, key)
      if node.left.right != null:
        node.left = rotateLeft(node.left)
    return rotateRight(node)

  else:
    if node.right == null:
      return node
    if key > node.right.key: // Zig-Zig
      node.right.right = splay(node.right.right, key)
      node = rotateLeft(node)
    else if key < node.right.key: // Zig-Zag
      node.right.left = splay(node.right.left, key)
      if node.right.left != null:
        node.right = rotateRight(node.right)
    return rotateLeft(node)
```

**Преимущества:**

- Простота реализации (нет необходимости хранить баланс-фактор или цвет узлов).
- Отличен для случаев, когда часто обращаются к одним и тем же элементам.
- Амортизированная эффективность `O(log n)`.

**Недостатки:**

- Худший случай одной операции может занимать `O(n)`.
- Нет строгой гарантии высоты, как в AVL или Red-Black Tree.

**Применение Splay-деревьев:**

- **Кэширование:** когда важны частые обращения к одним и тем же данным.
- **Словари и множества:** где важен быстрый доступ к недавно использованным элементам.
- **Системы, где важна амортизированная эффективность, а не строгие гарантии.**

### Trie (Prefix Tree)

<b class="text-on-semi-light">Trie</b> — специализированная структура данных для хранения и поиска строк (слов), которая особенно эффективна при работе с префиксами.

**Основные характеристики:**

1. **Иерархия символов:** каждая вершина соответствует символу, а путь от корня до узла формирует строку или её префикс.
2. **Корень пустой:** в корне не хранится символ, только указатели на потомков.
3. **Маркер конца слова:** специальные флаги отмечают, что данный путь соответствует полноценному слову.

**Операции и их сложность:**

- **Вставка (`insert`):** последовательный обход символов строки и добавление вершин при необходимости — `O(m)`, где `m` длина строки.
- **Поиск (`search`):** проверка наличия строки по символам и конечному маркеру — `O(m)`.
- **Проверка префикса (`startsWith`):** обход символов префикса и проверка существования пути — `O(m)`.

**Реализация на практике (JavaScript):**

```js
class TrieNode {
  constructor() {
    this.children = {}; // потомки по символам
    this.isEndOfWord = false; // маркер конца слова
  }
}

class Trie {
  constructor() {
    this.root = new TrieNode();
  }

  insert(word) {
    let node = this.root;
    for (let char of word) {
      if (!node.children[char]) {
        node.children[char] = new TrieNode();
      }
      node = node.children[char];
    }
    node.isEndOfWord = true;
  }

  search(word) {
    let node = this.root;
    for (let char of word) {
      if (!node.children[char]) return false;
      node = node.children[char];
    }
    return node.isEndOfWord;
  }

  startsWith(prefix) {
    let node = this.root;
    for (let char of prefix) {
      if (!node.children[char]) return false;
      node = node.children[char];
    }
    return true;
  }
}

const trie = new Trie();
trie.insert("apple");
console.log(trie.search("apple")); // true
console.log(trie.search("app")); // false
console.log(trie.startsWith("app")); // true
```

**Применение Trie:**

- Автодополнение слов и поиск по словарю.
- Проверка корректности слов (spell checker).
- Поиск по префиксу (например, в поисковых системах).
- Хранение и сжатие большого набора строк (DNS, IP-адреса).

**Недостатки:**

- Высокое потребление памяти (каждый символ требует отдельного узла).
- Может быть избыточным для маленьких наборов слов.
- Для экономии памяти используются варианты: сжатый Trie (Radix Tree), Ternary Search Tree.

### Fenwick Tree (Binary Indexed Tree)

<b class="text-on-semi-light">Fenwick Tree</b> — это структура данных, позволяющая эффективно выполнять операции с префиксными суммами (и другими ассоциативными операциями). Она занимает меньше памяти, чем Segment Tree, и проще в реализации.

**Основные характеристики:**

1. **Идея:** массив, где каждая ячейка хранит агрегированную информацию о диапазоне элементов.
2. **Размер:** структура занимает `O(n)` памяти.
3. **Построение:** `O(n log n)` или `O(n)` (оптимизированное).

**Операции и их сложность:**

- **Обновление (`update`):** добавление значения к элементу и обновление связанных диапазонов — `O(log n)`.
- **Запрос суммы (`prefixSum`):** сумма элементов от начала массива до индекса `i` — `O(log n)`.
- **Запрос диапазона (`rangeSum`):** сумма на подотрезке `[l, r]` вычисляется как `prefixSum(r) - prefixSum(l-1)`.

**Реализация на практике (JavaScript):**

```js
class FenwickTree {
  constructor(size) {
    this.n = size;
    this.tree = new Array(size + 1).fill(0);
  }

  update(i, delta) {
    while (i <= this.n) {
      this.tree[i] += delta;
      i += i & -i;
    }
  }

  prefixSum(i) {
    let sum = 0;
    while (i > 0) {
      sum += this.tree[i];
      i -= i & -i;
    }
    return sum;
  }

  rangeSum(l, r) {
    return this.prefixSum(r) - this.prefixSum(l - 1);
  }
}

const arr = [1, 3, 5, 7, 9, 11];
const ft = new FenwickTree(arr.length);

// Построение дерева через update
arr.forEach((val, i) => ft.update(i + 1, val));

console.log(ft.prefixSum(3)); // сумма [1,3,5] => 9
console.log(ft.rangeSum(2, 4)); // сумма [3,5,7] => 15
ft.update(2, 7); // arr[1] += 7 => arr[1] = 10
console.log(ft.rangeSum(2, 4)); // сумма [10,5,7] => 22
```

**Применение Fenwick Tree:**

- Быстрые запросы **сумм/минимумов/максимумов** на подотрезках.
- Используется в задачах на **онлайн-запросы** и **динамическое обновление данных**.
- Часто применяется в соревновательном программировании (например, для инверсий в массиве).

**Недостатки:**

- Поддерживает только **ассоциативные операции** (чаще всего сумму).
- Ограничен по функциональности по сравнению с Segment Tree.
- Для сложных агрегатов (например, gcd, lcm, минимум/максимум) лучше использовать Segment Tree.

### Segment Tree (Дерево отрезков)

<b class="text-on-semi-light">Segment Tree</b> — это структура данных, которая используется для эффективной работы с массивами, где часто выполняются операции на подотрезках: суммы, минимумы, максимумы или другие агрегаты.

**Основные характеристики:**

1. **Иерархичность:** строится бинарное дерево, где каждый узел хранит агрегированное значение для определённого диапазона массива.
2. **Размер дерева:** для массива длиной `n` дерево имеет около `4n` узлов.
3. **Время построения:** `O(n)`.

**Операции и их сложность:**

- **Построение дерева:** `O(n)`.
- **Запрос на диапазоне (range query):** получение суммы/минимума/максимума на подотрезке — `O(log n)`.
- **Обновление элемента:** изменение значения в массиве и обновление соответствующих узлов дерева — `O(log n)`.

**Реализация на практике (JavaScript):**

```js
class SegmentTree {
  constructor(arr) {
    this.n = arr.length;
    this.tree = new Array(4 * this.n);
    this.build(arr, 1, 0, this.n - 1);
  }

  build(arr, node, l, r) {
    if (l === r) {
      this.tree[node] = arr[l];
      return;
    }
    let mid = Math.floor((l + r) / 2);
    this.build(arr, node * 2, l, mid);
    this.build(arr, node * 2 + 1, mid + 1, r);
    this.tree[node] = this.tree[node * 2] + this.tree[node * 2 + 1];
  }

  query(node, l, r, ql, qr) {
    if (ql > r || qr < l) return 0;
    if (ql <= l && r <= qr) return this.tree[node];
    let mid = Math.floor((l + r) / 2);
    return (
      this.query(node * 2, l, mid, ql, qr) +
      this.query(node * 2 + 1, mid + 1, r, ql, qr)
    );
  }

  update(node, l, r, idx, val) {
    if (l === r) {
      this.tree[node] = val;
      return;
    }
    let mid = Math.floor((l + r) / 2);
    if (idx <= mid) this.update(node * 2, l, mid, idx, val);
    else this.update(node * 2 + 1, mid + 1, r, idx, val);
    this.tree[node] = this.tree[node * 2] + this.tree[node * 2 + 1];
  }
}

const arr = [1, 3, 5, 7, 9, 11];
const segTree = new SegmentTree(arr);
console.log(segTree.query(1, 0, arr.length - 1, 1, 3)); // сумма [3,5,7] => 15
segTree.update(1, 0, arr.length - 1, 1, 10); // arr[1] = 10
console.log(segTree.query(1, 0, arr.length - 1, 1, 3)); // сумма [10,5,7] => 22
```

**Применение Segment Tree:**

- Быстрые **range queries** в массивах (сумма, минимум, максимум).
- Используется в задачах на соревнованиях (dynamic queries).
- Применяется в базах данных и компиляторах для анализа диапазонов.

**Недостатки:**

- Занимает больше памяти (`4n`).
- Более сложная реализация, чем Fenwick Tree (BIT).
- Труднее расширять для произвольных операций.

## Кучи (Heaps)

**Кучи** — это специализированные структуры данных на основе _полного бинарного дерева_, где каждый узел подчиняется определённому правилу упорядоченности относительно своих потомков. В зависимости от этого правила различают **Min-Heap** (минимальная куча) и **Max-Heap** (максимальная куча).

**Основные идеи кучи:**

1. **Полнота:** дерево всегда заполняется уровнями слева направо без пропусков, что делает его компактным и легко хранимым в массиве.
2. **Свойство кучи:**
   - В **Min-Heap** значение в родителе ≤ значения в детях.
   - В **Max-Heap** значение в родителе ≥ значения в детях.

3. **Экстремум в корне:**
   - В Min-Heap минимальный элемент всегда в корне.
   - В Max-Heap максимальный элемент всегда в корне.

**Хранение в массиве**

Куча обычно реализуется на массиве без указателей:

- `left(i) = 2i + 1`
- `right(i) = 2i + 2`
- `parent(i) = (i - 1) // 2`

Пример массива и его представления в виде дерева:

```
Массив: [10, 7, 5, 3, 2]

      10
     /  \
    7    5
   / \
  3   2
```

**Heapify (балансировка)**

**Heapify** — это процесс восстановления свойства кучи после изменения её структуры.

- **Heapify-up (просеивание вверх):**
  Используется при вставке. Новый элемент добавляется в конец массива и поднимается вверх, пока не будет удовлетворено свойство кучи.

  Пример (Max-Heap):
  - Вставляем 12 в [10, 7, 5, 3, 2]
  - Новый массив: [10, 7, 5, 3, 2, 12]
  - 12 поднимается, так как больше родителя (5):

```
До:              После:

      10                10
     /  \              /  \
    7    5            7    12
   / \   /           / \   /
  3   2             3   2  5
```

- **Heapify-down (просеивание вниз):**
  Используется при удалении корня. Корень заменяется последним элементом массива и опускается вниз, пока не будет восстановлено свойство кучи.

  Пример (Max-Heap):
  - Удаляем 10 из [10, 7, 5, 3, 2, 12]
  - Новый корень = 12, но он больше детей, ничего делать не нужно.
  - Если бы был меньше — его меняли бы местами с большим потомком.

**Операции и сложность**

- **Вставка (`insert`):** `O(log n)` за счёт heapify-up.
- **Извлечение экстремума (`extractMin`/`extractMax`):** `O(log n)` за счёт heapify-down.
- **Просмотр экстремума (`peek`):** `O(1)`.
- **Построение кучи (`buildHeap`):** `O(n)`.

**Зачем нужны кучи?**

- **Приоритетные очереди:** быстрый доступ к элементу с наибольшим/наименьшим приоритетом.
- **Алгоритмы:**
  - Дейкстры (поиск кратчайших путей),
  - Хаффмана (сжатие данных),
  - A\* (поиск пути).

- **Сортировка:** HeapSort (`O(n log n)`), не требует доп. памяти.

### Min-Heap (минимальная куча)

<b class="text-on-semi-light">Min-Heap</b> — структура, в которой родительский узел всегда меньше или равен своим потомкам. Таким образом, минимальный элемент всегда находится в корне.

**Основные характеристики:**

1. **Свойство упорядоченности:** каждый родитель ≤ его дети.
2. **Полнота:** дерево заполняется уровнями слева направо без пропусков.
3. **Минимальный элемент:** всегда в корне, доступ за `O(1)`.

**Операции и их сложность:**

- **Вставка (`insert`):** элемент добавляется в конец массива (последний уровень дерева) и поднимается вверх (_heapify-up_) — `O(log n)`.
- **Извлечение минимума (`extractMin`):** корень удаляется, последний элемент ставится на его место и опускается вниз (_heapify-down_) — `O(log n)`.
- **Просмотр минимума (`peek`):** просто возвращается корень — `O(1)`.

**Реализация на практике (JavaScript):**

```js
class MinHeap {
  constructor() {
    this.heap = [];
  }

  getParentIndex(i) {
    return Math.floor((i - 1) / 2);
  }
  getLeftChildIndex(i) {
    return 2 * i + 1;
  }
  getRightChildIndex(i) {
    return 2 * i + 2;
  }

  insert(value) {
    this.heap.push(value);
    this.heapifyUp();
  }

  heapifyUp() {
    let index = this.heap.length - 1;
    while (index > 0) {
      let parentIndex = this.getParentIndex(index);
      if (this.heap[parentIndex] > this.heap[index]) {
        [this.heap[parentIndex], this.heap[index]] = [
          this.heap[index],
          this.heap[parentIndex],
        ];
        index = parentIndex;
      } else break;
    }
  }

  extractMin() {
    if (this.heap.length === 0) return null;
    if (this.heap.length === 1) return this.heap.pop();

    const min = this.heap[0];
    this.heap[0] = this.heap.pop();
    this.heapifyDown(0);
    return min;
  }

  heapifyDown(index) {
    let smallest = index;
    let left = this.getLeftChildIndex(index);
    let right = this.getRightChildIndex(index);

    if (left < this.heap.length && this.heap[left] < this.heap[smallest])
      smallest = left;
    if (right < this.heap.length && this.heap[right] < this.heap[smallest])
      smallest = right;

    if (smallest !== index) {
      [this.heap[smallest], this.heap[index]] = [
        this.heap[index],
        this.heap[smallest],
      ];
      this.heapifyDown(smallest);
    }
  }
}

const h = new MinHeap();
h.insert(5);
h.insert(3);
h.insert(8);
h.insert(1);

console.log(h.extractMin()); // 1
console.log(h.extractMin()); // 3
```

**Применение Min-Heap:**

- **Приоритетные очереди:** выбор элемента с наименьшим приоритетом.
- **Алгоритмы:** Дейкстры (поиск кратчайших путей), Хаффмана (кодирование).
- **Сортировка:** HeapSort (на базе кучи).

**Недостатки:**

- Неэффективен для произвольного поиска (`O(n)`).
- Требует балансировки при вставке/удалении.
- Сложнее, чем простые массивы или списки, для базового использования.

### Max-Heap (максимальная куча)

<b class="text-on-semi-light">Max-Heap</b> — структура, в которой родительский узел всегда больше или равен своим потомкам. Таким образом, максимальный элемент всегда находится в корне.

1. **Свойство упорядоченности:** каждый родитель ≥ его дети.
2. **Полнота:** дерево заполняется уровнями слева направо без пропусков.
3. **Максимальный элемент:** всегда в корне, доступ за `O(1)`.

**Операции и их сложность:**

- **Вставка (`insert`):** элемент добавляется в конец массива (последний уровень дерева) и поднимается вверх (_heapify-up_) — `O(log n)`.

- **Просмотр максимума (`peek`):** просто возвращается корень — `O(1)`.

### Fibonacci Heap

**Fibonacci Heap** — это усовершенствованная структура данных для реализации приоритетных очередей, которая улучшает асимптотическую сложность некоторых операций по сравнению с обычными бинарными и биномиальными кучами. Она особенно полезна в алгоритмах, где требуется множество операций объединения (merge) и уменьшения ключа (decreaseKey).

**Основные характеристики:**

1. **Структура:** набор деревьев (обычно представленных в виде корневого списка), где каждое дерево подчиняется свойству кучи (родитель ≤ дети).
2. **Гибкость:** узлы объединяются «лениво» — балансировка откладывается до вызова операций `extractMin`.
3. **Минимальный элемент:** хранится в указателе `min`, доступ за `O(1)`.

**Асимптотическая сложность операций:**

- **Вставка (`insert`):** `O(1)` амортизированно.
- **Объединение (`union` или `merge`):** `O(1)`.
- **Просмотр минимума (`findMin`):** `O(1)`.
- **Извлечение минимума (`extractMin`):** `O(log n)` амортизированно (требует перестройки деревьев).
- **Уменьшение ключа (`decreaseKey`):** `O(1)` амортизированно.
- **Удаление (`delete`):** `O(log n)`.

**Ключевая идея:**

- В отличие от бинарной кучи, где дерево всегда строгое, **Fibonacci Heap** позволяет временно хранить «расбалансированные» деревья и приводит их к упорядоченной форме только при необходимости.
- Это делает операции `insert` и `merge` чрезвычайно быстрыми (за `O(1)`).

**Пример структуры:**

Корневой список:

```
[min: 2] → 7 → 18 → 25
```

- Узел `2` указывает на минимальный элемент.
- У него могут быть дочерние деревья, образующие подкучи.

**Сравнение с бинарной кучей:**

- В бинарной куче `decreaseKey` = `O(log n)`, а в Fibonacci Heap — `O(1)`.
- Поэтому в алгоритмах, где часто нужно уменьшать ключи (например, Дейкстра, Прим), **Fibonacci Heap** даёт преимущество.

**Псевдокод вставки:**

```pseudo
function insert(H, x):
  x.degree = 0
  x.parent = NIL
  x.child = NIL
  x.mark = FALSE
  add x to the root list of H
  if H.min == NIL or x.key < H.min.key:
    H.min = x
  H.n = H.n + 1
```

**Применение:**

- Алгоритм Дейкстры (для графов с большим количеством вершин).
- Алгоритм Прима (минимальное остовное дерево).
- Любые задачи с частыми `decreaseKey` и `merge`.

**Недостатки:**

- Реализация гораздо сложнее, чем у бинарных куч.
- Большая константа в реальной производительности: на практике бинарные кучи часто быстрее для небольших данных.
- Используется в основном в теоретических и исследовательских целях.

## Графы (Graphs)

**Графы** — это фундаментальные структуры данных, где объекты представлены в виде **вершин** (nodes), а связи между ними — в виде **рёбер** (edges). В отличие от деревьев, графы не обязаны иметь корень и могут содержать циклы.

**Ключевые характеристики графов:**

- **Ориентированные и неориентированные:** рёбра могут иметь направление (стрелку) или быть двусторонними.
- **Взвешенные и невзвешенные:** рёбра могут иметь числовой вес (стоимость, расстояние) или быть равнозначными.
- **Связные и несвязные:** граф может представлять собой единую связанную структуру или состоять из нескольких компонент.
- **Плотность:** графы могут быть плотными (много рёбер) или разреженными (мало рёбер).

**Применение графов:**

- Социальные сети (пользователи и связи между ними).
- Карты и навигация (города и дороги).
- Компиляторы (зависимости между модулями).
- Поиск оптимальных путей (алгоритмы Дейкстры, A\*).
- Планирование задач (графы зависимостей).

**Представления графов:**

- **Матрица смежности:** таблица `n × n`, показывающая наличие или вес рёбер.
- **Список смежности:** для каждой вершины хранится список соседей, что эффективно для разреженных графов.

> Здесь не будет разбора алгоритмов Дейкстры, Беллмана-Форда, Флойда-Уоршелла и обходов графов в ширину и высоту, так как они будут разбираться в следуюших страницах.

### Матрица смежности (Adjacency Matrix)

<b class="text-on-semi-light">Матрица смежности</b> — способ представления графа с помощью квадратной матрицы `n × n`, где `n` — количество вершин. Каждая ячейка `matrix[i][j]` указывает, существует ли ребро между вершинами `i` и `j`, а в случае взвешенного графа хранит вес ребра.

**Основные характеристики:**

1. **Размер:** всегда `n × n`, независимо от количества рёбер.
2. **Доступ к рёбрам:** проверка наличия ребра выполняется за `O(1)`.
3. **Симметрия:** в неориентированном графе `matrix[i][j] = matrix[j][i]`.
4. **Память:** требует `O(n²)`, что делает её неэффективной для разреженных графов.

**Операции и их сложность:**

- **Проверка наличия ребра (`hasEdge(i, j)`):** `O(1)`.
- **Добавление/удаление ребра:** `O(1)`.
- **Перебор всех рёбер:** `O(n²)`.

**Реализация на практике (JavaScript):**

```js
class AdjacencyMatrix {
  constructor(size) {
    this.size = size;
    this.matrix = Array.from({ length: size }, () => Array(size).fill(0));
  }

  addEdge(i, j, weight = 1) {
    this.matrix[i][j] = weight;
    this.matrix[j][i] = weight; // для неориентированного графа
  }

  removeEdge(i, j) {
    this.matrix[i][j] = 0;
    this.matrix[j][i] = 0;
  }

  hasEdge(i, j) {
    return this.matrix[i][j] !== 0;
  }

  print() {
    console.log(this.matrix.map((row) => row.join(" ")).join("\n"));
  }
}

const g = new AdjacencyMatrix(3);
g.addEdge(0, 1);
g.addEdge(1, 2);
g.print();
```

**Пример вывода:**

```
0 1 0
1 0 1
0 1 0
```

**Применение матрицы смежности:**

- Подходит для **плотных графов**, где количество рёбер близко к `n²`.
- Используется в алгоритмах, требующих быстрый доступ к информации о наличии рёбер, например, в алгоритме Флойда–Уоршелла.

**Недостатки:**

- Требует слишком много памяти для больших разреженных графов.
- Перебор всех рёбер всегда занимает `O(n²)`, даже если рёбер мало.

### Список смежности (Adjacency List)

<b class="text-on-semi-light">Список смежности</b> — это структура представления графа, где для каждой вершины хранится список всех её соседей (вершин, с которыми она соединена ребром).

**Основные характеристики:**

1. **Хранение:** у каждой вершины есть массив/список, содержащий её смежные вершины.
2. **Память:** требует `O(V + E)`, где `V` — вершины, `E` — рёбра (эффективнее, чем `O(V²)` у матрицы).
3. **Доступ:** проверка наличия ребра занимает `O(deg(v))`, где `deg(v)` — степень вершины.

**Операции и их сложность:**

- **Добавить ребро:** `O(1)` (добавление в список).
- **Удалить ребро:** `O(deg(v))` (поиск и удаление).
- **Итерация по соседям вершины:** `O(deg(v))`.
- **Обход всех рёбер графа:** `O(V + E)`.

**Реализация на практике (JavaScript):**

```js
class Graph {
  constructor() {
    this.adjList = new Map();
  }

  addVertex(v) {
    if (!this.adjList.has(v)) {
      this.adjList.set(v, []);
    }
  }

  addEdge(v, w) {
    this.addVertex(v);
    this.addVertex(w);
    this.adjList.get(v).push(w);
    this.adjList.get(w).push(v); // если граф неориентированный
  }

  getNeighbors(v) {
    return this.adjList.get(v) || [];
  }
}

const g = new Graph();
g.addEdge("A", "B");
g.addEdge("A", "C");
g.addEdge("B", "D");

console.log(g.getNeighbors("A")); // [ 'B', 'C' ]
```

**Применение списка смежности:**

- **Социальные сети:** друзья/подписчики (редко все связаны со всеми).
- **Sparse-графы:** когда рёбер существенно меньше, чем `V²`.
- **Алгоритмы:** поиск в глубину (DFS), поиск в ширину (BFS), Дейкстра, Крускал.

**Недостатки:**

- Проверка существования конкретного ребра медленнее (`O(deg(v))`).
- Для очень плотных графов занимает больше времени на доступ к данным, чем матрица смежности.

### Directed Graph (Ориентированный граф)

<b class="text-on-semi-light">Directed Graph</b> — это граф, в котором каждое ребро имеет направление. Если существует ребро `A → B`, это не означает наличия обратного ребра `B → A`.

**Основные характеристики:**

1. **Направленные рёбра:** каждое соединение задаётся упорядоченной парой вершин.
2. **Ассиметричность связей:** наличие пути от `u` к `v` не гарантирует пути обратно.
3. **Типы графов:** может быть с циклами (например, веб-ссылки) или ациклическим (DAG — Directed Acyclic Graph).

**Операции и их свойства:**

- **Добавление вершины:** `O(1)` в матрице смежности, `O(1)` в списке смежности.
- **Добавление ребра (`u → v`):** `O(1)` в обеих структурах.
- **Проверка наличия ребра:** `O(1)` в матрице смежности, `O(deg(u))` в списке смежности.
- **Обход графа (DFS/BFS):** `O(V + E)`.

**Пример представления (Adjacency List, JavaScript):**

```js
class DirectedGraph {
  constructor() {
    this.adjList = new Map();
  }

  addVertex(v) {
    if (!this.adjList.has(v)) {
      this.adjList.set(v, []);
    }
  }

  addEdge(u, v) {
    this.addVertex(u);
    this.addVertex(v);
    this.adjList.get(u).push(v);
  }

  getNeighbors(v) {
    return this.adjList.get(v) || [];
  }
}

const g = new DirectedGraph();
g.addEdge("A", "B");
g.addEdge("A", "C");
g.addEdge("B", "D");

console.log(g.getNeighbors("A")); // ["B", "C"]
console.log(g.getNeighbors("B")); // ["D"]
```

**Применение Directed Graph:**

- **Зависимости:** граф вызовов функций, компиляторные зависимости (например, порядок сборки модулей).
- **Потоки данных:** модели сетевого трафика, процессы в операционных системах.
- **Web-граф:** ссылки между страницами.
- **Алгоритмы:** топологическая сортировка (для DAG), поиск сильносвязанных компонент (алгоритмы Косарайю, Тарьяна).

**Недостатки:**

- Сложнее визуализировать и отлаживать, чем неориентированные графы.
- Анализ сильносвязанных компонент и циклов требует дополнительных алгоритмов.
- Для некоторых задач поиск путей усложняется по сравнению с неориентированными структурами.

### Undirected Graph (Неориентированный граф)

<b class="text-on-semi-light">Undirected Graph</b> — это граф, в котором рёбра не имеют направления. Связь между вершинами двусторонняя: если существует ребро между `A` и `B`, то оно подразумевает как `A → B`, так и `B → A`.

**Основные характеристики:**

1. **Симметричность:** наличие ребра `(u, v)` означает, что вершины соединены в обе стороны.
2. **Хранение:**
   - В матрице смежности запись симметрична относительно главной диагонали.
   - В списке смежности каждая вершина содержит ссылку на соседа, и наоборот.

3. **Количество рёбер:** для `n` вершин максимальное число рёбер равно `n(n-1)/2`.

**Операции и их сложность:**

- **Проверка наличия ребра:**
  - Матрица смежности — `O(1)`.
  - Список смежности — `O(deg(v))`, где `deg(v)` — степень вершины.

- **Перебор соседей:**
  - Список смежности — `O(deg(v))`.
  - Матрица смежности — `O(n)`.

- **Добавление ребра:**
  - Матрица смежности — `O(1)`.
  - Список смежности — `O(1)` (добавить в два списка).

**Реализация на практике (JavaScript):**

```js
class UndirectedGraph {
  constructor() {
    this.adjList = new Map();
  }

  addVertex(v) {
    if (!this.adjList.has(v)) {
      this.adjList.set(v, []);
    }
  }

  addEdge(v, w) {
    this.addVertex(v);
    this.addVertex(w);
    this.adjList.get(v).push(w);
    this.adjList.get(w).push(v); // так как граф неориентированный
  }

  getNeighbors(v) {
    return this.adjList.get(v) || [];
  }
}

const g = new UndirectedGraph();
g.addEdge("A", "B");
g.addEdge("A", "C");
g.addEdge("B", "D");

console.log(g.getNeighbors("A")); // ["B", "C"]
console.log(g.getNeighbors("B")); // ["A", "D"]
```

**Применение Undirected Graph:**

- **Социальные сети:** дружба, где связь взаимна.
- **Дороги и транспорт:** если движение разрешено в обе стороны.
- **Сети:** peer-to-peer соединения без выделенного направления.

**Недостатки:**

- Не подходит для моделирования направленных зависимостей (например, очередности задач).
- При хранении

### Weighted Graph (Взвешенный граф)

<b class="text-on-semi-light">Weighted Graph</b> — это граф, в котором каждому ребру приписан определённый вес (числовое значение). Вес может отражать расстояние, стоимость, время или любую другую метрику, связанную с переходом между вершинами.

**Основные характеристики:**

1. **Рёбра с весами:** каждое ребро хранит не только связь между вершинами, но и числовой вес.
2. **Поддержка разных типов графов:** может быть как ориентированным, так и неориентированным.
3. **Моделирование реальности:** веса позволяют описывать «стоимость» перемещения или взаимодействия.

**Операции и их сложность:**

- **Доступ к весу ребра:** зависит от структуры хранения (матрица смежности — `O(1)`, список смежности — `O(deg(v))`).
- **Обновление веса:** также зависит от хранения (обычно `O(1)`).
- **Поиск кратчайшего пути:** Дейкстра (`O((V+E) log V)`), Беллман-Форд (`O(VE)`).

**Реализация на практике (JavaScript, список смежности):**

```js
class WeightedGraph {
  constructor() {
    this.adjList = new Map();
  }

  addVertex(v) {
    if (!this.adjList.has(v)) this.adjList.set(v, []);
  }

  addEdge(v1, v2, weight) {
    this.adjList.get(v1).push({ node: v2, weight });
    this.adjList.get(v2).push({ node: v1, weight }); // для неориентированного графа
  }

  getNeighbors(v) {
    return this.adjList.get(v);
  }
}

const g = new WeightedGraph();
g.addVertex("A");
g.addVertex("B");
g.addVertex("C");

g.addEdge("A", "B", 4);
g.addEdge("A", "C", 2);
g.addEdge("B", "C", 5);

console.log(g.getNeighbors("A"));
// [ { node: 'B', weight: 4 }, { node: 'C', weight: 2 } ]
```

**Применение Weighted Graph:**

- **Поиск кратчайших путей:** Дейкстра, A\*, Беллман-Форд.
- **Оптимизация:** маршрутизация в сетях, транспортные задачи.
- **Моделирование:** дорожные карты, логистика, энергосети.

**Недостатки:**

- Более сложные алгоритмы, чем для невзвешенных графов.
- Возможны отрицательные веса, что требует особых алгоритмов (например, Беллман-Форд вместо Дейкстры).

### Unweighted Graph (Невзвешенный граф)

<b class="text-on-semi-light">Unweighted Graph</b> — структура, в которой все рёбра имеют одинаковую стоимость (обычно принимается равной `1`). Здесь важно только наличие или отсутствие связи между вершинами, а не её «вес».

**Основные характеристики:**

1. **Рёбра без веса:** все связи считаются равнозначными.
2. **Простота модели:** граф отражает лишь факт соединения узлов.
3. **Кратчайший путь:** измеряется в количестве шагов (рёбер), а не в суммарной стоимости.

**Операции и их особенности:**

- **Поиск в ширину (BFS):** находит кратчайший путь по числу рёбер за `O(V + E)`.
- **Поиск в глубину (DFS):** используется для обхода и анализа структуры.
- **Добавление вершины/ребра:** зависит от представления (список или матрица смежности).

**Пример (список смежности, JavaScript):**

```js
class UnweightedGraph {
  constructor() {
    this.adjList = new Map();
  }

  addVertex(v) {
    if (!this.adjList.has(v)) this.adjList.set(v, []);
  }

  addEdge(v, w) {
    this.adjList.get(v).push(w);
    this.adjList.get(w).push(v); // для неориентированного графа
  }

  bfs(start) {
    const visited = new Set();
    const queue = [start];
    visited.add(start);

    while (queue.length) {
      const v = queue.shift();
      console.log(v);

      for (const neighbor of this.adjList.get(v)) {
        if (!visited.has(neighbor)) {
          visited.add(neighbor);
          queue.push(neighbor);
        }
      }
    }
  }
}

const g = new UnweightedGraph();
g.addVertex("A");
g.addVertex("B");
g.addVertex("C");

g.addEdge("A", "B");
g.addEdge("A", "C");

g.bfs("A");
```

**Применение Unweighted Graph:**

- **Социальные связи:** «друзья» в сетях без весов.
- **Грид-лабиринты:** поиск кратчайшего пути по клеткам.
- **BFS-кратчайшие пути:** когда важен только минимальный шаговый путь.

**Недостатки:**

- Нельзя отразить различие в «стоимости» связей.
- Модель слишком упрощает задачи, где важен вес (например, время или расстояние).

### DAG (Directed Acyclic Graph)

<b class="text-on-semi-light">DAG (Directed Acyclic Graph)</b> — это ориентированный граф, который **не содержит циклов**. Иными словами, невозможно начать в одной вершине и, двигаясь по рёбрам, вернуться обратно в неё же. Такое ограничение делает DAG особенно полезным в задачах, где важен порядок и зависимости.

**Основные характеристики:**

1. **Ориентированность:** каждое ребро имеет направление `u → v`.
2. **Ацикличность:** отсутствуют циклы, что гарантирует возможность упорядочивания вершин.
3. **Топологический порядок:** вершины можно расположить линейно так, что все рёбра идут «слева направо».
4. **Модель зависимостей:** вершины часто представляют задачи, а рёбра — зависимости между ними.

**Операции и их особенности:**

- **Топологическая сортировка:** выполняется за `O(V + E)` с помощью DFS или алгоритма Кана.
- **Динамическое программирование на DAG:** позволяет эффективно решать задачи поиска кратчайшего/длиннейшего пути.
- **Добавление вершины/ребра:** выполняется аналогично обычным графам, но нужно следить, чтобы не образовался цикл.

**Пример (DAG с топологической сортировкой, JavaScript):**

```js
class DAG {
  constructor() {
    this.adjList = new Map();
  }

  addVertex(v) {
    if (!this.adjList.has(v)) this.adjList.set(v, []);
  }

  addEdge(v, w) {
    this.adjList.get(v).push(w);
  }

  topologicalSort() {
    const visited = new Set();
    const stack = [];

    const dfs = (v) => {
      visited.add(v);
      for (const neighbor of this.adjList.get(v)) {
        if (!visited.has(neighbor)) dfs(neighbor);
      }
      stack.push(v);
    };

    for (const v of this.adjList.keys()) {
      if (!visited.has(v)) dfs(v);
    }

    return stack.reverse();
  }
}

const g = new DAG();
g.addVertex("A");
g.addVertex("B");
g.addVertex("C");
g.addVertex("D");

g.addEdge("A", "B");
g.addEdge("A", "C");
g.addEdge("B", "D");
g.addEdge("C", "D");

console.log(g.topologicalSort()); // [ 'A', 'C', 'B', 'D' ] или [ 'A', 'B', 'C', 'D' ]
```

**Применение DAG:**

- **Компиляторы:** определение порядка выполнения выражений.
- **Планировщики задач:** выполнение зависимых задач (например, в CI/CD системах).
- **Системы контроля версий (Git):** коммиты образуют DAG.
- **Data pipelines:** обработка данных с зависимостями между шагами.

**Недостатки:**

- DAG сложнее поддерживать: при добавлении рёбер нужно следить, чтобы не образовался цикл.
- Не всегда подходит для задач, где допустимы повторные связи.
- Может быть менее наглядным для очень больших сетей зависимостей.

## Критерии выбора структуры данных

1. Частота операций (чтение vs запись)

- Много поисков → Hash Table (`O(1)`)
- Нужны вставки + поддержка порядка → BST (`O(log n)`)

2. Время и память

- HashMap → быстрее, но больше памяти
- Linked List → экономия памяти, но медленный поиск

3. Параллелизм и многопоточность

- Concurrent HashMap
- Lock-free структуры
- Immutable деревья

4. Локальность доступа (Cache Friendliness)

- Массивы → элементы подряд в памяти → кеш работает эффективно
- Списки → элементы раскиданы → кеш работает хуже

5. Персистентность (Immutable структуры)

- Нужен "откат" или хранение истории (undo/redo, функциональные языки)
- Используются persistent trees

6. Внешняя память / диск

- В БД используют B-деревья
- Читают блоками, а не по одному элементу
