# Сборщик мусора

## Как работает Garbage Collector в JavaScript

### Общая идея

JavaScript использует **автоматическое управление памятью**, где выделение и освобождение памяти происходит без участия разработчика. В основе этой системы лежит **сборщик мусора (Garbage Collector)**, встроенный в движок (например, **V8** для Chrome и Node.js, **SpiderMonkey** для Firefox). Его задача — освободить память, занимаемую объектами, которые стали **недостижимыми**.

Ключевое понятие здесь — **достижимость (reachability)**. Любой объект, до которого можно добраться, начиная от корневых ссылок (**root references**), считается живым (используемым). Всё остальное считается мусором и подлежит удалению.

К корневым ссылкам относятся:

- глобальные объекты (`window`, `globalThis`, `global`);
- текущий стек вызовов (переменные и параметры активных функций);
- ссылки из активных замыканий;
- внутренние объекты, удерживаемые движком (например, объекты событий, таймеров, промисов).

---

### Алгоритм mark-and-sweep

Базовый механизм работы GC в JS — **mark-and-sweep** (пометка и зачистка). Это универсальный алгоритм, применяемый практически во всех современных реализациях.

#### Этап 1: Mark (пометка достижимых объектов)

GC начинает с обхода всех корней и рекурсивно помечает все объекты, которые можно достичь по ссылкам. Для этого движок хранит внутреннюю структуру связей между объектами в памяти (граф объектов).

1. Обход начинается с глобальных корней.
2. Каждый найденный объект получает метку «reachable».
3. Для каждого помеченного объекта GC проходит по всем его ссылкам и помечает связанные объекты.

Этот процесс продолжается, пока не будут пройдены все возможные пути достижимости.

#### Этап 2: Sweep (зачистка недостижимых объектов)

После маркировки движок проходит по куче (heap) и освобождает память от объектов, не имеющих метки достижимости. Освобождённая память возвращается в пул свободной памяти и может быть использована повторно при создании новых объектов.

#### Этап 3: Compact (компактация)

Некоторые реализации GC дополнительно проводят **компактацию** — физическое перемещение живых объектов в памяти, чтобы устранить фрагментацию и улучшить эффективность последующих аллокаций.

---

### Циклические ссылки и граф достижимости

Garbage Collector работает не по принципу подсчёта ссылок, а по **графу достижимости**. Благодаря этому он корректно обрабатывает даже **циклические ссылки**.

Пример:

```js
function createCycle() {
  const a = {};
  const b = {};
  a.link = b;
  b.link = a;
  return a;
}

let obj = createCycle();
obj = null;
```

Оба объекта (`a` и `b`) ссылаются друг на друга, но к ним невозможно добраться от корневых ссылок. GC определит их как недостижимые и освободит память.

---

### Поколенческая сборка (Generational GC)

Современные движки используют **поколенческую модель** управления памятью. Она основана на эмпирическом наблюдении, что:

> Большинство объектов живут недолго, а долгоживущие объекты составляют меньшинство.

#### Молодое и старое поколения

1. **Young Generation** (молодое поколение) — область памяти, куда попадают все новые объекты. Эта часть очищается часто и быстро, потому что большинство объектов исчезает вскоре после создания (например, временные массивы, контексты вызова функций).
2. **Old Generation** (старое поколение) — объекты, пережившие несколько сборок в молодом поколении. Эти объекты проверяются реже и более сложными алгоритмами.

Такой подход позволяет сильно сократить паузы, поскольку большая часть мусора собирается из молодого поколения, не затрагивая старое.

---

### Minor и Major GC

В движке V8 (используется в Chrome и Node.js) процесс делится на два типа сборки:

#### Minor GC

- Отвечает за молодое поколение.
- Работает часто и быстро.
- Использует **копирующий алгоритм (copying collector)**: память делится на две области (_from-space_ и _to-space_). При сборке живые объекты копируются из одной области в другую, а старая полностью очищается. Это избавляет от необходимости индивидуально удалять объекты и ускоряет работу.

#### Major GC

- Отвечает за старое поколение.
- Запускается значительно реже.
- Использует комбинацию **mark-and-sweep** и **compaction**.
- Основная цель — очистка долгоживущих, но уже неиспользуемых объектов.

---

### Инкрементальный и параллельный GC

Чтобы избежать длительных пауз, движки внедряют **инкрементальные** и **параллельные** алгоритмы сборки.

- **Incremental GC** — делит процесс на множество маленьких шагов, которые выполняются между другими операциями движка. Это снижает время блокировки основного потока.
- **Concurrent GC** — часть работы (например, маркировку) выполняет в отдельных потоках, параллельно с выполнением JavaScript.
- **Idle-time GC** — запускается, когда движок находится в состоянии покоя (например, браузер ждёт пользовательского ввода), чтобы не мешать основным вычислениям.

Такой подход делает процесс сборки практически незаметным для пользователя и уменьшает «фризы» интерфейса.

---

### Внутренние структуры и оптимизации движков

Современные движки используют комбинации сложных техник, чтобы повысить эффективность работы GC и минимизировать влияние на производительность.

- **Write Barriers** — механизмы, которые отслеживают, когда объект из старого поколения начинает ссылаться на объект из молодого. Это позволяет не сканировать всю кучу при каждом проходе.
- **Remembered Sets** — специальные таблицы, хранящие информацию о таких перекрёстных ссылках между поколениями.
- **Compacting GC** — алгоритм, сжимающий память после зачистки, устраняя фрагментацию.
- **Generational Promotion** — процесс, при котором объекты, пережившие несколько сборок, автоматически перемещаются в старшее поколение.

---

### Работа с лексическими окружениями и замыканиями

GC учитывает **лексическое окружение** (Lexical Environment) каждой функции. Если внутренние функции продолжают ссылаться на переменные внешней, это окружение остаётся достижимым.

```js
function outer() {
  const bigArray = new Array(1_000_000);
  return function inner() {
    console.log("Hello");
  };
}

const fn = outer();
```

В этом примере `bigArray` не используется внутри `inner`, но остаётся в памяти, потому что ссылка на лексическое окружение `outer` жива через `fn`. GC не может освободить память, пока `fn` существует.

---

### Влияние модели выполнения JavaScript

Работа GC тесно связана с моделью выполнения JavaScript:

- Каждый активный **контекст выполнения** (execution context) удерживает ссылки на свои локальные переменные и аргументы.
- **Стек вызовов** (call stack) и **heap** (куча) существуют независимо: стек хранит ссылки на объекты в куче.
- После завершения функции контекст снимается со стека, и если на его объекты не осталось других ссылок, они становятся кандидатами на удаление.

GC запускается автоматически, но движок может адаптировать частоту и стратегию в зависимости от доступной памяти, активности приложения и поведения кучи.

---

### Итоговая модель под капотом

1. В памяти движка существует **heap** — неструктурированная область, где живут все объекты.
2. **Roots** (корневые ссылки) — это точки входа, от которых начинается обход достижимости.
3. Сборщик мусора периодически приостанавливает выполнение кода (или частично, в инкрементальном режиме) и выполняет следующие шаги:
   - маркирует все объекты, достижимые из корней;
   - очищает память от всех остальных;
   - при необходимости сжимает (компактирует) память.

4. Алгоритмы Generational, Incremental и Concurrent GC работают совместно, чтобы минимизировать паузы и сделать управление памятью максимально прозрачным.

---

Таким образом, Garbage Collector в JavaScript — это не просто фоновая служба, а сложный, многослойный механизм, тесно интегрированный в архитектуру движка. Он сочетает граф достижимости, поколенческую модель, параллелизм и оптимизации уровня системного программирования, чтобы обеспечить баланс между скоростью и эффективным использованием памяти.
