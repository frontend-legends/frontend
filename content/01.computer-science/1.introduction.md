# Introduction!

Эта страница посвящена краткому введению в программирование и computer science.

## Что такое бит?

Бит — это наименьшая единица информации в компьютерных системах. Он может принимать только два значения:
- 0 (ложь, отсутствие сигнала, низкий уровень напряжения)
- 1 (истина, наличие сигнала, высокий уровень напряжения)

Зачем нужны биты?
Все данные в компьютере (числа, текст, изображения, видео) хранятся и обрабатываются в двоичной системе, где информация представляется последовательностью битов.

- **1 бит** = 0 или 1
- **1 байт** = **8 бит**
- **1 килобайт (KB)** = 1024 байта
- **1 мегабайт (MB)** = 1024 KB
- **1 гигабайт (GB)** = 1024 MB

## What is computer?

Компьютер — это электронное устройство, способное **принимать, обрабатывать, хранить и выводить данные**. Он работает на основе программного обеспечения (software) и аппаратных компонентов (hardware).

## What is web?

**Веб (Web, World Wide Web, WWW)** — это **глобальная система взаимосвязанных документов и ресурсов**, доступных через интернет с помощью веб-браузера.

- **Веб 1.0** – статические сайты, только чтение (1990-е).  
- **Веб 2.0** – интерактивные сайты, соцсети, динамический контент (2000-е).  
- **Веб 3.0** – децентрализованный интернет, блокчейн, AI (будущее).

### What is DNS?

**DNS (Domain Name System)** — это система, которая переводит **доменные имена** (например, `google.com`) в **IP-адреса** (например, `142.250.182.206`), чтобы компьютеры могли находить друг друга в интернете.

Типы DNS-записей:
- **A (Address)** – привязывает домен к IPv4-адресу.  
- **AAAA** – привязывает домен к IPv6-адресу.  
- **CNAME (Canonical Name)** – перенаправляет один домен на другой.  
- **MX (Mail Exchange)** – указывает серверы для почты.  
- **TXT** – текстовая информация (SPF, DKIM, верификация).

Как работает DNS?
- **Браузер ищет IP-адрес** в кеше (если есть, сразу загружает сайт).  
- **ОС проверяет локальный DNS-кеш** (если нет, передает запрос дальше).  
- **Запрос отправляется на рекурсивный DNS-сервер** (обычно у провайдера).  
- **Рекурсивный сервер спрашивает у корневых DNS-серверов** (`.`).  
- **Корневой сервер направляет к TLD-серверу** (`.com`, `.ru`).  
- **TLD-сервер указывает на авторитетный DNS-сервер** (`example.com`).  
- **Авторитетный сервер возвращает IP-адрес** (например, `192.168.1.1`).  
- **Рекурсивный сервер кеширует IP и передает браузеру**.  
- **Браузер подключается к серверу по IP и загружает сайт**.

Что такое CDN?

**CDN (Content Delivery Network, сеть доставки контента)** — это **глобальная сеть серверов**, которая хранит копии сайта и раздает их пользователям из ближайшего дата-центра. Это ускоряет загрузку страниц и снижает нагрузку на основной сервер.

What is SEO?

SEO (Search Engine Optimization) — это оптимизация сайта для поисковых систем (Google, Yandex, Bing) с целью улучшения позиций в результатах поиска и привлечения большего количества посетителей.

Как работает SEO?
- **Поисковый робот** сканирует сайт (индексирует контент).  
- **Алгоритмы анализируют сайт** (качество контента, скорость загрузки, ссылки и т. д.).  
- **Сайт ранжируется** в результатах поиска по ключевым запросам.  
- **Чем лучше оптимизация — тем выше сайт в поиске** и больше трафика.

Основные факторы SEO
**On-Page SEO (внутренняя оптимизация)**  
- Качественный и уникальный контент.  
- Использование ключевых слов.  
- Оптимизированные заголовки (`H1, H2, H3`).  
- Метатеги (`title`, `description`).  
- Читаемые URL (пример: `site.com/blog/seo`).  
- Оптимизированные изображения (`alt`-теги, сжатие).  
- Быстрая загрузка страниц.

**Off-Page SEO (внешняя оптимизация)**  
- Ссылки с других сайтов (беклинки).  
- Упоминания в соцсетях.  
- Репутация и доверие к сайту.

**Technical SEO (техническая оптимизация)**  
- Мобильная адаптация.  
- HTTPS (SSL-сертификат).  
- Карта сайта (`sitemap.xml`).  
- Файл `robots.txt` для управления индексацией.

Что происходит, когда пишешь google.com?

Когда ты вводишь `google.com` в адресной строке браузера и нажимаешь Enter, происходит целая цепочка технических процессов. Разберёмся по шагам.

**1. Ввод URL и проверка кеша**
Когда ты вводишь `google.com`, браузер сначала проверяет кеш:
- **DNS-кеш** – локальное хранилище браузера, где могут быть сохранены IP-адреса посещённых сайтов.
- **Кеш операционной системы** – если в браузере нет записи, проверяется кеш в системе.
- **Локальный hosts-файл** – ОС проверяет, нет ли в файле `/etc/hosts` (Linux/macOS) или `C:\Windows\System32\drivers\etc\hosts` (Windows) привязки `google.com` к IP.

Если IP-адрес найден, он используется сразу. Если нет — запускается DNS-запрос.

**2. DNS-резолвинг (поиск IP-адреса)**
Если кеша нет, браузер делает запрос к **DNS-серверу**, который переводит `google.com` в IP-адрес. Процесс:
1. Браузер запрашивает у **рекурсивного DNS-сервера** (обычно это серверы провайдера).
2. Сервер ищет в своём кешe, если нет — идёт к **корневым DNS-серверам**.
3. Корневой сервер перенаправляет к **серверу зоны .com**.
4. Сервер зоны `.com` указывает на **DNS Google (NS-серверы)**.
5. NS-серверы Google возвращают IP-адрес (например, `142.250.187.78`).

Теперь браузер знает, куда отправлять запрос.

**3. Установление TCP-соединения (трёхстороннее рукопожатие)**
Используется протокол **TCP (Transmission Control Protocol)**:
- **SYN** – клиент (браузер) отправляет серверу запрос на установку соединения.
- **SYN-ACK** – сервер отвечает согласием.
- **ACK** – клиент подтверждает соединение.

Теперь можно передавать данные. Если сайт использует HTTPS, дополнительно запускается **TLS Handshake**.

**4. TLS Handshake (если HTTPS)**
Если сайт использует **HTTPS**, происходит установление защищённого соединения:
1. Браузер отправляет **ClientHello** (список поддерживаемых шифров).
2. Сервер отвечает **ServerHello** (выбирает алгоритм шифрования).
3. Сервер отправляет свой **SSL-сертификат** (проверяется подлинность).
4. Генерируется **сессионный ключ**, которым шифруются дальнейшие данные.

Теперь данные передаются по **зашифрованному** соединению.

**5. Отправка HTTP-запроса**

Браузер отправляет **HTTP-запрос** на сервер:
```vbnet
GET / HTTP/1.1  
Host: google.com  
User-Agent: Chrome/120.0.0  
Accept: text/html  
```

**6. Сервер обрабатывает запрос**

На сервере Google работает **Nginx, Envoy или собственный веб-сервер**. Он делает следующее:
1. Проверяет, что запрашивает пользователь (HTML, CSS, JS, API).
2. Может запросить данные из базы данных (если нужно).
3. Генерирует HTML-страницу.
4. Отправляет **HTTP-ответ** браузеру.

**7. Браузер получает и обрабатывает ответ**

Сервер отвечает:
```yml
HTTP/1.1 200 OK  
Content-Type: text/html  
Content-Length: 1024  
```

Далее браузер:
1. Парсит HTML-код.
2. Загружает CSS, JS, изображения (если есть).
3. Отображает страницу на экране.

---

**Proxy-сервер** — это такой посредник между тобой (твоим компьютером) и интернетом. Когда ты отправляешь запрос, например, хочешь открыть сайт, этот запрос сначала идет на прокси-сервер, а уже он отправляет его дальше в интернет. А ответ от сервера он тоже сначала получает сам, а потом передает тебе.

Зачем он нужен:
- **Скрыть твой IP-адрес**. Сайт увидит IP прокси, а не твой.
- **Обход блокировок**. Можно заходить на сайты, закрытые в твоем регионе.
- **Фильтрация и контроль трафика**. Например, в офисах могут ограничивать доступ к определенным сайтам.
- **Кеширование**. Прокси может сохранять часто запрашиваемые страницы, чтобы быстрее отдавать их пользователям.
- **Повышение анонимности**.

Какие бывают:
- **Forward Proxy** — обычный прокси для клиентов.
- **Reverse Proxy** — ставится перед сервером и принимает запросы от интернета, защищая или балансируя нагрузку.
- **Open Proxy** — доступен для всех, но обычно небезопасен.

What is URL?

URL (Uniform Resource Locator) — это уникальный адрес ресурса в интернете, который указывает путь к веб-странице, изображению, файлу и т. д.

Структура URL
`https://example.com:8080/blog/article?id=10#comments

🔹 **`https://`** – протокол (HTTP или HTTPS).  
🔹 **`example.com`** – доменное имя (адрес сайта).  
🔹 **`:8080`** – порт (необязательный, обычно `80` для HTTP и `443` для HTTPS).  
🔹 **`/blog/article`** – путь к ресурсу на сервере.  
🔹 **`?id=10`** – параметры запроса (ключ=`значение`, используются для передачи данных).  
🔹 **`#comments`** – якорь (перенаправляет на нужный фрагмент страницы).

Что такое URI и отличие от URL?

**URI (Uniform Resource Identifier)** — это **универсальный идентификатор ресурса**, который определяет его местоположение или имя в системе.

**URL — это частный случай URI**, который указывает путь к ресурсу в интернете.

Пример:
- **URI:** `mailto:info@example.com` (идентифицирует почтовый адрес).
- **URL:** `https://example.com/blog` (указывает путь к странице).
- **Каждый URL — это URI, но не каждый URI — это URL.**

---

Query string — это часть URL, которая содержит параметры запроса в виде **ключ=значение** и передает данные на сервер. Она начинается с `?`, а несколько параметров разделяются `&`.

- Query string передает параметры в URL.  
- Символы `?`, `&`, `=` **кодируются**.  
- **Порядок параметров не важен**, но API могут обрабатывать их по-разному.  
- **Массивы и объекты** передаются разными способами.  
- **Ограниченная длина** — не стоит передавать большие данные.  
- **Чувствителен к регистру** (`Sort ≠ sort`).  
- **Может влиять на кеширование браузером**.

Хотя спецификация не ставит жесткий лимит, браузеры и серверы могут ограничивать длину query string. Например:
- **Chrome / Firefox** → ~2000 символов
- **IE** → ~2083 символа
- **Nginx** → лимит по умолчанию **8192 байта**

What is HTTP?
**HTTP (HyperText Transfer Protocol)** — это **протокол передачи данных** в интернете. Он определяет, **как клиент (браузер) и сервер обмениваются данными** (HTML, CSS, JS, JSON, изображения и т. д.).

 **Основные характеристики HTTP**
- **Клиент-серверная модель** → Браузер (клиент) отправляет запрос, сервер отвечает.
- **Без сохранения состояния (stateless)** → Каждый запрос обрабатывается **независимо** от других.
- **Текстовый протокол** → Читаемый формат запроса/ответа.
- **Расширяемый** → Поддерживает заголовки (`headers`) для передачи метаданных.

Как работает HTTP?
1. Клиент (браузер) отправляет HTTP-запрос к серверу.
2. Сервер принимает запрос, обрабатывает его и формирует HTTP-ответ.
3. Клиент получает ответ и отображает содержимое страницы.

Методы HTTP-запросов:
- `GET`: Метод GET используется для получения данных с сервера. Он не изменяет данные и считается безопасным.
- `POST`: Метод POST применяется для отправки данных на сервер (например, форма регистрации, создание ресурса).
- `PUT`: Метод PUT используется для полной замены ресурса на сервере.
- `PATCH`: Метод PATCH применяется для частичного изменения ресурса.
- `DELETE`: Метод DELETE удаляет указанный ресурс на сервере.
- `HEAD`: Метод HEAD аналогичен GET, но сервер отвечает только заголовками, без тела.
- `OPTIONS`: Метод OPTIONS позволяет клиенту запросить, какие методы доступны для ресурса. При определенных случаях, браузер сам отправляет этот запрос - `preflight request`, чтобы проверить, разрешает ли сервер доступ.
- `CONNECT`: Метод CONNECT используется для установления сетевого туннеля, чаще всего в прокси-серверах.
- `TRACE`: Метод TRACE выполняет диагностику соединения, возвращая запрос в его изначальном виде.

```http
GET /api/products?page=1&limit=10 HTTP/1.1
```

Версии HTTP:
- **HTTP/1.1** — добавил поддержку keep-alive для ускорения работы (1997).
- **HTTP/2** — улучшил многопоточный обмен данными, сжатие заголовков (2015).
- **HTTP/3** — использует протокол QUIC вместо TCP, повышая скорость и безопасность (TBA).

[learn more](https://http.dev/1.1)

HTTP keep-alive
Keep-Alive – это механизм HTTP, который позволяет повторно использовать одно и то же соединение для нескольких запросов, вместо того чтобы открывать новое соединение для каждого запроса.

Запрос клиента:
```http
GET /index.html HTTP/1.1
Host: example.com
Connection: keep-alive
```

Ответ сервера:
```http
HTTP/1.1 200 OK
Connection: keep-alive
Keep-Alive: timeout=10, max=100
```

- `timeout=10` — соединение будет поддерживаться **10 секунд** без активности.
- `max=100` — соединение может использоваться для **100 запросов** максимум.

HTTP Status codes

**1xx — Информационные** 🔹
- **100 Continue** — Сервер принял запрос, можно отправлять тело запроса.  
- **101 Switching Protocols** — Сервер меняет протокол (например, на WebSocket).  
- **103 Early Hints** — Подсказки для клиента (например, предзагрузка ресурсов).

**2xx — Успешные** ✅
- **200 OK** — Успешный запрос (страница, данные и т. д.).  
- **201 Created** — Успешно создан новый ресурс (например, POST-запрос).  
- **202 Accepted** — Запрос принят, но еще не обработан.  
- **204 No Content** — Запрос успешен, но ответа нет (например, DELETE).

**3xx — Перенаправления** ➡️
- **301 Moved Permanently** — Ресурс переехал навсегда (SEO важно!).  
- **302 Found** (или **"Moved Temporarily"**) — Временный редирект.  
- **304 Not Modified** — Контент не изменился (браузер использует кеш).  
- **307 Temporary Redirect** — Временный редирект без смены метода запроса.

**4xx — Ошибки клиента** ❌
- **400 Bad Request** — Некорректный запрос (неправильный JSON, параметры).  
- **401 Unauthorized** — Требуется аутентификация (нужен `Authorization` заголовок).  
- **403 Forbidden** — Доступ запрещен (даже с авторизацией).  
- **404 Not Found** — Ресурс не найден (битая ссылка, страница не существует).  
- **405 Method Not Allowed** — HTTP-метод запрещен (например, `POST` на `/users/1`).  
- **418 I'm a teapot** ☕ — Пасхалка из **RFC 2324** (чайник не умеет заваривать кофе).

**5xx — Ошибки сервера** 🔥
- **500 Internal Server Error** — Общая ошибка сервера (что-то сломалось).  
- **502 Bad Gateway** — Ошибка прокси/балансировщика (например, Nginx → Backend).  
- **503 Service Unavailable** — Сервер перегружен/на обслуживании.  
- **504 Gateway Timeout** — Таймаут у прокси/балансировщика.

Как отправляются изображения

- Отправка изображения в виде файла (через multipart/form-data)

Что такое multipart/form-data
Это специальный формат тела HTTP-запроса, который позволяет отправлять несколько частей данных в одном запросе: например, текстовые поля + файл. Запрос состоит из "частей", каждая из которых имеет свой заголовок и тело. Одна из частей может содержать изображение.

```http
POST /upload HTTP/1.1
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4YWxkTrZu0gW

------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="username"

myusername
------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="file"; filename="cat.png"
Content-Type: image/png

<байты изображения>
------WebKitFormBoundary7MA4YWxkTrZu0gW--
```

В HTML выглядит так:
```html
<form action="/upload" method="POST" enctype="multipart/form-data">
  <input type="file" name="image" />
  <button type="submit">Upload</button>
</form>
```

В JS выглядит так:
```js
const formData = new FormData();
formData.append('image', fileInput.files[0]);

fetch('/upload', {
  method: 'POST',
  body: formData
});
```

- Отправка изображения как base64-строку (в теле JSON)

Иногда вместо того, чтобы отправлять файл напрямую, картинку кодируют в **base64-строку** (специальный формат, который превращает байты в текст) и отправляют как обычную строку в JSON.

Как это выглядит:
```json
{
  "image": "data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAA..."
}
```

И реализация на фронте:
```js
const reader = new FileReader();

reader.onload = function () {
  const base64Image = reader.result;
  fetch('/upload', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({ image: base64Image })
  });
};

reader.readAsDataURL(fileInput.files[0]);
```

Оба варианта хороши, но в base64 в JSON больше на 30–40% из-за кодировки (по весу)

WebSockets

**WebSocket** — это протокол, разработанный для обеспечения постоянного двунаправленного соединения между клиентом и сервером поверх TCP.

- В отличие от классического HTTP, который по сути “один запрос — один ответ” и затем соединение закрывается, WebSocket держит канал открытым постоянно.
- Позволяет обмениваться данными в реальном времени, без накладных расходов на открытие новых соединений.

Как работает WebSocket?
1. Клиент отправляет серверу HTTP-запрос с заголовком `Upgrade: websocket` — запрос на апгрейд соединения.
2. Если сервер поддерживает WebSocket, он отвечает подтверждением и устанавливается постоянное соединение.
3. Теперь клиент и сервер могут **свободно обмениваться сообщениями** (текстовыми или бинарными) без повторного запроса.
4. Соединение держится открытым, пока не будет закрыто одной из сторон.

Что такое `wss`?

**HTTP polling** — это способ реализации связи клиента и сервера, при котором клиент регулярно (через определённые интервалы времени) отправляет HTTP-запросы к серверу, чтобы проверить, появились ли новые данные.

**Long polling** — это техника, которая позволяет серверу «задерживать» ответ на запрос клиента, пока не появятся новые данные или не истечёт таймаут.

В отличие от классического polling, где клиент просто опрашивает сервер с фиксированным интервалом, при long polling клиент отправляет запрос, а сервер держит соединение открытым до тех пор, пока:

- Не появятся новые данные, или
- Не закончится таймаут.

Когда данные появляются — сервер сразу их отправляет, клиент обрабатывает ответ и сразу же отправляет новый запрос, чтобы ждать следующие обновления.

**WebRTC** (Web Real-Time Communication) — это открытый стандарт и набор API, позволяющих веб-приложениям и мобильным приложениям осуществлять **прямую передачу аудио, видео и данных в реальном времени** между браузерами без необходимости в промежуточных серверах.

**WebRTC по умолчанию использует UDP** как основной транспорт для передачи медиа и данных между участниками. Если из-за **NAT**, firewall или корпоративных ограничений UDP нельзя использовать (например, заблокирован), WebRTC **падает на TURN**, а он уже может ретранслировать трафик как по **UDP**, так и по **TCP** (обычно это последняя надежда).

Этапы работы WebRTC:
1. **Signaling** — обмен SDP и ICE-кандидатами через сервер.
2. **SDP Offer/Answer** — согласование параметров соединения.
3. **ICE Gathering** — поиск возможных IP/портов для соединения.
4. **ICE Connectivity Checks** — проверка соединений между пир-to-пир.
5. **Выбор лучшего соединения**.
6. **Установка P2P соединения (DTLS, SRTP)**.
7. **Передача аудио, видео и данных**.

STUN (Session Traversal Utilities for NAT) - позволяет клиенту узнать свой публичный IP и порт за NAT. Используется для организации прямого P2P соединения.

TURN (Traversal Using Relays around NAT) - ретранслирует трафик, если прямое соединение невозможно (например, строгий NAT или firewall). Пакеты проходят через TURN-сервер.

ICE (Interactive Connectivity Establishment) - алгоритм, который собирает все возможные пути соединения (локальные IP, STUN, TURN) и выбирает лучший. Работает с STUN и TURN серверами.

**Trickle ICE** — это оптимизация процесса ICE (Interactive Connectivity Establishment) в WebRTC, при которой **ICE-кандидаты передаются по мере их появления**, а не все сразу после завершения сбора.

Безопасность:

- DTLS (Datagram Transport Layer Security) - обеспечивает шифрование сигналов и данных в WebRTC (аналог HTTPS ток для данных). 

- SRTP (Secure Real-time Transport Protocol) - защищённый протокол передачи аудио и видео в реальном времени. Работает поверх UDP. Использует ключи, согласованные через DTLS (для медиафайлов).

Виды взаимодействия WebRTC:
- P2P: Два клиента (браузера или приложений) устанавливают прямое соединение друг с другом.
- Mesh-сеть (сетевое P2P): Каждое устройство соединяется напрямую с каждым другим.
- SFU (Selective Forwarding Unit): Каждый клиент устанавливает одно соединение с сервером (SFU).
- MCU (Multipoint Control Unit): Клиенты отправляют потоки на сервер (MCU), который объединяет их в один поток и рассылает назад.

Что такое NAT?

NAT - это технология, которая позволяет устройствам внутри локальной сети (например, дома или в офисе) использовать **один общий внешний IP-адрес** для выхода в интернет, скрывая при этом свои **внутренние IP-адреса**.

Зачем нужен NAT:
1. **Экономия IP-адресов** — так как публичных IPv4-адресов мало.
2. **Безопасность** — устройства внутри сети не видны напрямую снаружи.
3. **Организация работы с внутренними адресами** — внутри могут быть приватные IP (например `192.168.x.x`, `10.x.x.x`)

OSI Layers

**Модель OSI (Open System Interconnection)**, или эталонная модель взаимодействия открытых систем описывает, как устройства в локальных и глобальных сетях обмениваются данными и что происходит с этими данными. 

Модель OSI включает семь слоёв, или уровней, — причём каждый из них выполняет определённую функцию: например, передать данные или представить их в понятном для человека виде на компьютере. Кстати, у каждого слоя — свой набор протоколов.

**1-й уровень OSI — физический**

На самом нижнем уровне модели OSI данные представляют собой физические объекты — ток, свет или радиоволны. Они передаются по проводам или с помощью беспроводных сигналов.

Самый известный протокол на физическом уровне — Ethernet. Он описывает, как сигналы кодируются и передаются по проводам. Кроме него есть Bluetooth, Wi-Fi и ИК-порт, которые также содержат инструкции для передачи данных.

2-й уровень OSI — канальный

Над физическим уровнем располагается канальный. Его задача — проверить целостность полученных данных и исправить ошибки. Этот уровень «поумнее» предыдущего: он уже понимает, что разные амплитуды напряжений отвечают разным битам — нулям и единицам. А ещё канальный уровень умеет кодировать сигналы в биты и передавать их дальше.

Получается что-то вроде почтового конверта. На лицевой стороне у него написано, от кого пришло письмо, а внутри находится само письмо (в нашем случае данные).

Лицевая сторона конверта — это MAC-адрес устройства, которое отправило нам информацию.

3-й уровень OSI — сетевой

Этот уровень отвечает за маршрутизацию данных внутри сети между компьютерами. Здесь уже появляются такие термины, как «маршрутизаторы» и «IP-адреса».

Маршрутизаторы позволяют разным сетям общаться друг с другом: они используют MAC-адреса, чтобы построить путь от одного устройства к другому.

Данные на сетевом уровне представляются в виде пакетов. Такие пакеты похожи на фреймы из канального уровня, но используют другие адреса получателя и отправителя — [IP-адреса](https://skillbox.ru/media/code/chto-takoe-ipadres-i-maska-podseti-i-zachem-oni-nuzhny/?utm_source=media&utm_medium=link&utm_campaign=all_all_media_links_links_articles_all_all_skillbox).

Чтобы получить IP-адрес обоих устройств (отправителя и получателя), существует протокол [ARP](https://ru.wikipedia.org/wiki/ARP) (address resolution protocol). Он умеет конвертировать MAC- в IP-адрес и наоборот.

4-й уровень OSI — транспортный

Из названия понятно, что на этом уровне происходит передача данных по сети. Так и есть. Два главных протокола здесь — [TCP](https://skillbox.ru/media/code/model-tcpip-chto-eto-takoe-i-kak-ona-rabotaet/?utm_source=media&utm_medium=link&utm_campaign=all_all_media_links_links_articles_all_all_skillbox) и [UDP](https://ru.wikipedia.org/wiki/UDP). Они как раз и отвечают за то, как именно будут передаваться данные.

Difference between TCP and UDP?

TCP (Transmission Control Protocol) и UDP (User Datagram Protocol) — два основных транспортных протокола в сети. Они используются для передачи данных, но работают по-разному.

TCP гарантирует **доставку** и **порядок пакетов**. Это как доставка заказного письма: если оно потеряется, почта вышлет его заново.

**Как TCP работает?**
1. **Устанавливает соединение (3-way handshake)**
    - Клиент отправляет **SYN** (начать соединение)
    - Сервер отвечает **SYN-ACK** (подтверждение)
    - Клиент отправляет **ACK** (начинаем передачу)
2. **Передает данные с контролем ошибок**
    - Если пакет потерялся, сервер **запрашивает повторную отправку**
    - Все пакеты приходят **по порядку**
3. **Закрывает соединение (4-way handshake)**

**Где используется TCP?**
- **Web (HTTP, HTTPS)** → страница должна загрузиться полностью
- **Email (SMTP, IMAP, POP3)** → нельзя терять письма
- **Файловые передачи (FTP, SFTP)** → файлы должны быть целыми
- **SSH** → для безопасного доступа к серверу

UDP просто отправляет пакеты **без подтверждения доставки**. Это как обычное письмо: если потеряется, никто не узнает.

**Как UDP работает?**
1. **Отправляет пакеты без установки соединения**
2. **Не ждет подтверждения доставки**
3. **Не гарантирует порядок пакетов**

**Где используется UDP?**
- **Стриминг (YouTube, Twitch, Netflix, IPTV)** → лучше пропустить кадр, чем тормозить
- **VoIP (Zoom, Skype, WhatsApp Calls)** → важно получать голос в реальном времени
- **Онлайн-игры (CS:GO, Fortnite, Dota 2)** → скорость важнее надежности
- **DNS-запросы** → быстрее проверить IP-адрес, чем ждать подтверждения

What is SSL?

SSL (Secure Sockets Layer) — это **протокол шифрования**, который раньше использовался для защиты данных в интернете. Он создавал **безопасное соединение** между браузером и сервером.

> *"SSL-сертификаты" — это просто название, внутри работает TLS.*

❌ **Важно:** SSL **устарел** и был заменен **TLS (Transport Layer Security)**.

---

What is TLS?

TLS (Transport Layer Security) — это **криптографический протокол**, который обеспечивает **безопасное шифрование данных** при передаче по сети. Он защищает данные от **перехвата, подмены и взлома**. TLS — это **улучшенная версия SSL**, которая **сейчас используется в HTTPS, VPN, email, VoIP и других сервисах**.

**1️⃣ Handshake (установление соединения)**
- Клиент (браузер) говорит: **"Привет, сервер! Давай зашифруем связь, вот список моих алгоритмов шифрования."**
- Сервер отвечает: **"Ок, давай использовать AES + SHA256, вот мой сертификат."**
- Клиент проверяет **сертификат** (он подтверждает, что сайт подлинный, не поддельный).

**2️⃣ Key Exchange (обмен ключами)**
- Клиент и сервер **создают общий секретный ключ**, чтобы зашифровать все последующие сообщения.
- Этот ключ создается с помощью **асимметричного шифрования** (RSA или ECDH).

**3️⃣ Шифрованная передача данных**
- Теперь вся переписка между клиентом и сервером **зашифрована**.
- Никто в сети (даже интернет-провайдер) не может подслушать данные.

What is IP?

**IP (Internet Protocol)** — это **протокол интернета**, который позволяет устройствам находить друг друга и обмениваться данными в сети.

Когда ты заходишь на сайт, отправляешь сообщение или смотришь видео, все данные передаются **с IP-адреса на IP-адрес**.

Каждое устройство в интернете (сервер, компьютер, телефон) имеет **уникальный IP-адрес**, который выглядит как:

- **IPv4**: `192.168.1.1` (четыре числа от 0 до 255)
- **IPv6**: `2001:db8::ff00:42:8329` (длинные строки с буквами и цифрами)

🔹 **IPv4 — старая версия (4.3 млрд адресов, заканчиваются).**  
🔹 **IPv6 — новая версия (почти бесконечно много адресов).**

**🛠 Как работает IP?**
- **Ты вводишь `google.com` в браузере.**  
- Компьютер запрашивает у DNS (системы доменных имен) **IP-адрес Google**.  
- DNS отвечает: `142.250.187.78` (пример).  
- Браузер соединяется с сервером Google **по этому IP** и загружает сайт.

📌 **Вот почему IP-адрес нужен даже если запрос идет на `google.com`!**

Больше про CRP (или как браузер рендерит сайт)?

CRP — это процесс преобразования HTML, CSS и JavaScript в отображаемую веб-страницу. Он включает несколько этапов, которые браузер проходит перед тем, как показать пользователю контент (см ниже).

Факторы, замедляющие CRP:
- Большие файлы CSS и JS
- Блокирующие ресурсы (синхронные скрипты, внешние CSS-файлы)
- Рендер-блокирующий JavaScript

Как оптимизировать CRP?
- Дроби большой JS на чанки (code splitting)
- Минимизировать и сжимать CSS, JS, HTML (Purge CSS, CSS Nano, TerserJS, UglifyJS, html-minifier + Gzip/minify)
- Откладывать загрузку неважных скриптов (**defer, async**)
- Использовать Critical CSS для быстрой отрисовки основного контента
- Загружать шрифты и изображения лениво (**lazy loading**)
- Mobile First / Desktop First

[learn more](https://habr.com/ru/articles/484900/)
[learn more](https://developer.mozilla.org/ru/docs/Web/Performance/Critical_rendering_path)

Стадии рендеринга (Critical Rendering Path):
- 🔄 **Получение HTML-документа** (по HTTP/HTTPS)  
    → начинается парсинг потока HTML.
- 🧱 **Парсинг HTML → создание `DOM` (Document Object Model)**  
    → создаётся структура из элементов и текста.
- 🎨 **Загрузка CSS-файлов** (они **блокируют рендеринг**!)
- 🧵 **Парсинг CSS → создание `CSSOM` (CSS Object Model)**  
    → CSS-правила собираются в дерево стилей.
- 🌲 **Объединение `DOM` + `CSSOM` → создание `Render Tree`**  
    → только видимые элементы (без `display: none`), с применёнными стилями.
- 📐 **Layout (или Reflow)**  
    → рассчитываются точные **позиции и размеры** элементов в пикселях.
- 🖌️ **Painting (или Repaint)**  
    → элементы превращаются в пиксели, происходит отрисовка на экране.
- 🧩 _(опционально)_ **Compositing**  
    → разные слои (например, с `transform`, `z-index`, `opacity`) объединяются в финальную картинку.

**Reflow** — это процесс, когда браузер **пересчитывает расположение и размеры элементов на странице**.

Это происходит когда:
- добавляется или удаляется элемент
- меняется размер окна
- меняется размер шрифта или padding/margin
- меняется `display`, `position`, `height`, `width` у элемента
- DOM-дерево изменилось

**Repaint** — это процесс, когда браузер **перерисовывает пиксели** элемента на экране, без изменения его размера или позиции.

Это происходит когда:
- изменяется цвет (`color`, `background-color`)
- прозрачность (`opacity`)
- тени
- границы (`border-color`)
- или любые другие **визуальные свойства**, не влияющие на размер и расположение

Это отдельный компонент браузера, который выполняет JavaScript-код. Он берет JS-код, парсит его, компилирует в машинный код (JIT - Just-In-Time Compilation) и выполняет.

Функции JS-движка:
- Парсинг и разбор JS-кода
- Оптимизация и компиляция (JIT)
- Выполнение кода в Call Stack
- Работа с Event Loop и Web API

Примеры JavaScript-движков:
- V8 (Chrome, Edge, Node.js)
- SpiderMonkey (Firefox)
- JavaScriptCore/Nitro (Safari)

Подробный разбор Storage API в браузере

Браузер предоставляет несколько способов хранения данных на клиенте. Основные хранилища:

- **LocalStorage** – долговременное хранилище.  
- **SessionStorage** – временное хранилище (на время сессии).  
- **IndexedDB** – база данных в браузере.  
- **Cookies** – небольшие данные, отправляемые с HTTP-запросами.  
- **Cache Storage** – кеширование данных для PWA.

**Cookies (куки)** — это небольшие файлы с данными, которые браузер сохраняет на компьютере пользователя. Они используются для хранения информации между сессиями, например:
- Авторизация (сохранение логина)
- Настройки пользователя (тема, язык)
- История посещений и корзина покупок
- Трекинг пользователей (аналитика, реклама)

Cookies используются для передачи данных серверу, а `LocalStorage` / `SessionStorage` — для хранения данных в браузере.

По назначению:
- **Функциональные** — хранят настройки пользователя (язык, тема).
- **Аутентификационные** — используются для логина (токены сессии).
- **Аналитические** — отслеживают активность пользователя (Google Analytics).
- **Рекламные (Tracking Cookies)** — собирают данные для таргетинга рекламы.

Основные свойства:
- **`Name=Value`** — ключ и значение cookie (например, `session=abc123`).
- **`Expires`** — дата истечения срока действия (например, `Expires=Wed, 21 Oct 2025 07:28:00 GMT`).
- **`Max-Age`** — время жизни в секундах (например, `Max-Age=3600` — 1 час).
- **`Domain`** — домен, для которого действует cookie (например, `Domain=example.com`).
- **`Path`** — путь, для которого cookie доступны (например, `Path=/dashboard` — только для `/dashboard`).

Безопасность:
- **`Secure`** — cookie передаётся только по HTTPS.
- **`HttpOnly`** — cookie недоступен из JavaScript (`document.cookie`).
- **`SameSite`** — предотвращает CSRF-атаки (`Strict`, `Lax`, `None`).

LocalStorage предназначен для **долговременного хранения данных** в браузере.
- Данные **не удаляются** после закрытия вкладки или браузера.
- Доступен **только на текущем домене**.
- Не передается с HTTP-запросами.
- Хранит **до 5-10 MB** данных.
- Доступ к данным осуществляется через **JavaScript API**.

```js
// Запись данных
localStorage.setItem("username", "John");

// Получение данных
console.log(localStorage.getItem("username")); // "John"

// Удаление данных
localStorage.removeItem("username");

// Очистка всех данных
localStorage.clear();
```

**Когда использовать?**
- Хранение **настроек пользователя** (тема, язык, фильтры).
- Локальный кеш данных (например, сохранение товаров в корзине).
- Хранение токенов или информации, которая должна **сохраняться при перезагрузке страницы**.

SessionStorage используется для **временного хранения данных** в браузере **только во время сеанса**.
- Данные удаляются **после закрытия вкладки**.
- Доступен **только на текущем домене**.
- Не передается с HTTP-запросами.
- Хранит **до 5 MB** данных.
- Доступ к данным через **JavaScript API**.

```js
// Запись данных
sessionStorage.setItem("sessionID", "abc123");

// Получение данных
console.log(sessionStorage.getItem("sessionID")); // "abc123"

// Удаление данных
sessionStorage.removeItem("sessionID");

// Очистка всех данных
sessionStorage.clear();
```

**Когда использовать?**
- Временные **формы, фильтры, параметры поиска**.
- Данные, которые **не должны сохраняться** после закрытия вкладки.
- Реализация механики "Продолжить позже" в пределах сессии.

Архитектура Electron:
- **Main Process (Node.js)**
  Главный процесс, который управляет окнами, системными API, меню и может выполнять любую серверную логику.
- **Renderer Process (Chromium + V8)**
  Отдельный процесс для каждого окна. По сути полноценный браузер Chromium, который рендерит твою страницу.
- **IPC (Inter-Process Communication)**
  Специальный механизм для общения между `Main` и `Renderer` процессами.

Архитектура Tauri:
- **Core (Rust)**
  Главный процесс написан на **Rust**, он отвечает за взаимодействие с системными API (файлы, окна, безопасность).
- **WebView (на системном движке)**
  Использует встроенный в ОС WebView:
    - Windows — **Edge WebView2 (Chromium-based)**
    - macOS — **WKWebView (WebKit)**
    - Linux — **WebKitGTK**
- **IPC (Tauri Commands)**
  Общение между фронтендом (в WebView) и бэкендом на Rust через безопасный IPC.

Apache Cordova / Ionic Capacitor:

- Ты пишешь приложение на **HTML/CSS/JS**, как обычный вебсайт.
- JavaScript-приложение запускается внутри **WebView** — встроенного браузера на мобильном устройстве (iOS/Android).
- JS код отображает интерфейс и реализует логику, как в обычном веб-приложении.
- Для доступа к нативным функциям (камера, геолокация, файловая система и др.) используется **bridge** — двунаправленный мост между JS и нативным кодом (Java/Swift/Obj-C).
- При вызове нативных API из JS (например, `Camera.getPhoto()`):
  - Запрос сериализуется и отправляется через bridge в нативный код.
  - Нативный код выполняет нужное действие (открывает камеру, получает данные).
  - Результат возвращается обратно в JS через bridge.
- В Cordova bridge основан на callback-ориентированном обмене сообщениями через WebView. В Capacitor используется современный подход с Promise и async/await, обеспечивающий более удобный и чистый API.

>Capacitor лучше интегрируется с современными инструментами и поддерживает работу с PWA и Electron.

Что такое интерпретатор?
Интерпретатор - пооператорная (покомандная, построчная) трансляция, анализ, обработка и тут же выполнение исходной программы или запроса.

Что такое компилятор?
Компилятор - трансляция программы, составленной на исходном языке высокого уровня, в эквивалентную программу на низкоуровневом языке и выполнение ее разом.

[learn more](https://blog.skillfactory.ru/glossary/interpretator/)

---

Что такое транспилятор?
Транспиляция — преобразование программы, написанной на одном языке программирования в качестве исходных данных, в эквивалентный код другой версии этого языка или в другой язык программирования того же уровня абстракции. Транспиляторы под капотом используют AST.

Стадии транспиляции JS-кода:
- Лексический анализ (Tokenization)
- Синтаксический анализ (Parsing) - создается AST
- Генерация байт-кода (Interpreter, Ignition в V8) - интерпретируется в байт-код
- JIT-компиляция (TurboFan в V8) - компиляция в машинный код (+оптимизация горячих мест)
- Исполнение
- Уборка мусора (Garbage Collector)

Стадии транспиляции WASM-кода:
- Компиляция исходного кода в WebAssembly (WAT → WASM)
- Загрузка и проверка в браузере
- Компиляция в машинный код (AOT, Ahead-of-Time)
- Оптимизации и выполнение

>Однако WASM не ускорит ваш код, так как он все равно работает с Javascript (DOM API и т.д.)

Data structures

- **Массив (Array)**  
    Упорядоченный набор элементов, доступных по индексу. Может быть фиксированного или динамического размера.
- **Связный список (Linked List)**  
    Последовательность узлов, каждый из которых содержит ссылку на следующий (и иногда на предыдущий). Удобен для вставки и удаления элементов.
- **Стек (Stack)**  
    Структура по принципу LIFO (последним пришёл — первым вышел). Операции: добавление (push) и удаление (pop) с вершины.
- **Очередь (Queue)**  
    Структура по принципу FIFO (первым пришёл — первым вышел). Операции: добавление (enqueue) в конец и удаление (dequeue) из начала.
- **Хеш-таблица (Hash Table, Hash Map)**  
    Хранит пары ключ-значение, обеспечивает быстрый доступ, вставку и удаление по ключу с помощью хеширования.
- **Дерево (Tree)**  
    Иерархическая структура с узлами, имеющими дочерние элементы. Виды: бинарное дерево, бинарное дерево поиска, куча, префиксное дерево (trie).
- **Граф (Graph)**  
    Множество вершин (узлов), соединённых рёбрами. Может быть ориентированным или неориентированным, взвешенным или невзвешенным.
- **Множество (Set)**  
    Коллекция уникальных элементов, поддерживающая быстрые операции проверки наличия элемента.

Imperative programming

Императивное программирование — это стиль написания кода, в котором программист явно указывает, КАК нужно выполнить задачу. Этот подход использует последовательные инструкции и управление состоянием.

- **Пошаговое выполнение** — код выполняется **строго сверху вниз**.
- **Изменение состояния** — в коде используются переменные, которые **изменяют свое значение**.
- **Использование циклов и условий** — `for`, `while`, `if/else`.
- **Близко к низкоуровневым языкам** — такой стиль похож на то, как работает процессор.

```js
function sumArray(arr) {
  let sum = 0; // Изменяемая переменная (состояние)
  for (let i = 0; i < arr.length; i++) { // Четкий контроль за итерацией
    sum += arr[i];
  }
  return sum;
}

console.log(sumArray([1, 2, 3, 4, 5])); // 15
```

Declarative programming

Декларативный стиль программирования описывает ЧТО нужно сделать, без указания КАК это должно выполняться. Вместо пошаговых инструкций мы описываем цель, а детали реализации скрыты.

- **Фокус на конечном результате** — мы описываем **ЧТО нужно**, а не **КАК это сделать**.
- **Минимизация побочных эффектов** — реже изменяются переменные и состояние.
- **Использование абстракций** — такие как `map`, `reduce`, SQL-запросы, CSS.
- **Читаемость и краткость** — код проще понять, так как в нем **меньше логики управления**.

```js
const sumArray = (arr) => arr.reduce((acc, num) => acc + num, 0);

console.log(sumArray([1, 2, 3, 4, 5])); // 15
```

Object-Oriented Programming

ООП — это парадигма программирования, основанная на создании и взаимодействии **объектов**, которые содержат данные и методы для работы с этими данными.  
Цель — моделировать реальный мир и создавать более удобный, понятный и поддерживаемый код.

Основные концепции (4):

Абстракция (Abstraction):
**Скрытие деталей реализации и предоставление только необходимых функций.**  
Позволяет пользователю взаимодействовать с объектами через **чёткий интерфейс**, не зная их внутреннее устройство.

```js
class Car {
  startEngine() {
    console.log("Двигатель запущен");
  }

  drive() {
    console.log("Машина едет");
  }
}

const myCar = new Car();
myCar.startEngine(); // "Двигатель запущен"
myCar.drive();       // "Машина едет"
```

**Абстракция** – пользователь вызывает `startEngine()`, но не знает, как работает двигатель внутри.

---

Наследование (Inheritance):
**Передача свойств и методов от одного класса другому.**  
Позволяет избежать дублирования кода и расширять функциональность.

```js
class Animal {
  constructor(name) {
    this.name = name;
  }

  speak() {
    console.log(`${this.name} издаёт звук`);
  }
}

class Dog extends Animal {
  speak() {
    console.log(`${this.name} лает`);
  }
}

const dog = new Dog("Шарик");
dog.speak(); // "Шарик лает"
```

**Наследование** – `Dog` унаследовал `name` и `speak()` от `Animal`, но переопределил `speak()`.

---

Полиморфизм (Polymorphism):
**Один интерфейс – разное поведение.**  
Позволяет переопределять методы, сохраняя общую структуру кода.

```js
class Bird {
  speak() {
    console.log("Птица чирикает");
  }
}

class Parrot extends Bird {
  speak() {
    console.log("Попугай говорит");
  }
}

class Crow extends Bird {
  speak() {
    console.log("Ворон каркает");
  }
}

const birds = [new Bird(), new Parrot(), new Crow()];
birds.forEach(bird => bird.speak());
```

**Полиморфизм** – все объекты вызывают `speak()`, но делают это **по-разному**.

---

Инкапсуляция:
**Сокрытие данных и ограничение прямого доступа к ним.**  
Позволяет защитить данные от случайного или нежелательного изменения.

```js
class BankAccount {
  #balance = 0; // Приватное свойство

  deposit(amount) {
    if (amount > 0) {
      this.#balance += amount;
      console.log(`Пополнение на ${amount}, баланс: ${this.#balance}`);
    }
  }

  withdraw(amount) {
    if (amount > this.#balance) {
      console.log("Недостаточно средств");
      return;
    }
    this.#balance -= amount;
    console.log(`Снятие ${amount}, баланс: ${this.#balance}`);
  }
}

const account = new BankAccount();
account.deposit(100);
account.withdraw(50);
account.withdraw(100); // "Недостаточно средств"
// account.#balance = 1000; // ❌ Ошибка, приватное свойство
```

**Инкапсуляция** – защищает `#balance`, доступен только через `deposit()` и `withdraw()`.

>как запомнить?
- абстракция - машина и двигатель
- наследование - собака и энимал
- полиморфизм - разные птицы
- инкапсуляция - банк и баланс

What is SOLID?
SOLID principles are five guidelines for crafting robust and agile object-oriented code.

S — Single Responsibility Principle (Принцип единственной ответственности)
Каждый модуль/класс должен **иметь только одну причину для изменения**.

```js
// ❌ **Плохо** (один класс делает слишком много)
class User {
  constructor(name, email) {
    this.name = name;
    this.email = email;
  }

  saveToDatabase() {
    console.log("Saving user to database...");
  }
}

// ✅ **Хорошо** (разделяем логику)
class User {
  constructor(name, email) {
    this.name = name;
    this.email = email;
  }
}

class UserRepository {
  save(user) {
    console.log(`Saving ${user.name} to database...`);
  }
}
```

O — Open/Closed Principle (Принцип открытости/закрытости)
Код должен быть открыт для расширения, но закрыт для изменения.

```js
// ❌ **Плохо** (каждый раз меняем код при добавлении нового типа сообщения)
class Notification {
  send(type, message) {
    if (type === "email") {
      console.log("Sending email: " + message);
    } else if (type === "sms") {
      console.log("Sending SMS: " + message);
    }
  }
}

// ✅ **Хорошо** (теперь добавление новых уведомлений не требует изменения существующего кода.)
class Notification {
  send(message) {
    throw new Error("send() must be implemented");
  }
}

class EmailNotification extends Notification {
  send(message) {
    console.log("Sending email: " + message);
  }
}

class SMSNotification extends Notification {
  send(message) {
    console.log("Sending SMS: " + message);
  }
}
```

L — Liskov Substitution Principle (Принцип подстановки Барбары Лисков)
Объекты должны заменять родительские классы без изменения поведения.

```js
❌ Плохо (наследование нарушает принцип)
class Bird {
  fly() {
    console.log("Flying...");
  }
}

class Penguin extends Bird {
  fly() {
    throw new Error("Penguins can't fly!");
  }
}

// ✅ **Хорошо** (разделяем интерфейсы)
class Bird {}
class FlyingBird extends Bird {
  fly() {
    console.log("Flying...");
  }
}
class Penguin extends Bird {
  swim() {
    console.log("Swimming...");
  }
}

```

I - Interface Segregation Principle (Принцип разделения интерфейсов)
Не заставляйте классы реализовывать методы, которые им не нужны.

```js
 // ❌ **Плохой пример (Нарушение ISP)**
class Worker {
  work() {
    console.log("Работает...");
  }

  eat() {
    console.log("Ест...");
  }
}

class Robot extends Worker {
  eat() {
    throw new Error("Роботы не едят!");
  }
}

const robot = new Robot();
robot.eat();


// ✅ **Правильный вариант (Исправляем нарушение)**
class Workable {
  work() {
    throw new Error("Метод work() должен быть реализован");
  }
}

class Eatable {
  eat() {
    throw new Error("Метод eat() должен быть реализован");
  }
}

class Human extends Workable {
  work() {
    console.log("Человек работает...");
  }
}

class HumanWithFood extends Human {
  eat() {
    console.log("Человек ест...");
  }
}

class Robot extends Workable {
  work() {
    console.log("Робот работает...");
  }
}

const worker = new HumanWithFood();
worker.work(); // Человек работает...
worker.eat();  // Человек ест...

const robot = new Robot();
robot.work();  // Робот работает...
```

D - Dependency Inversion Principle (Принцип инверсии зависимостей)
Высокоуровневые модули не должны зависеть от низкоуровневых. Оба должны зависеть от абстракций (интерфейсов). Абстракции не должны зависеть от деталей. Детали должны зависеть от абстракций.

Просто говоря: **не стоит жестко привязывать один класс к другому, лучше использовать интерфейсы или зависимости через DI (Dependency Injection).**

Functional programming (FP)

Функциональное программирование (FP) — это парадигма, основанная на **чистых функциях**, **неизменяемости данных** и **отсутствии побочных эффектов**.

Основные принципы:
- Чистые функции (Pure Functions)
- Неизменяемость (Immutability)
- Функции — объекты первого класса (HoF - функции высшего порядка)
- Функциональная композиция (+рекурсии)
- Чистота и отсутствие побочных эффектов
- Каррирование (Currying) и частичное применение

---

Функциональная композиция — это техника программирования, при которой **несколько функций объединяются в одну**, создавая **новые функции без изменения их исходного кода**.

Если есть две функции `f(x)` и `g(x)`, то их композиция `h(x)` будет:

`h(x) = g(f(x))`.

Например
```js
const toUpper = (str) => str.toUpperCase();
const exclaim = (str) => str + "!";

const shout = (str) => exclaim(toUpper(str));

console.log(shout("hello")); // "HELLO!"
```

- Разработчики должны избегать ненужных усложнений.

What is DI?

**Dependency Injection (DI)** — это **паттерн проектирования**, который **разделяет создание объекта и его использование**. Вместо того чтобы создавать зависимости внутри класса, мы передаём их **извне**, что делает код более гибким, тестируемым и удобным в поддержке.

- **Ослабляет связность (Loose Coupling)** → Классы не зависят от конкретных реализаций, а только от абстракций.
- **Облегчает тестирование** → Можно подменять зависимости моками (mock).
- **Упрощает поддержку и расширение** → Легко менять поведение, подставляя другие реализации.

Пример без DI (Жёсткая зависимость)
```js
class Database {
  query(sql) {
    console.log(`Executing query: ${sql}`);
  }
}

class UserService {
  constructor() {
    this.db = new Database(); // 🔴 Жёсткая зависимость!
  }

  getUser(id) {
    this.db.query(`SELECT * FROM users WHERE id = ${id}`);
  }
}
```

Пример с DI (Гибкая зависимость)
```js
class UserService {
  constructor(database) {
    this.db = database; // ✅ Внедрение зависимости
  }

  getUser(id) {
    this.db.query(`SELECT * FROM users WHERE id = ${id}`);
  }
}

// Создаём зависимость отдельно
const database = new Database(); 
const userService = new UserService(database); // Передаём зависимость

userService.getUser(1);
```

What is API?

**API** (Application Programming Interface) — это **интерфейс для взаимодействия между программами**.

Грубо говоря, это **набор правил и команд**, с помощью которых одна программа (например, твое фронтенд-приложение) может общаться с другой программой (например, сервером или сервисом вроде OpenAI, Spotify или GitHub).

Web API - Общение между клиентом и сервером через HTTP
Browser API - Доступ к функциям браузера (например, `localStorage`, `fetch`)
3rd-party API - API сторонних сервисов (например, OpenAI API или Google Maps API)

_API — это как официант в ресторане, который передаёт твой заказ на кухню (сервер), а потом приносит тебе еду (ответ с данными)._

REST API / RESTful application

- **REST (Representational State Transfer)** — архитектурный стиль для создания веб-сервисов.
- **REST API** — API, которое следует принципам REST.

**7 принципов REST:**
1. **Клиент-сервер (Client-Server)**  
    Клиент и сервер отделены, взаимодействуют через стандартизированный интерфейс.   
2. **Отсутствие состояния (Stateless)**  
    Каждый запрос содержит всю необходимую информацию, сервер не хранит состояние клиента.
3. **Кэшируемость (Cacheable)**  
    Ответы должны явно указывать, можно ли кэшировать данные, чтобы улучшить производительность.
4. **Единый интерфейс (Uniform Interface)**  
    Определён набор правил взаимодействия между клиентом и сервером (URI, методы HTTP, представления ресурсов).
5. **Слойность системы (Layered System)**  
    Архитектура может состоять из нескольких слоев (прокси, шлюзы), которые не влияют на взаимодействие.
6. **Код по требованию (Code on Demand) — опционально**  
    Сервер может передавать клиенту исполняемый код (например, скрипты) для расширения функционала.
7. **Идентификация ресурсов (Identification of Resources)**  
    Ресурсы однозначно идентифицируются через URI.

>**RESTful App** — приложение, которое максимально соответствует этим принципам.

Big O notation — это **математическая нотация**, которая показывает, **как быстро растет время выполнения алгоритма в зависимости от размера входных данных (n).**

O(1) — Константная сложность, не зависит от размера данных

Пример: доступ к элементу массива (`arr[i]`)
```js
function getFirst(arr) {
  return arr[0]; // всегда выполняется за 1 операцию
}
```

O(log n) – Логарифмическое время (Бинарный поиск)

```js
function binarySearch(arr, target) {
  let left = 0, right = arr.length - 1;
  while (left <= right) {
    let mid = Math.floor((left + right) / 2);
    if (arr[mid] === target) return mid;
    if (arr[mid] < target) left = mid + 1;
    else right = mid - 1;
  }
  return -1;
}
```

O(n) — Линейная сложность, растет пропорционально `n`

Пример: линейный поиск в массиве
```js
function linearSearch(arr, target) {
  for (let i = 0; i < arr.length; i++) {
    if (arr[i] === target) return i;
  }
  return -1;
}
```

O(n log n) – Квазилинейное время (Быстрая сортировка, Merge Sort)

```js
function quickSort(arr) {
  if (arr.length <= 1) return arr;
  let pivot = arr[Math.floor(arr.length / 2)];
  let left = arr.filter(x => x < pivot);
  let right = arr.filter(x => x > pivot);
  return [...quickSort(left), pivot, ...quickSort(right)];
}
```

O(n²) – Квадратичное время (Bubble sort)

```js
function bubbleSort(arr) {
  for (let i = 0; i < arr.length; i++) {
    for (let j = 0; j < arr.length - 1; j++) {
      if (arr[j] > arr[j + 1]) {
        [arr[j], arr[j + 1]] = [arr[j + 1], arr[j]];
      }
    }
  }
  return arr;
}
```

O(2ⁿ) – Экспоненциальное время (Фибоначчи через рекурсию)

```js
function fibonacci(n) {
  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}
```

Space/Time complexity

Когда мы анализируем алгоритмы, важно понимать **две ключевые метрики производительности**:

- **Time Complexity (Временная сложность)** – **сколько операций** выполняет алгоритм в зависимости от входных данных.  
- **Space Complexity (Пространственная сложность)** – **сколько памяти** (RAM) требуется для работы алгоритма.

**TDD (Test-Driven Development) — Разработка через тестирование**

**Суть:**
1. **Сначала пишем тест** (который заведомо падает).
2. **Пишем минимальный код**, чтобы тест прошел.

4. **Повторяем** цикл.

BDD расширяет TDD, но фокусируется на **поведении системы**, а не на ее внутренней реализации.

**📌 Отличие от TDD:**
- В BDD пишем тесты не на функции, а на **поведение** приложения.
- Тесты описываются на **человеческом языке** (используется Gherkin-синтаксис).

**XSS (Cross-Site Scripting)** — Межсайтовый скриптинг

XSS — это атака, при которой злоумышленник внедряет **вредоносный JavaScript-код** на страницу, который выполняется у других пользователей.

**Как работает?**
1. Пользователь вводит **скрипт** в форму на сайте.
2. Этот скрипт **сохраняется** и отображается другим пользователям.
3. Код **выполняется в их браузерах** (например, крадёт cookies или делает фейковые запросы).

Если сайт **не фильтрует вводимые данные**, можно вставить JavaScript-код:
```js
<input type="text" value='<script>alert("XSS Attack!");</script>'>
```

**Как защититься?**

✔ **Экранировать (sanitize) ввод**:
- В HTML **преобразовывать** `<`, `>`, `&`, `'`, `"` в `&lt;`, `&gt;` и т. д. 
✔ **Content Security Policy (CSP)** — ограничить выполнение сторонних скриптов.  
- **Использовать `innerText` вместо `innerHTML`**, если вставляете данные в DOM.  
- **Не доверять пользовательскому вводу**, фильтровать его.

---

**CSRF (Cross-Site Request Forgery)** — Подделка межсайтовых запросов

Атака CSRF заставляет пользователя неосознанно выполнить действие на сайте от его имени.

**Как работает?**
1. Пользователь **авторизован** на сайте (`example.com`).
2. Злоумышленник отправляет **скрытый запрос** на `example.com` (например, перевод денег).
3. Сервер **не проверяет источник запроса** → команда выполняется.

Злоумышленник вставляет `<img>` на другой сайт:
```js
<img src="https://bank.com/transfer?to=attacker&amount=1000" />
```

**Как защититься?**
✔ **CSRF-токены**: каждый запрос должен содержать **уникальный токен**.  
✔ **Проверять заголовок `Origin` или `Referer`**.  
✔ **Отключать cookies для CORS-запросов (`SameSite=Strict`)**.  
✔ **Требовать повторного ввода пароля для важных действий**.

---

**Man-in-the-Middle (MITM)** — Атака "человек посередине"

MITM — это перехват данных между пользователем и сервером.

**Как работает?**
1. Пользователь подключается к **незащищённой сети (Wi-Fi в кафе)**.
2. Злоумышленник подменяет запросы и ворует пароли, данные карт.

**Как защититься?**
✔ **Использовать HTTPS**  
✔ **Не передавать чувствительные данные по HTTP**  
✔ **Не использовать открытые Wi-Fi без VPN**

---

**SQL Injection** — Внедрение SQL-кода

SQL Injection — это атака, при которой злоумышленник **вставляет SQL-код** в запрос, чтобы изменить базу данных.

**Как работает?**
1. Входные данные **не фильтруются**.
2. Пользователь вводит SQL-код, который выполняется на сервере.

```sql
const query = `SELECT * FROM users WHERE username = '${username}' AND password = '${password}'`;
```

**Как защититься?**
✔ **Использовать подготовленные запросы (Prepared Statements)**  
✔ **Фильтровать ввод пользователя**  
✔ **Не возвращать детализированные ошибки (stack traces)**

---

**Clickjacking** — Перехват кликов

Злоумышленник вставляет **ваш сайт в `iframe`**, а пользователь кликает **не туда, куда хотел**.

**Как работает?**
1. Жертве показывают **поддельную страницу**, где сверху наложен `iframe` с реальным сайтом.
2. Она кликает "Принять" или "Оплатить", а на самом деле **совершает покупку** или **меняет пароль**.

```html
<iframe src="https://bank.com/transfer?to=attacker&amount=1000" style="opacity: 0; position: absolute;"></iframe>
```

**Как защититься?**
✔ **HTTP-заголовок `X-Frame-Options: DENY`**  
✔ **CSP (Content Security Policy): `frame-ancestors 'none'`**  
✔ **Защищать важные кнопки `double-click` или `captcha`**

**Обфускация (Obfuscation) в JavaScript**

📌 **Что это?**  
Обфускация — это **процесс преобразования исходного кода** в менее читаемый, но **функционально эквивалентный** вариант. Основная цель — **защитить код** от реверс-инжиниринга, копирования или модификации.

До:
```js
function getUserName(id) { return users[id].name; }
```

**После**:
```js
function a(b) { return c[b].d; }
```

**Привязка к домену (Domain Locking)**

📌 **Что это?**  
Привязка к домену — это **механизм, который ограничивает выполнение кода** только на разрешенных доменах. Если код запущен на другом сайте, он **не работает**.

```js
(function() {
    const allowedDomain = "example.com";
    
    if (window.location.hostname !== allowedDomain) {
        document.body.innerHTML = "Доступ запрещен";
        throw new Error("Unauthorized domain");
    }
})();
```

Аутентификация (authentication) — это процесс подтверждения личности пользователя. Веб-приложения используют разные способы аутентификации.

**1. Password-based (аутентификация по паролю)**
Самый распространённый способ аутентификации, где пользователь вводит логин и пароль.

**Как работает:**
1. Пользователь вводит логин и пароль.
2. Сервер проверяет пароль, **сравнивая его хеш** в базе данных.
3. Если пароль верный — пользователь получает сессию или токен.

```js
const bcrypt = require('bcrypt');
const saltRounds = 10;
const password = "my_secure_password";

bcrypt.hash(password, saltRounds, (err, hash) => {
    console.log(hash); // Хеш пароля, который можно сохранить в БД
});
```

**2. OTP (One-Time Password – одноразовые пароли)**
Использует **одноразовый код**, отправленный пользователю (SMS, email, app). Код может быть **TOTP (временной)** или **HOTP (счётный)**.

**Как работает:**
1. Пользователь вводит логин.
2. Сервер отправляет **одноразовый код**.
3. Пользователь вводит код, сервер **проверяет его** и подтверждает вход.

```js
const { authenticator } = require('otplib');
const secret = authenticator.generateSecret(); // Генерация секрета
const token = authenticator.generate(secret); // Генерация OTP-кода
console.log(token);
```

**3. JWT (JSON Web Token – аутентификация через токены)**
**JWT – это самодостаточный токен**, содержащий в себе данные о пользователе. Подписывается **секретным ключом** или **асимметричной криптографией**. Используется в **REST API** для аутентификации без сессий.

**Как работает:**
1. Пользователь вводит логин и пароль.
2. Сервер **создаёт JWT-токен**, подписанный секретным ключом.
3. Клиент сохраняет токен и отправляет его в **заголовке запроса** (`Authorization: Bearer token`).
4. Сервер **декодирует токен** и проверяет подпись.

```js
const jwt = require('jsonwebtoken');
const token = jwt.sign({ userId: 123 }, 'secret_key', { expiresIn: '1h' });

console.log(token); // JWT-токен
```

**4. OAuth 2.0 (аутентификация через соцсети и сервисы)**
OAuth 2.0 позволяет **авторизоваться через Google, GitHub, Facebook** и другие сервисы. Использует **access tokens** вместо пароля.

**Как работает:**
1. Клиент перенаправляет пользователя на страницу авторизации сервиса (например, Google).
2. Пользователь соглашается предоставить доступ.
3. Сервис выдаёт **access token**, который клиент использует для запросов к API.

**5. OpenID Connect (на основе OAuth 2.0, но для идентификации пользователя)**
OpenID Connect (OIDC) — расширение OAuth 2.0, **которое возвращает не только токен, но и информацию о пользователе**. Используется для **единого входа (SSO)**.

**Как работает:**
1. Клиент перенаправляет пользователя на OpenID Provider (например, Google).
2. OpenID Provider **аутентифицирует пользователя и возвращает ID Token (JWT)**.
3. Клиент использует ID Token для идентификации пользователя.

**6. WebAuthn (Аутентификация без пароля)**
WebAuthn — это стандарт аутентификации **без паролей**, использующий **биометрию, ключи безопасности (YubiKey) или TPM-чипы**. Работает через **криптографические ключи, привязанные к устройству**.

**Как работает:**
1. Пользователь регистрирует устройство (например, сканер отпечатка пальца).
2. При входе браузер запрашивает **биометрию или аппаратный ключ**.
3. Сервер проверяет цифровую подпись и аутентифицирует пользователя.

**7. Magic Links (Аутентификация по email-ссылке)**
**Magic Links** — это способ входа **без пароля**, где пользователь получает **уникальную ссылку** на email.

**Как работает:**
1. Пользователь вводит email.
2. Сервер отправляет **одноразовую ссылку**.
3. Пользователь открывает ссылку, сервер проверяет токен, выполняется вход.

**Авторизация** — это процесс **определения прав и доступа пользователя** после успешной аутентификации.

- **Аутентификация** (Authentication) → проверяет, **кто ты** (логин/пароль, JWT, OAuth и т. д.).  
- **Авторизация** (Authorization) → проверяет, **что ты можешь делать** (доступ к данным, ролям, ресурсам).

Виды авторизации:

**1. Ролевая (Role-Based Access Control, RBAC)**
Пользователь получает права на основе своей роли (Admin, User, Manager и т. д.)

**2. Доступ на основе полномочий (Permission-Based Access Control, PBAC)**
Каждый пользователь получает не роль, а конкретные права (create, read, update, delete).

**3. Авторизация на основе атрибутов (Attribute-Based Access Control, ABAC)**
Доступ зависит от конкретных условий (времени, местоположения, устройства, статуса подписки и т. д.)

---

**SSO (Единый вход)** — это механизм аутентификации, который позволяет пользователю **входить в несколько сервисов, вводя данные только один раз**.  
После авторизации в одном сервисе пользователь автоматически получает доступ к другим, без повторного ввода логина и пароля.

**Как работает:**
1. Пользователь заходит на сайт (например, корпоративный портал).
2. Сайт перенаправляет пользователя на **SSO-провайдера** (например, Google, Okta, Auth0).
3. Пользователь **проходит аутентификацию один раз**.
4. SSO-провайдер выдаёт **токен** или **сессию**.
5. При последующих входах на другие сервисы пользователь **автоматически аутентифицирован**.

---

**JWT (JSON Web Token)** — это компактный, безопасный токен аутентификации, который используется для передачи информации между клиентом и сервером в формате JSON.

Он **подписан** (HMAC или RSA), что гарантирует целостность данных, но **не зашифрован** (если не использовать дополнительное шифрование).

JWT часто применяется в **OAuth 2.0, OpenID Connect, аутентификации API и веб-приложениях**.

**Как работает JWT?**
1. Клиент отправляет **логин/пароль** или аутентифицируется через OAuth.
2. Сервер **генерирует JWT**, подписывает его и отправляет клиенту.
3. Клиент **хранит JWT** (обычно в **localStorage**, **sessionStorage** или **HTTP-only cookie**).
4. При каждом запросе клиент **отправляет JWT в заголовке** `Authorization: Bearer <token>`.
5. Сервер **проверяет подпись** и, если JWT валиден, предоставляет доступ.

**Структура JWT:**
JWT состоит из **трёх частей**, разделённых точками:
- Header (заголовок) - метаданные: алгоритм подписи и тип токена.
```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```
- Payload (полезные данные) - содержит информацию (например, ID пользователя, роли, срок действия).
```json
{
  "sub": "1234567890",
  "name": "John Doe",
  "iat": 1710023600,
  "exp": 1710030000
}
```
- Signature (подпись) - Обеспечивает целостность токена.
```js
HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)
```

Протоколы авторизации

| Протокол       | Цель                                 | Тип данных               | Пример использования                      |
| -------------- | ------------------------------------ | ------------------------ | ----------------------------------------- |
| OAuth 2.0      | Авторизация, управление доступом     | Токены (Access, Refresh) | Доступ к API, социальная авторизация      |
| OpenID Connect | Аутентификация поверх OAuth          | ID токен (JWT)           | "Вход с Google", "Вход с Facebook"        |
| SAML           | Аутентификация и авторизация (SSO)   | Утверждение (XML)        | Корпоративные SSO системы                 |
| Kerberos       | Аутентификация и шифрование          | Билеты (Tickets)         | Корпоративные сети (Active Directory)     |
| LDAP           | Аутентификация через директорию      | Учетные данные           | Корпоративные базы данных пользователей   |
| RADIUS         | Аутентификация и управление доступом | Учетные данные           | Управление сетевыми ресурсами             |
| CAS            | Единый вход (SSO)                    | Токены SSO               | Университетские или корпоративные системы |

**1. OAuth 2.0**
- **Тип**: Протокол авторизации
- **Цель**: Предоставить сторонним приложениям доступ к ресурсам пользователя без необходимости раскрывать его учетные данные.
- **Ключевая особенность**: Использует токены доступа (access tokens) для предоставления ограниченного доступа к защищённым ресурсам.
- **Механизм**: После успешной авторизации OAuth 2.0 сервер выдаёт **токен доступа**, который затем может быть использован для запросов к API от имени пользователя.
- **Пример использования**: Авторизация через Google или Facebook (когда вы входите в стороннее приложение с помощью учетной записи Google/Facebook).

Основные этапы:
1. Приложение направляет пользователя на сервер авторизации.
2. Пользователь соглашается предоставить доступ к своим ресурсам.
3. Сервер выдает **токен доступа**, который используется для доступа к API.

**2. OpenID Connect (OIDC)**
- **Тип**: Протокол аутентификации, построенный поверх OAuth 2.0
- **Цель**: Авторизовать пользователя и получить его идентификационные данные.
- **Ключевая особенность**: OpenID Connect расширяет OAuth 2.0 для поддержки аутентификации (проверки личности пользователя), предоставляя так называемый **ID токен**.
- **Механизм**: Помимо токенов доступа, OpenID Connect предоставляет **ID токен**, который содержит информацию о пользователе, такую как его уникальный идентификатор, email и т.д.
- **Пример использования**: "Вход с помощью Google" — это классический пример использования OIDC.

Основные этапы:
1. OAuth 2.0 используется для получения токена доступа.
2. OIDC добавляет получение **ID токена**, который содержит данные аутентификации пользователя.

**3. SAML (Security Assertion Markup Language)**
- **Тип**: Протокол аутентификации и авторизации на основе XML.
- **Цель**: Обеспечить единую точку входа (SSO) для доступа к нескольким приложениям.
- **Ключевая особенность**: Использует формат XML для обмена данными между поставщиком идентификации (IdP) и поставщиком услуг (SP).
- **Механизм**: Поставщик идентификации (например, корпоративный сервер) аутентифицирует пользователя, а затем выдает **утверждение** (assertion), которое передается на сайт или сервис для авторизации.
- **Пример использования**: Корпоративные SSO системы, когда один логин дает доступ ко многим корпоративным приложениям (например, Microsoft Active Directory).

Основные этапы:
1. Пользователь аутентифицируется на IdP (например, в корпоративной сети).
2. IdP отправляет SAML утверждение поставщику услуг (например, веб-сайту).
3. Поставщик услуг предоставляет доступ пользователю на основе полученного утверждения.

**4. Kerberos**
- **Тип**: Протокол аутентификации с распределением секретных ключей.
- **Цель**: Защищённая аутентификация и авторизация в сетевых системах.
- **Ключевая особенность**: Использует симметричные ключи и требует наличия доверенного третьего лица — **Центра распределения ключей (KDC)**.
- **Механизм**: Клиент и сервер получают билеты от KDC, которые используются для подтверждения личности и авторизации доступа.
- **Пример использования**: Используется в корпоративных сетях, например, в Microsoft Active Directory.

Основные этапы:
1. Клиент запрашивает аутентификацию у KDC.
2. KDC выдает **билет** для клиента, который клиент использует для доступа к серверу.
3. Сервер проверяет билет и предоставляет доступ.

**5. LDAP (Lightweight Directory Access Protocol)**
- **Тип**: Протокол доступа к каталогам данных.
- **Цель**: Аутентификация и авторизация через централизованную базу данных, такую как директория пользователей.
- **Ключевая особенность**: Использует древовидную структуру для хранения данных пользователей и прав доступа.
- **Механизм**: Приложения и службы обращаются к LDAP-серверу для аутентификации пользователей и получения информации о них.
- **Пример использования**: Корпоративные сети для централизованной аутентификации пользователей, управления доступом к сетевым ресурсам.

Основные этапы:
1. Клиент запрашивает доступ к директории на сервере LDAP.
2. Сервер проверяет учетные данные клиента и предоставляет соответствующие данные или разрешения.

**6. RADIUS (Remote Authentication Dial-In User Service)**
- **Тип**: Протокол для централизованной аутентификации, авторизации и учёта.
- **Цель**: Управление доступом к сетевым ресурсам через централизованный сервер.
- **Ключевая особенность**: Часто используется для управления доступом в корпоративных сетях, обеспечивая аутентификацию пользователей и устройств.
- **Механизм**: Клиенты отправляют запросы на доступ к серверу RADIUS, который проверяет учетные данные и отправляет ответ.
- **Пример использования**: Сетевые компании используют RADIUS для управления доступом к Wi-Fi сетям, VPN и другим сетевым ресурсам.

**7. CAS (Central Authentication Service)**
- **Тип**: Протокол единого входа (SSO).
- **Цель**: Обеспечить единый вход для аутентификации пользователя через несколько приложений.
- **Ключевая особенность**: После того как пользователь аутентифицируется один раз через CAS-сервер, он может получать доступ к различным сервисам без повторного ввода учетных данных.
- **Пример использования**: Используется в университетах и крупных организациях для управления доступом к разным внутренним системам.
