# Объектно-ориентированное программирование

### Определение

Объектно-ориентированное программирование (ООП) — это парадигма программирования, ориентированная на **объекты**, которые объединяют:

- **состояние (поля)**
- **поведение (методы)**

В отличие от подходов, ориентированных на функции или простые данные, ООП позволяет моделировать реальные сущности, их взаимодействие и изменения состояния во времени. Каждый объект может представлять что угодно: пользователя, документ, устройство, процесс и т.д.

Объекты взаимодействуют друг с другом через **вызовы методов** и обмен сообщениями, что обеспечивает высокую гибкость и структурированность программы. Такой подход упрощает понимание сложных систем, делает их более предсказуемыми и облегчает командную разработку.

### Основная цель

- **Упрощение разработки и поддержки больших систем.**
  Большие проекты делятся на отдельные объекты с чёткими обязанностями, что облегчает понимание, тестирование и рефакторинг кода.

- **Моделирование реальных сущностей.**
  Программисты могут создавать объекты, отражающие реальные вещи или процессы, что улучшает визуализацию и концептуальное понимание системы.

- **Изоляцию деталей реализации.**
  Скрытие внутренних деталей (инкапсуляция) позволяет менять или расширять объекты без влияния на остальную систему.

### Дополнительные аспекты

- **Объекты как самостоятельные сущности.** Каждый объект существует независимо и управляет своим состоянием.

- **Взаимодействие через сообщения.** Объекты не изменяют внутренние данные других напрямую, а используют методы для запроса действий или информации.

- **Динамическое поведение.** Методы объектов могут изменять состояние в зависимости от условий, что позволяет моделировать реальные процессы.

- **Организация больших проектов.** Системы можно разложить на множество объектов и подсистем, каждая из которых выполняет конкретную роль, облегчая командную работу и поддержку.

## Из чего состоит ООП

ООП можно рассматривать как совокупность нескольких ключевых элементов, которые образуют основу любой объектно-ориентированной системы:

#### 1. Классы

Класс — это **шаблон (чертёж)**, который описывает структуру и поведение объектов. Он определяет, какие поля (данные) и методы (функции) будут у экземпляров этого класса. Класс не существует в памяти до тех пор, пока из него не будет создан объект.

```js
class User {
  constructor(name, email) {
    this.name = name;
    this.email = email;
    this.isLoggedIn = false;
  }

  login() {
    this.isLoggedIn = true;
    console.log(`${this.name} вошёл в систему.`);
  }
}
```

#### 2. Объекты

Объект — это **экземпляр класса**, то есть конкретная реализация шаблона. Он существует в памяти, хранит собственные значения данных и может выполнять методы, определённые в классе.

```js
const user1 = new User("Alex", "alex@mail.com");
const user2 = new User("Mila", "mila@mail.com");

user1.login(); // Alex вошёл в систему.
user2.login(); // Mila вошёл в систему.
```

#### 3. Атрибуты (свойства, поля)

Это **данные**, которые описывают состояние объекта. Каждый объект хранит собственные значения атрибутов, даже если создан из одного класса.

```js
console.log(user1.name); // Alex
console.log(user2.name); // Mila
```

---

#### 4. Методы

Методы — это **функции**, которые определяют поведение объекта. Они позволяют объектам взаимодействовать с другими объектами и изменять своё состояние.

```js
user1.login(); // Alex вошёл в систему.
console.log(user1.isLoggedIn); // true
```

---

#### 5. Взаимодействие между объектами

Объекты не существуют изолированно — они **взаимодействуют** друг с другом, обмениваясь сообщениями (вызовами методов). Такое взаимодействие позволяет моделировать сложные процессы.

```js
class Order {
  constructor(user, total) {
    this.user = user;
    this.total = total;
  }

  complete() {
    console.log(
      `Заказ на сумму ${this.total}$ оформлен пользователем ${this.user.name}.`
    );
  }
}

const order = new Order(user1, 250);
order.complete(); // Заказ на сумму 250$ оформлен пользователем Alex.
```

---

#### 6. Модули и пространства имён

В крупных системах классы и объекты объединяются в **модули, пакеты или пространства имён** для организации структуры проекта. Это помогает управлять зависимостями и логически группировать компоненты.

```js
// user.js
export class User {
  constructor(name) {
    this.name = name;
  }
}

// order.js
import { User } from "./user.js";

const user = new User("Eva");
console.log(`Создан новый пользователь: ${user.name}`);
```

## Основные принципы

- **Абстракция:** <span class="text-gray">автомобиль и двигатель</span>
- **Инкапсуляция:** <span class="text-gray">банк и баланс</span>
- **Наследование:** <span class="text-gray">животное и собака</span>
- **Полиморфизм:** <span class="text-gray">разное чирикание у птиц</span>

### Абстракция

Абстракция позволяет сосредоточиться только на **существенных характеристиках** объекта, скрывая сложные внутренние детали. Она помогает проектировать систему на уровне интерфейсов и моделей, не вдаваясь в конкретные реализации.

<b class="text-on-semi-light">Ключевая идея:</b> Скрываем сложность, показываем только важное.

<b class="text-on-semi-light">Как запомнить:</b> Представь автомобиль — ты не знаешь, как работает двигатель, но знаешь, что есть педаль газа и тормоз. Интерфейс — это педали, а реализация — двигатель.

<b class="text-on-semi-light">Формула:</b> Интерфейс → скрывает детали реализации.

<b class="text-on-semi-light">На фронте:</b> бизнес-логика не зависит от конкретного API — используем абстрактный слой (hooks, сервисы, адаптеры), который можно заменить без изменений UI.

```js
class Car {
  startEngine() {
    throw new Error("Method must be implemented");
  }
}

class ElectricCar extends Car {
  startEngine() {
    console.log("Starting silent electric engine");
  }
}

class GasolineCar extends Car {
  startEngine() {
    console.log("Starting roaring gasoline engine");
  }
}

const myCar = new ElectricCar();
myCar.startEngine(); // Starting silent electric engine
```

Абстракция делает код гибким: мы можем заменить реализацию двигателя, не меняя интерфейс автомобиля.

---

### Инкапсуляция

Инкапсуляция скрывает внутренние данные объекта и предоставляет **контролируемый доступ** к ним через методы (геттеры и сеттеры). Это защищает состояние объекта и предотвращает ошибки.

<b class="text-on-semi-light">Ключевая идея:</b> Прячем внутренние данные и разрешаем доступ к ним только через методы.

<b class="text-on-semi-light">Как запомнить:</b> Представь банковский счёт — клиент не может напрямую изменить баланс, только внести или снять деньги через банк.

<b class="text-on-semi-light">Формула:</b> Данные → защищены методами.

<b class="text-on-semi-light">На фронте:</b> состояние и логика компонента скрыты от внешнего кода — доступ только через публичные пропсы и методы.

```js
class BankAccount {
  #balance = 0; // приватное поле

  deposit(amount) {
    if (amount > 0) this.#balance += amount;
  }

  withdraw(amount) {
    if (this.#balance >= amount) this.#balance -= amount;
  }

  getBalance() {
    return this.#balance;
  }
}

const account = new BankAccount();
account.deposit(1000);
account.withdraw(300);
console.log(`Balance: $${account.getBalance()}`); // Balance: $700
```

Инкапсуляция обеспечивает надёжность: нельзя напрямую изменить баланс и нарушить внутреннюю логику.

---

### Наследование

Наследование позволяет одному классу **наследовать свойства и методы другого**, расширяя или переопределяя их поведение.

<b class="text-on-semi-light">Ключевая идея:</b> Позволяет одному классу использовать код и поведение другого.

<b class="text-on-semi-light">Как запомнить:</b> Собака наследует общие черты животного (дышать, двигаться), но реализует свои особенности (лаять).

<b class="text-on-semi-light">Формула:</b> Общий функционал → передаётся потомкам.

<b class="text-on-semi-light">На фронте:</b> переиспользуем общие компоненты или классы (например, `BaseButton`, `BaseService`), расширяя их поведение.

```js
class Animal {
  speak() {
    console.log("Some generic animal sound");
  }
}

class Dog extends Animal {
  speak() {
    console.log("Woof! Woof!");
  }
}

const rex = new Dog();
rex.speak(); // Woof! Woof!
```

Наследование помогает избежать дублирования кода и выстраивает иерархию, где базовый класс определяет общие черты, а потомки — конкретное поведение.

### Полиморфизм

Полиморфизм позволяет использовать **один интерфейс для разных реализаций** — объекты разных классов могут вести себя по-разному при одинаковых вызовах.

<b class="text-on-semi-light">Ключевая идея:</b> Один интерфейс — много реализаций.

<b class="text-on-semi-light">Как запомнить:</b> Представь птиц — все они умеют чирикать, но каждый делает это по-своему: воробей щебечет, сова ухает, ворон каркает. Метод один — `chirp()`, но реализация у каждого класса своя.

<b class="text-on-semi-light">Формула:</b> Один метод → разное поведение у разных объектов.

<b class="text-on-semi-light">На фронте:</b> один интерфейс — разные реализации: компоненты с одинаковыми пропсами (`value`, `onChange`), но разным рендерингом или логикой.

```js
class Bird {
  chirp() {
    console.log("Some bird sound");
  }
}

class Sparrow extends Bird {
  chirp() {
    console.log("Chirp chirp!");
  }
}

class Owl extends Bird {
  chirp() {
    console.log("Hoot hoot!");
  }
}

class Crow extends Bird {
  chirp() {
    console.log("Caw caw!");
  }
}

const birds = [new Sparrow(), new Owl(), new Crow()];
birds.forEach((bird) => bird.chirp());
```

Полиморфизм делает код гибким и расширяемым: можно добавлять новые классы птиц без изменения основной логики.

## Принципы проектирования (SOLID)

Принципы **SOLID** — это фундаментальные правила, которые лежат в основе **объектно-ориентированного проектирования (ООП)**. Они помогают создавать **гибкие, масштабируемые и легко поддерживаемые системы**, минимизируя дублирование и зависимость между компонентами.

Эти принципы не просто дополняют ООП — они объясняют, **как правильно применять** его базовые идеи (инкапсуляцию, наследование, полиморфизм и абстракцию) на практике. Если ООП даёт инструменты, то SOLID показывает, **как их использовать, чтобы проект не превратился в хаос**.

Каждый принцип решает конкретную архитектурную проблему:

- **SRP** — борется с избыточной ответственностью и сложностью классов;
- **OCP** — помогает расширять систему без переписывания существующего кода;
- **LSP** — гарантирует корректность наследования и заменяемость объектов;
- **ISP** — предотвращает избыточные зависимости между компонентами;
- **DIP** — делает архитектуру независимой от конкретных реализаций.

### Single Responsibility Principle

<b class="text-on-semi-light">Single Responsibility Principle (SRP) — Принцип единственной ответственности</b>

Каждый класс должен иметь **только одну причину для изменения**. То есть выполнять только одну задачу.

<b class="text-on-semi-light">Ключевая идея:</b> Один класс — одна ответственность.

<b class="text-on-semi-light">Как запомнить:</b> Представь работника: если он и код пишет, и кофе варит, и бухгалтерию ведёт — любая проблема в этих областях затронет его. Разделяй обязанности.

<b class="text-on-semi-light">Формула:</b> Одна причина → одно изменение.

<b class="text-on-semi-light">На фронте:</b> компонент отвечает только за отображение, а не за логику запроса; отдельный хук или сервис обрабатывает данные.

**Плохо:**

```java
class UserService {
    void createUser(String name) {
        // логика создания пользователя
    }

    void sendEmail(String email) {
        // логика отправки email
    }
}
```

**Хорошо:**

```java
class UserService {
    void createUser(String name) {}
}

class EmailService {
    void sendEmail(String email) {}
}
```

Каждый класс делает своё — проще тестировать и менять.

### Open/Closed Principle

<b class="text-on-semi-light">Open/Closed Principle (OCP) — Принцип открытости/закрытости</b>

Классы должны быть **открыты для расширения, но закрыты для изменения**. То есть не нужно переписывать старый код — лучше добавить новый.

<b class="text-on-semi-light">Ключевая идея:</b> Изменяем поведение через расширение, не трогая существующий код.

<b class="text-on-semi-light">Как запомнить:</b> Не ломай старое — добавь новое. Как в розетку: не нужно разбирать стену, чтобы вставить переходник.

<b class="text-on-semi-light">Формула:</b> Добавляем → не переписываем.

<b class="text-on-semi-light">На фронте:</b> добавляем новые темы, типы карточек или варианты UI через пропсы или конфигурации, не переписывая базовый компонент.

**Плохо:**

```java
class PaymentService {
    void pay(String method) {
        if (method.equals("paypal")) {
            // PayPal
        } else if (method.equals("stripe")) {
            // Stripe
        }
    }
}
```

**Хорошо:**

```java
interface PaymentMethod {
    void pay();
}

class PayPalPayment implements PaymentMethod {
    public void pay() { System.out.println("Paid via PayPal"); }
}

class StripePayment implements PaymentMethod {
    public void pay() { System.out.println("Paid via Stripe"); }
}

class PaymentService {
    void process(PaymentMethod method) { method.pay(); }
}
```

Теперь можно добавлять новые способы оплаты, не изменяя `PaymentService`.

### Liskov Substitution Principle

<b class="text-on-semi-light">Liskov Substitution Principle (LSP) — Принцип подстановки Барбары Лисков</b>

Подкласс должен **полностью заменять родительский класс** без нарушения логики программы.

<b class="text-on-semi-light">Ключевая идея:</b> Наследник должен вести себя как родитель.

<b class="text-on-semi-light">Как запомнить:</b> Если в коде вместо родителя поставить ребёнка — всё должно работать.

<b class="text-on-semi-light">Формула:</b> Дочерний класс = Родительский класс.

<b class="text-on-semi-light">На фронте:</b> любой кастомный Input должен работать там же, где используется базовый Input — с теми же пропсами и поведением.

**Плохо:**

```java
class Bird {
    void fly() {}
}

class Penguin extends Bird {
    void fly() { throw new UnsupportedOperationException(); }
}
```

**Хорошо:**

```java
class Bird {
    void makeSound() {}
}

class Sparrow extends Bird {
    void makeSound() { System.out.println("Chirp"); }
}

class Penguin extends Bird {
    void makeSound() { System.out.println("Honk"); }
}
```

Теперь пингвин корректно заменяет птицу, не ломая поведение.

### Interface Segregation Principle

<b class="text-on-semi-light">Interface Segregation Principle (ISP) — Принцип разделения интерфейсов</b>

Клиенты не должны зависеть от интерфейсов, которые они **не используют**. Лучше иметь несколько маленьких интерфейсов, чем один огромный.

<b class="text-on-semi-light">Ключевая идея:</b> Разделяй обязанности интерфейсов.

<b class="text-on-semi-light">Как запомнить:</b> Не заставляй рыбу летать. Если интерфейс обязывает к ненужным методам — он слишком общий.

<b class="text-on-semi-light">Формула:</b> Один интерфейс — одна ответственность.

<b class="text-on-semi-light">На фронте:</b> разделяем большие API-интерфейсы или пропсы-компонентов на меньшие; компонент не должен требовать ненужные данные.

**Плохо:**

```java
interface Worker {
    void work();
    void eat();
}

class Robot implements Worker {
    public void work() {}
    public void eat() { throw new UnsupportedOperationException(); }
}
```

**Хорошо:**

```java
interface Workable { void work(); }
interface Eatable { void eat(); }

class Human implements Workable, Eatable {
    public void work() {}
    public void eat() {}
}

class Robot implements Workable {
    public void work() {}
}
```

Теперь каждый класс реализует только нужное поведение.

### Dependency Inversion Principle

<b class="text-on-semi-light">Dependency Inversion Principle (DIP) — Принцип инверсии зависимостей</b>

Модули верхнего уровня не должны зависеть от модулей нижнего уровня — **оба должны зависеть от абстракций**.

<b class="text-on-semi-light">Ключевая идея:</b> Завись от интерфейсов, а не от конкретных классов.

<b class="text-on-semi-light">Как запомнить:</b> Если ресторан зависит от конкретного повара, всё рухнет, когда тот уволится. Пусть ресторан работает с интерфейсом “повар”.

<b class="text-on-semi-light">Формула:</b> Абстракция > Реализация.

<b class="text-on-semi-light">На фронте:</b> бизнес-логика не зависит от конкретного API — используем абстрактный слой (hooks, сервисы, адаптеры), который можно заменить без изменений UI.

**Плохо:**

```java
class MySQLDatabase {
    void save(String data) {}
}

class UserService {
    private MySQLDatabase db = new MySQLDatabase();
    void saveUser(String data) { db.save(data); }
}
```

**Хорошо:**

```java
interface Database {
    void save(String data);
}

class MySQLDatabase implements Database {
    public void save(String data) {}
}

class UserService {
    private Database db;

    UserService(Database db) {
        this.db = db;
    }

    void saveUser(String data) { db.save(data); }
}
```

Теперь `UserService` можно использовать с любым типом базы данных.

## Паттерны проектирования

Паттерны проектирования появились как **естественное продолжение идей ООП**. Когда разработчики начали активно использовать классы, объекты и принципы вроде инкапсуляции и полиморфизма, стало ясно: во многих проектах повторяются одни и те же архитектурные решения.

ООП породило целое направление — **Design Patterns (GoF)**:

- **Порождающие:** Singleton, Factory, Builder...
- **Структурные:** Adapter, Decorator, Composite...
- **Поведенческие:** Strategy, Observer, Command...

Паттерны — это способ **правильно применять ООП на практике**, чтобы код был читаемым, расширяемым и устойчивым к изменениям.

> Более подробно о паттернах проектирования написано здесь — [Паттерны проектирования](/content/architecture/design-patterns).

## Типичные проблемы и критика

Хотя ООП десятилетиями оставалось доминирующей парадигмой, в современном фронтенде оно часто подвергается критике — особенно после перехода экосистемы **React** и **Vue** от классов и наследования к **функциональному стилю**.

Основные проблемы классического ООП:

- **Избыточная иерархия классов** — глубокие деревья наследования усложняют поддержку и расширение.
- **Сильная связность** — изменение одного класса часто требует изменений в других.
- **Сложность тестирования** — поведение объектов скрыто за состоянием, что делает unit-тесты менее предсказуемыми.
- **"Божественные объекты" (God Objects)** — классы, которые делают слишком много.
- **Нарушение SRP и DIP** — при росте проекта классы начинают смешивать ответственность и зависеть от деталей.

### Почему фронтенд отходит от ООП

- **React** изначально поддерживал классовые компоненты (`class Component`), но со временем перешёл на **функциональные компоненты с хуками** — чистые функции без состояния, с декларативным управлением жизненным циклом.
- **Vue 3** продвигает **Composition API**, который ближе к функциональному стилю, чем к традиционному ООП.
- В современном фронтенде **функциональная композиция, чистые функции, иммутабельность и реактивность** оказались более удобными для работы с UI, чем наследование и классовая иерархия.

ООП по-прежнему полезно — например, для проектирования SDK, библиотек, сложных бизнес-доменов или игровых движков. Но в UI-разработке оно всё чаще заменяется **FP-подходами** (функции, композиция, хуки, реактивные потоки).

---

**Сравнение OOP и FP:**

| Критерий          | ООП                              | FP                                    |
| ----------------- | -------------------------------- | ------------------------------------- |
| Модель            | Сущности и взаимодействия        | Потоки данных и функции               |
| Состояние         | Изменяемое                       | Неизменяемое                          |
| Поведение         | Методы объектов                  | Композиция функций                    |
| Переиспользование | Наследование, интерфейсы         | Высшие функции, композиция            |
| Пример (React)    | `class App extends Component {}` | `function App() { return <UI /> }`    |
| Главный принцип   | Инкапсуляция и структура         | Прозрачность и предсказуемость данных |

> Подробнее про [функциональное программирование](/content/computer-science/fp).

## Применение ООП в индустрии

- **Enterprise backend:** Java, C#, Spring, .NET.
- **GameDev:** Unity (C#), Unreal (C++).
- **Mobile:** Android (Kotlin/Java), iOS (Swift, Objective-C).
- **Frontend:** React классовые компоненты, TypeScript модели данных.
- **AI/ML:** Python-классы для моделей, пайплайнов.

ООП остаётся стандартом проектирования сложных систем, особенно там, где важно моделировать сложные домены и взаимодействия между сущностями.
