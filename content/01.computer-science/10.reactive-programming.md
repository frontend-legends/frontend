# Реактивное программирование

Реактивное программирование — это парадигма, в которой данные рассматриваются как **потоки значений во времени**, а изменения **автоматически распространяются** по зависимым вычислениям. Вместо того чтобы вручную обновлять каждую зависимую переменную, ты описываешь связи между данными — а система сама следит за синхронизацией.

Для фронтенд-разработчика это центральная тема: **вся современная UI-разработка построена на реактивности** — от `ref()` в Vue до `useState()` в React, от RxJS в Angular до Signals в Solid. Понимание принципов реактивности позволяет осознанно выбирать инструменты и не бороться с фреймворком.

> Подробнее о месте реактивного программирования среди других парадигм — [Парадигмы программирования](/content/computer-science/paradigms).

## Императивный подход vs реактивный

<b class="text-on-semi-light">Императивный: ты обновляешь вручную</b>

```js
let a = 1;
let b = 2;
let sum = a + b; // 3

a = 10;
// sum по-прежнему 3 — нужно вручную пересчитать
sum = a + b; // 12
```

<b class="text-on-semi-light">Реактивный: обновление происходит автоматически</b>

```js
const a = ref(1);
const b = ref(2);
const sum = computed(() => a.value + b.value); // 3

a.value = 10;
// sum.value автоматически стал 12
```

В реактивной модели `sum` — это **не значение**, а **вычисление, привязанное к источникам**. Когда `a` или `b` меняются, `sum` пересчитывается без явного вызова.

## Ключевые концепции

### Push vs Pull

Два фундаментальных подхода к распространению изменений.

| Модель | Кто инициирует | Как работает | Примеры |
|---|---|---|---|
| **Pull** | Потребитель | Потребитель запрашивает данные, когда они нужны | Iterator, Generator, React render |
| **Push** | Производитель | Производитель уведомляет потребителей при изменении | Observable, EventEmitter, Vue reactivity |

<span class="text-gray">**Pull**: ты спрашиваешь «какое сейчас значение?» — и получаешь ответ в этот момент. Генераторы и итераторы работают так: `.next()` вытягивает следующее значение.</span>

<span class="text-gray">**Push**: источник сам говорит «я изменился» — и все подписчики получают обновление. EventEmitter, промисы (одноразовый push), Observable (многоразовый push) — всё это push-модели.</span>

```text
Pull:   Consumer ──запрос──> Producer ──ответ──> Consumer
Push:   Producer ──событие──> Consumer
                 ──событие──> Consumer
                 ──событие──> Consumer
```

> Promise — это **push с одним значением**: производитель resolve'ит, потребитель получает через `.then()`. Observable — **push с множеством значений** во времени.

### Observer Pattern (паттерн Наблюдатель)

Фундамент реактивности. Subject (субъект) хранит список подписчиков и уведомляет их при изменении состояния.

```ts
class Subject<T> {
  private observers: Array<(value: T) => void> = [];

  subscribe(fn: (value: T) => void) {
    this.observers.push(fn);
    return () => {
      this.observers = this.observers.filter(ob => ob !== fn);
    };
  }

  notify(value: T) {
    this.observers.forEach(fn => fn(value));
  }
}

const price = new Subject<number>();

const unsub = price.subscribe(v => console.log('Цена:', v));

price.notify(100);  // Цена: 100
price.notify(250);  // Цена: 250
unsub();            // отписка
price.notify(300);  // тишина
```

<span class="text-gray">Этот паттерн лежит в основе `EventEmitter` в Node.js, `addEventListener` в DOM, реактивных систем Vue/Solid и библиотеки RxJS. Различия — в уровне абстракции и возможностях композиции.</span>

### Dependency Tracking (отслеживание зависимостей)

Реактивная система должна знать, **какие вычисления зависят от каких данных**. Два основных подхода:

<b class="text-on-semi-light">Явная подписка</b>

<span class="text-gray">Разработчик сам указывает, за чем следить. Пример — массив зависимостей в React `useEffect`:</span>

```js
useEffect(() => {
  document.title = `Count: ${count}`;
}, [count]); // явно указываем зависимость
```

<b class="text-on-semi-light">Автоматическое отслеживание</b>

<span class="text-gray">Система сама определяет зависимости во время выполнения. При первом запуске вычисления она запоминает, какие реактивные переменные были **прочитаны**, и подписывается на них. Так работают Vue `computed`, Solid `createEffect`, MobX `autorun`:</span>

```js
// Vue: зависимости определяются автоматически
const fullName = computed(() => {
  // Vue отслеживает, что были прочитаны firstName.value и lastName.value
  return `${firstName.value} ${lastName.value}`;
});
```

| Подход | Плюсы | Минусы | Примеры |
|---|---|---|---|
| Явные зависимости | Предсказуемость, контроль | Легко забыть зависимость, stale closures | React hooks |
| Автотрекинг | Нет ручной работы, нет ошибок | Магия, сложнее дебажить | Vue, Solid, MobX |

## Reactive Primitives (реактивные примитивы)

Современные фреймворки предоставляют набор базовых примитивов для построения реактивных систем.

### Signal (сигнал)

Signal — это **ячейка с реактивным значением**. При чтении сигнала внутри реактивного контекста автоматически создаётся подписка. При записи — все зависимые вычисления пересчитываются.

```ts
// Solid.js
const [count, setCount] = createSignal(0);

createEffect(() => {
  console.log(count()); // автоподписка на count
});

setCount(1); // эффект перезапустится
setCount(2); // и ещё раз
```

Сигналы сейчас реализованы или предложены во многих фреймворках:

| Фреймворк | Примитив | Синтаксис |
|---|---|---|
| **Vue 3** | `ref()` / `reactive()` | `ref(0)`, `reactive({})` |
| **Solid** | `createSignal()` | `const [get, set] = createSignal(0)` |
| **Angular 16+** | `signal()` | `signal(0)` |
| **Svelte 5** | `$state` | `let count = $state(0)` |
| **Preact** | `@preact/signals` | `signal(0)` |
| **TC39 Proposal** | `Signal` | `new Signal.State(0)` (stage 1) |

### Computed / Derived (вычисляемое значение)

Значение, которое **автоматически пересчитывается** при изменении зависимостей. Обычно кэшируется — если зависимости не изменились, повторного вычисления не происходит (memoization).

```ts
// Vue
const firstName = ref('John');
const lastName = ref('Doe');

const fullName = computed(() => `${firstName.value} ${lastName.value}`);
// fullName.value === 'John Doe'

firstName.value = 'Jane';
// fullName.value автоматически === 'Jane Doe'
```

### Effect (эффект / побочное действие)

Функция, которая выполняется **при изменении зависимостей** и производит побочные эффекты: обновляет DOM, отправляет запрос, пишет в лог.

```ts
// Solid
createEffect(() => {
  // Выполнится при каждом изменении count()
  document.title = `Счётчик: ${count()}`;
});
```

```ts
// Vue
watch(count, (newVal, oldVal) => {
  console.log(`Было: ${oldVal}, стало: ${newVal}`);
});
```

### Реактивный граф

Signal, Computed и Effect формируют **направленный ациклический граф (DAG)** зависимостей:

```text
  Signal A ──┐
             ├──> Computed C ──> Effect (DOM update)
  Signal B ──┘
                  ↑
          Computed D ──> Effect (log)
```

Когда Signal A меняется, система обходит граф и пересчитывает **только** затронутые узлы. Это называется **fine-grained reactivity** — обновляется ровно то, что нужно, без лишних вычислений.

## Fine-Grained vs Coarse-Grained реактивность

Ключевое архитектурное различие между подходами фреймворков.

<b class="text-on-semi-light">Fine-grained (мелкозернистая)</b>

<span class="text-gray">Каждый реактивный примитив точно знает, кто от него зависит. При изменении обновляется **только конкретный** участок DOM или вычисление. Нет Virtual DOM, нет diffing. Минимальная работа.</span>

Примеры: **Vue**, **Solid**, **Svelte**, **Angular Signals**.

<b class="text-on-semi-light">Coarse-grained (крупнозернистая)</b>

<span class="text-gray">При изменении состояния компонент **целиком** перевычисляется (re-render). Фреймворк сравнивает новый и старый Virtual DOM (diffing) и применяет минимальные изменения к реальному DOM.</span>

Примеры: **React** (Virtual DOM + reconciliation).

| Характеристика | Fine-grained | Coarse-grained (VDOM) |
|---|---|---|
| Обновление | Точечное, только затронутые узлы | Re-render компонента + diffing |
| Overhead | Подписки и граф зависимостей | VDOM diff и reconciliation |
| Когда быстрее | Частые, мелкие обновления | Массовые обновления (batching) |
| Пример | Vue `ref()` обновляет один `<span>` | React re-render всего компонента |
| Ментальная модель | Данные → DOM напрямую | UI = f(state), render при каждом изменении |

> React не является «реактивным» в строгом смысле. Он использует **pull-модель**: компонент вызывается заново (re-render), и React «вытягивает» текущие значения state. Vue, Solid и Svelte используют **push-модель**: изменение данных само «проталкивает» обновление в DOM.

## Observables и RxJS

Observable — обобщённый реактивный примитив, представляющий **поток значений во времени**. В отличие от Promise (одно значение, eager), Observable может emit'ить **множество значений**, и подписка ленивая (lazy) — поток начинает работать только при подписке.

### Observable: базовая механика

```ts
import { Observable } from 'rxjs';

const stream$ = new Observable<number>(subscriber => {
  subscriber.next(1);
  subscriber.next(2);
  subscriber.next(3);

  setTimeout(() => {
    subscriber.next(4);
    subscriber.complete(); // завершение потока
  }, 1000);
});

// Подписка запускает поток
const subscription = stream$.subscribe({
  next: val => console.log(val),     // 1, 2, 3, ...через 1с... 4
  complete: () => console.log('done'),
  error: err => console.error(err),
});

// Отписка — важно для предотвращения утечек
subscription.unsubscribe();
```

### Hot vs Cold Observable

| Тип | Когда начинает emit | Подписчики | Пример |
|---|---|---|---|
| **Cold** | При каждой подписке (lazy) | Каждый получает свой поток | HTTP-запрос, timer |
| **Hot** | Независимо от подписок | Все получают одни и те же данные | WebSocket, DOM events, Subject |

```ts
// Cold: каждый подписчик получает свой поток
const cold$ = new Observable(sub => {
  sub.next(Math.random()); // разное значение для каждого подписчика
});

// Hot: один источник, все получают одинаковые данные
const subject = new Subject<number>();
subject.subscribe(v => console.log('A:', v));
subject.subscribe(v => console.log('B:', v));
subject.next(42); // A: 42, B: 42
```

### Операторы: композиция потоков

Сила RxJS — в операторах, которые позволяют **трансформировать, фильтровать и комбинировать** потоки декларативно. Операторы не мутируют исходный поток, а создают новый (как `map`/`filter` для массивов, но для значений во времени).

```ts
import { fromEvent, map, filter, debounceTime, switchMap } from 'rxjs';

// Поиск с debounce: реальный паттерн из продакшена
const search$ = fromEvent<InputEvent>(input, 'input').pipe(
  map(e => (e.target as HTMLInputElement).value),   // достать текст
  filter(query => query.length >= 3),                // минимум 3 символа
  debounceTime(300),                                 // подождать 300мс без ввода
  switchMap(query => fetchResults(query)),            // отменить предыдущий запрос
);

search$.subscribe(results => renderResults(results));
```

<b class="text-on-semi-light">Ключевые категории операторов</b>

| Категория | Операторы | Что делают |
|---|---|---|
| Трансформация | `map`, `scan`, `switchMap`, `mergeMap`, `concatMap` | Преобразуют значения или переключают потоки |
| Фильтрация | `filter`, `distinctUntilChanged`, `take`, `skip`, `first` | Пропускают/отбрасывают значения |
| Комбинирование | `merge`, `combineLatest`, `zip`, `forkJoin`, `withLatestFrom` | Объединяют несколько потоков |
| Управление временем | `debounceTime`, `throttleTime`, `delay`, `timeout` | Контролируют timing |
| Обработка ошибок | `catchError`, `retry`, `retryWhen` | Восстановление после ошибок |

### switchMap vs mergeMap vs concatMap

Самый частый вопрос на интервью по RxJS.

```text
switchMap:  A──1──2──B──3──4──
            Результат: A:1, A:2, B:3, B:4
            → Отменяет предыдущий внутренний поток при новом значении.
            Когда: поиск, навигация (нужен только последний запрос)

mergeMap:   A──1──2──B──3──4──
            Результат: A:1, B:3, A:2, B:4 (параллельно)
            → Все внутренние потоки работают одновременно.
            Когда: параллельные независимые операции

concatMap:  A──1──2──B──3──4──
            Результат: A:1, A:2, B:3, B:4 (строго последовательно)
            → Ждёт завершения предыдущего перед запуском следующего.
            Когда: очередь операций, порядок важен
```

## Backpressure

Ситуация, когда производитель генерирует данные **быстрее**, чем потребитель может их обработать.

```text
Producer:  ██████████████████  (1000 событий/сек)
Consumer:  ██░░██░░██░░██░░██  (100 событий/сек)
           → 900 событий теряются или переполняют буфер
```

**Стратегии:**

- **Throttle / Debounce** — ограничить частоту (наиболее частый подход во фронтенде).
- **Buffer** — накопить значения и обработать пачкой.
- **Sample** — брать только последнее значение с заданным интервалом.
- **Drop** — отбрасывать, если потребитель занят.

```ts
// Пример: throttle скролла
fromEvent(window, 'scroll').pipe(
  throttleTime(100), // максимум 1 событие в 100мс
).subscribe(handleScroll);
```

## Реактивность во фреймворках

### Vue 3

Vue использует **Proxy-based** реактивность. При чтении свойства внутри `computed` или `watch` — автоматическая подписка (dependency tracking через `track`). При записи — уведомление зависимых (`trigger`).

```ts
import { ref, computed, watch, watchEffect } from 'vue';

const count = ref(0);
const doubled = computed(() => count.value * 2);

// watchEffect: автотрекинг зависимостей
watchEffect(() => {
  console.log(`Count: ${count.value}, Doubled: ${doubled.value}`);
});

// watch: явное наблюдение за конкретным источником
watch(count, (newVal, oldVal) => {
  console.log(`${oldVal} → ${newVal}`);
});

count.value++; // оба watcher'а сработают
```

<span class="text-gray">Под капотом: `ref()` оборачивает значение в объект с getter/setter. `reactive()` создаёт Proxy вокруг объекта. При первом запуске `computed` или `watchEffect` Vue запоминает, какие ref'ы были прочитаны (track), и при их изменении (trigger) — пересчитывает.</span>

### React

React **не является реактивным** в классическом смысле. Это **pull-based** система с UI = f(state):

```tsx
function Counter() {
  const [count, setCount] = useState(0);
  const doubled = useMemo(() => count * 2, [count]);

  useEffect(() => {
    document.title = `Count: ${count}`;
  }, [count]); // зависимости указываются вручную

  return <button onClick={() => setCount(c => c + 1)}>{doubled}</button>;
}
```

<span class="text-gray">`useState` — не сигнал. Вызов `setCount` ставит компонент в очередь на re-render. При re-render'е вся функция компонента выполняется заново, `useMemo` проверяет зависимости, `useEffect` запускается после paint. Это **scheduling + reconciliation**, а не push-based реактивность.</span>

### Solid.js

Solid реализует **true fine-grained reactivity** без Virtual DOM. Компонент выполняется **один раз** — только для настройки реактивного графа. Дальше обновления идут напрямую в DOM.

```tsx
function Counter() {
  const [count, setCount] = createSignal(0);
  const doubled = createMemo(() => count() * 2);

  createEffect(() => {
    document.title = `Count: ${count()}`;
  });

  // JSX компилируется в прямые DOM-операции, а не VDOM
  return <button onClick={() => setCount(c => c + 1)}>{doubled()}</button>;
}
```

### Angular (Signals + RxJS)

Angular исторически построен на RxJS, а с версии 16 добавляет Signals для синхронной реактивности:

```ts
// Signals (Angular 16+)
const count = signal(0);
const doubled = computed(() => count() * 2);

effect(() => {
  console.log('Count:', count());
});

count.set(5);
count.update(v => v + 1);
```

```ts
// RxJS — для асинхронных потоков (HTTP, WebSocket, routing)
this.http.get<User[]>('/api/users').pipe(
  map(users => users.filter(u => u.active)),
  catchError(() => of([])),
).subscribe(users => this.users = users);
```

### Сравнение реактивных моделей

| Фреймворк | Модель | Tracking | VDOM | Гранулярность |
|---|---|---|---|---|
| **Vue 3** | Push (Proxy) | Авто | Есть (но оптимизирован) | Fine-grained |
| **React** | Pull (re-render) | Ручная (`[deps]`) | Есть | Coarse (компонент) |
| **Solid** | Push (Signals) | Авто | Нет | Fine-grained |
| **Svelte 5** | Push ($state) | Авто (компиляция) | Нет | Fine-grained |
| **Angular** | Push (Signals + RxJS) | Авто / Ручная | Нет (с Signals) | Зависит от подхода |

## Распространённые паттерны

<b class="text-on-semi-light">Derived State (вычисляемое состояние)</b>

Не храни то, что можно вычислить. Если `fullName` — это `firstName + lastName`, сделай его `computed`, а не отдельным `ref`.

```ts
// Плохо
const firstName = ref('John');
const lastName = ref('Doe');
const fullName = ref('John Doe'); // дублирование, рассинхронизация

// Хорошо
const fullName = computed(() => `${firstName.value} ${lastName.value}`);
```

<b class="text-on-semi-light">Unsubscribe / Cleanup</b>

Подписки без отписки — источник утечек памяти. В компонентных фреймворках cleanup привязан к жизненному циклу.

```ts
// Vue
onMounted(() => {
  const subscription = stream$.subscribe(handler);
  onUnmounted(() => subscription.unsubscribe());
});

// React
useEffect(() => {
  const subscription = stream$.subscribe(handler);
  return () => subscription.unsubscribe(); // cleanup
}, []);
```

> Подробнее об утечках памяти из-за подписок — [Memory](/content/computer-science/memory).

<b class="text-on-semi-light">Batching (группировка обновлений)</b>

<span class="text-gray">Если изменяется несколько реактивных значений одновременно, реактивная система группирует обновления, чтобы зависимые вычисления и DOM-обновления произошли **один раз**, а не на каждое изменение.</span>

```ts
// Vue: обновления батчатся автоматически внутри одного tick
firstName.value = 'Jane';
lastName.value = 'Smith';
// computed fullName пересчитается один раз, не два
```

```tsx
// React 18+: автоматический batching в event handlers, promises, timeouts
setCount(1);
setFlag(true);
// один re-render, не два
```

<b class="text-on-semi-light">Glitch-free updates</b>

<span class="text-gray">«Глитч» — это ситуация, когда зависимое вычисление видит **частично обновлённое** состояние. Качественные реактивные системы (Vue, Solid) гарантируют, что computed-значение пересчитывается **только после обновления всех его зависимостей** — это называется topological sorting графа зависимостей.</span>

```text
Глитч:    A=1→2, B=A+1 (считает до обновления A) → неконсистентное состояние
Без глитча: A=1→2, потом B пересчитывается с новым A → консистентность
```
