# Введение в базы данных

**База данных (БД)** — это упорядоченная коллекция данных и механизмов для их хранения, поиска, обновления и защиты. Для инженера важно понимать не только синтаксис SQL или API конкретной СУБД, но и архитектурные принципы, модели данных, ограничения и паттерны масштабирования — чтобы строить надёжные, быстрые и безопасные приложения.

Здесь написан поверхностный обзор: модели данных, реляционная теория в кратком виде, NoSQL-альтернативы, транзакции и изоляция, индексы, оптимизация запросов, масштабирование (репликация, шардинг), бэкапы и восстановление, безопасность и практические рекомендации.

---

## 1. Типы баз данных и модели данных

### 1.1 Реляционные базы данных (RDBMS)

- Модель: таблицы (relations) со строками и столбцами; строгая схема (schema).
- Язык: SQL (Structured Query Language).
- Примеры: PostgreSQL, MySQL (InnoDB), MariaDB, Oracle, SQL Server.
- Подходящие задачи: транзакционный учёт (OLTP), строгая целостность данных, сложные JOIN-вычисления.

### 1.2 NoSQL базы данных — категории

- **Key–Value**: простейшая пара «ключ → значение». Очень быстрые. Примеры: Redis, DynamoDB.
- **Document (документные)**: хранят JSON-подобные документы; гибкая схема. Примеры: MongoDB, Couchbase.
- **Column-Family**: оптимизированы для хранения колонок и больших объёмов (wide rows). Примеры: Cassandra, HBase.
- **Graph**: для данных с богатой моделью связей. Примеры: Neo4j, JanusGraph.

### 1.3 NewSQL и распределённые транзакционные СУБД

- Комбинация горизонтального масштабирования и поддержки транзакций/SQL: CockroachDB, YugabyteDB, Google Spanner.

---

## 2. Реляционная модель — ключевые идеи (на высоком уровне)

- Таблица = множество записей (строк).
- Каждый столбец имеет тип и ограничения (NOT NULL, UNIQUE, FOREIGN KEY).
- Первичный ключ (PRIMARY KEY) — уникальный идентификатор строки.
- Внешние ключи (FOREIGN KEY) — механизмы поддержания целостности между таблицами.
- Нормализация — процесс уменьшения дублирования данных (1NF, 2NF, 3NF, BCNF).

---

## 3. SQL — базовые операции и примеры

- DDL (Data Definition Language): `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE`.
- DML (Data Manipulation Language): `INSERT`, `UPDATE`, `DELETE`, `SELECT`.
- DCL (Data Control Language): `GRANT`, `REVOKE`.
- Пример создания таблицы и простых запросов (Postgres/MySQL):

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  username VARCHAR(50) UNIQUE NOT NULL,
  email VARCHAR(255) NOT NULL,
  created_at TIMESTAMP DEFAULT now()
);

INSERT INTO users (username, email) VALUES ('alice', 'a@example.com');

SELECT id, username FROM users WHERE email LIKE '%@example.com';
```

### JOIN-ы — кратко

- `INNER JOIN` — только строки, у которых есть совпадение в обеих таблицах.
- `LEFT/RIGHT OUTER JOIN` — все строки левой (правой) таблицы + совпадающие.
- `FULL OUTER JOIN` — объединение обеих.

Пример:

```sql
SELECT u.username, o.total
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE o.total > 100;
```

## 4. Транзакции, ACID и уровни изоляции

### 4.1 Транзакция

- Транзакция — последовательность операций, которая выполняется атомарно: либо всё успешно, либо ничего не меняется.
- Команды: `BEGIN`/`START TRANSACTION`, `COMMIT`, `ROLLBACK`.

### 4.2 ACID

- **Atomicity** — атомарность.
- **Consistency** — согласованность (правила целостности сохраняются).
- **Isolation** — изоляция между параллельными транзакциями.
- **Durability** — стойкость: после `COMMIT` изменения сохраняются даже при сбое.

### 4.3 Уровни изоляции (SQL standard)

- **READ UNCOMMITTED** — могут происходить грязные чтения (dirty reads).
- **READ COMMITTED** — не видим неподтверждённые изменения, но возможны non-repeatable reads и phantom reads.
- **REPEATABLE READ** — повторяемые чтения; предотвращает non-repeatable reads, но может не защищать от phantom reads в некоторых СУБД.
- **SERIALIZABLE** — строго последовательное исполнение — самый сильный уровень.

Примеры аномалий:

- **Dirty read**: транзакция A читает данные, изменённые транзакцией B, которая потом откатывается.
- **Non-repeatable read**: между двумя чтениями транзакция видит разные значения, потому что другая транзакция изменила данные.
- **Phantom read**: повторный `SELECT` возвращает больше/меньше строк из-за вставки/удаления другой транзакцией.

## 5. Индексы и структуры хранения

### 5.1 Почему индексы

- Индексы ускоряют поиск, сортировку и фильтрацию, но замедляют `INSERT`/`UPDATE`/`DELETE` и занимают место.

### 5.2 Типы индексов

- **B‑tree** — широкораспространённый для диапазонных запросов и сортировки.
- **Hash index** — быстрое точное совпадение, плохо для диапазонов.
- **GiST / SP-GiST / GIN (Postgres)** — индексы для полей типа JSON, полнотекстового поиска, геоданных.
- **Bitmap index** — эффективны для OLAP/колоночных СУБД при низкой кардинальности.

### 5.3 Понятия

- **Кластерный индекс (clustered)** — физический порядок строк соответствует индексу (например, InnoDB кластеризует по первичному ключу).
- **Покрывающий индекс (covering index)** — индекс содержит все запрашиваемые столбцы, чтение без обращения к таблице.
- **Составные индексы** — порядок колонок имеет значение: индекс по (a,b) помогает WHERE a=.. AND b=.. и ORDER BY a,b.

### 5.4 Как правильно индексировать

- Индексировать колонки, которые участвуют в WHERE, JOIN, ORDER BY.
- Не индексировать колонки с очень низкой кардинальностью (например, boolean), если это не оправдано.
- Следить за индексной селективностью (selectivity).

## 6. Нормализация и денормализация

### 6.1 Нормализация

- 1NF: атомарные значения (нет повторяющихся групп).
- 2NF: каждая колонка зависит от всего составного ключа, а не от части.
- 3NF: нет транзитивных зависимостей; каждая неключевая колонка зависит только от ключа.
- BCNF: усиленная версия 3NF.

Цель: уменьшить дублирование и аномалии обновления.

### 6.2 Денормализация

- Иногда ради производительности (меньше JOIN-ов) данные сознательно дублируют.
- Применяется в OLAP, read-heavy системах, кэширующих слоях.
- Важно иметь механизмы согласования/обновления дублированных данных.

---

## 7. Оптимизация запросов и план выполнения

### 7.1 EXPLAIN / EXPLAIN ANALYZE

- `EXPLAIN` показывает план выполнения запроса: последовательность операций, какие индексы используются, оценки затрат.
- `EXPLAIN ANALYZE` выполняет запрос и показывает реальные времена.

### 7.2 Практические советы

- Избегать `SELECT *` в продакшне.
- Использовать LIMIT для больших выборок, страничную загрузку (pagination).
- Индексировать условия WHERE, JOIN и ORDER BY.
- Объединять мелкие транзакции в одну, если это допустимо.
- Убрать функции вокруг колонок в WHERE (они мешают использованию индекса).
- Профилировать медленные запросы и смотреть план выполнения.

## 8. Масштабирование: репликация и шардинг

### 8.1 Репликация

- **Master–Slave (Primary–Replica)**: одна запись (master), множество реплик для чтения.
  - Преимущества: масштаб читающих нагрузок, резервирование.
  - Минусы: задержка репликации (replication lag), возможная потеря данных при failover.

- **Master–Master (Multi-Primary)**: записи возможны на нескольких нодах — сложнее поддерживать согласованность.
- **Streaming replication (Postgres)**, Binlog (MySQL).

### 8.2 Шардирование (Sharding)

- Горизонтальное разбиение таблицы на несколько нод по ключу (shard key).
- Примеры шардирования: по user_id % N, по диапазонам дат.
- Сложности: выбор ключа, перекоманды при росте, межшардовые JOIN-ы.

### 8.3 Схемы репликации и согласованности

- Асинхронная репликация: высокая производительность, возможна задержка.
- Синхронная репликация: гарантирует, что данные на реплике подтверждены, но медленнее.

---

## 9. CAP theorem и распределённые системы

- **CAP**: Consistency, Availability, Partition tolerance — можно обеспечить только два из трёх в условиях сетевых разделений.
- На практике системы выбирают между сильной согласованностью (CP) и высокой доступностью/разрешёнными конфликтами (AP). Например:
  - CP: HBase, MongoDB (в некоторых конфигурациях);
  - AP: Cassandra, DynamoDB (в некоторых конфигурациях).

---

## 10. Хранилища и движки

- **InnoDB** (MySQL): транзакционный, поддерживает MVCC, кластеризованный индекс.
- **MyISAM** (MySQL): без транзакций — устаревший.
- **Postgres storage**: MVCC, расширяемость индексов.
- **RocksDB / LevelDB**: LSM-tree на основе key-value, часто используется в распределённых системах.

Понимание движка важно для тонкой оптимизации, отката, политик хранения и поведения при сбоях.

---

## 11. Резервное копирование и восстановление (Backups & Recovery)

- **Logical backups**: pg_dump, mysqldump — удобны для переносов, медленнее.
- **Physical backups / Snapshots**: копирование файлов данных / snapshot хранилища, быстрее восстановление.
- **PITR (Point-in-Time Recovery)**: журнал транзакций (WAL), позволяет откат к моменту времени.
- Регулярно тестировать восстановление — очень важный пункт.

---

## 12. Безопасность и доступ

- Принцип наименьших привилегий (least privilege).
- Хранение паролей, шифрование данных в покое (at rest) и в канале (TLS).
- Аудит, логирование доступа.
- Защита от SQL-инъекций: параметризованные запросы / prepared statements.

## 13. NoSQL: практические примеры и паттерны

### 13.1 Document store (MongoDB) — примеры

```js
// вставка в MongoDB
db.users.insertOne({
  username: "alice",
  email: "a@example.com",
  tags: ["admin", "beta"],
});

// запрос
db.users.find({ tags: "admin" });
```

Паттерны: вложенные документы, массивы, обновление частичных документов (`$set`, `$push`).

### 13.2 Key-Value (Redis) — примеры

```bash
SET session:abc123 "{...json...}"
GET session:abc123
LPUSH myqueue "task1"
BRPOP myqueue 0
```

### 13.3 Column-Family (Cassandra) — особенности

- Хороша для write-heavy и линейного масштабирования.
- Дизайн ориентирован на чтение по заранее известным паттернам.

### 13.4 Graph DB (Neo4j) — пример

```cypher
CREATE (a:User {name: 'Alice'})-[:FRIENDS_WITH]->(b:User {name: 'Bob'});
MATCH (u:User)-[:FRIENDS_WITH]->(v:User) WHERE u.name='Alice' RETURN v;
```

## 14. Транзакции в распределённых системах

- **Two‑phase commit (2PC)**: координатор собирает подтверждения от участников и затем либо коммитит, либо откатывает.
- **Three‑phase commit (3PC)**: попытка избежать блокировок 2PC, сложнее и реже используется.
- Современные подходы: использование Consensus протоколов (Paxos, Raft) и распределённых транзакций на уровне кооперации.
- **Eventual consistency** и компенсационные транзакции как альтернатива строгим распределённым транзакциям.

---

## 15. ORMs, миграции и операционная практика

- ORM (Object-Relational Mapping): удобство разработки (ActiveRecord, SQLAlchemy, Hibernate), но возможные потери в производительности и непредвиденные запросы (N+1 problem).
- Миграции схемы (Flyway, Liquibase, Django/Migrations) — контроль версий схемы.
- Connection pooling (pgBouncer, HikariCP) — важен для пропускной способности.

---

## 16. Мониторинг, измерения и SLO

- Мониторить: latency, QPS, cache hit ratio, replication lag, open connections.
- Инструменты: Prometheus + Grafana, ELK/EFK (logging), pg_stat_statements.
- Определять SLO/SLI/ SLA: как быстро и как часто система должна отвечать.

---

## 17. Типичные ошибки и антипаттерны

- Недостаточное тестирование восстановления из бэкапа.
- Выбор неверной модели данных (например, использование RDBMS там, где лучше документный store, и наоборот).
- Индексация «всё подряд» — много памяти и медленные вставки.
- Неправильный шард-ключ — горячие шарды.
- Полагаться на утилиты ОС для бэкапов без согласованности (crash-consistent vs application-consistent snapshots).

---

## 18. Практические рекомендации

1. Начинайте с нормализованной схемы для OLTP; профилируйте и денормализуйте целенаправленно.
2. Индексируйте только по необходимости и следите за размером индексов.
3. Используйте транзакции для критичных операций; избегайте долгих транзакций.
4. Разделяйте чтение и запись через репликацию, если это допустимо.
5. Тестируйте бэкап и восстановление регулярно.
6. Следите за метриками и логами.
