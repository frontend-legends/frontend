# Введение в базы данных

**База данных (БД)** — это упорядоченная коллекция данных и механизмов для их хранения, поиска, обновления и защиты. Даже если ты работаешь исключительно на фронтенде, знание баз данных помогает: осознанно проектировать API, понимать почему запрос «тормозит», общаться с бэкенд-командой на одном языке и проходить system design интервью. А если ты fullstack — это must-have.

Здесь разбираем модели данных, реляционную теорию, NoSQL-альтернативы, транзакции и изоляцию, индексы, оптимизацию, масштабирование и практические аспекты для веб-разработки.

---

## Типы баз данных

### Реляционные (RDBMS)

<span class="text-gray">Модель: таблицы (relations) со строками и столбцами, строгая схема. Язык запросов — SQL. Основа для большинства бизнес-приложений с транзакционными требованиями.</span>

Примеры: **PostgreSQL**, **MySQL** (InnoDB), **MariaDB**, **SQLite**.

### NoSQL

Семейство систем, отказавшихся от строгой реляционной модели в пользу масштабируемости, гибкости или специфичных паттернов доступа.

| Категория | Модель данных | Сильные стороны | Примеры |
|---|---|---|---|
| **Key-Value** | Ключ → значение (blob) | Минимальная задержка, простота | Redis, DynamoDB |
| **Document** | JSON/BSON-документы | Гибкая схема, вложенные структуры | MongoDB, Couchbase |
| **Column-Family** | Wide rows с колонками | Write-heavy, линейное масштабирование | Cassandra, HBase |
| **Graph** | Узлы + рёбра + свойства | Глубокие связи, traversal | Neo4j, JanusGraph |

### NewSQL

<span class="text-gray">Комбинация горизонтального масштабирования и полноценных ACID-транзакций + SQL: **CockroachDB**, **YugabyteDB**, **Google Spanner**. Появились как ответ на ограничения NoSQL, когда бизнес-логика требует и масштаба, и строгой согласованности.</span>

## Обзор популярных баз данных

### PostgreSQL

<b class="text-on-semi-light">Тип:</b> Реляционная (RDBMS), объектно-реляционная.

<span class="text-gray">Самая продвинутая open-source реляционная БД. Де-факто стандарт для production-приложений. Поддерживает JSONB, полнотекстовый поиск, GIS (PostGIS), расширения, мощный оптимизатор запросов.</span>

| Преимущества | Недостатки |
|---|---|
| Полный ACID, строгие типы, constraints | Сложнее в настройке, чем MySQL/SQLite |
| JSONB — документная работа внутри SQL | Vertically scales (горизонтальное — через расширения: Citus, Supabase) |
| Расширения: PostGIS, pg_trgm, pgvector (AI embeddings) | Более высокое потребление памяти на idle |
| MVCC — отличная конкурентность чтения | Write-heavy нагрузка может требовать тюнинга |
| Мощный `EXPLAIN ANALYZE`, оконные функции, CTE, lateral joins | |

<b class="text-on-semi-light">Используется для:</b> SaaS-приложения, API-бэкенды, финтех (транзакции), геоданные (PostGIS), AI/ML (pgvector для embeddings), fullstack (Supabase, Neon).

### MySQL

<b class="text-on-semi-light">Тип:</b> Реляционная (RDBMS).

<span class="text-gray">Одна из самых распространённых БД в мире. Движок InnoDB — ACID-совместимый. Простая в развёртывании, огромная экосистема. Исторически — выбор для PHP/WordPress стека.</span>

| Преимущества | Недостатки |
|---|---|
| Простота установки и настройки | Менее мощный SQL (нет lateral join, ограниченные CTE до 8.0) |
| Широкая поддержка хостингов | JSONB менее развит, чем в Postgres |
| Быстрые простые запросы (SELECT по PK) | GROUP BY с неагрегированными столбцами — неожиданное поведение |
| Репликация из коробки | Слабее оконные функции и подзапросы |
| Огромное комьюнити, совместимость с MariaDB | |

<b class="text-on-semi-light">Используется для:</b> CMS (WordPress, Drupal), e-commerce, legacy-приложения, простые CRUD API, read-heavy приложения.

### SQLite

<b class="text-on-semi-light">Тип:</b> Встроенная реляционная (embedded RDBMS).

<span class="text-gray">Файловая БД без сервера — вся база в одном `.db` файле. Не требует установки, процесс приложения напрямую читает/пишет файл. Идеальна для локального хранения, прототипов и edge-приложений.</span>

| Преимущества | Недостатки |
|---|---|
| Zero-config, нет сервера | Не подходит для concurrent writes (один writer) |
| Один файл — легко копировать, бэкапить | Нет сетевого доступа (embedded only) |
| Очень быстрая для read-heavy | Нет user management / permissions |
| Используется в мобильных и desktop приложениях | Ограниченные типы данных |
| Идеальна для тестов и прототипов | |

<b class="text-on-semi-light">Используется для:</b> Мобильные приложения (Android, iOS), Electron apps, embedded systems, @nuxt/content и другие SSG/SSR frameworks (build-time данные), локальный кэш, тесты.

### MongoDB

<b class="text-on-semi-light">Тип:</b> Document store (NoSQL).

<span class="text-gray">Хранит данные как JSON/BSON-документы. Гибкая схема — каждый документ может иметь разную структуру. Мощный query language, aggregation pipeline. Горизонтально масштабируется через шардирование.</span>

| Преимущества | Недостатки |
|---|---|
| Гибкая схема — быстрый старт, быстрые итерации | Нет JOIN'ов (есть `$lookup`, но медленнее SQL JOIN) |
| Вложенные документы — одним запросом получаешь всё | Денормализация → дублирование данных |
| Горизонтальное масштабирование (sharding) из коробки | Транзакции появились в 4.0, но менее зрелые, чем в RDBMS |
| Мощный aggregation pipeline | Потребляет больше места (BSON overhead) |
| Atlas — managed cloud с search, analytics, triggers | Schema validation опционален — легко «загрязнить» данные |

<b class="text-on-semi-light">Используется для:</b> CMS, каталоги с переменной структурой, IoT, real-time analytics, мобильные бэкенды, прототипы (быстрый старт без миграций).

### Redis

<b class="text-on-semi-light">Тип:</b> Key-Value (in-memory NoSQL).

<span class="text-gray">In-memory хранилище данных. Всё в оперативной памяти → наносекундные задержки. Поддерживает разные структуры: строки, списки, множества, sorted sets, hashes, streams. Часто используется как кэш-слой перед основной БД.</span>

| Преимущества | Недостатки |
|---|---|
| Сверхбыстрый (in-memory, ~100μs latency) | Данные ограничены объёмом RAM |
| Множество структур данных (не только key-value) | Persistence опционален — при крэше можно потерять данные |
| TTL — автоматическое истечение ключей | Не подходит как primary database для сложных запросов |
| Pub/Sub, Streams — real-time messaging | Без persistence — волатильное хранилище |
| Lua scripting для atomic операций | Single-threaded (I/O multiplexing) |

<b class="text-on-semi-light">Используется для:</b> Кэширование (API responses, DB query cache), сессии, rate limiting, лидерборды (sorted sets), очереди задач, pub/sub messaging, distributed locks.

### Elasticsearch

<b class="text-on-semi-light">Тип:</b> Search engine (document store с инвертированным индексом).

<span class="text-gray">Распределённый поисковый движок на базе Apache Lucene. Хранит JSON-документы и строит инвертированные индексы для быстрого полнотекстового поиска. Часто используется как дополнение к основной БД.</span>

| Преимущества | Недостатки |
|---|---|
| Быстрый полнотекстовый поиск, fuzzy matching | Не ACID, не primary database |
| Автокомплит, подсветка результатов | Высокое потребление ресурсов (CPU, RAM, disk) |
| Агрегации и аналитика в реальном времени | Сложность администрирования кластера |
| Горизонтальное масштабирование | Данные нужно синхронизировать с основной БД |
| Kibana — визуализация логов и метрик | Не подходит для частых обновлений |

<b class="text-on-semi-light">Используется для:</b> Поиск по каталогу/сайту, логирование (ELK stack: Elasticsearch + Logstash + Kibana), мониторинг, аналитика, автокомплит.

### Cassandra

<b class="text-on-semi-light">Тип:</b> Column-Family (wide-column NoSQL).

<span class="text-gray">Распределённая БД, спроектированная для огромных объёмов записи. Нет single point of failure — все ноды равноправны (peer-to-peer). Линейное масштабирование: добавляешь ноды → пропускная способность растёт пропорционально.</span>

| Преимущества | Недостатки |
|---|---|
| Линейное горизонтальное масштабирование | Нет JOIN'ов, ограниченные запросы (query-driven model) |
| Нет single point of failure | Eventual consistency по умолчанию |
| Оптимизирована для write-heavy нагрузок | Сложная модель данных (нужно проектировать под запросы) |
| Multi-datacenter репликация | Нет транзакций (lightweight transactions ограничены) |
| Высокая доступность (AP в CAP) | Компакция и ремонт требуют ресурсов |

<b class="text-on-semi-light">Используется для:</b> Логи, метрики, IoT-данные, messaging (Discord использует Cassandra), временные ряды, системы с огромным write throughput.

### Neo4j

<b class="text-on-semi-light">Тип:</b> Graph database.

<span class="text-gray">Хранит данные как графы: узлы (nodes), рёбра (relationships) и свойства (properties). Обход связей за O(k) (k = количество связей), а не O(n) как JOIN в RDBMS. Язык запросов — Cypher.</span>

| Преимущества | Недостатки |
|---|---|
| Traversal связей за O(k), не O(n) | Не подходит для табличных данных и агрегаций |
| Интуитивная визуализация связей | Масштабирование сложнее, чем у document stores |
| Cypher — выразительный язык для графов | Меньшее комьюнити, чем у RDBMS/MongoDB |
| Pattern matching для рекомендаций | Высокое потребление памяти для больших графов |

<b class="text-on-semi-light">Используется для:</b> Социальные сети (друзья друзей), рекомендательные системы, fraud detection, knowledge graphs, dependency analysis, network topology.

### ClickHouse

<b class="text-on-semi-light">Тип:</b> Column-oriented OLAP database.

<span class="text-gray">Колоночная аналитическая БД от Яндекса. Молниеносные агрегации на миллиардах строк. Не для OLTP (отдельные INSERT/UPDATE), а для аналитических запросов.</span>

| Преимущества | Недостатки |
|---|---|
| Агрегации на миллиардах строк за секунды | Не подходит для point lookups (WHERE id = ?) |
| Колоночное хранение — сжатие до 10x | UPDATE/DELETE дорогие (append-only модель) |
| SQL-совместимый синтаксис | Нет полноценных транзакций |
| Линейное масштабирование | Не для OLTP-нагрузок |

<b class="text-on-semi-light">Используется для:</b> Аналитика, дашборды, логи и метрики, A/B тестирование, BI-системы, мониторинг (Grafana + ClickHouse).

### Что выбрать

| Задача | Подходящая БД | Почему |
|---|---|---|
| Пользователи, заказы, транзакции | PostgreSQL, MySQL | ACID, JOIN'ы, строгая схема |
| Сессии, кэш, очереди | Redis | Скорость, TTL, pub/sub |
| Контент с гибкой структурой (CMS, каталоги) | MongoDB | Гибкая схема, вложенные документы |
| Социальные связи, рекомендации | Neo4j | Traversal графа за O(k), не O(n) |
| Логи, метрики, IoT | Cassandra, ClickHouse | Write-heavy, append-only |
| Полнотекстовый поиск | Elasticsearch | Инвертированный индекс, fuzzy search |

## Реляционная модель

<b class="text-on-semi-light">Таблица (Relation)</b>

<span class="text-gray">Множество строк (записей) с фиксированным набором столбцов. Каждый столбец имеет тип данных и может иметь ограничения.</span>

<b class="text-on-semi-light">Primary Key (PK)</b>

<span class="text-gray">Уникальный идентификатор строки. Может быть числовым (auto-increment) или UUID. На практике: `SERIAL` / `BIGSERIAL` в Postgres, `AUTO_INCREMENT` в MySQL, или `uuid` / `cuid` сгенерированный приложением.</span>

<b class="text-on-semi-light">Foreign Key (FK)</b>

<span class="text-gray">Ссылка на PK другой таблицы. Обеспечивает **ссылочную целостность**: нельзя вставить запись, ссылающуюся на несуществующий ID, и нельзя удалить запись, на которую кто-то ссылается (без `CASCADE`).</span>

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  username VARCHAR(50) UNIQUE NOT NULL,
  email VARCHAR(255) NOT NULL,
  created_at TIMESTAMP DEFAULT now()
);

CREATE TABLE orders (
  id SERIAL PRIMARY KEY,
  user_id INT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  total DECIMAL(10,2) NOT NULL,
  created_at TIMESTAMP DEFAULT now()
);
```

### Нормализация

Процесс организации данных для уменьшения дублирования и аномалий обновления.

| Форма | Правило | Пример нарушения |
|---|---|---|
| **1NF** | Атомарные значения, нет повторяющихся групп | `tags: "js,ts,vue"` в одном поле |
| **2NF** | Каждая колонка зависит от **всего** ключа | Составной PK (order_id, product_id), а `product_name` зависит только от product_id |
| **3NF** | Нет транзитивных зависимостей | `city_name` зависит от `zip_code`, а не от PK |
| **BCNF** | Каждый детерминант — суперключ | Усиленная 3NF для edge cases |

<b class="text-on-semi-light">Денормализация</b>

<span class="text-gray">Иногда ради скорости чтения (меньше JOIN'ов) данные сознательно дублируют. Применяется в read-heavy системах, аналитике (OLAP), кэширующих слоях. Важно иметь механизм согласования дублированных данных.</span>

## SQL — основные операции

### DDL, DML, DCL

```sql
-- DDL (Data Definition Language)
CREATE TABLE products (id SERIAL PRIMARY KEY, name TEXT NOT NULL, price DECIMAL(10,2));
ALTER TABLE products ADD COLUMN stock INT DEFAULT 0;
DROP TABLE IF EXISTS products;

-- DML (Data Manipulation Language)
INSERT INTO users (username, email) VALUES ('alice', 'a@example.com');
UPDATE users SET email = 'new@example.com' WHERE id = 1;
DELETE FROM users WHERE id = 1;
SELECT id, username FROM users WHERE email LIKE '%@example.com';

-- DCL (Data Control Language)
GRANT SELECT ON users TO readonly_role;
REVOKE DELETE ON users FROM app_role;
```

### JOIN'ы

| Тип | Что возвращает | Когда использовать |
|---|---|---|
| `INNER JOIN` | Только строки, у которых есть совпадение в **обеих** таблицах | Нужны данные, существующие в обеих таблицах |
| `LEFT JOIN` | **Все** строки из левой таблицы + совпадающие из правой (`NULL` если нет) | «Все пользователи, даже без заказов» |
| `RIGHT JOIN` | **Все** строки из правой таблицы + совпадающие из левой | Используется редко — обычно переписывают через LEFT JOIN |
| `FULL OUTER JOIN` | **Все** строки из обеих таблиц, `NULL` где нет совпадений | Сравнение двух наборов данных, поиск расхождений |

```sql
-- INNER JOIN: пользователи с заказами
SELECT u.username, o.total
FROM users u
JOIN orders o ON o.user_id = u.id;

-- LEFT JOIN: все пользователи, даже без заказов
SELECT u.username, COALESCE(SUM(o.total), 0) AS total_spent
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.username;

-- Подзапрос: пользователи с суммой заказов выше средней
SELECT username FROM users
WHERE id IN (
  SELECT user_id FROM orders
  GROUP BY user_id
  HAVING SUM(total) > (SELECT AVG(total) FROM orders)
);
```

## Транзакции и ACID

### Транзакция

Последовательность операций, выполняемых **атомарно**: либо все применяются, либо ни одна.

```sql
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
-- Если что-то пошло не так: ROLLBACK;
```

### ACID

| Свойство | Что гарантирует | Пример |
|---|---|---|
| **Atomicity** | Всё или ничего | Перевод денег: списание + зачисление — неделимо |
| **Consistency** | Данные всегда валидны | FK не может ссылаться на удалённую запись |
| **Isolation** | Параллельные транзакции не мешают | Два перевода одновременно не создадут «фантомных» денег |
| **Durability** | После COMMIT данные сохранены | Даже если сервер упадёт через 1мс после коммита |

### Уровни изоляции

<span class="text-gray">Чем строже изоляция — тем меньше аномалий, но больше блокировок и ниже пропускная способность.</span>

| Уровень | Dirty Read | Non-repeatable Read | Phantom Read |
|---|---|---|---|
| **READ UNCOMMITTED** | Возможен | Возможен | Возможен |
| **READ COMMITTED** | Нет | Возможен | Возможен |
| **REPEATABLE READ** | Нет | Нет | Возможен* |
| **SERIALIZABLE** | Нет | Нет | Нет |

<span class="text-gray">*В MySQL InnoDB REPEATABLE READ защищает и от phantom reads через gap locks. В PostgreSQL по умолчанию READ COMMITTED, но REPEATABLE READ реализован через MVCC (Multiversion Concurrency Control) — каждая транзакция видит snapshot данных на момент своего начала.</span>

<b class="text-on-semi-light">Аномалии — что это</b>

- **Dirty read**: читаем данные, изменённые другой транзакцией, которая потом откатится.
- **Non-repeatable read**: между двумя чтениями одной строки другая транзакция изменила значение.
- **Phantom read**: между двумя одинаковыми SELECT'ами появились/исчезли строки из-за INSERT/DELETE другой транзакции.

## Индексы

Индекс — дополнительная структура данных, ускоряющая поиск. Аналогия: предметный указатель в книге — вместо перелистывания 500 страниц, сразу переходишь к нужной.

### Типы индексов

<b class="text-on-semi-light">B-tree</b>

<span class="text-gray">Дефолтный индекс в большинстве СУБД. Сбалансированное дерево, эффективен для: точного поиска (`=`), диапазонов (`<`, `>`, `BETWEEN`), сортировки (`ORDER BY`), prefix-поиска (`LIKE 'abc%'`). Не эффективен для: `LIKE '%abc'` (нет prefix), полнотекстового поиска.</span>

<b class="text-on-semi-light">Hash index</b>

<span class="text-gray">O(1) для точного совпадения (`=`). Не поддерживает диапазоны и сортировку. Используется в Redis, PostgreSQL (ручное создание).</span>

<b class="text-on-semi-light">GIN (Generalized Inverted Index)</b>

<span class="text-gray">Postgres-специфичный. Эффективен для: JSONB-полей, полнотекстового поиска (`tsvector`), массивов. Каждое значение внутри документа/массива индексируется отдельно.</span>

### Важные концепции

| Концепция | Что значит |
|---|---|
| **Clustered index** | Физический порядок строк = порядок индекса. В InnoDB — всегда по PK |
| **Covering index** | Индекс содержит все нужные столбцы → чтение без обращения к таблице |
| **Composite index** | Индекс по (a, b, c) — порядок колонок важен (leftmost prefix rule) |
| **Selectivity** | Доля уникальных значений. Высокая (email) — хороший кандидат. Низкая (boolean) — обычно нет |

```sql
-- Составной индекс: помогает для WHERE user_id = ? AND status = ?
-- и для WHERE user_id = ?, но НЕ для WHERE status = ? (нет leftmost prefix)
CREATE INDEX idx_orders_user_status ON orders (user_id, status);

-- Покрывающий индекс: Postgres не обращается к таблице
CREATE INDEX idx_users_email_name ON users (email) INCLUDE (username);
```

### EXPLAIN — план выполнения

```sql
EXPLAIN ANALYZE
SELECT u.username, COUNT(o.id)
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.created_at > '2025-01-01'
GROUP BY u.id;
```

<span class="text-gray">Ключевые показатели: **Seq Scan** (полный перебор — обычно плохо для больших таблиц), **Index Scan** (использует индекс — хорошо), **Nested Loop** vs **Hash Join** (способ JOIN'а), **actual time** (реальное время в мс). Если видите Seq Scan на большой таблице с фильтром — скорее всего нужен индекс.</span>

### Когда индексы вредят

- Каждый индекс замедляет `INSERT`/`UPDATE`/`DELETE` — данные нужно обновить и в индексе.
- Индексы занимают место на диске и в памяти.
- Функция вокруг столбца убивает индекс: `WHERE LOWER(email) = 'a@b.com'` — Seq Scan. Решение: functional index `CREATE INDEX ON users (LOWER(email))`.

## CAP-теорема

В распределённой системе при сетевом разделении (Partition) невозможно гарантировать одновременно **Consistency** и **Availability** — приходится выбирать.

<span class="text-gray">**Consistency** — все ноды видят одинаковые данные в один момент. **Availability** — каждый запрос получает ответ (не ошибку). **Partition Tolerance** — система продолжает работать при разрыве сети между нодами. На практике partition tolerance обязателен (сети ненадёжны), поэтому реальный выбор всегда между CP и AP.</span>

| Выбор | Поведение при network split | Примеры |
|---|---|---|
| **CP** | Отклоняет запросы, пока не восстановит согласованность | HBase, MongoDB (majority write concern) |
| **AP** | Отвечает, но данные могут быть устаревшими (eventual consistency) | Cassandra, DynamoDB |
| **CA** | Невозможно в реальных сетях (нет partition tolerance) | Только single-node системы |

<span class="text-gray">Многие системы позволяют настраивать уровень согласованности: MongoDB можно настроить как CP (`w: majority`, `readConcern: majority`) или AP (`w: 1`). Cassandra позволяет выбирать consistency level на каждый запрос (`QUORUM`, `ONE`, `ALL`).</span>

## Масштабирование

### Репликация

<b class="text-on-semi-light">Primary-Replica (Master-Slave)</b>

<span class="text-gray">Одна нода принимает записи (primary), остальные — реплики только для чтения. Все записи идут на primary, который асинхронно (или синхронно) передаёт изменения на реплики. Клиенты читают с реплик, снижая нагрузку на primary.</span>

**Преимущества:** масштабирование read-нагрузки (добавить реплик), отказоустойчивость (если primary упал — promote реплику).
**Проблемы:** replication lag (реплика отстаёт — клиент может прочитать устаревшие данные), при failover возможна потеря последних записей, если реплика не успела их получить.

<b class="text-on-semi-light">Multi-Primary (Master-Master)</b>

<span class="text-gray">Записи на нескольких нодах. Сложнее: конфликты при одновременной записи одних данных. Используется, когда нужна запись в нескольких регионах.</span>

### Шардирование (Sharding)

<span class="text-gray">Горизонтальное разбиение данных между нодами по ключу (shard key). Каждый шард хранит подмножество строк. Например, при `shard_key = user_id % 3`: пользователи 0, 3, 6, 9... идут на шард 0; пользователи 1, 4, 7, 10... — на шард 1; и так далее. Каждый шард — самостоятельная база, которая обрабатывает только свои данные.</span>

| Стратегия шардирования | Как работает | Когда подходит |
|---|---|---|
| **Hash-based** (`id % N`) | Равномерное распределение по хэшу ключа | Равномерная нагрузка, нет range-запросов |
| **Range-based** (по дате, ID) | Данные из одного диапазона на одном шарде | Временные ряды, логи, партиционирование |
| **Directory-based** | Lookup-таблица определяет шард | Гибкость, но single point of failure |

**Сложности:**
- Выбор shard key: плохой ключ → «горячие шарды» (один шард получает непропорционально много запросов).
- Cross-shard JOIN'ы и транзакции — дорогие или невозможные.
- Ребалансировка при добавлении шардов.

## NoSQL: паттерны и примеры

### MongoDB (Document store)

```js
// Гибкая схема — каждый документ может иметь разные поля
db.users.insertOne({
  username: 'alice',
  email: 'a@example.com',
  profile: { bio: 'Frontend dev', links: ['github.com/alice'] },
  tags: ['admin', 'beta'],
});

// Запрос по вложенному полю
db.users.find({ 'profile.bio': /frontend/i });

// Обновление: $set меняет поле, $push добавляет в массив
db.users.updateOne(
  { username: 'alice' },
  { $set: { 'profile.bio': 'Fullstack dev' }, $push: { tags: 'v2' } }
);
```

### Redis (Key-Value)

```bash
# Строки с TTL (идеально для сессий и кэша)
SET session:abc123 '{"userId":42}' EX 3600   # истечёт через 1 час
GET session:abc123

# Pub/Sub (реальное время)
SUBSCRIBE notifications
PUBLISH notifications '{"type":"new_order","id":123}'

# Sorted Set (лидерборд)
ZADD leaderboard 1500 "alice"
ZADD leaderboard 2100 "bob"
ZREVRANGE leaderboard 0 9 WITHSCORES   # топ-10
```

### Neo4j (Graph)

```cypher
// Создать связь
CREATE (a:User {name: 'Alice'})-[:FRIENDS_WITH]->(b:User {name: 'Bob'});

// Найти друзей друзей (2 шага)
MATCH (u:User {name: 'Alice'})-[:FRIENDS_WITH*2]->(fof:User)
WHERE fof.name <> 'Alice'
RETURN DISTINCT fof.name;
```

## ORM и работа с БД из JavaScript/TypeScript

На фронтенде и в fullstack-разработке на JS/TS взаимодействие с БД происходит через ORM или query builder'ы.

### Популярные инструменты

| Инструмент | Тип | БД | Особенности |
|---|---|---|---|
| **Prisma** | ORM (schema-first) | Postgres, MySQL, SQLite, MongoDB | Генерация типов из схемы, миграции |
| **Drizzle** | Query builder (TS-first) | Postgres, MySQL, SQLite | SQL-like API, полный type-safety |
| **Kysely** | Query builder | Postgres, MySQL, SQLite | Типизированный SQL, без кодогенерации |
| **TypeORM** | ORM (class-based) | Postgres, MySQL, SQLite | Декораторы, Active Record/Data Mapper |
| **Mongoose** | ODM | MongoDB | Схемы, валидация, middleware |

### N+1 Problem

Классическая проблема ORM: для вывода списка пользователей с их заказами ORM может сделать **1 запрос на пользователей + N запросов на заказы** вместо одного JOIN'а.

```ts
// Плохо: N+1 (Prisma без include)
const users = await prisma.user.findMany();
for (const user of users) {
  const orders = await prisma.order.findMany({ where: { userId: user.id } });
}
// 1 + N запросов к БД

// Хорошо: eager loading (1-2 запроса)
const users = await prisma.user.findMany({
  include: { orders: true },
});
```

## Безопасность

<b class="text-on-semi-light">SQL-инъекции</b>

<span class="text-gray">Если пользовательский ввод подставляется в SQL без экранирования — атакующий может выполнить произвольный запрос.</span>

```js
// УЯЗВИМО: конкатенация строк
const query = `SELECT * FROM users WHERE username = '${input}'`;
// input: ' OR '1'='1
// → SELECT * FROM users WHERE username = '' OR '1'='1' — все пользователи

// БЕЗОПАСНО: параметризованный запрос
const result = await db.query(
  'SELECT * FROM users WHERE username = $1',
  [input]
);
```

<b class="text-on-semi-light">Принцип наименьших привилегий</b>

<span class="text-gray">Приложение должно подключаться к БД с минимально необходимыми правами. Веб-приложение не должно иметь `DROP TABLE` или `GRANT` — только `SELECT`, `INSERT`, `UPDATE`, `DELETE` на нужных таблицах.</span>

<b class="text-on-semi-light">Шифрование</b>

<span class="text-gray">Данные в транзите — TLS-соединение с БД. Данные в покое — шифрование на уровне диска или отдельных столбцов. Подробнее — [Шифрование](/content/computer-science/encryption).</span>

## Распределённые транзакции

<b class="text-on-semi-light">Two-Phase Commit (2PC)</b>

<span class="text-gray">Координатор спрашивает всех участников «готовы?» (prepare phase). Если все ответили «да» — commit. Если хоть один «нет» — rollback. Проблема: если координатор упадёт между фазами — участники заблокированы.</span>

<b class="text-on-semi-light">Saga Pattern</b>

<span class="text-gray">Альтернатива 2PC для микросервисов. Длинная транзакция разбивается на серию локальных транзакций. Если одна фейлится — выполняются компенсирующие действия (rollback каждого предыдущего шага). Используется в event-driven архитектурах.</span>

<b class="text-on-semi-light">Eventual Consistency</b>

<span class="text-gray">Система не гарантирует мгновенную согласованность, но все реплики **в конечном итоге** придут к одному состоянию. Подходит для систем, где допустима задержка (ленты соцсетей, счётчики лайков, рекомендации).</span>

## БД на фронте

Фронтенд-разработчик напрямую не пишет SQL (обычно), но работает с данными, которые приходят из БД. Знание баз данных помогает:

- **Проектировать API** — понимать, почему endpoint возвращает nested объект (денормализация) или требует 2 запроса (нормализация)
- **Оптимизировать запросы** — знать, что `GET /users?include=orders` может быть N+1 на бэкенде
- **Client-side storage** — IndexedDB (мини-БД в браузере), localStorage, sessionStorage, Cache API
- **Fullstack** — Prisma/Drizzle + Next.js/Nuxt = фронтендер работает с БД напрямую
- **System Design** — на senior-собесе спрашивают: «как бы ты спроектировал X?» — нужно выбрать подходящую БД

### Браузерные базы данных

| Хранилище | Тип | Лимит | Когда использовать |
|---|---|---|---|
| `localStorage` | Key-value (синхронный) | ~5–10 MB | Настройки, тема, язык |
| `sessionStorage` | Key-value (синхронный) | ~5–10 MB | Данные на время сессии |
| `IndexedDB` | Document store (асинхронный) | Сотни MB+ | Offline data, кэш API-ответов, файлы |
| `Cache API` | Request → Response | Сотни MB+ | Service Worker кэш для PWA |

<span class="text-gray">`localStorage` блокирует main thread при записи — для больших объёмов данных используй IndexedDB. Библиотеки: **idb** (обёртка над IndexedDB), **Dexie.js** (IndexedDB с query API).</span>

## Частые вопросы на собесе

**SQL vs NoSQL — когда что?**
SQL (PostgreSQL, MySQL) — когда нужны JOIN'ы, строгая схема, транзакции (финансы, заказы, пользователи). NoSQL — когда гибкая схема (MongoDB для каталогов), скорость (Redis для кэша), масштаб записи (Cassandra для логов), связи (Neo4j для графов).

**Что такое ACID?**
Atomicity (всё или ничего), Consistency (данные валидны), Isolation (транзакции не мешают друг другу), Durability (данные сохранены после commit). Гарантируют надёжность в реляционных БД.

**Что такое N+1 problem?**
ORM делает 1 запрос на список + N запросов на связанные данные для каждого элемента. Решение: eager loading (`include` в Prisma, `populate` в Mongoose, `JOIN` в SQL).

**Зачем нужны индексы и когда они вредят?**
Индексы ускоряют SELECT (B-tree для поиска, сортировки, диапазонов). Но замедляют INSERT/UPDATE/DELETE (нужно обновлять индекс) и занимают место. Индексировать стоит колонки в WHERE, JOIN, ORDER BY с высокой selectivity.

**Что такое CAP-теорема?**
В распределённой системе при network partition нельзя гарантировать и Consistency, и Availability одновременно. CP (HBase, MongoDB majority) — отклоняет запросы ради согласованности. AP (Cassandra, DynamoDB) — отвечает, но данные могут быть устаревшими.

**PostgreSQL vs MySQL — что выбрать?**
PostgreSQL — мощнее (JSONB, оконные функции, extensions, pgvector), лучший оптимизатор. MySQL — проще в setup, быстрее для простых запросов, огромная экосистема (WordPress). Для нового проекта — PostgreSQL в большинстве случаев.
