# Функциональное программирование

## Введение

Функциональное программирование (FP) — это парадигма, в которой вычисления рассматриваются как применение **функций к данным**, а не последовательность команд. Она опирается на **математические принципы лямбда-исчисления** и противопоставляется императивным стилям, где основой является изменение состояния.

Ключевые принципы:

- **Функции как объекты первого класса** — их можно передавать, возвращать и хранить.
- **Чистые функции** — одинаковый результат при одинаковых входных данных, без побочных эффектов.
- **Иммутабельность** — данные не изменяются, а создаются новые копии при модификации.

## Функции как объекты первого класса

Одна из фундаментальных идей функционального программирования — это то, что **функции являются объектами первого класса** (First-Class Functions).

Это означает, что функции можно:

- **сохранять в переменных**;
- **передавать как аргументы** другим функциям;
- **возвращать из других функций**;
- **хранить в структурах данных** (массивах, объектах и т.д.);
- **создавать динамически**.

Проще говоря — функции в JavaScript (и других FP-языках) ведут себя так же, как и любые другие значения.

```js
// Сохраняем функцию в переменную
const greet = (name) => `Hello, ${name}!`;

// Передаём как аргумент
function run(fn) {
  console.log(fn("world"));
}

run(greet); // "Hello, world!"

// Возвращаем функцию
function multiplier(factor) {
  return (x) => x * factor;
}

const double = multiplier(2);
console.log(double(5)); // 10
```

Благодаря этому мы можем строить **абстракции поведения**, создавать **функции высшего порядка**, и в целом — мыслить программой как комбинацией функций, а не последовательных команд.

> Без концепции "функций первого класса" не было бы ни функций высшего порядка, ни композиции, ни каррирования — это основа функционального стиля программирования.

## Основные концепции FP

- Чистые функции (Pure Functions)
- Функции высшего порядка (HoC)
- Композиция функций (Function Composition)
- Каррирование (Currying)
- Рекурсия (Recursion)
- Частичное применение (Partial Application)
- Иммутабельность (Immutability)
- Ленивые вычисления (Lazy Evaluation)
- Функторы и монады (Functors & Monads)
- Сопоставление с образцом (Pattern Matching)

### Чистые функции (Pure Functions)

**Чистая функция** — это функция, которая при одинаковых входных данных **всегда возвращает один и тот же результат** и **не имеет побочных эффектов**. Иными словами, она зависит только от своих аргументов и не изменяет ничего за пределами своей области.

Функция `f(x)` считается чистой, если выполняются два условия:

1. **Детерминированность** — для одинаковых входных аргументов результат всегда один и тот же.
2. **Отсутствие побочных эффектов** — выполнение функции не изменяет внешнее состояние (глобальные переменные, файлы, БД и т.д.).

#### Пример чистой функции

```js
// Пример: чистая функция
function add(a, b) {
  return a + b;
}

console.log(add(2, 3)); // 5
console.log(add(2, 3)); // 5 — всегда одинаковый результат
```

- Результат зависит только от аргументов.
- Не изменяет ничего вне функции.
- Можно безопасно вызывать много раз.

#### Почему чистые функции важны

1. **Предсказуемость** — их поведение не зависит от контекста.
2. **Тестируемость** — легко протестировать, так как нет зависимостей от окружения.
3. **Упрощённая отладка** — отсутствие скрытых состояний делает логику прозрачной.
4. **Memoization (кэширование)** — результат можно запоминать, т.к. он всегда одинаковый для одинаковых аргументов.
5. **Параллелизация** — чистые функции безопасно выполнять одновременно (нет гонок данных).

#### Побочные эффекты (side effects)

Побочный эффект — это любое изменение состояния за пределами функции:

- изменение глобальной переменной;
- мутация аргументов;
- взаимодействие с внешним миром (файлы, сеть, база данных, DOM, консоль и т.п.);
- генерация случайных чисел (`Math.random()`), получение текущего времени (`Date.now()`).

> Полностью чистый код невозможен — где-то нужно взаимодействовать с внешним миром. Но **идея FP** в том, чтобы:
>
> - изолировать нечистые операции (I/O, API) от чистой логики;
> - писать большую часть кода как чистые функции;
> - использовать композицию для объединения чистых функций в цепочки преобразований данных.

### Функции высшего порядка

**Функции высшего порядка (Higher-Order Functions)** или HoC — это функции, которые **принимают другие функции как аргументы и/или возвращают их как результат**. Такой подход позволяет создавать гибкие и выразительные абстракции, сводя дублирование логики к минимуму.

```js
const apply = (fn, value) => fn(value);
const double = (x) => x * 2;

apply(double, 4); // 8
```

#### Основная идея

Функции высшего порядка дают возможность **манипулировать поведением** так же, как данными. Вместо того чтобы писать десятки циклов и условий, мы можем выразить логику более декларативно: _что сделать_, а не _как сделать_.

Пример на JavaScript:

```js
const numbers = [1, 2, 3, 4, 5];

// Императивно
const doubled = [];
for (let i = 0; i < numbers.length; i++) {
  doubled.push(numbers[i] * 2);
}

// Функционально
const doubledFP = numbers.map((x) => x * 2);
```

Во втором случае `map` — это функция высшего порядка: она **принимает функцию обратного вызова (callback)** и применяет её к каждому элементу.

#### Примеры встроенных функций высшего порядка

- **`map(fn)`** — применяет функцию к каждому элементу и возвращает новый массив.
- **`filter(fn)`** — возвращает элементы, для которых функция вернула `true`.
- **`reduce(fn, acc)`** — сводит массив к одному значению.
- **`forEach(fn)`** — выполняет побочное действие для каждого элемента (не возвращает значение).

```js
const users = [
  { name: "Alice", age: 25 },
  { name: "Bob", age: 30 },
  { name: "Charlie", age: 35 },
];

const names = users.map((u) => u.name);
const adults = users.filter((u) => u.age >= 30);
const avgAge = users.reduce((sum, u) => sum + u.age, 0) / users.length;
```

#### Возврат функций

Функции высшего порядка часто **возвращают новые функции**. Это используется для создания **замыканий**, **декораторов** и **каррирования**.

```js
function multiplier(factor) {
  return function (x) {
    return x * factor;
  };
}

const double = multiplier(2);
const triple = multiplier(3);

double(4); // 8
triple(4); // 12
```

#### Применение

1. **Абстракция повторяющихся шаблонов** — например, при работе с асинхронными запросами, логами или ошибками.
2. **Декораторы и middleware** — добавляют дополнительное поведение, не изменяя исходную функцию.
3. **Функциональные цепочки (pipelines)** — объединяют последовательность преобразований.

```js
const log = (fn) => (x) => {
  const result = fn(x);
  console.log("Input:", x, "Output:", result);
  return result;
};

const square = (x) => x * x;

const loggedSquare = log(square);
loggedSquare(3); // Input: 3 Output: 9
```

#### Преимущества

- Повышают **переиспользуемость** и **декларативность** кода.
- Упрощают обработку коллекций и потоков данных.
- Позволяют работать с поведением как с данными (функции — тоже значения).

#### Недостатки

- Сложнее понимать новичкам, особенно при вложенных вызовах.
- При чрезмерном использовании возможна потеря читаемости.
- Отладка цепочек функций может быть нетривиальной.

### Композиция функций

**Композиция функций** — это процесс объединения нескольких функций таким образом, чтобы результат одной функции автоматически становился входом для следующей. Это фундаментальная идея функционального программирования, позволяющая строить декларативные, предсказуемые и легко читаемые цепочки преобразований данных.

```js
const compose = (f, g) => (x) => f(g(x));
const toUpper = (s) => s.toUpperCase();
const exclaim = (s) => s + "!";

const shout = compose(exclaim, toUpper);
shout("hello"); // "HELLO!"
```

#### Почему это важно

Композиция функций позволяет строить программы как **конвейеры обработки данных** — данные проходят через последовательность чистых функций, каждая из которых выполняет одну задачу. Это способствует:

- **Декларативности** — вместо пошагового описания действий, вы описываете, _что_ должно произойти с данными;
- **Переиспользованию** — каждая функция выполняет минимальную задачу и может использоваться в разных контекстах;
- **Тестируемости** — каждая часть конвейера можно проверять изолированно;
- **Отсутствию побочных эффектов** — чистые функции не изменяют данные, а создают новые.

#### Пример на практике

```js
const trim = (s) => s.trim();
const toLower = (s) => s.toLowerCase();
const replaceSpaces = (s) => s.replace(/\s+/g, "-");

// Ручная запись (императивно):
const slugify1 = (text) => replaceSpaces(toLower(trim(text)));

// Через композицию:
const compose =
  (...fns) =>
  (x) =>
    fns.reduceRight((v, f) => f(v), x);
const slugify2 = compose(replaceSpaces, toLower, trim);

slugify2("  Hello World  "); // "hello-world"
```

Здесь `compose` строит _pipeline_ преобразований, выполняемых справа налево. Существуют также функции `pipe`, которые делают то же самое, но читаются слева направо:

```js
const pipe =
  (...fns) =>
  (x) =>
    fns.reduce((v, f) => f(v), x);

const slugify = pipe(trim, toLower, replaceSpaces);
slugify("  Hello FP World  "); // "hello-fp-world"
```

#### Композиция в библиотечных решениях

- **Ramda** — популярная JS-библиотека с функциями `R.compose` и `R.pipe`.
- **Lodash/fp** — функциональная версия Lodash с поддержкой композиции.
- **RxJS** — использует композицию операторов для построения реактивных потоков.

#### Преимущества композиции

- Повышает читаемость и декларативность кода;
- Упрощает тестирование и сопровождение;
- Позволяет легко масштабировать логику без изменения существующих функций;
- Является ключом к **чистому, предсказуемому** программированию.

#### Недостатки

- Может быть труднее отлаживать из-за вложенности вызовов;
- Требует привычки к функциональному стилю мышления (особенно при вложенных композициях);
- Ошибки типов и несоответствия аргументов проявляются только во время выполнения (в JavaScript).

### Каррирование

**Каррирование (Currying)** — это техника преобразования функции, принимающей несколько аргументов, в последовательность функций, каждая из которых принимает только один аргумент и возвращает новую функцию, пока не будут получены все аргументы.

```js
const add = (a) => (b) => a + b;
add(2)(3); // 5
```

На практике это позволяет **создавать более гибкие и переиспользуемые функции**. Например, можно заранее задать часть контекста:

```js
const multiply = (a) => (b) => a * b;
const double = multiply(2);
const triple = multiply(3);

double(5); // 10
triple(5); // 15
```

**Преимущества каррирования:**

- Повышает **переиспользуемость** функций.
- Облегчает **композицию** и функциональные цепочки.
- Позволяет **избегать дублирования логики**, фиксируя часть аргументов.
- Делает код более **декларативным** и читаемым.

### Рекурсия вместо циклов

В функциональном программировании (FP) **рекурсия** используется вместо императивных циклов (`for`, `while`), поскольку изменяемое состояние (например, счётчик цикла) противоречит идее **иммутабельности**.

Рекурсия — это процесс, при котором функция вызывает саму себя до тех пор, пока не будет достигнуто базовое условие (остановка). Каждая итерация рекурсии возвращает результат, основанный на предыдущих вызовах.

```js
// Классический пример: факториал
const factorial = (n) => (n <= 1 ? 1 : n * factorial(n - 1));

console.log(factorial(5)); // 120
```

#### Почему рекурсия важна в ФП

1. **Без побочных эффектов:** нет изменяемых переменных или счётчиков.
2. **Декларативность:** мы описываем _что_ делаем, а не _как_.
3. **Композиционность:** рекурсивные функции легко комбинировать с другими функциями.

#### Хвостовая рекурсия (Tail Recursion)

Хвостовая рекурсия — форма рекурсии, при которой **последняя операция функции — это вызов самой себя**. Это позволяет интерпретатору (в некоторых языках) оптимизировать выполнение и не создавать новые кадры стека.

```js
// Хвостовая рекурсия для вычисления факториала
const factorialTail = (n, acc = 1) =>
  n <= 1 ? acc : factorialTail(n - 1, acc * n);

console.log(factorialTail(5)); // 120
```

> В языках вроде **Scala**, **Haskell**, **Elixir** и **Clojure** хвостовая рекурсия оптимизируется на уровне компилятора (Tail Call Optimization, TCO).

### Частичное применение

**Частичное применение (Partial Application)** — это создание новой функции на основе существующей, в которой **часть аргументов уже предопределена**, но функция всё ещё может принимать оставшиеся.

```js
function multiply(a, b) {
  return a * b;
}

const double = multiply.bind(null, 2);
console.log(double(5)); // 10
```

В отличие от каррирования, частичное применение **не требует одной функции на каждый аргумент** — вы можете зафиксировать любое количество аргументов сразу.

```js
const log = (level, message) => console.log(`[${level}] ${message}`);

const info = log.bind(null, "INFO");
const error = log.bind(null, "ERROR");

info("Server started"); // [INFO] Server started
error("Something went wrong"); // [ERROR] Something went wrong
```

**Разница между каррированием и частичным применением:**

| Характеристика   | Каррирование                        | Частичное применение                             |
| ---------------- | ----------------------------------- | ------------------------------------------------ |
| Возвращает       | Цепочку функций по одному аргументу | Одну функцию с частью зафиксированных аргументов |
| Используется для | Функциональных цепочек, композиции  | Преднастройки функций и переиспользования        |
| Пример           | `f(a)(b)(c)`                        | `f(a, b)` -> `g(c)`                              |

### Иммутабельность

**Иммутабельность** (immutability) — это свойство объекта, при котором его состояние нельзя изменить после создания. Вместо изменения данных создаются их новые копии с учётом нужных изменений.

Пример на JavaScript:

```js
const user = { name: "Alice", age: 25 };
const updatedUser = { ...user, age: 26 };

console.log(user.age); // 25
console.log(updatedUser.age); // 26
```

Здесь объект `user` остаётся неизменным — создаётся новый объект `updatedUser` с обновлённым значением.

Иммутабельность важна, потому что:

1. **Предсказуемость** — объекты не меняются неожиданно в разных частях программы.
2. **Упрощение отладки** — можно легко отследить, где и когда создаются новые данные.
3. **Безопасность в многопоточности** — неизменяемые структуры данных не требуют блокировок.

Во многих языках иммутабельность используется по умолчанию (например, в функциональных языках), а в JavaScript или Python — это практика, которой придерживаются для надёжности и чистоты кода.

### Ленивые вычисления

**Ленивые вычисления (Lazy Evaluation)** — это стратегия исполнения, при которой выражения не вычисляются сразу, а только тогда, когда их значение действительно необходимо. Такой подход позволяет экономить ресурсы и работать даже с потенциально бесконечными структурами данных.

#### Принцип работы

- При ленивых вычислениях создаётся **отложенное выражение** (thunk).
- Выражение вычисляется **по требованию**, а не заранее.
- Повторные обращения могут кэшироваться (memoization).

```js
function* naturalNumbers() {
  let i = 0;
  while (true) yield i++;
}

const gen = naturalNumbers();
console.log(gen.next().value); // 0
console.log(gen.next().value); // 1
```

#### Преимущества

- **Экономия ресурсов** — неиспользуемые вычисления не происходят.
- **Возможность бесконечных структур** — например, бесконечные списки чисел.
- **Повышение декларативности** — код описывает «что» нужно, а не «как».

#### Недостатки

- **Сложнее отладка** — порядок вычислений не всегда очевиден.
- **Потребление памяти** — при накоплении слишком многих отложенных вычислений (thunks).

#### Примеры в разных языках

- **Haskell** — использует ленивую стратегию по умолчанию.
- **JavaScript** — реализует её через генераторы (`function*`) и итераторы.
- **Python** — поддерживает ленивые вычисления через `yield` и генераторные выражения.

### Функторы и монады

<b class="text-on-semi-light">Функтор</b> — это структура данных, к которой можно применить функцию, не разрушая саму структуру. Формально функтор реализует операцию `map` (или `fmap`) с двумя фундаментальными законами:

1. **Закон тождества:** `F.map(x => x) === F` — применение тождественной функции не меняет функтор.
2. **Закон композиции:** `F.map(x => f(g(x))) === F.map(g).map(f)` — композиция функций эквивалентна последовательному применению.

**Примеры:** массивы (`Array.map`), `Promise` (частично: `then` похож на `map`), `Maybe/Option`, `Result/Either`.

**Простая иллюстрация (JS):**

```js
const numbers = [1, 2, 3];
const doubled = numbers.map((x) => x * 2); // Array — функтор
```

**Пользовательский функтор (микро-реализация Maybe):**

```js
class Maybe {
  constructor(value) {
    this._value = value;
  }
  static of(value) {
    return new Maybe(value);
  }

  // map применяет функцию к внутреннему значению, если оно не null/undefined
  map(fn) {
    if (this._value == null) return Maybe.of(null);
    return Maybe.of(fn(this._value));
  }
}

Maybe.of(2).map((x) => x + 1); // Maybe(3)
Maybe.of(null).map((x) => x + 1); // Maybe(null)
```

Функтор даёт единообразный способ «подвешивать» вычисления на контейнеры, не беспокоясь о внутренней форме.

---

<b class="text-on-semi-light">Монада</b> — это расширение идеи функтора: это структура с операцией `of/return` (встраивание значения в контекст) и оператором связывания (`flatMap`, `bind`, `chain`), который позволяет последовательно выполнять вычисления, где каждая функция возвращает контекстную (обёрнутую) сущность.

Ключевые законы монады:

1. **Left identity:** `M.of(a).flatMap(f) ≡ f(a)`
2. **Right identity:** `m.flatMap(M.of) ≡ m`
3. **Associativity:** `m.flatMap(f).flatMap(g) ≡ m.flatMap(x => f(x).flatMap(g))`

**Почему flatMap?** Если `map` вернёт вложенный контейнер (`F<F<T>>`), то он усложнит композицию. `flatMap` выравнивает (flatten) уровни и даёт «плоскую» последовательность вычислений.

**Примеры монад:**

- `Promise` — асинхронный контекст (`then` ≈ `flatMap`),
- `Maybe/Option` — контекст возможного отсутствия значения,
- `Either/Result` — контекст, отражающий успех/ошибку,
- `List` — монада списков (комбинирует результаты),
- `IO` (в теоретическом FP) — для отложенных побочных эффектов.

**JS-пример с Promise:**

```js
Promise.resolve(2)
  .then((x) => x * 2) // then — связывает вычисления и возвращает новый Promise
  .then(console.log); // 4
```

**Maybe как монада:**

```js
class Maybe {
  constructor(x) {
    this._x = x;
  }
  static of(x) {
    return new Maybe(x);
  }
  map(fn) {
    return this._x == null ? Maybe.of(null) : Maybe.of(fn(this._x));
  }
  flatMap(fn) {
    return this._x == null ? Maybe.of(null) : fn(this._x);
  }
}

const safeDivide = (a, b) => (b === 0 ? Maybe.of(null) : Maybe.of(a / b));

Maybe.of(10)
  .flatMap((x) => safeDivide(x, 2)) // Maybe(5)
  .flatMap((x) => safeDivide(x, 0)); // Maybe(null)
```

Монады помогают структурировать последовательности операций с учётом контекста (ошибки, отсутствующие значения, асинхронность) и делают код композиционным и безопасным.

#### Когда использовать функторы/монады

- Когда нужно **элегантно обработать отсутствие значения** (Maybe) без `if`/`throw`.
- Для **цепочек асинхронных действий** (Promise).
- Для **композиции операций с возможными ошибками** (Either/Result).
- В системах трансформации данных, где нужен «контекст» (логирование, состояние, кэширование).

### Сопоставление с образом

**Определение.** Pattern matching — язык-синтаксический механизм для проверки структуры данных и извлечения её компонентов в зависимости от формы.

**Преимущества:**

- более выразительное, компактное и безопасное ветвление по структуре;
- возможность **исчерпывающей проверки** (компилятор может сигнализировать о неучтённых вариантах);
- удобная работа с ADT (алгебраическими типами данных).

Haskell:

```haskell
case xs of
  [] -> "empty"
  (x:xs') -> "head: " ++ show x
```

Scala:

```scala
x match {
  case Nil => "empty"
  case head :: tail => s"head: $head"
}
```

**JS-подходы (эмуляция):**

- `switch`/`if` + `typeof` / `Array.isArray` / деструктуризация — ограниченная замена;
- библиотеки: `ts-pattern` (TypeScript), `match-iz` — предлагают синтаксис pattern matching и статические проверки;
- использование ADT-стиля с `tag` / `type` полем и `switch` по `tag`.

**Пример ADT в JS (Either) и эмуляция паттерн-матчинга:**

```js
const Right = (v) => ({ tag: "Right", value: v });
const Left = (e) => ({ tag: "Left", error: e });

function match(either) {
  switch (either.tag) {
    case "Right":
      return `OK: ${either.value}`;
    case "Left":
      return `Error: ${either.error}`;
  }
}

console.log(match(Right(42))); // OK: 42
console.log(match(Left("not found"))); // Error: not found
```

**Почему pattern matching важен в FP:**

- Является естественным инструментом для работы с ADT (например, `Maybe`, `Either`, `Result`, `Option`), где данные представляют собой конечное множество форм.
- Позволяет переносить логику разбора структуры из кода в синтаксис языка, делая её более декларативной и безопасной.

## Функциональные типы данных (ADT)

Алгебраические типы данных (Algebraic Data Types, ADT) — это способ описывать структуру данных с помощью комбинации других типов. Они позволяют явно моделировать все возможные состояния программы.

Существуют два основных вида ADT:

### Product types (произведения)

Описывают **сочетание** нескольких полей. Это как запись (record) или объект.

Пример: структура пользователя, где есть и `name`, и `age`:

```js
// Product type — комбинация данных
const User = (name, age) => ({ name, age });

const user = User("Alice", 25);
console.log(user); // { name: 'Alice', age: 25 }
```

Математически — произведение, потому что количество возможных значений = произведение вариантов каждого поля.

### Sum types (суммы)

Описывают **варианты** данных — одно из нескольких состояний. Это как `enum`, но с данными.

Пример: результат операции может быть успешным или с ошибкой:

```js
// Sum type — одно из состояний
const Ok = (value) => ({ type: "Ok", value });
const Err = (error) => ({ type: "Err", error });

function divide(a, b) {
  return b === 0 ? Err("Division by zero") : Ok(a / b);
}

const result = divide(10, 0);
console.log(result.type === "Ok" ? result.value : result.error);
```

Математически — сумма, потому что количество возможных значений = сумма вариантов.

### Комбинирование

ADT можно комбинировать: один тип может включать другие. Например, `Result` может содержать `Maybe` внутри.

```js
// Пример: Result с вложенным Maybe
const Just = (v) => ({ type: "Just", v });
const Nothing = { type: "Nothing" };

const Ok = (v) => ({ type: "Ok", v });
const Err = (e) => ({ type: "Err", e });

const getUserAge = (user) =>
  user ? Ok(Just(user.age)) : Err("User not found");
```

Зачем нужны ADT

- Явно описывают **все возможные состояния** программы.
- Упрощают **pattern matching** и обработку ошибок.
- Исключают `null`, `undefined`, `try/catch`.
- Повышают надёжность и выразительность типов.

ADT — один из краеугольных камней функционального программирования, потому что они делают код декларативным и безопасным по типам.

## Функциональные операции над коллекциями

В функциональном программировании коллекции рассматриваются как **неизменяемые потоки данных**, к которым применяются функции трансформации. Вместо явных циклов и счётчиков разработчик описывает _что_ нужно сделать, а не _как_.

### Основные операции

- **`map(fn)`** — применяет функцию к каждому элементу коллекции и возвращает новую коллекцию.

  ```js
  const nums = [1, 2, 3];
  const doubled = nums.map((x) => x * 2);
  console.log(doubled); // [2, 4, 6]
  ```

- **`filter(fn)`** — выбирает элементы, удовлетворяющие условию.

  ```js
  const ages = [15, 22, 18, 30];
  const adults = ages.filter((x) => x >= 18);
  console.log(adults); // [22, 18, 30]
  ```

- **`reduce(fn, init)`** — сворачивает коллекцию в одно значение, проходя по всем элементам.

  ```js
  const sum = [1, 2, 3, 4].reduce((acc, x) => acc + x, 0);
  console.log(sum); // 10
  ```

- **`flatMap(fn)`** — применяет функцию, возвращающую коллекцию, и «разворачивает» результат.

  ```js
  const words = ["hello", "world"];
  const letters = words.flatMap((word) => word.split(""));
  console.log(letters); // ['h','e','l','l','o','w','o','r','l','d']
  ```

### Ключевые идеи

- Все функции **не изменяют** исходную коллекцию — они возвращают новую.
- Можно **комбинировать** операции:

  ```js
  const users = [
    { name: "Alice", age: 25 },
    { name: "Bob", age: 30 },
    { name: "Carol", age: 20 },
  ];

  const adultNames = users
    .filter((u) => u.age >= 21)
    .map((u) => u.name)
    .sort();

  console.log(adultNames); // ['Alice', 'Bob']
  ```

### Почему это важно

- **Декларативность** — фокус на цели, а не на механике.
- **Иммутабельность** — предотвращает побочные эффекты и ошибки состояний.
- **Композиция** — операции можно объединять в конвейеры обработки данных.

Эти принципы лежат в основе функциональных библиотек вроде **Ramda**, **Lodash/fp** и реактивных инструментов вроде **RxJS**.

## Управление побочными эффектами

**Побочные эффекты** — это любые действия, которые выходят за рамки чистых вычислений и влияют на внешний мир или зависят от него: ввод/вывод, работа с сетью, файловая система, мутации данных, логирование и т.д.

В **чистом функциональном программировании** задача — изолировать такие эффекты, чтобы остальной код оставался **детерминированным** и **тестируемым**. Для этого эффекты не выполняются напрямую, а **инкапсулируются** в специальные структуры, например:

- `IO` — откладывает выполнение эффекта до явного вызова.
- `Task` — отложенное вычисление, которое может завершиться успешно или с ошибкой.
- `Future`, `Promise` — представляют значение, которое появится в будущем.

---

### Пример на JavaScript

```js
// Простая реализация IO-моноиды
const IO = (effect) => ({
  run: effect,
  map: (fn) => IO(() => fn(effect())),
  flatMap: (fn) => fn(effect()),
});

// Эффекты
const readFile = IO(() => "file content");
const log = (msg) => IO(() => console.log(msg));

// Комбинируем эффекты без их немедленного выполнения
const program = readFile.map((content) => content.toUpperCase()).flatMap(log);

// Побочные эффекты выполняются только при вызове run()
program.run();
```

---

### Ключевые идеи

1. **Отложенность выполнения**
   Побочный эффект не выполняется, пока явно не вызван `run()`.

2. **Композиция без побочек**
   Мы можем комбинировать эффекты (через `map`, `flatMap`), не теряя чистоты функций.

3. **Тестируемость и предсказуемость**
   Код можно тестировать без фактического ввода/вывода, заменяя эффекты заглушками.

4. **Прозрачность**
   Все функции остаются чистыми, а эффекты явно видны в типах и структуре программы.

---

### Пример из функциональных библиотек

В TypeScript/FP-средах (`fp-ts`, `effect`, `folktale`) IO и Task — стандартные абстракции для управления побочными эффектами:

```ts
import { IO } from "fp-ts/IO";

const readFile: IO<string> = () => "file content";
const log =
  (msg: string): IO<void> =>
  () =>
    console.log(msg);

const program = () => {
  const content = readFile();
  log(content.toUpperCase())(); // выполняем эффекты
};
```

Управление побочными эффектами — это не запрет на них, а **контроль и изоляция**, чтобы логика программы оставалась чистой, предсказуемой и удобной для тестирования.

## Point-free стиль и композиция

**Point-free стиль** (или _безаргументный стиль_) — это способ написания функций, где вы **не указываете явно параметры**, а строите выражения через **композицию других функций**. Этот подход особенно распространён в функциональных языках (Haskell, Elm, F#), но также применим и в JavaScript.

### Основная идея

Вместо того чтобы фокусироваться на данных (что мы передаём), мы описываем **поток преобразований**. То есть — не _что делаем с аргументом_, а _как сочетаются функции_.

```js
// Обычный (pointed) стиль
const toUpper = (s) => s.toUpperCase();
const exclaim = (s) => s + "!";
const shout = (s) => exclaim(toUpper(s));

// Point-free
const compose = (f, g) => (x) => f(g(x));
const shoutPF = compose(exclaim, toUpper);

console.log(shoutPF("hello")); // HELLO!
```

### Зачем использовать point-free стиль

1. **Меньше шума** — меньше переменных и аргументов, чище код.
2. **Декларативность** — вы описываете процесс, а не шаги.
3. **Композиционность** — функции легко комбинировать и переиспользовать.
4. **Тестируемость** — каждая функция независима, её можно проверить изолированно.

### Когда не стоит использовать point-free стиль

Иногда чрезмерное увлечение point-free делает код **менее читаемым**:

```js
// Слишком абстрактно
const transform = compose(map(toUpperCase), filter(isLetter));
```

> Но е сли программисту приходится тратить больше времени на то, чтобы понять логику, чем при обычной записи — значит, стоит вернуться к более явному стилю.
