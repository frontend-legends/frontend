# Потоки и параллелизм

Понимание потоков, процессов и моделей параллелизма — одна из ключевых тем на техническом интервью. Даже если ты пишешь фронтенд, вопросы про event loop, Web Workers, race conditions и разницу между concurrency и parallelism встречаются регулярно. Этот раздел разбирает всё по порядку — от базовых понятий до практических паттернов.

---

## Процесс vs Поток

<b class="text-on-semi-light">Процесс (Process)</b>

<span class="text-gray">Процесс — это экземпляр запущенной программы. У каждого процесса **собственное адресное пространство**: своя heap, свой стек, свои файловые дескрипторы. Процессы изолированы друг от друга — один процесс не может напрямую читать память другого. Взаимодействие между процессами (IPC) возможно через механизмы ОС: pipe, socket, shared memory, message queue.</span>

<b class="text-on-semi-light">Поток (Thread)</b>

<span class="text-gray">Поток — это единица выполнения **внутри** процесса. Потоки одного процесса разделяют общее адресное пространство (heap, глобальные переменные, файловые дескрипторы), но у каждого потока — свой стек вызовов и свой instruction pointer. Это делает потоки «легче» процессов: создание и переключение потоков дешевле, но общая память порождает проблемы синхронизации.</span>

| Характеристика | Процесс | Поток |
|---|---|---|
| Память | Изолированная (своё адресное пространство) | Общая heap, свой стек |
| Создание | Дорогое (fork, clone) | Дешевое |
| Переключение контекста | Медленное (смена page tables) | Быстрое (смена стека и регистров) |
| Изоляция | Полная (crash одного не затрагивает другой) | Нет (crash потока убивает процесс) |
| Коммуникация | IPC (pipe, socket, shared memory) | Общая память напрямую |

> Браузер — мультипроцессная система. Каждая вкладка работает в отдельном процессе (site isolation), а внутри каждого процесса — один main thread и, при необходимости, worker threads.

## Concurrency vs Parallelism

Эти два понятия часто путают, но это фундаментально разные вещи.

<b class="text-on-semi-light">Concurrency (конкурентность)</b>

<span class="text-gray">Управление несколькими задачами, которые **могут перекрываться во времени**, но не обязательно выполняются одновременно. Один процессор может переключаться между задачами так быстро, что создаётся иллюзия параллельности. Главное — **структура** программы позволяет работать с несколькими задачами.</span>

<b class="text-on-semi-light">Parallelism (параллелизм)</b>

<span class="text-gray">Буквальное **одновременное** выполнение нескольких задач на нескольких ядрах/процессорах. Parallelism невозможен на одном ядре — это физическое свойство оборудования.</span>

```text
Concurrency (1 ядро):
  Task A: ██░░██░░██
  Task B: ░░██░░██░░
  → задачи чередуются

Parallelism (2 ядра):
  Core 1: ██████████  (Task A)
  Core 2: ██████████  (Task B)
  → задачи выполняются одновременно
```

> JavaScript в браузере — **concurrent, но не parallel** (в рамках main thread). Один поток обрабатывает и рендеринг, и пользовательские события, и таймеры, переключаясь между ними через event loop. Реальный параллелизм достигается только через Web Workers.

## Многозадачность

Операционная система управляет тем, как процессы и потоки получают доступ к CPU. Есть два основных подхода.

<b class="text-on-semi-light">Вытесняющая многозадачность (Preemptive)</b>

<span class="text-gray">ОС сама решает, когда прервать выполнение потока и передать CPU другому. Поток не может «захватить» процессор навсегда. Используется во всех современных ОС (Linux, Windows, macOS). Планировщик ОС выделяет каждому потоку квант времени (time slice), после чего происходит переключение контекста.</span>

<b class="text-on-semi-light">Кооперативная многозадачность (Cooperative)</b>

<span class="text-gray">Задача сама «отдаёт» управление, когда считает нужным. Если задача зависнет — зависнет всё. Именно так работает **JavaScript event loop**: каждая задача (callback, промис, обработчик) выполняется до конца, прежде чем event loop возьмёт следующую. Долгий синхронный код блокирует весь main thread — UI замирает, события не обрабатываются.</span>

## JavaScript: однопоточная модель

JavaScript выполняется в **одном потоке** (main thread). Это касается и браузера, и Node.js. Но «однопоточный» не значит «медленный» — архитектура event loop позволяет эффективно обрабатывать тысячи конкурентных операций.

### Из чего состоит runtime

```text
┌──────────────────────────────────────────────┐
│                  JS Engine (V8)               │
│  ┌──────────┐  ┌──────────────────────────┐  │
│  │ Call Stack│  │        Heap              │  │
│  │           │  │   (объекты, замыкания)   │  │
│  └──────────┘  └──────────────────────────┘  │
└──────────────────────────────────────────────┘
         │
         ▼
┌──────────────────────────────────────────────┐
│              Event Loop                       │
│  ┌─────────────────┐  ┌───────────────────┐  │
│  │  Macrotask Queue │  │ Microtask Queue   │  │
│  │  (setTimeout,    │  │ (Promise.then,    │  │
│  │   setInterval,   │  │  queueMicrotask,  │  │
│  │   I/O callbacks) │  │  MutationObserver)│  │
│  └─────────────────┘  └───────────────────┘  │
└──────────────────────────────────────────────┘
```

### Порядок выполнения event loop

1. Выполнить весь синхронный код в call stack.
2. Очистить **всю** microtask queue (Promise callbacks, queueMicrotask).
3. Выполнить **одну** macrotask (setTimeout, setInterval, I/O).
4. Снова очистить microtask queue.
5. Если нужно — выполнить rendering (requestAnimationFrame, layout, paint).
6. Повторить.

```js
console.log('1'); // sync

setTimeout(() => console.log('2'), 0); // macrotask

Promise.resolve().then(() => console.log('3')); // microtask

console.log('4'); // sync

// Вывод: 1, 4, 3, 2
```

<b class="text-on-semi-light">Почему microtasks выполняются раньше macrotasks?</b>

<span class="text-gray">Microtask queue полностью опустошается после каждого шага call stack и перед следующей macrotask. Это гарантирует, что `.then()` промиса выполнится до следующего таймера или I/O callback.</span>

### Блокировка main thread

Любой синхронный код блокирует весь поток — UI зависает, анимации останавливаются, клики не обрабатываются.

```js
// Плохо: блокирует main thread на ~секунду
function heavySync() {
  const arr = [];
  for (let i = 0; i < 10_000_000; i++) {
    arr.push(Math.sqrt(i));
  }
  return arr;
}

// Хорошо: разбить на чанки через scheduler
async function heavyAsync() {
  const arr = [];
  for (let i = 0; i < 10_000_000; i++) {
    arr.push(Math.sqrt(i));
    if (i % 100_000 === 0) {
      await new Promise(resolve => setTimeout(resolve, 0)); // даём event loop «дышать»
    }
  }
  return arr;
}
```

> Более подробно async/await, Promise и event loop разбираются в -> [7. JavaScript -> Async](/content/javascript/async)

## Web Workers: реальный параллелизм в браузере

Web Worker — это **отдельный поток** с собственным event loop и глобальным контекстом. Worker не имеет доступа к DOM, `window`, `document` — он полностью изолирован от main thread. Общение происходит исключительно через `postMessage` / `onmessage` (message passing).

### Типы Worker'ов

| Тип | Область видимости | Lifetime | Использование |
|---|---|---|---|
| **Dedicated Worker** | Один скрипт-создатель | Пока не вызван `terminate()` | Тяжёлые вычисления, парсинг |
| **Shared Worker** | Несколько вкладок/iframe одного origin | Пока есть хотя бы одно соединение | Общий кэш, WebSocket между вкладками |
| **Service Worker** | Весь origin (proxy между сетью и браузером) | Управляется браузером | Offline, push-уведомления, кэширование |

### Dedicated Worker: пример

```js
// main.js
const worker = new Worker('worker.js');

worker.postMessage({ numbers: [1, 2, 3, 4, 5] });

worker.onmessage = (e) => {
  console.log('Сумма:', e.data.result); // 15
};

worker.onerror = (e) => {
  console.error('Worker error:', e.message);
};
```

```js
// worker.js
self.onmessage = (e) => {
  const sum = e.data.numbers.reduce((a, b) => a + b, 0);
  self.postMessage({ result: sum });
};
```

### Transferable Objects

По умолчанию `postMessage` **копирует** данные (structured clone). Для больших бинарных данных это дорого. `Transferable` позволяет **передать владение** буфером — данные перемещаются без копирования, но становятся недоступны в отправляющем контексте.

```js
const buffer = new ArrayBuffer(1024 * 1024); // 1 МБ

// Передаём владение — buffer становится neutered (пустой) в main thread
worker.postMessage({ data: buffer }, [buffer]);

console.log(buffer.byteLength); // 0 — буфер передан
```

### SharedArrayBuffer и Atomics

`SharedArrayBuffer` — разделяемая память между main thread и worker'ами. В отличие от `Transferable`, данные **не перемещаются**, а доступны обоим потокам одновременно. Это порождает все классические проблемы многопоточности — race conditions. Для безопасной работы используется `Atomics`.

```js
// main.js
const sab = new SharedArrayBuffer(4); // 4 байта = 1 int32
const view = new Int32Array(sab);

worker.postMessage(sab);

// Атомарная запись
Atomics.store(view, 0, 42);

// Ожидание: поток засыпает, пока значение по индексу 0 равно 42
Atomics.wait(view, 0, 42);
```

```js
// worker.js
self.onmessage = (e) => {
  const view = new Int32Array(e.data);

  // Атомарное чтение
  const value = Atomics.load(view, 0); // 42

  // Атомарное изменение
  Atomics.store(view, 0, value + 1);

  // Пробудить ожидающий поток
  Atomics.notify(view, 0, 1);
};
```

> `SharedArrayBuffer` требует COOP/COEP заголовков (`Cross-Origin-Opener-Policy: same-origin`, `Cross-Origin-Embedder-Policy: require-corp`).

## Node.js: worker_threads

Node.js — тоже однопоточный (main event loop), но модуль `worker_threads` даёт настоящий параллелизм. В отличие от `child_process` (отдельный процесс), worker_threads работают **в одном процессе** с общей памятью (через `SharedArrayBuffer`).

```js
// main.mjs
import { Worker } from 'node:worker_threads';

const worker = new Worker('./heavy.mjs');

worker.postMessage({ n: 45 });

worker.on('message', (result) => {
  console.log('Fibonacci:', result);
});
```

```js
// heavy.mjs
import { parentPort } from 'node:worker_threads';

function fib(n) {
  if (n <= 1) return n;
  return fib(n - 1) + fib(n - 2);
}

parentPort.on('message', (msg) => {
  parentPort.postMessage(fib(msg.n));
});
```

Когда использовать `worker_threads` вместо `child_process`:

| Критерий | worker_threads | child_process |
|---|---|---|
| Overhead | Низкий (общий процесс) | Высокий (fork процесса) |
| Shared memory | Да (SharedArrayBuffer) | Нет |
| Изоляция | Частичная | Полная |
| Когда использовать | CPU-bound задачи | Запуск внешних программ, полная изоляция |

## Проблемы многопоточности

Когда несколько потоков работают с общей памятью, возникают классические проблемы. Эти проблемы актуальны при использовании `SharedArrayBuffer` в браузере и `worker_threads` в Node.js.

### Race Condition (гонка данных)

Два или более потока одновременно читают и пишут в одну область памяти. Результат зависит от порядка выполнения, который **недетерминирован**.

```js
// SharedArrayBuffer: race condition
// Оба потока делают: value = load(view, 0); store(view, 0, value + 1);
//
// Ожидаемый результат: 2
// Возможный результат: 1
//
// Поток A: load → 0
// Поток B: load → 0       (прочитал ДО того, как A записал)
// Поток A: store → 1
// Поток B: store → 1      (перезаписал результат A)
```

**Решение:** использовать `Atomics.add(view, 0, 1)` — атомарная операция read-modify-write.

### Deadlock (взаимная блокировка)

Два потока ждут друг друга и не могут продвинуться. Каждый держит ресурс, который нужен другому.

```text
Поток A: захватил Lock 1, ждёт Lock 2
Поток B: захватил Lock 2, ждёт Lock 1
→ оба заблокированы навсегда
```

**Условия возникновения (все четыре одновременно):**
1. Mutual exclusion — ресурс эксклюзивен.
2. Hold and wait — поток держит один ресурс и ждёт другой.
3. No preemption — ресурс нельзя отобрать принудительно.
4. Circular wait — циклическая зависимость.

**Предотвращение:** фиксированный порядок захвата ресурсов, timeout'ы, отказ от вложенных блокировок.

### Livelock

Потоки не блокируются, но постоянно реагируют друг на друга и не продвигаются — как два человека в коридоре, которые одновременно уступают друг другу дорогу.

### Starvation (голодание)

Поток с низким приоритетом не получает доступ к ресурсу, потому что высокоприоритетные потоки постоянно его захватывают.

## Механизмы синхронизации

<b class="text-on-semi-light">Mutex (Mutual Exclusion)</b>

<span class="text-gray">Простейший примитив: только один поток может владеть мьютексом в любой момент. Остальные потоки блокируются до освобождения. В JavaScript нет встроенного mutex, но можно реализовать через `Atomics.wait` / `Atomics.notify`.</span>

<b class="text-on-semi-light">Семафор (Semaphore)</b>

<span class="text-gray">Обобщение мьютекса: разрешает доступ **N потокам** одновременно. Мьютекс — это семафор с N=1. Применяется для ограничения одновременных подключений, пулов потоков и т.д.</span>

<b class="text-on-semi-light">Спинлок (Spinlock)</b>

<span class="text-gray">Поток в цикле проверяет, свободен ли ресурс (busy-waiting). Не отдаёт управление ОС, поэтому эффективен только для **очень коротких** блокировок. На одном ядре — бесполезен (тратит CPU впустую).</span>

<b class="text-on-semi-light">Condition Variable</b>

<span class="text-gray">Позволяет потоку **заснуть** до наступления определённого условия и быть разбуженным другим потоком. В JS: `Atomics.wait()` (поток спит) + `Atomics.notify()` (пробуждение).</span>

### Пример: простой mutex на Atomics

```js
const lock = new Int32Array(new SharedArrayBuffer(4));

function acquire(lock) {
  // Пытаемся атомарно сменить 0 → 1
  while (Atomics.compareExchange(lock, 0, 0, 1) !== 0) {
    // Ждём, пока кто-то освободит (значение станет 0)
    Atomics.wait(lock, 0, 1);
  }
}

function release(lock) {
  Atomics.store(lock, 0, 0);
  Atomics.notify(lock, 0, 1);
}
```

## Модели параллелизма

Существует несколько фундаментальных подходов к организации параллельного кода.

### Shared Memory (общая память)

Потоки читают и пишут в общую память. Требует синхронизации (mutex, semaphore, atomics). Используется в: C/C++ (pthreads), Java (synchronized), JavaScript (SharedArrayBuffer).

**Плюсы:** быстрый обмен данными, нет overhead на сериализацию.
**Минусы:** race conditions, deadlocks, сложная отладка.

### Message Passing (передача сообщений)

Потоки/процессы общаются только через сообщения, не имея общей памяти. Данные **копируются** или **передаются** (transfer). Используется в: JavaScript (`postMessage`), Erlang, Elixir.

**Плюсы:** нет shared state → нет race conditions, проще рассуждать о коде.
**Минусы:** overhead на копирование, сложнее координировать доступ к общим данным.

### CSP (Communicating Sequential Processes)

Модель, где процессы общаются через **каналы** (channels). Отправка и получение — блокирующие операции: отправитель ждёт, пока получатель не прочитает. Используется в Go (goroutines + channels).

```go
ch := make(chan int)
go func() { ch <- 42 }() // отправить
val := <-ch                // получить (блокирует до отправки)
```

### Actor Model (модель акторов)

Каждый «актор» — изолированная сущность со своим состоянием и mailbox'ом. Акторы общаются **только** через асинхронные сообщения. Нет общей памяти, нет блокировок. Используется в: Erlang/OTP, Akka (Scala/Java).

| Модель | Общая память | Синхронизация | Примеры |
|---|---|---|---|
| Shared Memory | Да | Mutex, Atomics | C++, Java, SharedArrayBuffer |
| Message Passing | Нет | Не нужна | JS Workers, Erlang |
| CSP | Нет | Каналы (blocking) | Go |
| Actor Model | Нет | Mailbox (async) | Erlang, Akka |

> JavaScript использует **message passing** как основную модель (postMessage), но поддерживает **shared memory** через SharedArrayBuffer + Atomics для high-performance сценариев.

## Практическое применение на фронтенде

<b class="text-on-semi-light">Когда выносить в Worker</b>

- Парсинг и обработка больших JSON/CSV/XML.
- Обработка изображений (Canvas → pixel manipulation).
- Crypto-операции (хэширование, шифрование).
- Компиляция/транспиляция (WASM, markdown → HTML).
- Поиск и фильтрация по большим спискам.
- Тяжёлые алгоритмы (сортировка, pathfinding).

<b class="text-on-semi-light">Когда НЕ стоит использовать Worker</b>

- Для простых операций — overhead на создание worker'а и сериализацию данных не оправдан.
- Когда нужен доступ к DOM — worker'ы не имеют доступа к `document`, `window`, `localStorage`.
- Когда данные маленькие — `postMessage` с structured clone для 10 элементов медленнее, чем обработка в main thread.

<b class="text-on-semi-light">Паттерн: Worker Pool</b>

Создание worker'а — не бесплатная операция. Если задачи частые, используют пул заранее созданных worker'ов, распределяя задачи между ними.

```js
class WorkerPool {
  #workers;
  #queue = [];

  constructor(script, size = navigator.hardwareConcurrency || 4) {
    this.#workers = Array.from({ length: size }, () => {
      const w = new Worker(script);
      w.busy = false;
      return w;
    });
  }

  run(data) {
    return new Promise((resolve, reject) => {
      const idle = this.#workers.find(w => !w.busy);
      if (idle) {
        this.#dispatch(idle, data, resolve, reject);
      } else {
        this.#queue.push({ data, resolve, reject });
      }
    });
  }

  #dispatch(worker, data, resolve, reject) {
    worker.busy = true;
    worker.onmessage = (e) => {
      worker.busy = false;
      resolve(e.data);
      if (this.#queue.length) {
        const next = this.#queue.shift();
        this.#dispatch(worker, next.data, next.resolve, next.reject);
      }
    };
    worker.onerror = (e) => {
      worker.busy = false;
      reject(e);
    };
    worker.postMessage(data);
  }
}
```

## Потоки в других языках: краткий обзор

Для общего понимания — как выглядят потоки за пределами JavaScript.

### Go (goroutines)

Горутины — легковесные потоки, управляемые Go runtime (не ОС). Тысячи горутин работают поверх нескольких OS-потоков. Общение — через каналы (CSP).

```go
func worker(id int, ch chan<- int) {
    ch <- id * 2
}

func main() {
    ch := make(chan int, 3)
    for i := 0; i < 3; i++ {
        go worker(i, ch)
    }
    for i := 0; i < 3; i++ {
        fmt.Println(<-ch)
    }
}
```

### Rust (std::thread + ownership)

Rust предотвращает data races **на этапе компиляции** через ownership и borrow checker. Если данные передаются в другой поток, они **перемещаются** (move) — исходный поток теряет к ним доступ.

```rust
use std::thread;

fn main() {
    let data = vec![1, 2, 3];

    let handle = thread::spawn(move || {
        println!("{:?}", data); // data перемещена сюда
    });

    // println!("{:?}", data); // ошибка компиляции: data moved
    handle.join().unwrap();
}
```

### Java (Thread + synchronized)

Классическая модель с OS-потоками. Синхронизация через `synchronized`, `Lock`, `Atomic*`, `volatile`.

```java
class Counter {
    private int count = 0;

    public synchronized void increment() {
        count++;
    }

    public synchronized int getCount() {
        return count;
    }
}
```
