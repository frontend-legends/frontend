# Шифрование и безопасность данных

Шифрование — это процесс преобразования данных в нечитаемую форму, доступную только тому, у кого есть ключ для расшифровки. Для фронтенд-разработчика это не абстрактная теория: HTTPS, JWT, хэширование паролей, цифровые подписи и Web Crypto API — повседневные инструменты, а вопросы про TLS handshake и разницу между encoding/encryption/hashing задают на каждом втором собеседовании.

---

## Encoding vs Hashing vs Encryption

Три фундаментально разных операции, которые часто путают.

| Операция | Обратимость | Ключ | Цель | Примеры |
|---|---|---|---|---|
| **Encoding** | Да, свободно | Нет | Представление данных в другом формате | Base64, URL-encoding, UTF-8 |
| **Hashing** | Нет (одностороннее) | Нет | Получение «отпечатка» фиксированной длины | SHA-256, bcrypt, MD5 |
| **Encryption** | Да, с ключом | Да | Защита конфиденциальности | AES, RSA, ChaCha20 |

<b class="text-on-semi-light">Encoding</b>

<span class="text-gray">Это **не защита**. Base64, hex, URL-encoding — это способ представить данные в другом формате для передачи. Любой может декодировать без секрета. Подробнее о кодировках — [Представление информации](/content/computer-science/data-representation).</span>

```js
btoa('Hello');         // "SGVsbG8=" — Base64 encode
atob('SGVsbG8=');      // "Hello"   — Base64 decode
// Никакого секрета — это просто преобразование формата
```

<b class="text-on-semi-light">Hashing</b>

<span class="text-gray">Односторонняя функция: из входных данных любой длины получается «отпечаток» (digest) фиксированной длины. Невозможно восстановить исходные данные из хэша. Даже минимальное изменение входа полностью меняет результат (лавинный эффект).</span>

<b class="text-on-semi-light">Encryption</b>

<span class="text-gray">Обратимое преобразование с использованием ключа. Только обладатель ключа может расшифровать данные. Бывает симметричное (один ключ) и асимметричное (пара ключей).</span>

## Хэширование

### Свойства криптографической хэш-функции

1. **Детерминированность** — одинаковый вход всегда даёт одинаковый выход.
2. **Быстрое вычисление** — хэш считается за разумное время.
3. **Необратимость (preimage resistance)** — по хэшу нельзя восстановить вход.
4. **Лавинный эффект** — изменение одного бита входа полностью меняет хэш.
5. **Устойчивость к коллизиям** — практически невозможно найти два разных входа с одинаковым хэшем.

```js
// Web Crypto API
async function sha256(message) {
  const data = new TextEncoder().encode(message);
  const hash = await crypto.subtle.digest('SHA-256', data);
  return Array.from(new Uint8Array(hash))
    .map(b => b.toString(16).padStart(2, '0'))
    .join('');
}

await sha256('Hello');
// "185f8db32271fe25f561a6fc938b2e264306ec304eda518007d1764826381969"

await sha256('Hello!'); // один символ — полностью другой хэш
// "334d016f755cd6dc58c53a86e183882f8ec14f52fb05345887c8a5edd42c87b7"
```

### Алгоритмы хэширования

| Алгоритм | Длина выхода | Статус | Применение |
|---|---|---|---|
| **MD5** | 128 бит | Сломан (коллизии) | Только checksum файлов, не для безопасности |
| **SHA-1** | 160 бит | Сломан | Не рекомендуется |
| **SHA-256** | 256 бит | Безопасен | Подписи, целостность, JWT, блокчейн |
| **SHA-3** | 256/512 бит | Безопасен | Альтернатива SHA-2 (другая внутренняя структура) |
| **bcrypt** | 184 бит | Безопасен | Хэширование паролей (медленный by design) |
| **Argon2** | Настраивается | Безопасен | Хэширование паролей (победитель PHC) |

### Хэширование паролей

Обычные хэш-функции (SHA-256) **слишком быстрые** для паролей — атакующий может перебирать миллиарды вариантов в секунду. Для паролей используют **специально медленные** функции: bcrypt, scrypt, Argon2.

```text
Обычный хэш:
  SHA-256("password123") → всегда одинаковый результат
  → уязвим к rainbow tables (предвычисленные хэши)

Хэш с солью (salt):
  bcrypt("password123", random_salt) → уникальный результат каждый раз
  → salt хранится рядом с хэшем, делает rainbow tables бесполезными
```

<b class="text-on-semi-light">Salt (соль)</b>

<span class="text-gray">Случайная строка, добавляемая к паролю перед хэшированием. Даже если два пользователя выбрали одинаковый пароль, их хэши будут разными. Salt хранится в открытом виде рядом с хэшем — её задача не секретность, а уникальность.</span>

<b class="text-on-semi-light">Pepper</b>

<span class="text-gray">Секретная строка, общая для всей системы, добавляемая к паролю **помимо** соли. Хранится отдельно от базы данных (в env-переменной или secret manager). Если утекла БД, но не pepper — хэши бесполезны для атакующего.</span>

### Применения хэширования

- **Проверка целостности** — checksum скачанного файла (`sha256sum file.iso`).
- **Хранение паролей** — хранится хэш, не пароль.
- **Цифровые подписи** — подписывается хэш документа, а не весь документ.
- **Дедупликация** — одинаковые файлы имеют одинаковый хэш.
- **Content-addressable storage** — Git хранит объекты по SHA-1 хэшу содержимого.
- **Subresource Integrity (SRI)** — браузер проверяет хэш загруженного скрипта/стиля.

```html
<!-- SRI: браузер проверит, что файл не изменён -->
<script
  src="https://cdn.example.com/lib.js"
  integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K/uxy9rx7HNQlGYl1kPzQho1wx4JwY8w"
  crossorigin="anonymous"
></script>
```

## Симметричное шифрование

Один и тот же ключ используется для шифрования и расшифровки.

```text
Plaintext ──[ключ K]──> Ciphertext ──[ключ K]──> Plaintext
```

<b class="text-on-semi-light">Преимущества:</b> быстрое, эффективное для больших объёмов данных.
<b class="text-on-semi-light">Проблема:</b> как безопасно передать ключ второй стороне? Если ключ перехвачен — шифрование бесполезно.

### Алгоритмы

| Алгоритм | Тип | Размер ключа | Статус |
|---|---|---|---|
| **AES** (Rijndael) | Блочный | 128/192/256 бит | Стандарт, безопасен |
| **ChaCha20** | Потоковый | 256 бит | Безопасен, быстр на мобильных |
| **DES** | Блочный | 56 бит | Сломан, устарел |
| **3DES** | Блочный | 168 бит | Устарел, медленный |

<b class="text-on-semi-light">Блочные vs потоковые шифры</b>

<span class="text-gray">**Блочный шифр** (AES) обрабатывает данные фиксированными блоками (128 бит). Если данные короче блока — добавляется padding. Если длиннее — разбиваются на блоки и применяется режим работы (CBC, GCM и др.).</span>

<span class="text-gray">**Потоковый шифр** (ChaCha20) генерирует поток псевдослучайных байт и XOR'ит с данными побайтно. Не требует padding, хорошо работает с данными произвольной длины.</span>

### Режимы работы блочных шифров

| Режим | Описание | Безопасность |
|---|---|---|
| **ECB** | Каждый блок шифруется независимо | Небезопасен — одинаковые блоки дают одинаковый ciphertext |
| **CBC** | Каждый блок XOR'ится с предыдущим ciphertext + IV | Безопасен, но не параллелизуется |
| **GCM** | Counter mode + аутентификация (AEAD) | Стандарт для TLS, параллелизуется, проверяет целостность |

> В вебе: TLS 1.3 использует **AES-128-GCM**, **AES-256-GCM** и **ChaCha20-Poly1305**.

### Пример: AES-GCM в браузере (Web Crypto API)

```js
async function encrypt(plaintext, key) {
  const iv = crypto.getRandomValues(new Uint8Array(12)); // 96-bit IV
  const encoded = new TextEncoder().encode(plaintext);

  const ciphertext = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv },
    key,
    encoded
  );

  return { iv, ciphertext };
}

async function decrypt({ iv, ciphertext }, key) {
  const decrypted = await crypto.subtle.decrypt(
    { name: 'AES-GCM', iv },
    key,
    ciphertext
  );

  return new TextDecoder().decode(decrypted);
}

// Генерация ключа
const key = await crypto.subtle.generateKey(
  { name: 'AES-GCM', length: 256 },
  true,
  ['encrypt', 'decrypt']
);

const encrypted = await encrypt('Секретное сообщение', key);
const decrypted = await decrypt(encrypted, key);
// "Секретное сообщение"
```

## Асимметричное шифрование

Используется **пара ключей**: публичный (public) и приватный (private). Что зашифровано одним — расшифровывается только другим.

```text
Шифрование:     Plaintext ──[public key]──>  Ciphertext
Расшифровка:    Ciphertext ──[private key]──> Plaintext

Подпись:        Hash ──[private key]──>  Signature
Проверка:       Signature ──[public key]──> Hash (сравнить)
```

<b class="text-on-semi-light">Преимущества:</b> не нужно передавать секретный ключ — публичный ключ можно раздать всем.
<b class="text-on-semi-light">Проблема:</b> медленнее симметричного в 100–1000 раз. Поэтому на практике используют **гибридную схему**: асимметричное шифрование для обмена симметричным ключом, далее — симметричное шифрование для данных.

### Алгоритмы

| Алгоритм | Основа | Применение |
|---|---|---|
| **RSA** | Факторизация больших чисел | Шифрование, подписи, TLS (legacy) |
| **ECDSA** | Эллиптические кривые | Подписи (TLS, JWT, Bitcoin) |
| **ECDH** | Эллиптические кривые | Обмен ключами (TLS) |
| **Ed25519** | Кривая Curve25519 | Подписи (SSH, современные системы) |

<span class="text-gray">Эллиптические кривые (ECC) обеспечивают такую же безопасность при **значительно меньшем** размере ключа: 256-битный ECC-ключ ~ 3072-битный RSA-ключ.</span>

## Цифровая подпись

Цифровая подпись подтверждает **авторство** и **целостность** данных. Это не шифрование — данные остаются открытыми, но получатель может убедиться, что они не изменены и отправлены именно владельцем приватного ключа.

```text
Отправитель:
  1. Вычислить hash(document)
  2. Зашифровать hash приватным ключом → signature

Получатель:
  1. Вычислить hash(document) самостоятельно
  2. Расшифровать signature публичным ключом → original_hash
  3. Сравнить: hash === original_hash? → подпись верна
```

**Где используется:**
- TLS-сертификаты (подтверждение подлинности сервера).
- JWT (подпись токена).
- npm/git: подпись пакетов и коммитов.
- Code signing: подпись приложений и расширений.

## TLS/SSL и HTTPS

TLS (Transport Layer Security) — протокол, обеспечивающий шифрование, аутентификацию и целостность данных при передаче по сети. HTTPS = HTTP + TLS.

### TLS 1.3 Handshake

```text
Client                                Server
  │                                     │
  │──── ClientHello ──────────────────>│  (поддерживаемые cipher suites,
  │     + key_share (ECDH public)      │   случайное число, supported versions)
  │                                     │
  │<──── ServerHello ─────────────────│  (выбранный cipher suite,
  │      + key_share (ECDH public)     │   certificate, verify)
  │      + EncryptedExtensions         │
  │      + Certificate                 │
  │      + CertificateVerify           │
  │      + Finished                    │
  │                                     │
  │──── Finished ──────────────────────>│
  │                                     │
  │<════ Encrypted Application Data ══>│  (AES-GCM / ChaCha20)
```

<span class="text-gray">TLS 1.3 требует **1 RTT** (round-trip) для установки соединения (в 1.2 было 2 RTT). Поддерживает только безопасные cipher suites: ECDHE для обмена ключами, AES-GCM или ChaCha20-Poly1305 для шифрования. Устаревшие алгоритмы (RSA key exchange, CBC, RC4) полностью удалены.</span>

### Сертификаты и цепочка доверия

```text
Root CA (зашит в браузере/ОС)
  └── Intermediate CA (подписан Root CA)
        └── Сертификат сайта (подписан Intermediate CA)
```

<span class="text-gray">Браузер проверяет цепочку: сертификат сайта подписан Intermediate CA, тот подписан Root CA, а Root CA заранее встроен в ОС/браузер. Если цепочка не выстраивается — соединение отклоняется. Бесплатные сертификаты выдаёт **Let's Encrypt** через автоматический протокол ACME.</span>

## JWT (JSON Web Token)

JWT — формат токена для передачи подписанных данных между сторонами. Широко используется для аутентификации и авторизации в веб-приложениях.

### Структура

```text
header.payload.signature

eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjQyLCJyb2xlIjoiYWRtaW4ifQ.signature
```

```js
// Header (алгоритм подписи)
{ "alg": "HS256", "typ": "JWT" }

// Payload (данные / claims)
{ "userId": 42, "role": "admin", "exp": 1700000000 }

// Signature
HMAC-SHA256(base64url(header) + "." + base64url(payload), secret)
```

<b class="text-on-semi-light">JWT подписан, но НЕ зашифрован</b>

<span class="text-gray">Payload закодирован в Base64 — его может прочитать **любой**. Подпись гарантирует только то, что данные не изменены и созданы владельцем секрета. **Никогда** не помещайте в JWT чувствительные данные (пароли, номера карт), если не используете JWE (JSON Web Encryption).</span>

### Алгоритмы подписи JWT

| Алгоритм | Тип | Ключ | Когда использовать |
|---|---|---|---|
| **HS256** | Симметричный (HMAC) | Общий secret | Один сервис (бэкенд сам создаёт и проверяет) |
| **RS256** | Асимметричный (RSA) | Private / Public | Микросервисы (один подписывает, другие проверяют) |
| **ES256** | Асимметричный (ECDSA) | Private / Public | Как RS256, но компактнее и быстрее |

> Подробнее об аутентификации и сессиях — [Auth & Sessions](/content/network-api/auth-sessions).

## Web Crypto API

Браузерный API для криптографических операций. Работает с `ArrayBuffer` и `CryptoKey`, операции асинхронные. Доступен **только** в secure context (HTTPS или localhost).

| Метод | Назначение |
|---|---|
| `crypto.subtle.digest()` | Хэширование (SHA-256, SHA-384, SHA-512) |
| `crypto.subtle.encrypt()` / `decrypt()` | Шифрование/расшифровка (AES-GCM, RSA-OAEP) |
| `crypto.subtle.sign()` / `verify()` | Цифровая подпись (HMAC, ECDSA, RSA-PSS) |
| `crypto.subtle.generateKey()` | Генерация ключей |
| `crypto.subtle.deriveKey()` | Деривация ключа из пароля (PBKDF2, HKDF) |
| `crypto.subtle.importKey()` / `exportKey()` | Импорт/экспорт ключей |
| `crypto.getRandomValues()` | Криптографически стойкие случайные числа |

```js
// Генерация криптографически стойкого случайного ID
const array = crypto.getRandomValues(new Uint8Array(16));
const id = Array.from(array, b => b.toString(16).padStart(2, '0')).join('');
// "3a7f2b..." (32 hex символа = 128 бит)
```

> `Math.random()` **не** криптографически стойкий — не используйте его для токенов, сессий или чего-либо, связанного с безопасностью.

## Практические аспекты для фронтенда

<b class="text-on-semi-light">HTTPS everywhere</b>

<span class="text-gray">Без HTTPS: данные передаются открытым текстом, возможна подмена (MITM), не работают Service Workers, Geolocation API, Web Crypto API и другие secure-only API.</span>

<b class="text-on-semi-light">Content Security Policy (CSP)</b>

<span class="text-gray">HTTP-заголовок, ограничивающий откуда можно загружать скрипты, стили, изображения. Защищает от XSS-инъекций.</span>

```text
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com
```

<b class="text-on-semi-light">Subresource Integrity (SRI)</b>

<span class="text-gray">Хэш в атрибуте `integrity` тега `<script>` или `<link>`. Если CDN скомпрометирован и файл подменён — браузер отклонит загрузку.</span>

<b class="text-on-semi-light">Хранение секретов на клиенте</b>

<span class="text-gray">Фронтенд — это **недоверенная среда**. Любой код, ключ или секрет в JavaScript доступен пользователю через DevTools. API-ключи, secret'ы подписи и приватные ключи должны жить **только** на бэкенде.</span>
