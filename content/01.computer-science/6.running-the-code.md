# Как выполняется программа

Когда разработчик запускает программу — будь то скрипт на Python, приложение на Java или веб-приложение на JavaScript — начинается сложный, но логичный процесс, состоящий из множества этапов. Чтобы понимать, как работает код и почему он ведёт себя именно так, важно понимать, что происходит под капотом и как устроен flow выполнения.

## Основной цикл выполнения

Когда программа запущена, в действие вступает так называемый **основной цикл выполнения** (execution cycle). Он описывает, как процессор или виртуальная машина (например, JVM, CPython, V8) поэтапно обрабатывает каждую инструкцию программы.

Этот цикл обычно состоит из четырёх шагов:

1. **Fetch (извлечение)** — процессор или интерпретатор получает следующую инструкцию из памяти.
2. **Decode (декодирование)** — определяет, что именно нужно сделать: арифметическую операцию, переход, вызов функции и т.д.
3. **Execute (исполнение)** — выполняет инструкцию. Это может быть вычисление, сравнение, изменение регистра или переход к другому адресу в памяти.
4. **Store (сохранение)** — результат операции записывается обратно в память, регистр или стек.

Такой цикл повторяется миллионы раз в секунду, формируя основу работы любого вычислительного устройства.

## Как JS превращается в выполняемый код

Современные JavaScript-движки (например, **V8** от Google, **SpiderMonkey** от Mozilla, **JavaScriptCore** от Apple) — это сложные системы, сочетающие принципы компиляторов и интерпретаторов. Они обеспечивают высокую скорость выполнения, используя динамическую компиляцию, оптимизацию и генерацию машинного кода на лету.

Процесс выполнения JavaScript-кода в движке **V8** можно условно разделить на несколько последовательных этапов:

1. **Парсинг и лексический анализ (Tokenizer / Parser)**
2. **Построение AST (Abstract Syntax Tree)**
3. **Создание промежуточного представления (Bytecode / IR)**
4. **Интерпретация и сбор профиля исполнения**
5. **JIT-компиляция (Just-In-Time)**
6. **Оптимизация и генерация машинного кода**
7. **Деоптимизация при изменении предположений**

### Лексический анализ и парсинг (Tokenizer / Parser)

Когда JavaScript-файл загружается в браузер или среду Node.js, движок V8 начинает выполнение программы с **анализа исходного кода**. Этот процесс делится на два ключевых этапа:

#### 1. Лексический анализ (Tokenization / Lexing)

На этом шаге движок получает строку исходного кода и **разбивает её на токены** — минимальные смысловые единицы языка.

Примеры токенов:

- Ключевые слова: `let`, `const`, `if`, `return`
- Идентификаторы: `x`, `myFunction`
- Литералы: `42`, `'hello'`
- Операторы и разделители: `+`, `=`, `{`, `}`

**Лексер (Tokenizer)** проходит по коду посимвольно, определяя, какие группы символов соответствуют валидным токенам. Ошибки вроде незакрытых строк или неизвестных символов выявляются уже здесь.

**Пример:**

```js
let x = 2 + 3;
```

Лексер превращает это выражение в последовательность токенов:

```
['let', 'x', '=', '2', '+', '3', ';']
```

#### 2. Синтаксический анализ (Parsing)

После токенизации начинается **парсинг** — процесс построения **дерева синтаксиса (AST, Abstract Syntax Tree)**.

**Парсер** принимает поток токенов и проверяет, соответствует ли их порядок и структура грамматике языка JavaScript. Если код синтаксически неверный (например, `if (x {}`), парсер выбросит ошибку.

AST — это древовидная структура, которая отражает иерархию выражений и операторов программы.

**Пример:**

```js
let x = 2 + 3;
```

Упрощённое AST-представление:

```
VariableDeclaration
├─ Identifier(x)
└─ BinaryExpression(+)
   ├─ Literal(2)
   └─ Literal(3)
```

**Зачем нужно AST:**

- На основе AST выполняются оптимизации.
- Генерация байткода или машинного кода.
- Инструменты вроде Babel, ESLint или TypeScript анализируют и преобразуют код именно через AST.

#### 3. Проверка корректности и подготовка к компиляции

После построения AST движок:

- Проверяет корректность объявления переменных и областей видимости.
- Готовит структуру программы к последующей **компиляции и интерпретации** (создание Execution Context, Scope Chain и т. д.).

Таким образом, **лексер** и **парсер** — это два первых и критически важных шага в работе движка JavaScript, без которых невозможно дальнейшее выполнение программы.

### Промежуточное представление (IR) и байткод (Ignition)

После построения AST движок передаёт дерево интерпретатору **Ignition**, который компилирует его в **байткод (bytecode)** — компактное промежуточное представление, более удобное для исполнения и анализа.

Этот байткод:

- состоит из простых инструкций (например, `LdaSmi`, `Add`, `StaGlobal`),
- оптимизирован под виртуальную машину V8,
- содержит **точки сбора профиля** (inline caches), которые позволяют анализировать поведение кода при исполнении.

Пример байткода для `let x = 2 + 3;`:

```
LdaSmi [2]     ; загрузить число 2
LdaSmi [3]     ; загрузить число 3
Add            ; сложить
StaGlobal [x]  ; сохранить результат в переменной x
```

Ignition выполняет байткод и **собирает профиль исполнения** — сколько раз выполняется функция, какие типы данных встречаются, какие ветви кода чаще всего активны и т. д.

### Интерпретация и сбор профиля исполнения

Пока код исполняется в Ignition, движок отслеживает динамическое поведение программы. Это нужно для **адаптивной оптимизации**:

- какие функции вызываются часто ("hot functions"),
- какие типы данных передаются в аргументах,
- какие выражения можно упростить или предсказать.

Когда функция достигает определённого порога частоты вызовов, Ignition передаёт её в **оптимизирующий компилятор Turbofan**.

### JIT-компиляция (Just-In-Time) — Turbofan

**Turbofan** — это мощный JIT-компилятор, который преобразует часто выполняемые участки байткода в **высокооптимизированный машинный код**. Он работает на основе собранного профиля, что позволяет принимать решения на основе реального исполнения.

Основные этапы работы Turbofan:

- **Построение графа потока данных и управления (CFG)**.
- **Типовая специализация** — если аргументы функции всегда числа, проверки типов можно удалить.
- **Инлайнинг (Inlining)** — подстановка тела часто вызываемых функций прямо в место вызова.
- **Генерация оптимизированного машинного кода**, который напрямую исполняется процессором.

Пример:

```js
function sum(a, b) {
  return a + b;
}
```

Если `a` и `b` всегда числа, JIT создаёт специализированный машинный код, исключающий проверки типов. Это ускоряет выполнение в разы.

### Оптимизация и генерация машинного кода

После анализа профиля и применения оптимизаций Turbofan генерирует машинные инструкции для конкретной архитектуры (x86, ARM и т. д.). Этот код:

- исполняется напрямую процессором,
- хранится в памяти процесса,
- может использоваться повторно, пока не изменятся допущения о типах или структуре кода.

Отличие от классических компиляторов (C/C++, Rust):

- здесь машинный код **не сохраняется** в исполняемый файл;
- весь процесс компиляции и линковки происходит **во время выполнения**;
- управление полностью динамическое и зависит от профиля исполнения.

### Деоптимизация (Deoptimization)

Если предположения, на которых строилась оптимизация, перестают быть верными, движок возвращается к предыдущему состоянию — **деоптимизирует** код.

Пример:

```js
sum(1, 2); // числа — оптимизация активна
sum(1, "2"); // строка — деоптимизация
```

V8 откатывает выполнение к байткоду Ignition, пересобирает профиль и при необходимости снова компилирует функцию с новыми типами данных.

### Итоговый поток исполнения

1. **Исходный JS-код** → **Tokenizer** → **Parser** → **AST**.
2. **AST** → **Ignition** → **Байткод**.
3. **Исполнение байткода** + **сбор профиля**.
4. **Turbofan** → **JIT-компиляция** → **Машинный код**.
5. **Деоптимизация**, если типы или логика изменились.

### Пример: путь JS-кода в браузере

1. **Загрузка:** браузер получает `.js` файл через сетевой стек.
2. **Парсинг:** движок (V8, SpiderMonkey и др.) строит AST.
3. **Байткод:** Ignition (или аналог) создаёт промежуточный байткод.
4. **Исполнение:** байткод интерпретируется.
5. **JIT:** часто вызываемые функции компилируются в машинный код.
6. **Оптимизация:** профиль используется для ускорения.
7. **Деоптимизация:** при изменении типов или поведения.

## Исполнение кода в других языках

Механизм выполнения программ различается в зависимости от парадигмы и архитектуры языка. Ниже — обзор основных подходов в популярных экосистемах.

**C / C++ — компиляция в машинный код:**

C и C++ используют классическую модель **ahead-of-time компиляции (AOT)**. Код полностью компилируется до машинных инструкций ещё до запуска:

1. **Компилятор (clang/gcc)** — переводит исходный код в **объектные файлы (.o)**.
2. **Линковщик (linker)** — объединяет объектные файлы и библиотеки в один исполняемый файл.
3. **Исполнение** — CPU выполняет бинарный код напрямую, без промежуточных слоёв.

Преимущество — высокая производительность. Недостаток — сложность портирования и отладки.

**Java — байткод и JVM:**

Java использует **виртуальную машину (JVM)**, где код сначала компилируется в **байткод (.class)**, а уже затем исполняется JVM на любой платформе.

- JVM содержит **интерпретатор и JIT-компилятор (HotSpot)**.
- Часто выполняемые участки кода оптимизируются «на лету» в машинный код.

Такой подход даёт баланс между переносимостью и производительностью.

**Python — интерпретация и байткод:**

Python не исполняет исходный код напрямую. Интерпретатор (например, **CPython**) делает следующее:

1. Парсит исходный код и создаёт **байткод (.pyc)**.
2. Этот байткод интерпретируется **виртуальной машиной Python (PVM)**.

Благодаря этому Python гибкий и прост в использовании, но уступает в скорости компилируемым языкам.

**Rust — компиляция в нативный код:**

Rust, как и C/C++, использует **AOT-компиляцию** через LLVM. Код полностью превращается в машинные инструкции до запуска.

- Rust делает упор на **безопасность памяти** без garbage collector.
- В отличие от C++, компилятор Rust проводит строгий **семантический анализ** на этапе компиляции.

**Go — компиляция с встроенным рантаймом:**

Go компилируется в нативный код, но поставляется со своим **встроенным рантаймом**, включающим планировщик горутин и garbage collector.

- Код компилируется быстро и создаёт статически связанный бинарник.
- Выполнение происходит напрямую на CPU, но рантайм управляет потоками и памятью.

**.NET (C#, F#) — CLR и IL-код:**

Как и Java, экосистема .NET компилирует исходный код в **Intermediate Language (IL)**.

- При запуске CLR (Common Language Runtime) использует **JIT-компиляцию**.
- Современные версии .NET могут также использовать **AOT (NativeAOT)** для ускорения запуска.

**WebAssembly — универсальный бинарный формат:**

**WebAssembly (Wasm)** — низкоуровневый формат, работающий в браузере и вне его.

- Код компилируется из языков вроде C, Rust, Go в Wasm-модуль.
- Браузер (или рантайм) выполняет его почти с нативной скоростью.

Wasm объединяет преимущества производительности нативного кода и безопасности веб-песочницы.

**Ruby, PHP, Lua — интерпретируемые языки:**

Эти языки, как и Python, работают через интерпретаторы:

- Исходный код компилируется в **байткод**, который выполняется виртуальной машиной.
- В некоторых реализациях (например, Ruby YARV или PHP OPcache) добавлены оптимизации и JIT.

Главное преимущество — гибкость и простота. Недостаток — ограниченная производительность.

## Поток выполнения программы

После компиляции или интерпретации программа начинает выполняться — пошагово, инструкция за инструкцией. Процессор считывает команды из памяти, выполняет их и переходит к следующим. Этот процесс может быть полностью управляем самим процессором (в низкоуровневых языках, как C/C++) или с помощью специальной **среды выполнения (runtime)** — в языках вроде JavaScript, Python, Java.

### Что такое runtime

**Runtime** — это программная среда, которая обеспечивает выполнение кода и взаимодействие программы с системой. Она управляет:

- выделением и освобождением памяти (через **heap** и **garbage collector**),
- обработкой ошибок и исключений,
- планированием задач и событий,
- взаимодействием с внешними API (файлы, сеть, UI и т. д.),
- выполнением абстракций, которых нет на уровне процессора (например, замыканий, async/await, классов).

Примеры runtime:

- **JavaScript:** V8 (в браузерах и Node.js) + Web API / Node API.
- **Python:** CPython runtime.
- **Java:** JVM (Java Virtual Machine).
- **.NET:** CLR (Common Language Runtime).

Runtime можно представить как “мини-операционную систему” для языка программирования — он управляет жизненным циклом вашего кода, следит за памятью и событиями.

### Компоненты выполнения в JavaScript

В случае V8 и браузера, выполнение кода опирается на несколько ключевых механизмов:

- **Call Stack** — стек вызовов. Каждая активная функция помещается в стек при вызове и удаляется, когда завершается.
- **Heap** — динамическая память, где хранятся объекты, массивы, функции и всё, что не имеет фиксированного размера.
- **Event Loop** — бесконечный цикл, который проверяет, есть ли задачи для выполнения, и управляет взаимодействием между стеком и очередями задач.
- **Task Queues** — очереди задач, где хранятся запланированные действия:
  - **Macrotasks:** `setTimeout`, `setInterval`, `fetch`, I/O.
  - **Microtasks:** `Promise.then`, `queueMicrotask`, `MutationObserver`.

> Но об этом более детально разберем в главе про [7. Javascript](/content/javascript)

### Асинхронность и события

Асинхронность — это механизм, который позволяет программе не ждать завершения долгих операций. Вместо блокировки выполнения, задача отправляется в систему (например, сетевой запрос), а код продолжает выполняться. Когда результат готов, событие попадает в очередь, и **event loop** решает, когда его обработать.

Это позволяет JavaScript эффективно работать с пользовательским интерфейсом, не замораживая страницу, и обрабатывать тысячи событий без создания дополнительных потоков.

### Ошибки времени выполнения

Даже при корректной компиляции программа может упасть во время исполнения. Это называется **runtime error** — ошибка, произошедшая уже в процессе выполнения:

- деление на ноль,
- обращение к несуществующему свойству,
- выход за границы массива,
- утечки памяти.

В разных языках они проявляются по-разному:

- JavaScript — `RuntimeError`, `TypeError`, `ReferenceError`.
- Python — `Exception`, `ValueError`.
- C — `segmentation fault`.

Чтобы такие ошибки не останавливали всю программу, используются конструкции вроде `try/catch`, обработчики событий ошибок или логирование.

### Отладка и анализ потока выполнения

Понимание **execution flow** (потока выполнения) — важнейший навык разработчика. Это способность проследить, как данные и функции взаимодействуют в программе, какие операции выполняются синхронно, а какие — асинхронно.

Для этого используются инструменты:

- **DevTools (Chrome, Firefox)** — позволяет пошагово отслеживать выполнение JS-кода, смотреть стек вызовов и состояние переменных.
- **GDB / LLDB** — для низкоуровневых языков.
- **PyCharm Debugger**, **VS Code Debugger** — для Python и TypeScript.

Анализируя поток, разработчик может понять не только _что_ происходит, но и _почему_ — где функция зависает, почему не срабатывает колбэк, как распределяются ресурсы и память.

## Поток и порядок выполнения кода

Каждая программа — это набор инструкций, которые выполняются в определённом порядке. Этот порядок называют **потоком управления (control flow)**. Он определяет, какая строка кода будет выполнена следующей.

- **Последовательность**.
- **Ветвления**.
- **Циклы**.
- **Функции и вызовы**.
- **Исключения и ошибки**.
- **Асинхронность**.

### Последовательность

Код по умолчанию выполняется сверху вниз — инструкция за инструкцией. Например:

```js
let x = 5;
let y = x + 2;
console.log(y);
```

Каждая строка исполняется строго после предыдущей, пока не встретится оператор, который изменяет этот порядок — например, условие или цикл.

### Ветвления

Ветвления позволяют выполнять разные участки кода в зависимости от условий. Используются конструкции `if`, `else if`, `switch` и тернарные операторы.

```js
if (x > 10) {
  console.log("Большое число");
} else {
  console.log("Маленькое число");
}
```

Благодаря ветвлениям программа реагирует на данные и принимает решения во время выполнения.

### Циклы

Циклы позволяют выполнять один и тот же блок кода несколько раз. Это важно для обработки массивов, ожидания событий и повторяющихся вычислений.

```js
for (let i = 0; i < 5; i++) {
  console.log(i);
}
```

В JavaScript и других языках также есть `while`, `do...while`, а иногда и функциональные аналоги вроде `.forEach()` или `.map()`.

### Функции и вызовы

Функции — это именованные блоки кода, которые можно вызывать многократно. При вызове функции создаётся новая запись в **стеке вызовов (Call Stack)**.

```js
function greet(name) {
  console.log("Hello, " + name);
}

greet("Alice");
```

Когда функция завершается, её контекст удаляется из стека, и выполнение возвращается к месту вызова.

### Исключения и отклонения потока

Иногда поток может быть **прерван** из-за ошибки или намеренного исключения (`throw`). При этом выполнение текущего блока останавливается, и управление передаётся в ближайший обработчик (`try...catch`).

```js
try {
  throw new Error("Something went wrong");
} catch (err) {
  console.error(err.message);
}
```

Так реализуется **нестандартное ветвление** — управление выходит за пределы обычного линейного потока, чтобы обработать сбой или восстановить стабильное состояние программы.

Это механизм **обработки исключений**, позволяющий программе продолжать выполнение, даже если в одной из частей произошёл сбой.

### Асинхронность

Асинхронность нарушает линейный поток выполнения — код может «прыгать во времени». В JavaScript это реализуется через **колбэки**, **промисы** и **async/await**.

```js
console.log("Start");

setTimeout(() => {
  console.log("Async");
}, 1000);

console.log("End");
```

Результат выполнения:

```
Start
End
Async
```

### Переходы и `goto`

На низком уровне (в ассемблере или C) существует инструкция `goto`, которая позволяет **прыгать** на произвольную часть программы. Это прямое управление потоком, минуя привычные структуры вроде `if` или `for`.

```c
start:
  printf("Looping...\n");
  goto start;
```

Хотя такие переходы гибкие, они делают код трудно читаемым и непредсказуемым. В современных языках `goto` практически не используется — вместо этого применяются **структурированные потоки**: функции, условия, исключения, циклы и обработка событий.

### 1. Лексический анализ (лексер, tokenizer)

Цель: Разбить поток символов исходного текста на последовательность _токенов_ — минимальных осмысленных единиц языка: ключевые слова, идентификаторы, литералы, операторы, разделители.

Что делает:

- Убирает пробелы и комментарии (во многих языках).
- Выделяет токены по правилам грамматики (обычно описанным регулярными выражениями).
- Генерирует позиционную информацию (номер строки, столбец) — важна для качественных сообщений об ошибках.

Примеры токенов:

```
int main() { return 0; }
```

→ `INT` `IDENT(main)` `LPAREN` `RPAREN` `LBRACE` `RETURN` `INT_LITERAL(0)` `SEMICOLON` `RBRACE`

### Технические детали

- Лексер часто реализуют как конечный автомат (DFA) построенный из регулярных выражений.
- Поддерживает экранирование строк, многострочные литералы, числовые форматы (целые, с плавающей точкой, hex), символьные литералы.
- Обрабатывает предобработчики (например, `#include`/`#define` в C) на отдельной стадии или как часть лексера.

### Ошибки на этом этапе

- Неправильный токен (неожиданный символ).
- Некорректно закрытая строка или комментарий.

---

## 2) Синтаксический анализ (парсер)

### Цель

Проверить, что последовательность токенов соответствует синтаксису языка, и построить структурное представление — **Abstract Syntax Tree (AST)**.

### Что делает

- Собирает токены в выражения, операторы, объявления и блоки кода.
- Строит дерево, где узлы — конструкции языка (if, for, call, assignment, function, class и т.д.).

### Виды парсеров

- **LL-парсеры** (top-down, например, recursive descent) — читают токены слева направо и строят дерево сверху вниз. Просты и удобны для ручной реализации.
- **LR-парсеры** (bottom-up, например, LALR, Canonical LR) — более мощные, способны разбирать большинство контекстно-свободных грамматик и часто используются в генераторах парсеров (Yacc/Bison).

### Пример AST (псевдо-формат)

Исходник:

```
if (x > 0) {
  y = x + 1;
} else {
  y = 0;
}
```

AST:

```
If
├─ Condition: BinOp(>, Ident(x), Int(0))
├─ Then: Block( Assign(y, BinOp(+, Ident(x), Int(1))) )
└─ Else: Block( Assign(y, Int(0)) )
```

### Обработка неоднозначностей

- Парсер применяет правила приоритета операторов и ассоциативности.
- Для некоторых языков требуется разрешение неоднозначностей на этапе семантики (например, `typedef` vs `identifier` в C).

### Ошибки на этапе парсинга

- Синтаксические ошибки: пропущенные `;`, скобки, неверная структура.
- Сообщения должны содержать ожидаемые токены и позицию.

## 3) Семантический анализ

### Цели

- Проверить смысловую корректность программы (типизация, видимость имён, корректность вызовов).
- Построить таблицы символов (symbol tables), связи имя → определение.
- Выполнить простые проверки времени компиляции (const correctness, final, access modifiers).

### Основные операции

- **Resolution / Name binding** — связывание всех идентификаторов с их объявлениями (переменные, функции, классы, модули).
- **Type checking** — статическая проверка типов (явная/неявная), вывод типов (type inference) при поддержке языком.
- **Проверка областей видимости (scopes)** — локальные, глобальные, лексические замыкания.
- **Проверка корректности вызовов** — сопоставление аргументов и параметров, перегрузки.
- **Проверки правил языка** — final/const, модификаторы доступа, правила наследования.

### Примеры ошибок

- Необъявленная переменная.
- Несовместимые типы в присвоении.
- Неверное число аргументов у функции.

### Дополнительные шаги

- Построение и аннотирование AST информацией о типах и связях.
- Валидация атрибутов (аннотаций), проверка макросов и расширений.

## 4) Промежуточное представление (IR) и трансформации

### Почему нужен IR

AST удобен для синтаксических и семантических проверок, но плохо подходит для оптимизаций и генерации кода. IR — это уровень между синтаксисом и машинным кодом, более низкоуровневый и формализованный.

### Виды IR

- **Three-Address Code (TAC)** — инструкции вида `t1 = a + b`.
- **Static Single Assignment (SSA)** — форма, в которой каждая переменная присваивается один раз (позволяет упростить анализ и оптимизации).
- **Bytecode** — упакованный, платформонезависимый набор инструкций (пример: JVM bytecode).
- **Lower-level IR** — похож на ассемблер, но независим от целевой архитектуры.

### Пример трансляции выражения

Исходник:

```
z = (a + b) * (c - d);
```

TAC:

```
t1 = a + b
t2 = c - d
z = t1 * t2
```

## 5) Оптимизации

Оптимизации делят на уровни: локальные (внутри basic block), глобальные (на уровне функции), межпрограммные (между модулями), а также target-specific.

### Локальные оптимизации

- **Peephole optimization** — небольшие локальные замены последовательностей инструкций для упрощения.
- **Constant folding** — вычисление констант на этапе компиляции (`2+3 -> 5`).
- **Constant propagation** — распространение значений констант через переменные.

### Глобальные оптимизации

- **Dead code elimination (DCE)** — удаление кода, который никогда не выполняется или не влияет на результат.
- **Common subexpression elimination (CSE)** — вычисления, повторяющиеся в разных местах, выносятся в переменную.
- **Loop invariant code motion** — перемещение выражений из цикла, если их значение не меняется.
- **Inlining** — разворачивание вызовов функций, чтобы убрать накладные расходы на вызов.
- **Loop unrolling** — разворачивание итераций цикла для уменьшения overhead.

### SSA и анализ данных

- Преобразование в SSA упрощает многие оптимизации.
- **Data-flow analysis** (например, live variable analysis) используется для определения, какие переменные живы и где возможна DCE.

### Межпроцедурные оптимизации

- **Interprocedural analysis** — оптимизации с учётом нескольких функций/модулей (inline across modules, constant propagation across calls).
- **Escape analysis** — решение о выделении объектов в куче vs на стеке.

### Целевые оптимизации

- Vectorization (SIMD), auto-parallelization — преобразования для ускорения на конкретных процессорах.

## 6) Построение графа потока управления (CFG) и SSA

### CFG

- Basic block — последовательность инструкций без ветвлений внутри.
- Узлы CFG — basic blocks; ребра — возможные переходы управления.
- CFG используется для анализа путей выполнения, поиска циклов, определения достижимости.

### SSA

- Каждой переменной даётся уникальное имя при присвоении. Это облегчает анализ зависимостей и применение оптимизаций.
- Вводится оператор `phi` для слияния значений из разных ветвей.

## 7) Генерация кода / бекэнд

### Что происходит

- IR трансформируется в инструкции, специфичные для целевой архитектуры (x86-64, ARM, WebAssembly и т.д.).
- Выполняется **selection** (подбор инструкций), **scheduling** и **алокация регистров**.

### Аллокация регистров

- Проблема: у ограниченного числа регистров нужно разместить больше виртуальных переменных.
- Частый алгоритм: **graph coloring** на interference graph (переменные, мешающие друг другу, не могут использовать один регистр).
- Альтернатива: spill — выгружать часть переменных в стек.

### Calling conventions и ABI

- Генерация кода должна учитывать соглашения о вызове: какие аргументы передаются в регистрах, какие — через стек, кто отвечает за очистку стека.
- ABI (Application Binary Interface) гарантирует совместимость с другими объектными файлами и библиотеками.

### Пример трёхадресного кода → инструкции

TAC:

```
t1 = a + b
t2 = t1 * c
```

x86-псевдо-инструкции:

```
mov rax, [a]
add rax, [b]
imul rax, [c]
mov [result], rax
```

## 8) Сборка объектных файлов и линковка

### Объектные файлы

- Каждый скомпилированный модуль обычно превращается в объектный файл (.o, .obj) содержащий машинный код, таблицу символов и записи для релокации.

### Линковка

- **Static linking** — все нужные объекты и библиотеки объединяются в один исполняемый файл.
- **Dynamic linking** — генерируются ссылки на внешние библиотеки (shared libs / DLLs). Разрешение символов происходит во время загрузки или исполнения.
- **Relocation** — изменение адресов в машинном коде в соответствии с реальными адресами сегментов при сборке или загрузке.

### Форматы

- Популярные форматы: **ELF** (Linux), **PE** (Windows), **Mach-O** (macOS). Они содержат разделы, таблицы символов, релокации и метаданные.

## Ошибки и сообщения компилятора

Хороший компилятор даёт полезные сообщения:

- Позиционная информация (файл:строка:столбец).
- Контекст (строка исходника с указателем позиции).
- Подсказки (возможные исправления, похожие имена символов).

Примеры ошибок: синтаксические, типовые, link-time errors (не найден символ).

---

Оптимизации на этапе линковки (LTO)

**Link Time Optimization (LTO)** — позволяет выполнять оптимизации между модулями на этапе линковки, когда доступен весь код программы. Это даёт дополнительные возможности inline, DCE и интерпроцедурных преобразований.

---

Дополнительные понятия

- **Position Independent Code (PIC)** — код, который можно загружать по разным адресам (важно для shared libraries).
- **Debug info (DWARF)** — метаданные для отладки: соответствие между машинным кодом и исходным исходником.
- **Profile-guided optimization (PGO)** — компилятор оптимизирует код, опираясь на реальные профили выполнения.
