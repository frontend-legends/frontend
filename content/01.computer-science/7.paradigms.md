# Парадигмы программирования

Парадигма программирования — это не конкретный язык или фреймворк, а **способ мышления** и **метод организации кода**. Она определяет, как программист формулирует алгоритмы, описывает данные и взаимодействует с системой.

Понимание парадигм помогает инженеру мыслить шире:

- выбирать правильный инструмент под задачу;
- осознанно проектировать архитектуру;
- писать читаемый, поддерживаемый и расширяемый код;
- свободно переключаться между стилями, не застревая в одном подходе.

Современная разработка — особенно фронтенд — мультипарадигменна. Одни и те же библиотеки (например, React или Vue) объединяют декларативные, функциональные, реактивные и объектные идеи.

---

## Исторический контекст

Парадигмы программирования эволюционировали вместе с развитием вычислительной техники и ростом сложности программ.

- **1950–1960-е**: эпоха **императивного программирования**. Появились Fortran, Lisp, C — программисты напрямую управляли машиной, описывая последовательность инструкций.
- **1970–1980-е**: формируется **объектно-ориентированное мышление** (Smalltalk, C++). Мир стремился к модульности и повторному использованию кода.
- **1990-е**: вместе с ростом веба появляются **событийные и реактивные** подходы — обработка пользовательских событий и асинхронности.
- **2000–2010-е**: идеи **функционального программирования** (чистые функции, иммутабельность, композиция) проникают в популярные языки: JavaScript, Python, Java, C#.
- **2020-е**: границы между парадигмами размываются. Языки становятся гибридными, совмещая императивность, декларативность и функциональные концепции (например, Rust, Kotlin, TypeScript).

---

## Базовые категории парадигм

Все парадигмы можно условно разделить на две мета-группы:

| Категория         | Описание                                                                                       | Примеры                  |
| ----------------- | ---------------------------------------------------------------------------------------------- | ------------------------ |
| **Императивные**  | Определяют _**как**_ достичь результата. Программист управляет состоянием и шагами выполнения. | C, C++, Java, Go, Python |
| **Декларативные** | Определяют _**что**_ нужно получить, не указывая пошагово, как это сделать.                    | SQL, HTML, Haskell       |

Остальные парадигмы (ООП, ФП, реактивное, процедурное и др.) — конкретные реализации внутри этих подходов.

Пример:

- ООП и процедурное программирование — подтипы **императивного**.
- ФП и реактивное — чаще встречаются в **декларативных** контекстах.

Схематично это выглядит так:

```
Программирование
├── Императивное
│   ├── Процедурное
│   ├── Объектно-ориентированное
│   └── Реактивное (частично)
└── Декларативное
    ├── Функциональное
    ├── Логическое (Prolog)
    └── Реактивное (частично)
```

## Императивное программирование

Императивное программирование — один из старейших и наиболее интуитивных подходов. Программист **пошагово описывает процесс** выполнения программы: какие данные использовать, в каком порядке и как их изменять.

Примеры в JavaScript:

```js
let total = 0;
for (let i = 0; i < 5; i++) {
  total += i;
}
```

Здесь мы явно управляем состоянием (`total`) и последовательностью действий (инициализация, цикл, приращение).

**Во фронтенде:**

- манипуляции DOM напрямую: `document.querySelector().appendChild()`;
- imperative API в Canvas или WebGL.

**Плюсы:**

- полный контроль над процессом выполнения;
- высокая производительность в простых задачах.

**Минусы:**

- трудно масштабировать и поддерживать большие системы;
- код часто перегружен деталями реализации.

---

## Декларативное программирование

Декларативное программирование описывает **что** нужно получить, а не **как** это достичь. Программист определяет результат, а конкретные шаги берёт на себя язык, фреймворк или движок.

Примеры:

- React — UI как функция состояния:

  ```jsx
  <TodoList items={todos.filter((t) => !t.completed)} />
  ```

- SQL — запрос данных без указания механизма выборки:

  ```sql
  SELECT name FROM users WHERE active = true;
  ```

**Во фронтенде:** Vue, React, Svelte и Solid используют декларативные шаблоны и реактивность.

**Плюсы:**

- лаконичность, читаемость, меньше побочных эффектов;
- проще тестировать и масштабировать.

**Минусы:**

- меньше контроля над внутренними механизмами;
- иногда сложно оптимизировать или отлаживать.

## Процедурное программирование

Процедурное программирование — это один из подтипов **императивного** подхода, где программа делится на **последовательности процедур (функций)**, каждая из которых выполняет отдельную задачу.

Процедура (или функция) — это именованный блок кода, который можно вызывать из разных мест программы, что помогает **избегать дублирования** и **улучшает читаемость**.

**Ключевые идеи:**

- программа состоит из набора функций (процедур);
- используется передача данных через параметры и возвращаемые значения;
- управление потоком осуществляется вызовами функций и условными операторами.

**Пример (JavaScript):**

```js
function calculateSum(a, b) {
  return a + b;
}

function printResult() {
  const result = calculateSum(3, 5);
  console.log("Сумма:", result);
}

printResult();
```

Здесь каждая функция выполняет конкретную задачу, а программа управляется последовательными вызовами этих процедур.

**Преимущества:**

- простота понимания и обучения;
- структурированность и переиспользование кода;
- хорошо подходит для небольших и средних проектов.

**Недостатки:**

- с ростом проекта управление зависимостями и состоянием усложняется;
- слабая модульность по сравнению с ООП (данные и функции разделены);
- трудно масштабировать без строгой архитектурной дисциплины.

**Где используется:**

- в системном программировании (C);
- в утилитах и скриптах, где важна простота и скорость написания;
- во фронтенде — в виде вспомогательных функций и утилитарных модулей.

## Функциональное программирование (FP)

Функциональное программирование основано на математических принципах. Программа рассматривается как комбинация функций, не изменяющих внешнее состояние.

**Ключевые принципы:**

- чистые функции (без побочных эффектов);
- иммутабельность данных;
- композиция и функции высшего порядка;
- ленивые вычисления.

**Примеры во фронтенде:**

- использование `map`, `filter`, `reduce` вместо циклов;
- библиотека Ramda, Immer или lodash/fp;
- Redux — чистые редьюсеры для управления состоянием.

**Плюсы:**

- предсказуемость и простота тестирования;
- высокая переиспользуемость и читаемость.

**Минусы:**

- требует абстрактного мышления и дисциплины;
- может быть сложен для начинающих.

> Подробнее про функциональное программирование — [Functional Programming](/content/computer-science/fp).

## Объектно-ориентированное программирование (OOP)

ООП строится вокруг концепции **объектов**, которые объединяют данные (состояние) и методы (поведение).

**Основные принципы:**

1. **Инкапсуляция** — сокрытие деталей реализации внутри класса.
2. **Наследование** — возможность создавать новые классы на основе существующих.
3. **Полиморфизм** — разные объекты могут иметь общий интерфейс с разным поведением.

**Примеры:**

- Классы и интерфейсы в TypeScript;
- Классовые компоненты в React (до Hooks);
- Модели доменных сущностей в backend-разработке.

**Плюсы:**

- понятная организация кода;
- подходит для сложных бизнес-доменов.

**Минусы:**

- избыточность, склонность к "overengineering";
- может мешать гибкости при быстром прототипировании.

> Подробнее про объектно-ориентированное — [Object Oriented Programming](/content/computer-science/oop).

## Реактивное программирование

Реактивное программирование рассматривает программу как **поток данных во времени**. Состояние не меняется вручную — оно реагирует на события.

**Примеры:**

- RxJS в Angular;
- Signals во Vue 3.4+;
- React Hooks и context updates — формы реактивности.

**Плюсы:**

- естественная работа с асинхронностью;
- декларативное управление потоками событий.

**Минусы:**

- высокий порог вхождения;
- сложность отладки при большом количестве потоков.

---

## Смешанные подходы

Современные языки редко следуют одной парадигме:

- **JavaScript / TypeScript** — мультипарадигменные, поддерживают императивный, объектный и функциональный стили.
- **React** — декларативность + функциональные принципы.
- **Vue** — реактивность + шаблоны + возможность императивного кода.
- **Rust** — строгая типизация, иммутабельность, но императивный контроль.

Эта гибкость позволяет адаптировать стиль под задачу, но требует от разработчика осознанности в выборе подхода.

## Сравнение языков по парадигмам

| Язык                        | Парадигмы                            | Особенности                                             |
| --------------------------- | ------------------------------------ | ------------------------------------------------------- |
| **JavaScript / TypeScript** | Императивная, декларативная, ООП, ФП | Универсальные, но эклектичные.                          |
| **Python**                  | Императивная, ООП, ФП                | Прост в освоении, гибкий, с элементами декларативности. |
| **Java / C#**               | ООП → с элементами ФП                | Сильная типизация, эволюция в сторону функциональности. |
| **Haskell / Elm**           | Чистое ФП                            | Математически строгие, декларативные языки.             |
| **Rust**                    | Императивная + ФП                    | Безопасность памяти + иммутабельность.                  |
| **Go**                      | Императивная                         | Простота, минимализм, отказ от тяжёлых концепций.       |

## Заключение

Современный инженер не ограничен одной парадигмой. Напротив — понимание их взаимодействия делает вас сильнее: вы начинаете осознанно выбирать **не только язык**, но и **стиль мышления**, соответствующий задаче.

Понимание парадигм — это шаг от написания кода к **проектированию систем**.
