# Архитектура операционных систем

Операционная система (ОС) — это слой между оборудованием и приложениями, который управляет процессами, памятью, файлами, устройствами и безопасностью. Для фронтенд-разработчика знание ОС помогает понимать, как браузер взаимодействует с системой, почему process isolation важен для безопасности вкладок, как работает виртуальная память и что происходит, когда Node.js делает файловый I/O.

---

## Основные функции ОС

| Функция                     | Что делает                                             | Пример для веб-разработчика                         |
| --------------------------- | ------------------------------------------------------ | --------------------------------------------------- |
| **Управление процессами**   | Создание, планирование, завершение процессов и потоков | Каждая вкладка Chrome — отдельный процесс           |
| **Управление памятью**      | Виртуальная память, выделение/освобождение, защита     | V8 запрашивает память у ОС для heap                 |
| **Файловая система**        | Хранение, организация, права доступа                   | `fs.readFile()` в Node.js → системный вызов         |
| **Управление устройствами** | Драйверы, абстракция оборудования                      | Доступ к камере через `getUserMedia()`              |
| **Сеть**                    | TCP/IP стек, сокеты, DNS resolver                      | `fetch()` → сокет → TCP → IP → сеть                 |
| **Безопасность**            | Права, изоляция, шифрование                            | Sandbox браузера, запрет доступа к файловой системе |

## Пространство ядра и пространство пользователя

Фундаментальное разделение в архитектуре любой ОС.

<b class="text-on-semi-light">Kernel Space (пространство ядра)</b>

<span class="text-gray">Привилегированный режим процессора. Код ядра имеет полный доступ к оборудованию: памяти, дискам, сетевым интерфейсам, процессорным инструкциям. Ошибка в kernel space = kernel panic / BSOD — падает вся система.</span>

<b class="text-on-semi-light">User Space (пространство пользователя)</b>

<span class="text-gray">Непривилегированный режим. Все обычные программы (браузер, Node.js, терминал) работают здесь. Доступ к оборудованию — только через **системные вызовы** (syscalls), которые передают управление ядру.</span>

```text
┌───────────────────────────────────┐
│         User Space                │
│  ┌──────┐ ┌──────┐ ┌──────────┐  │
│  │Chrome│ │Node.js│ │ Terminal │  │
│  └──┬───┘ └──┬───┘ └────┬─────┘  │
│     │        │           │        │
│─────┼────────┼───────────┼────────│  ← системные вызовы (syscalls)
│     ▼        ▼           ▼        │
│           Kernel Space            │
│  ┌─────────────────────────────┐  │
│  │  Процессы │ Память │ FS │ I/O │  │
│  └─────────────────────────────┘  │
│              │                    │
│         Hardware                  │
│  CPU   RAM   Disk   Network      │
└───────────────────────────────────┘
```

### Системные вызовы (Syscalls)

Каждый раз, когда приложение обращается к ресурсам ОС, происходит системный вызов — переход из user space в kernel space и обратно. Это **дорогая операция** (переключение контекста), поэтому минимизация syscalls — один из способов оптимизации.

| Действие приложения    | Syscall (Linux)                              | Что делает ядро                   |
| ---------------------- | -------------------------------------------- | --------------------------------- |
| `fs.readFile()`        | `open()`, `read()`, `close()`                | Обращение к файловой системе      |
| `fetch()`              | `socket()`, `connect()`, `write()`, `read()` | TCP-соединение через сетевой стек |
| `new ArrayBuffer(1MB)` | `mmap()` или `brk()`                         | Выделение виртуальной памяти      |
| `console.log()`        | `write()` (fd=1)                             | Запись в stdout                   |

## Архитектурные модели ядра

<b class="text-on-semi-light">Монолитное ядро</b>

<span class="text-gray">Все сервисы ОС (управление памятью, файловая система, драйверы, сетевой стек, планировщик) работают в **одном адресном пространстве** ядра. Обмен данными между компонентами — через вызовы функций (быстро), но ошибка в любом компоненте может обрушить всю систему.</span>

Примеры: **Linux**, **FreeBSD**.

<b class="text-on-semi-light">Микроядро</b>

<span class="text-gray">В ядре — минимум: управление памятью, IPC (межпроцессное взаимодействие), базовое планирование. Все остальные сервисы (драйверы, файловая система, сетевой стек) — отдельные процессы в user space. Обмен — через сообщения (IPC). Надёжнее (crash драйвера не убивает ядро), но медленнее из-за overhead IPC.</span>

Примеры: **Minix**, **QNX**, **seL4**.

<b class="text-on-semi-light">Гибридное ядро</b>

<span class="text-gray">Компромисс: часть сервисов в ядре (для производительности), часть — в user space (для надёжности). Большинство коммерческих ОС используют этот подход.</span>

Примеры: **Windows NT/10/11**, **XNU (macOS, iOS)**.

| Модель     | Производительность  | Надёжность                 | Примеры        |
| ---------- | ------------------- | -------------------------- | -------------- |
| Монолитное | Высокая (нет IPC)   | Ниже (одна ошибка = panic) | Linux, FreeBSD |
| Микроядро  | Ниже (IPC overhead) | Высокая (изоляция)         | QNX, seL4      |
| Гибридное  | Средняя             | Средняя                    | Windows, macOS |

## Управление процессами

### Процесс

Процесс — экземпляр запущенной программы. У каждого процесса:

- **PID** (Process ID) — уникальный идентификатор.
- **Адресное пространство** — изолированная виртуальная память (code, heap, stack, data).
- **Ресурсы** — файловые дескрипторы, сокеты, таймеры.
- **Состояние** — набор регистров процессора, instruction pointer.

### Жизненный цикл процесса

```text
    ┌──────┐   fork/exec   ┌───────┐
    │ New  │──────────────>│ Ready │◄────────────┐
    └──────┘               └───┬───┘             │
                               │ scheduler       │ I/O complete /
                               ▼ dispatch        │ event
                          ┌─────────┐        ┌───┴──────┐
                          │ Running │───────>│ Waiting  │
                          └────┬────┘ I/O /  └──────────┘
                               │      event
                               ▼
                         ┌────────────┐
                         │ Terminated │
                         └────────────┘
```

### Планировщики (Schedulers)

Планировщик решает, какой процесс/поток получит CPU.

<b class="text-on-semi-light">Round Robin</b>

<span class="text-gray">Каждый процесс получает равный квант времени (time slice, обычно 1–10 мс). По истечении — переключение на следующий. Справедливо, но не учитывает приоритеты.</span>

<b class="text-on-semi-light">Priority Scheduling</b>

<span class="text-gray">Процессы с высоким приоритетом обслуживаются первыми. Проблема: starvation — низкоприоритетные процессы могут никогда не получить CPU. Решение: aging (повышение приоритета со временем ожидания).</span>

<b class="text-on-semi-light">Completely Fair Scheduler (CFS)</b>

<span class="text-gray">Современный планировщик Linux. Использует red-black tree для отслеживания «виртуального времени» каждого процесса. Выбирает процесс с наименьшим виртуальным временем — тот, кто получил CPU меньше всех. Обеспечивает честное распределение даже при разных приоритетах.</span>

### Переключение контекста (Context Switch)

<span class="text-gray">Когда ОС переключает CPU с одного процесса/потока на другой, она сохраняет состояние текущего (регистры, instruction pointer, стек) и загружает состояние следующего. Это занимает время (микросекунды), поэтому слишком частое переключение снижает производительность. Подробнее о потоках — [Потоки и параллелизм](/content/computer-science/threads).</span>

## Управление памятью

### Виртуальная память

Каждый процесс «думает», что у него есть непрерывный блок памяти от адреса 0 до максимума. На самом деле ОС отображает **виртуальные адреса** на **физические** через таблицу страниц (page table).

```text
Процесс A:                  Физическая RAM:
┌──────────────┐            ┌──────────────┐
│ 0x0000: code │───────────>│ Frame 5      │
│ 0x1000: heap │───────────>│ Frame 12     │
│ 0x2000: stack│───────────>│ Frame 3      │
└──────────────┘            │ Frame 7      │◄── Процесс B
                            │ ...          │
                            └──────────────┘
```

<b class="text-on-semi-light">Зачем виртуальная память?</b>

- **Изоляция**: процесс A не может прочитать память процесса B — у них разные page tables.
- **Абстракция**: программа не думает о физических адресах.
- **Overcommit**: ОС может «обещать» больше памяти, чем есть физически (подкачка на диск).
- **Shared memory**: разные процессы могут отображать одни и те же физические страницы (для IPC).

### Page Fault

Когда процесс обращается к виртуальному адресу, который **не отображён** на физическую память:

1. CPU генерирует **page fault** (исключение).
2. Ядро проверяет: это валидное обращение?
3. Если да — загружает страницу (из файла, swap, или выделяет новую) и обновляет page table.
4. Если нет — **segmentation fault** (SIGSEGV), процесс завершается.

<span class="text-gray">Page faults — нормальная часть работы: при первом обращении к malloc'd памяти, при lazy loading библиотек, при memory-mapped файлах. Но частые page faults с подкачкой на диск (thrashing) = серьёзная деградация производительности.</span>

### Адресное пространство процесса

```text
High addresses
┌──────────────────┐
│      Stack       │  ← растёт вниз (локальные переменные, return addresses)
│        ↓         │
│                  │
│        ↑         │
│       Heap       │  ← растёт вверх (malloc, new, объекты)
├──────────────────┤
│   BSS (uninit)   │  ← неинициализированные глобальные переменные
├──────────────────┤
│   Data (init)    │  ← инициализированные глобальные переменные
├──────────────────┤
│      Code        │  ← машинный код программы (read-only)
└──────────────────┘
Low addresses
```

> В контексте JavaScript: V8 запрашивает у ОС большие блоки памяти (через `mmap`) и управляет heap самостоятельно, деля его на поколения (Young Generation, Old Generation) для своего garbage collector. Подробнее — [Память и управление ею](/content/computer-science/memory).

## Файловые системы

Файловая система — абстракция ОС над физическим хранилищем: плоский массив блоков на диске превращается в иерархию файлов и каталогов с именами, правами и метаданными.

### Ключевые концепции

<b class="text-on-semi-light">Inode</b>

<span class="text-gray">Структура данных, хранящая метаданные файла: размер, владелец, права, timestamps, указатели на блоки данных. Имя файла хранится **не в inode**, а в каталоге — это запись «имя → номер inode». Поэтому у файла может быть несколько имён (hard links).</span>

<b class="text-on-semi-light">Файловые дескрипторы (FD)</b>

<span class="text-gray">Целое число, которое ОС присваивает открытому файлу/сокету/pipe. Стандартные: `0` = stdin, `1` = stdout, `2` = stderr. Когда Node.js делает `fs.open()` — ядро возвращает FD, через который происходят все дальнейшие операции.</span>

<b class="text-on-semi-light">Журналирование (Journaling)</b>

<span class="text-gray">Перед изменением файловой системы операция записывается в журнал. Если система упадёт в процессе — журнал позволит восстановить консистентное состояние без полной проверки диска.</span>

### Сравнение файловых систем

| FS        | ОС            | Особенности                                      |
| --------- | ------------- | ------------------------------------------------ |
| **ext4**  | Linux         | Журналирование, extents, 1 EiB макс. размер      |
| **APFS**  | macOS/iOS     | Copy-on-write, нативное шифрование, snapshots    |
| **NTFS**  | Windows       | Журналирование, ACL, compression, EFS encryption |
| **Btrfs** | Linux         | Copy-on-write, snapshots, RAID, subvolumes       |
| **ZFS**   | FreeBSD/Linux | Copy-on-write, checksums, inline compression     |

## Ввод-вывод (I/O)

### Модели I/O

Модели ввода-вывода — ключевой вопрос для понимания того, как работает Node.js.

<b class="text-on-semi-light">Blocking I/O (синхронный)</b>

<span class="text-gray">Поток вызывает `read()` и **блокируется** до получения данных. Простая модель, но один поток обслуживает одну операцию. Для множества соединений нужен поток на каждое — дорого по памяти.</span>

<b class="text-on-semi-light">Non-blocking I/O + Event Loop</b>

<span class="text-gray">Вызов `read()` возвращается сразу (с `EAGAIN`, если данных нет). Программа опрашивает готовность множества дескрипторов через мультиплексор (**epoll** в Linux, **kqueue** в macOS, **IOCP** в Windows) и обрабатывает только готовые. Именно так работает **libuv** — библиотека, лежащая в основе Node.js event loop.</span>

```text
Blocking (1 поток = 1 соединение):
  Thread 1: ████░░░░████░░░░  (wait → read → wait → read)
  Thread 2: ░░░░████░░░░████
  Thread 3: ██░░░░░░████░░░░
  → 10K соединений = 10K потоков = ~10 ГБ RAM (стек ~1 МБ/поток)

Non-blocking + epoll (1 поток, N соединений):
  Event Loop: check → ready? → handle → check → ready? → handle
  → 10K соединений = 1 поток, epoll следит за всеми FD
```

### Node.js и libuv

```text
┌──────────────────────────────────┐
│           Node.js                │
│  ┌────────────┐ ┌─────────────┐ │
│  │    V8      │ │  Node APIs  │ │
│  │ (JS engine)│ │ (fs, net..) │ │
│  └─────┬──────┘ └──────┬──────┘ │
│        │               │        │
│        ▼               ▼        │
│  ┌──────────────────────────┐   │
│  │         libuv            │   │
│  │  ┌──────┐  ┌──────────┐ │   │
│  │  │epoll/│  │Thread Pool│ │   │
│  │  │kqueue│  │ (fs, dns) │ │   │
│  │  └──────┘  └──────────┘ │   │
│  └──────────────────────────┘   │
└──────────────────────────────────┘
```

<span class="text-gray">libuv использует **epoll/kqueue** для сетевого I/O (неблокирующий) и **thread pool** (по умолчанию 4 потока) для операций, которые не имеют async API на уровне ОС: файловый I/O, DNS lookup, некоторые crypto-операции. Поэтому `fs.readFile()` не блокирует event loop — работа выполняется в потоке из пула.</span>

## Безопасность и изоляция

### User Mode vs Kernel Mode

<span class="text-gray">Процессор работает в двух режимах. В **kernel mode** доступны все инструкции и вся память. В **user mode** — ограниченный набор инструкций и только своё виртуальное адресное пространство. Попытка выполнить привилегированную инструкцию из user mode вызывает исключение. Это — базовый уровень защиты.</span>

### Права доступа (UNIX)

```text
-rwxr-xr--  1 user group  4096 Jan 15  file.txt
 │││ ││││││
 │││ │││└┴┴─ Others: r-- (read only)
 │││ └┴┴──── Group:  r-x (read + execute)
 └┴┴──────── Owner:  rwx (read + write + execute)
```

| Бит | Значение   | Число |
| --- | ---------- | ----- |
| `r` | Чтение     | 4     |
| `w` | Запись     | 2     |
| `x` | Выполнение | 1     |

`chmod 755` = owner: rwx (7), group: r-x (5), others: r-x (5).

### Sandboxing в браузере

<span class="text-gray">Браузер — одно из самых сложных приложений с точки зрения безопасности: он исполняет произвольный код из интернета. Chromium использует многоуровневую изоляцию.</span>

- **Process isolation**: каждый сайт (origin) — в отдельном процессе. Crash или эксплойт в одной вкладке не затрагивает другие.
- **Seccomp-BPF** (Linux): ограничивает набор syscalls, доступных процессу renderer'а. Renderer не может открывать файлы, сокеты, fork'ать процессы.
- **Filesystem sandbox**: renderer-процесс не имеет доступа к файловой системе — запросы проксируются через привилегированный browser-процесс.
- **Site Isolation**: каждый cross-site iframe — в отдельном процессе (защита от Spectre).

## Виртуализация и контейнеры

<b class="text-on-semi-light">Виртуальная машина (VM)</b>

<span class="text-gray">Полная эмуляция оборудования. Гипервизор (VMware, KVM, Hyper-V) запускает несколько ОС на одном физическом сервере. Каждая VM имеет своё ядро, свои процессы, свою память. Тяжёлая изоляция, но большой overhead (гигабайты RAM на каждую VM).</span>

<b class="text-on-semi-light">Контейнер (Docker)</b>

<span class="text-gray">Использует **ядро хост-системы**, но изолирует процессы через механизмы Linux: **namespaces** (изоляция PID, сети, файловой системы, пользователей) и **cgroups** (ограничение CPU, памяти, I/O). Контейнер стартует за секунды и потребляет на порядок меньше ресурсов, чем VM.</span>

```text
VM:                          Container:
┌─────────────────┐         ┌─────────────────┐
│   App A │ App B │         │   App A │ App B │
│   Libs  │ Libs  │         │   Libs  │ Libs  │
│  Guest  │ Guest │         ├─────────────────┤
│   OS    │  OS   │         │   Container     │
├─────────────────┤         │   Runtime       │
│   Hypervisor    │         ├─────────────────┤
├─────────────────┤         │   Host OS       │
│   Host OS       │         │   (shared kernel│)
├─────────────────┤         ├─────────────────┤
│   Hardware      │         │   Hardware      │
└─────────────────┘         └─────────────────┘
```

| Характеристика | VM                       | Container                              |
| -------------- | ------------------------ | -------------------------------------- |
| Изоляция       | Полная (своё ядро)       | Частичная (общее ядро)                 |
| Старт          | Минуты                   | Секунды                                |
| Размер образа  | Гигабайты                | Мегабайты                              |
| Overhead       | Высокий                  | Минимальный                            |
| Безопасность   | Сильнее (отдельное ядро) | Слабее (escape из контейнера возможен) |
