# Теория вычислений

<span class="text-gray">Теория вычислений изучает, что можно вычислить, насколько эффективно и какие модели вычислений существуют. Для senior frontend-разработчика это не абстрактная теория — конечные автоматы стоят за каждым regex, state machines моделируют ваш UI, формальные грамматики определяют, как Babel/PostCSS парсят код, а классы сложности объясняют, почему некоторые оптимизации фундаментально невозможны.</span>

---

## Модели вычислений

<b class="text-on-semi-light">Конечные автоматы (Finite Automata)</b>

<span class="text-gray">Простейшая модель вычислений — фиксированное число состояний, переходы по входным символам. Никакой памяти, кроме текущего состояния. Бывают двух видов:</span>

- **DFA (Deterministic Finite Automaton)** — для каждого состояния и символа ровно один переход. Быстрый (`O(n)`), но может требовать много состояний.
- **NFA (Non-deterministic Finite Automaton)** — допускает несколько переходов из одного состояния. Компактнее, но при выполнении может потребовать перебора путей.

<span class="text-gray">Любой NFA можно преобразовать в эквивалентный DFA (но число состояний может вырасти экспоненциально). Именно этот выбор между DFA и NFA стоит за разницей в производительности regex-движков.</span>

**Связь с фронтендом: Regular Expressions**

Каждый regex — это конечный автомат. Когда браузер выполняет `/^[a-z]+@[a-z]+\.[a-z]{2,}$/`, он внутри строит конечный автомат.

```js
// Regex = конечный автомат
const emailPattern = /^[a-z]+@[a-z]+\.[a-z]{2,}$/;
emailPattern.test('user@example.com'); // true

// Catastrophic backtracking: плохо написанный regex может работать экспоненциально,
// потому что NFA перебирает слишком много путей
const evil = /^(a+)+$/; // ⚠️ O(2^n) на входе 'aaaaaaaaaaX'
```

**Связь с фронтендом: State Machines для UI**

UI-состояния (idle → loading → success/error) — это конечные автоматы. Библиотеки вроде XState формализуют этот подход.

```ts
// State machine для загрузки данных
type State = 'idle' | 'loading' | 'success' | 'error';
type Event = 'FETCH' | 'RESOLVE' | 'REJECT' | 'RESET';

function transition(state: State, event: Event): State {
  switch (state) {
    case 'idle':    return event === 'FETCH' ? 'loading' : state;
    case 'loading': return event === 'RESOLVE' ? 'success'
                         : event === 'REJECT' ? 'error' : state;
    case 'success':
    case 'error':   return event === 'RESET' ? 'idle' : state;
  }
}
```

<span class="text-gray">Где state machines встречаются во фронтенде:</span>

| Где | Что | Почему |
|---|---|---|
| `useReducer` / Redux | Reducer = transition function | State + Action → New State |
| XState | Explicit state machines/statecharts | Сложные UI-флоу (визарды, формы) |
| CSS animations | `@keyframes` состояния | Idle → active → done |
| Component lifecycle | mount → update → unmount | React/Vue lifecycle |

<b class="text-on-semi-light">Автоматы с магазинной памятью (Pushdown Automata)</b>

<span class="text-gray">Добавляем стек к конечному автомату → можно обрабатывать вложенные структуры.</span>

**Связь с фронтендом**: HTML/XML-парсинг, проверка скобок, шаблонизаторы. Парсер HTML в браузере — это по сути pushdown-автомат, он использует стек для отслеживания открытых тегов.

```js
// Проверка скобок = pushdown-автомат
function isBalanced(str) {
  const stack = [];
  const pairs = { '(': ')', '[': ']', '{': '}' };
  for (const ch of str) {
    if (ch in pairs) stack.push(pairs[ch]);
    else if (')]}'.includes(ch)) {
      if (stack.pop() !== ch) return false;
    }
  }
  return stack.length === 0;
}
```

<b class="text-on-semi-light">Машина Тьюринга (кратко)</b>

<span class="text-gray">Бесконечная лента + головка чтения/записи + набор состояний. Может вычислить всё, что вообще вычислимо. Любой язык программирования (JS, Python, C) является Turing-complete — все они решают один и тот же класс задач.</span>

**Что делает систему Turing-complete?** Условия (if/else), циклы (while/for), произвольная память. Даже CSS + HTML формально не Turing-complete (без JS), а TypeScript система типов — Turing-complete (можно писать программы только в типах).

```ts
// TypeScript type system = Turing-complete
// Можно реализовать арифметику, логику и даже интерпретатор на уровне типов
type IsString<T> = T extends string ? true : false;
type A = IsString<'hello'>; // true
type B = IsString<42>;      // false
```

> «JavaScript — Turing-complete, но это не значит, что у каждой задачи есть решение в JS (или в любом другом языке).»

## Формальные грамматики и парсинг

<span class="text-gray">Иерархия Хомского связывает типы грамматик с моделями автоматов:</span>

| Тип | Грамматика | Автомат | Пример во фронтенде |
|---|---|---|---|
| Type 3 | Регулярная | Конечный автомат | Regex, токенизация |
| Type 2 | Контекстно-свободная | Pushdown-автомат | HTML/CSS/JS парсинг |
| Type 1 | Контекстно-зависимая | Линейно-ограниченный | Некоторые фичи TypeScript |
| Type 0 | Неограниченная | Машина Тьюринга | Общее вычисление |

**Связь с фронтендом**: каждый инструмент в вашем pipeline использует формальные грамматики:

- **Babel** — парсит JS/TS по контекстно-свободной грамматике → AST → трансформации → генерация кода
- **PostCSS** — парсит CSS → AST → плагины модифицируют → вывод
- **Template compilers** (Vue SFC, Svelte) — парсят шаблоны в render-функции
- **ESLint** — анализирует AST для поиска паттернов

<span class="text-gray">AST (Abstract Syntax Tree) — это мост между теорией грамматик и практическим тулингом.</span>

## Вычислимость и её пределы

<b class="text-on-semi-light">Проблема остановки (Halting Problem)</b>

<span class="text-gray">Не существует алгоритма, который для ЛЮБОЙ программы определит, завершится она или зациклится навсегда. Доказано Тьюрингом в 1936 году.</span>

Упрощённое доказательство от противного: допустим, существует функция `halts(fn)` → `true/false`. Тогда:

```js
function paradox() {
  if (halts(paradox)) {
    while (true) {} // если halts говорит "завершится" — зацикливаемся
  }
  return; // если halts говорит "зациклится" — завершаемся
}
// Противоречие: halts(paradox) не может дать правильный ответ
```

**Связь с фронтендом** — именно поэтому:

- TypeScript не может поймать ВСЕ ошибки типов (type checking в общем случае неразрешим)
- ESLint не может найти ВСЕ баги — только паттерны, на которые он запрограммирован
- Компиляторы не могут полностью оптимизировать любой код — часть оптимизаций требует решения неразрешимых задач
- Обнаружение бесконечных циклов в JS-движках — эвристическое, а не гарантированное

---

<b class="text-on-semi-light">Разрешимые vs неразрешимые задачи</b>

<span class="text-gray">**Разрешимая (decidable)** — существует алгоритм, который всегда завершается и даёт ответ «да» или «нет». **Неразрешимая (undecidable)** — такого алгоритма не существует. Проблема остановки — классический пример неразрешимой задачи.</span>

| Задача | Разрешима? | Почему важно |
|---|---|---|
| «Является ли строка валидным JSON?» | Да | `JSON.parse()` всегда даст ответ |
| «Завершится ли эта JS-функция?» | Нет | Проблема остановки |
| «Эквивалентны ли два regex?» | Да | Сравнение минимальных DFA |
| «Эквивалентны ли две произвольные программы?» | Нет | Теорема Райса |

## Теория сложности (практический взгляд)

<b class="text-on-semi-light">Big O — напоминание</b>

| Сложность | Название | Пример во фронтенде |
|---|---|---|
| O(1) | Константная | `Map.get()`, `Set.has()`, доступ по индексу |
| O(log n) | Логарифмическая | `Array.sort()` внутри (Timsort), бинарный поиск |
| O(n) | Линейная | `Array.find()`, `Array.filter()`, один проход по DOM |
| O(n log n) | Линейно-логарифмическая | `Array.sort()`, build heap |
| O(n²) | Квадратичная | Вложенные циклы, наивное сравнение всех пар |
| O(2ⁿ) | Экспоненциальная | Brute-force, catastrophic regex backtracking |

---

<b class="text-on-semi-light">P vs NP</b>

- **P** — задачи, решаемые за полиномиальное время. Большинство того, что мы делаем каждый день.
- **NP** — задачи, решение которых можно ПРОВЕРИТЬ за полиномиальное время, но НАЙТИ его — возможно, нет.
- **NP-complete** — самые сложные задачи в NP. Если решите одну быстро → все NP решены быстро.
- **P = NP?** — крупнейший открытый вопрос в CS. Большинство считают, что P ≠ NP.

**Связь с фронтендом** — когда задача «ощущается» невозможно медленной:

- CSS layout со сложными constraints → в общем случае NP-hard (браузеры используют эвристики)
- Оптимальное code splitting / размер бандла → вариант задачи bin packing (NP-hard)
- Regex с backtracking → может быть экспоненциальным

<span class="text-gray">Урок: если задача NP-hard, не пытайтесь решить её оптимально — используйте эвристики, аппроксимации или ограничьте размер входных данных.</span>

## AI и ML: что нужно знать фронтенду

<span class="text-gray">Краткий обзор на уровне awareness — не глубокое погружение, а понимание, что стоит за терминами.</span>

- **AI** — широкая область: машины, выполняющие задачи, требующие «интеллекта»
- **ML** — подмножество AI: системы учатся на данных. Supervised (classification, regression), unsupervised (clustering), reinforcement learning
- **Deep learning** — нейросети с множеством слоёв. Стоит за: распознавание изображений, NLP, генеративный AI
- **Для фронтенда**: WebNN API, TensorFlow.js, ONNX Runtime Web — inference прямо в браузере. Use cases: классификация изображений, анализ текста, рекомендации — без отправки данных на сервер
- **LLM и генеративный AI**: ChatGPT, Claude, встроенный AI в браузерах (`window.ai`). Интеграция во фронтенд: streaming responses, token-by-token rendering, управление контекстом

```js
// Пример: streaming response от LLM API
async function streamLLMResponse(prompt) {
  const response = await fetch('/api/chat', {
    method: 'POST',
    body: JSON.stringify({ prompt }),
  });
  const reader = response.body.getReader();
  const decoder = new TextDecoder();

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    const chunk = decoder.decode(value);
    appendToUI(chunk); // token-by-token rendering
  }
}
```

<span class="text-gray">Ключевое понимание: ML-модели работают внутри тех же вычислительных рамок — они не «преодолевают» проблему остановки и не решают NP-задачи за полиномиальное время. Но они хороши в аппроксимациях и эвристиках на практических данных.</span>

| «Как моделировать сложный UI-флоу?» | State machines, конечные автоматы |
