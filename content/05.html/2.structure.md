# Структура документа и рендеринг

Знание структуры HTML-документа — это не про запоминание тегов `<html>`, `<head>`, `<body>`. На senior-уровне ключевое — это понимание **Critical Rendering Path (CRP)**: как байты HTML превращаются в пиксели на экране, что блокирует рендеринг и как это оптимизировать.

---

## Базовая структура документа

Минимальный валидный HTML5-документ:

```html
<!DOCTYPE html>
<html lang="ru">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Заголовок страницы</title>
</head>
<body>
  <h1>Привет, мир</h1>
</body>
</html>
```

<b class="text-on-semi-light">Обязательные элементы:</b>

| Элемент | Зачем нужен |
|---|---|
| `<!DOCTYPE html>` | Включает standards mode (без него — quirks mode) |
| `<html lang="...">` | Корневой элемент. `lang` — язык документа, критичен для a11y и SEO |
| `<meta charset="UTF-8">` | Кодировка. Должен быть в первых 1024 байтах документа |
| `<meta name="viewport">` | Управление масштабированием на мобильных устройствах |
| `<title>` | Заголовок вкладки. Обязателен по спецификации. Влияет на SEO |

<span class="text-gray">Интересно: `<html>`, `<head>`, `<body>` технически необязательны — парсер вставит их автоматически. Но явно писать их — хорошая практика для читаемости и предсказуемости.</span>

## DOM-дерево

**DOM (Document Object Model)** — это древовидное представление HTML-документа в памяти браузера. Каждый элемент, текст, комментарий — это **узел (node)** дерева.

### Типы узлов

| Тип | nodeType | Описание | Пример |
|---|---|---|---|
| **Document** | 9 | Корень дерева | `document` |
| **Element** | 1 | HTML-элемент | `<div>`, `<p>`, `<a>` |
| **Text** | 3 | Текстовое содержимое | `"Привет"` |
| **Comment** | 8 | HTML-комментарий | `<!-- коммент -->` |
| **DocumentType** | 10 | DOCTYPE | `<!DOCTYPE html>` |
| **DocumentFragment** | 11 | Лёгкий контейнер | `document.createDocumentFragment()` |

### Пример DOM-дерева

```html
<body>
  <header>
    <h1>Заголовок</h1>
  </header>
  <main>
    <p>Текст <a href="#">ссылка</a></p>
  </main>
</body>
```

```text
Document
 └── html
      ├── head
      └── body
           ├── header
           │    └── h1
           │         └── #text "Заголовок"
           └── main
                └── p
                     ├── #text "Текст "
                     └── a [href="#"]
                          └── #text "ссылка"
```

<b class="text-on-semi-light">На фронте:</b> React, Vue, Svelte — все они в конечном счёте модифицируют этот DOM. Virtual DOM (React) — это абстракция, которая минимизирует прямые манипуляции с реальным DOM. Но реальный DOM — это всегда конечная точка.

## Critical Rendering Path (CRP)

**Critical Rendering Path** — это последовательность шагов, которую браузер выполняет для превращения HTML, CSS и JavaScript в пиксели на экране. Понимание CRP — это ключ к оптимизации производительности.

### Полный pipeline

```text
     HTML                          CSS
      │                             │
      ▼                             ▼
   Парсинг                       Парсинг
      │                             │
      ▼                             ▼
    DOM ─────────────┬──────────── CSSOM
                     │
                     ▼
              Render Tree
              (только видимые узлы)
                     │
                     ▼
                  Layout
              (вычисление позиций
               и размеров)
                     │
                     ▼
                   Paint
              (заполнение пикселей)
                     │
                     ▼
                Composite
              (объединение слоёв)
                     │
                     ▼
                 Пиксели
```

### Этапы подробно

#### 1. Построение DOM

Байты HTML декодируются в символы, токенизируются и преобразуются в дерево DOM-узлов. Подробнее → [Основы HTML5](/content/html/markup-language).

#### 2. Построение CSSOM

Браузер парсит все CSS (внешние файлы, `<style>`, inline-стили) и строит **CSSOM (CSS Object Model)** — дерево, где каждому узлу присвоены вычисленные стили.

<span class="text-gray">CSSOM строится по принципу каскада: стили наследуются сверху вниз, специфичность определяет приоритет, и `!important` переопределяет каскад. Пока CSSOM не построен полностью, рендеринг заблокирован.</span>

#### 3. Render Tree

DOM + CSSOM объединяются в **Render Tree** — дерево, содержащее только **видимые** элементы с их вычисленными стилями.

Что **не попадает** в Render Tree:
- Элементы с `display: none` (но `visibility: hidden` — попадают, занимая место)
- Содержимое `<head>`, `<script>`, `<meta>`
- Элементы с `aria-hidden` по-прежнему в Render Tree (они не скрыты визуально)

#### 4. Layout (Reflow)

Вычисление **точных позиций и размеров** каждого элемента на экране. Учитывает viewport, box model, flex/grid, float, позиционирование.

#### 5. Paint

Заполнение пикселей: отрисовка текста, цветов, границ, теней, изображений. Происходит послойно (painting layers).

#### 6. Composite

Объединение слоёв в финальное изображение. Элементы с `transform`, `opacity`, `will-change` выносятся на отдельные **композитные слои** и обрабатываются GPU.

## Render-blocking ресурсы

Не все ресурсы одинаково влияют на рендеринг. Понимание того, что и как блокирует — основа оптимизации.

### CSS блокирует рендеринг

CSS — **render-blocking** ресурс. Браузер не покажет контент, пока не построит CSSOM. Это логично: без стилей текст мигнёт нестилизованным (FOUC — Flash of Unstyled Content).

```html
<!-- Блокирует рендер до загрузки и парсинга -->
<link rel="stylesheet" href="styles.css">

<!-- Условная загрузка: не блокирует, если media не совпадает -->
<link rel="stylesheet" href="print.css" media="print">
```

### JavaScript блокирует парсинг

`<script>` без атрибутов — **parser-blocking**: парсер HTML останавливается, ждёт загрузки и выполнения скрипта.

```html
<!-- Блокирует парсинг HTML -->
<script src="app.js"></script>

<!-- НЕ блокирует парсинг -->
<script src="app.js" defer></script>
<script src="app.js" async></script>
```

<span class="text-gray">Подробное сравнение `async`, `defer` и `type="module"` → [Содержимое head](/content/html/head-content).</span>

### Схема блокировки

```text
HTML парсинг:   ████████░░░░░░████████████
CSS загрузка:   ──────████████
                         ↑
                  render blocked до этого момента

JS без defer:   ████████░░░░████████
                        ↑    ↑
                     парсинг заблокирован
```

## Reflow vs Repaint

Две самые «дорогие» операции после начального рендеринга.

| Операция | Что происходит | Когда вызывается | Стоимость |
|---|---|---|---|
| **Reflow (Layout)** | Пересчёт позиций и размеров элементов | Изменение размеров, добавление/удаление элементов, изменение шрифта | Высокая |
| **Repaint** | Перерисовка пикселей без изменения layout | Изменение цвета, тени, visibility | Средняя |
| **Composite** | Перекомпозиция слоёв | Изменение transform, opacity | Низкая (GPU) |

### Что вызывает reflow

- Изменение `width`, `height`, `padding`, `margin`, `border`
- Изменение `font-size`, `font-family`
- Добавление/удаление DOM-элементов
- Чтение layout-свойств: `offsetHeight`, `scrollTop`, `getBoundingClientRect()`
- Изменение `display`, `position`, `float`
- Resize окна

<b class="text-on-semi-light">Ключевая ловушка:</b> Чтение layout-свойств **между записями** форсирует **синхронный reflow** (forced synchronous layout). Это одна из самых частых причин тормозов в UI.

**Плохо:**

```js
// Forced synchronous layout — каждая итерация вызывает reflow
for (const el of elements) {
  el.style.width = container.offsetWidth + 'px'; // read → write → read → write...
}
```

**Хорошо:**

```js
// Batch: сначала читаем, потом пишем
const width = container.offsetWidth; // один read
for (const el of elements) {
  el.style.width = width + 'px'; // только write
}
```

### Оптимизация: Composite-only свойства

Свойства `transform` и `opacity` **не вызывают** reflow и repaint — они обрабатываются на этапе composite (GPU). Поэтому анимации через `transform` всегда быстрее, чем через `top`/`left`.

```css
/* Плохо: вызывает reflow на каждый кадр */
.animate { left: 100px; transition: left 0.3s; }

/* Хорошо: composite-only, GPU-ускорение */
.animate { transform: translateX(100px); transition: transform 0.3s; }
```

## Инкрементальный рендеринг

Браузер **не ждёт** загрузки всего HTML, чтобы начать рендеринг. Он рисует контент **по мере получения данных** (streaming/incremental rendering).

Это значит:

- Первый контент может появиться до загрузки всего HTML
- Ресурсы в `<head>` задерживают **первый paint** (LCP, FCP)
- `<link>` к CSS в середине `<body>` вызовет дополнительную блокировку рендеринга для контента ниже
- Серверный рендеринг (SSR) с потоковой отдачей HTML даёт лучший TTFB и FCP

<b class="text-on-semi-light">На фронте:</b> Это основа понимания Web Vitals. **LCP (Largest Contentful Paint)** зависит от того, как быстро browser получит HTML + CSS для самого крупного видимого элемента. **CLS (Cumulative Layout Shift)** возникает, когда элементы появляются с задержкой и двигают layout.

## Частые вопросы на собесе

**Что такое Critical Rendering Path?**
Последовательность: HTML → DOM, CSS → CSSOM, DOM + CSSOM → Render Tree → Layout → Paint → Composite. Оптимизация CRP — это уменьшение количества render-blocking ресурсов и их размера.

**Что происходит, когда вводишь URL в браузер?**
DNS-резолвинг → TCP-соединение → TLS handshake → HTTP-запрос → получение HTML → парсинг HTML → построение DOM → загрузка CSS/JS → CSSOM → Render Tree → Layout → Paint → Composite. Параллельно: preload scanner запускает загрузку ресурсов.

**Чем reflow отличается от repaint?**
Reflow пересчитывает геометрию (позиции, размеры) — дорого. Repaint перерисовывает пиксели без изменения layout — дешевле. Composite (transform, opacity) — дешевле всего, обрабатывается GPU.

**Почему `transform` лучше `top`/`left` для анимаций?**
`top`/`left` вызывают reflow → repaint → composite на каждый кадр. `transform` обрабатывается только на этапе composite (GPU), минуя reflow и repaint. Разница особенно заметна на мобильных устройствах.

**Что такое forced synchronous layout?**
Когда JavaScript читает layout-свойство (`offsetHeight`, `getBoundingClientRect`) после записи стиля, браузер вынужден выполнить reflow **синхронно**, чтобы вернуть актуальное значение. В цикле это приводит к layout thrashing — каскаду ненужных reflow.

**CSS блокирует рендеринг или парсинг?**
CSS блокирует **рендеринг** (browser не покажет контент до построения CSSOM), но не блокирует парсинг HTML. JS без `async`/`defer` блокирует **парсинг** HTML.
