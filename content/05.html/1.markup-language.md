# Основы HTML5

HTML5 — это не просто «пятая версия HTML». С 2019 года HTML развивается как **Living Standard** — живой стандарт, который обновляется непрерывно. Понимание того, как устроен стандарт, какие правила вложенности существуют и как браузер парсит разметку — это фундамент, без которого невозможно глубоко разбираться в веб-платформе.

---

## HTML5 Living Standard

Исторически HTML развивался через пронумерованные версии (HTML 2.0, 3.2, 4.01, XHTML 1.0). В 2004 году группа разработчиков браузеров создала **WHATWG** (Web Hypertext Application Technology Working Group), недовольная направлением W3C в сторону XHTML 2.0.

С 2019 года **WHATWG** — единственный хранитель стандарта HTML. W3C больше не публикует собственные версии.

| Организация | Роль | Статус |
|---|---|---|
| **WHATWG** | Развивает HTML Living Standard | Основной стандарт с 2019 |
| **W3C** | Исторически публиковал HTML-спецификации | Передал контроль WHATWG |

<b class="text-on-semi-light">Что это значит на практике:</b> Нет «HTML6». Новые фичи (элемент `<dialog>`, атрибут `inert`, Popover API) добавляются в живой стандарт и постепенно реализуются браузерами. Актуальная спецификация всегда одна — [html.spec.whatwg.org](https://html.spec.whatwg.org/).

## Content Models (модели содержимого)

В HTML5 каждый элемент принадлежит одной или нескольким **категориям содержимого**. Эти категории определяют, **какие элементы могут быть вложены** друг в друга. Это замена устаревшего деления на «block» и «inline» на уровне спецификации (block/inline остались только в CSS).

### Категории

| Категория | Описание | Примеры элементов |
|---|---|---|
| **Flow** | Большинство элементов, используемых в `<body>` | `<div>`, `<p>`, `<section>`, `<a>`, `<img>` |
| **Phrasing** | «Текстовые» элементы, аналог старого inline | `<span>`, `<a>`, `<em>`, `<strong>`, `<code>`, `<img>` |
| **Interactive** | Элементы, с которыми пользователь взаимодействует | `<a>`, `<button>`, `<input>`, `<select>`, `<details>` |
| **Embedded** | Встраивают внешний контент | `<img>`, `<video>`, `<audio>`, `<canvas>`, `<iframe>` |
| **Heading** | Заголовки секций | `<h1>`–`<h6>`, `<hgroup>` |
| **Sectioning** | Определяют области документа | `<article>`, `<section>`, `<nav>`, `<aside>` |
| **Metadata** | Метаинформация, не отображаются | `<meta>`, `<link>`, `<title>`, `<style>`, `<script>` |

### Правила вложенности

Категории определяют, **что может быть внутри чего**. Нарушение этих правил заставляет парсер «чинить» разметку, что приводит к непредсказуемому DOM.

<b class="text-on-semi-light">Ключевые правила:</b>

- `<p>` может содержать только **phrasing content** → `<div>` внутри `<p>` невалиден
- `<a>` не может содержать **interactive content** → `<a>` внутри `<a>` или `<button>` внутри `<a>` невалидны
- `<button>` не может содержать **interactive content** → `<a>` внутри `<button>` невалиден
- `<h1>`–`<h6>` могут содержать только **phrasing content**
- `<ul>` / `<ol>` принимают только `<li>` как прямых потомков

**Плохо:**

```html
<!-- Парсер закроет <p> перед <div>, создавая два <p> -->
<p>Текст <div>блок</div> ещё текст</p>

<!-- Интерактивный элемент внутри интерактивного -->
<a href="/page">
  <button>Нажми</button>
</a>
```

**Хорошо:**

```html
<p>Текст <span>инлайн</span> ещё текст</p>

<a href="/page">Перейти</a>
<button>Нажми</button>
```

> Валидатор [validator.w3.org](https://validator.w3.org/) и browser DevTools помогают выявить ошибки вложенности. Браузер **не покажет ошибку** — он молча «починит» разметку, но результат может отличаться от ожидаемого.

## DOCTYPE и режимы парсинга

`<!DOCTYPE html>` — это не HTML-тег, а **инструкция для парсера**, которая определяет режим рендеринга.

### Три режима

| Режим | Когда активируется | Поведение |
|---|---|---|
| **Standards (no-quirks)** | `<!DOCTYPE html>` | Полное соответствие стандартам CSS и HTML |
| **Almost Standards** | DOCTYPE HTML 4.01 Transitional | Как standards, но с quirk для line-height в ячейках таблиц |
| **Quirks Mode** | Нет DOCTYPE или устаревший DOCTYPE | Эмулирует баги IE5/Navigator 4 для совместимости |

<b class="text-on-semi-light">На практике:</b> Всегда используй `<!DOCTYPE html>` — это кратчайший DOCTYPE, который включает standards mode. Любой другой DOCTYPE нужен только для legacy-страниц.

```html
<!DOCTYPE html>
<html lang="ru">
  <!-- Standards mode активирован -->
</html>
```

<span class="text-gray">Quirks mode меняет box model (width включает padding/border, как `box-sizing: border-box`), поведение `margin: auto`, парсинг inline-элементов и многое другое. Это источник багов, которые невозможно воспроизвести на современных страницах.</span>

## Как браузер парсит HTML

Парсинг HTML — это **не просто чтение файла сверху вниз**. HTML-парсер работает по сложному алгоритму, описанному в спецификации, и умеет обрабатывать ошибки автоматически.

### Этапы парсинга

```text
Bytes (сеть)
  → Characters (декодирование, напр. UTF-8)
    → Tokens (токенизация: start tag, end tag, character, comment, DOCTYPE)
      → Nodes (создание DOM-узлов)
        → DOM Tree (построение дерева)
```

### Tokenizer (токенизатор)

Разбивает поток символов на токены. Работает как **конечный автомат (state machine)** с ~80 состояниями.

Пример для `<p class="text">Hello</p>`:

```text
StartTag: p, attrs: [{class: "text"}]
Character: "Hello"
EndTag: p
```

### Tree Construction (построение дерева)

Получает токены и строит DOM-дерево. Использует **стек открытых элементов** для отслеживания вложенности. Именно здесь происходит «авто-исправление» невалидной разметки:

- Незакрытый `<p>` перед `<div>` → парсер закроет `<p>` автоматически
- `<table>` с текстом напрямую → текст будет вынесен перед таблицей (foster parenting)
- Пропущенные `<html>`, `<head>`, `<body>` → парсер вставит их сам

### Speculative Parsing (спекулятивный парсинг)

Когда парсер встречает `<script>` без `async`/`defer`, он **останавливается** и ждёт загрузки и выполнения скрипта (script blocks parsing). Но браузер продолжает **сканировать разметку вперёд** (preload scanner / speculative parser), чтобы начать загрузку ресурсов (`<img>`, `<link>`, `<script>`) заранее.

```text
Основной парсер:  HTML ──> <script src="app.js"> ──СТОП── ожидание ──> продолжение
Preload scanner:  ──────────────────────────────> сканирует дальше, находит <img>, <link>
                                                   начинает загрузку параллельно
```

<b class="text-on-semi-light">На фронте:</b> Вот почему `<script>` в конце `<body>` или с атрибутом `defer` — стандартная рекомендация. Подробнее про `async`/`defer` → [Содержимое head](/content/html/head-content).

## HTML entities

HTML entities — это специальные последовательности для отображения символов, которые имеют особое значение в HTML или не вводятся с клавиатуры.

| Entity | Символ | Когда нужен |
|---|---|---|
| `&lt;` | < | В тексте, чтобы не интерпретировался как тег |
| `&gt;` | > | Для читаемости (не обязателен в большинстве контекстов) |
| `&amp;` | & | Когда `&` может быть началом entity |
| `&nbsp;` | неразрывный пробел | Предотвращает перенос строки между словами |
| `&copy;` | © | Символ копирайта |
| `&#8212;` / `&mdash;` | — | Длинное тире |

<span class="text-gray">В UTF-8 большинство символов можно вставлять напрямую (©, «, →). Entities нужны только для `<`, `>`, `&` в контексте HTML и для неразрывных пробелов.</span>

## Частые вопросы на собесе

**Что такое Living Standard и почему нет HTML6?**
HTML развивается как живой стандарт WHATWG — новые фичи добавляются непрерывно, без деления на версии. «HTML5» — это маркетинговый термин, актуальный стандарт один.

**Зачем нужен DOCTYPE?**
Определяет режим парсинга. Без DOCTYPE браузер включает quirks mode — эмуляцию багов 20-летней давности. `<!DOCTYPE html>` — минимальная форма для standards mode.

**Почему `<div>` внутри `<p>` — ошибка?**
`<p>` может содержать только phrasing content. `<div>` — flow content. Парсер автоматически закроет `<p>` перед `<div>`, создавая два отдельных параграфа, что ломает ожидаемую структуру DOM.

**Что такое speculative parsing?**
Когда основной парсер заблокирован выполнением `<script>`, браузер продолжает сканировать HTML вперёд, чтобы заранее начать загрузку ресурсов (изображений, стилей, скриптов). Это оптимизация, работающая прозрачно.

**Как проверить валидность HTML?**
Через [validator.w3.org](https://validator.w3.org/), расширения для IDE (HTMLHint), или Lighthouse audit. Браузер не показывает ошибки HTML — он молча «чинит» их, что может привести к багам.
