# Содержимое `<head>`

`<head>` — это «командный центр» страницы. Здесь определяется кодировка, viewport, SEO-метаданные, стратегия загрузки ресурсов и политика безопасности. На senior-собесе вопросы про `<head>` — это прежде всего вопросы про **производительность** (resource hints, async/defer) и **SEO** (meta, Open Graph).

---

## `<meta>` теги

Meta-теги передают метаинформацию браузеру, поисковикам и социальным сетям.

### Обязательные

```html
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width, initial-scale=1.0">
```

<span class="text-gray">`charset` должен быть в первых 1024 байтах документа — иначе браузер может неправильно определить кодировку. `viewport` критичен для мобильных: без него страница рендерится в ~980px ширины и масштабируется.</span>

### SEO и описание

```html
<title>Структура HTML — Frontend Legends</title>
<meta name="description" content="Разбираем структуру HTML-документа и Critical Rendering Path">
<meta name="robots" content="index, follow">
<link rel="canonical" href="https://frontend-legends.com/html/structure">
```

| Тег | Влияние на SEO | Описание |
|---|---|---|
| `<title>` | Прямое (ranking factor) | Отображается во вкладке и поисковой выдаче. До 60 символов |
| `meta description` | Косвенное (CTR) | Сниппет в выдаче. До 160 символов |
| `meta robots` | Прямое | Управляет индексацией: `noindex`, `nofollow`, `nosnippet` |
| `link canonical` | Прямое | Указывает основную версию страницы при дубликатах |

### Open Graph и Twitter Cards

```html
<!-- Open Graph (Facebook, Telegram, LinkedIn) -->
<meta property="og:title" content="Структура HTML">
<meta property="og:description" content="CRP, DOM, рендеринг">
<meta property="og:image" content="https://example.com/og.jpg">
<meta property="og:url" content="https://example.com/html/structure">
<meta property="og:type" content="article">

<!-- Twitter Cards -->
<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:title" content="Структура HTML">
<meta name="twitter:image" content="https://example.com/og.jpg">
```

<b class="text-on-semi-light">На фронте:</b> В SPA (React, Vue, Nuxt, Next.js) мета-теги нужно менять динамически при навигации. Nuxt использует `useHead()`, Next.js — `<Head>` / Metadata API. Без SSR или пререндеринга поисковые боты могут не увидеть динамические мета-теги.

### Другие полезные meta

```html
<meta name="theme-color" content="#1a1a2e">  <!-- Цвет адресной строки на мобильных -->
<meta name="color-scheme" content="dark light"> <!-- Поддержка тёмной/светлой темы -->
<meta http-equiv="X-UA-Compatible" content="IE=edge"> <!-- Legacy, для IE -->
```

## `<link>` и ресурсные подсказки

`<link>` подключает внешние ресурсы: стили, шрифты, иконки. Но его главная сила — **resource hints**, которые позволяют управлять приоритетом и моментом загрузки ресурсов.

### Базовое использование

```html
<link rel="stylesheet" href="styles.css">
<link rel="icon" href="/favicon.ico" sizes="32x32">
<link rel="icon" href="/icon.svg" type="image/svg+xml">
<link rel="apple-touch-icon" href="/apple-touch-icon.png">
```

### Resource Hints

| Hint | Что делает | Когда использовать | Пример |
|---|---|---|---|
| `preload` | Загружает ресурс с высоким приоритетом **для текущей страницы** | Критичные шрифты, CSS, hero image | `<link rel="preload" href="font.woff2" as="font" crossorigin>` |
| `prefetch` | Загружает ресурс с низким приоритетом **для будущей навигации** | Ресурсы следующей страницы | `<link rel="prefetch" href="/next-page.js">` |
| `preconnect` | Устанавливает соединение (DNS + TCP + TLS) заранее | CDN, API, шрифтовые серверы | `<link rel="preconnect" href="https://fonts.googleapis.com">` |
| `dns-prefetch` | Только DNS-резолвинг | Домены, к которым обратишься позже | `<link rel="dns-prefetch" href="https://analytics.example.com">` |
| `modulepreload` | Загружает ES-модуль и его зависимости | Для `<script type="module">` | `<link rel="modulepreload" href="/app.js">` |

<b class="text-on-semi-light">Ключевые правила:</b>

- `preload` **обязан** быть использован на странице — если загруженный ресурс не нужен, браузер выдаст warning
- `preconnect` ограничивай 2–4 доменами — каждое соединение стоит ресурсов
- `prefetch` имеет самый низкий приоритет — не мешает загрузке текущей страницы
- Шрифты требуют `crossorigin` даже с того же домена (CORS requirement для шрифтов)

```html
<!-- Типичный набор для продакшена -->
<link rel="preconnect" href="https://fonts.bunny.net">
<link rel="preload" href="/fonts/SpaceGrotesk.woff2" as="font" type="font/woff2" crossorigin>
<link rel="prefetch" href="/next-page-bundle.js">
```

## `<script>` — async, defer, module

Способ подключения скрипта определяет, **когда он загружается и выполняется** относительно парсинга HTML. Это один из самых частых вопросов на собеседованиях.

### Схема загрузки

```text
<script src="app.js">
HTML:  ████████░░░░░░░░░░████████
                ↑ загрузка ↑ выполнение
                парсинг заблокирован

<script src="app.js" async>
HTML:  ████████████████░░████████
                ↑ загрузка ↑ выполнение (как только загружен)
                параллельно   парсинг заблокирован на момент выполнения

<script src="app.js" defer>
HTML:  ██████████████████████████ → DOMContentLoaded
                ↑ загрузка       ↑ выполнение (после парсинга)
                параллельно       в порядке объявления

<script type="module">
HTML:  ██████████████████████████ → DOMContentLoaded
                ↑ загрузка       ↑ выполнение
                параллельно       как defer + strict mode + свой scope
```

### Сравнение

| Атрибут | Загрузка | Выполнение | Порядок | Блокирует парсинг |
|---|---|---|---|---|
| _(без атрибутов)_ | Последовательно | Сразу | В порядке объявления | Да |
| `async` | Параллельно | Как только загружен | **Не гарантирован** | На момент выполнения |
| `defer` | Параллельно | После парсинга HTML | В порядке объявления | Нет |
| `type="module"` | Параллельно | После парсинга HTML | В порядке объявления | Нет (как defer) |

<b class="text-on-semi-light">Когда что использовать:</b>

- **`defer`** — основной выбор для большинства скриптов. Гарантирует порядок и не блокирует парсинг
- **`async`** — для независимых скриптов, которым не важен порядок: аналитика, виджеты, A/B тесты
- **`type="module"`** — для ES-модулей. Ведёт себя как `defer`, но с `import`/`export`, strict mode и собственным scope
- **Без атрибутов** — почти никогда. Только если скрипт должен выполниться синхронно (inline polyfills)

> `async` на `type="module"` — допустимая комбинация. Модуль загрузится параллельно и выполнится сразу, без ожидания парсинга.

## CSP через `<meta>`

**Content Security Policy (CSP)** — механизм защиты от XSS и инъекций. Может быть задан HTTP-заголовком или `<meta>`.

```html
<meta http-equiv="Content-Security-Policy"
      content="default-src 'self'; script-src 'self' 'nonce-abc123'; style-src 'self' 'unsafe-inline'">
```

| Директива | Что контролирует |
|---|---|
| `default-src` | Fallback для всех типов ресурсов |
| `script-src` | Откуда можно загружать скрипты |
| `style-src` | Откуда можно загружать стили |
| `img-src` | Откуда можно загружать изображения |
| `connect-src` | Куда можно делать fetch/XHR/WebSocket |
| `frame-src` | Какие URL можно загружать в iframe |

<b class="text-on-semi-light">`nonce`-based CSP:</b>

```html
<meta http-equiv="Content-Security-Policy"
      content="script-src 'nonce-r4nd0m'">

<!-- Только скрипты с совпадающим nonce выполнятся -->
<script nonce="r4nd0m">console.log('OK')</script>
<script>console.log('BLOCKED')</script> <!-- Заблокирован CSP -->
```

<span class="text-gray">CSP через `<meta>` имеет ограничения: нельзя использовать `frame-ancestors`, `report-uri`, `sandbox`. Для полноценной CSP лучше HTTP-заголовок. Но `<meta>` удобен для SPA без контроля над сервером.</span>

## `<style>`, `<base>`, `<noscript>`

### `<style>`

Inline CSS. Блокирует рендеринг так же, как `<link rel="stylesheet">`.

```html
<style>
  /* Critical CSS: только то, что нужно для first paint */
  .hero { font-size: 2rem; color: var(--primary); }
</style>
```

<b class="text-on-semi-light">Паттерн Critical CSS:</b> инлайнить только стили для above-the-fold контента в `<style>`, а остальные загружать через `<link>` с `preload`. Это ускоряет FCP.

### `<base>`

Устанавливает базовый URL для всех относительных ссылок в документе.

```html
<base href="https://example.com/docs/">
<!-- <a href="page.html"> → https://example.com/docs/page.html -->
```

<span class="text-gray">Используется редко. Может сломать якорные ссылки (`#section`) и ожидаемое поведение relative URLs. В SPA обычно не нужен.</span>

### `<noscript>`

Отображается только если JavaScript отключён.

```html
<noscript>
  <p>Для работы приложения необходим JavaScript</p>
</noscript>
```

## Частые вопросы на собесе

**В чём разница между `async` и `defer`?**
Оба загружают скрипт параллельно с парсингом HTML. `async` выполняет сразу после загрузки (порядок не гарантирован, блокирует парсинг на момент выполнения). `defer` выполняет после завершения парсинга HTML, в порядке объявления, перед `DOMContentLoaded`.

**Что блокирует рендеринг страницы?**
CSS блокирует рендеринг (browser не покажет контент без CSSOM). JS без `async`/`defer` блокирует парсинг HTML. Шрифты могут вызвать FOIT (Flash of Invisible Text). Resource hints (`preload`, `preconnect`) помогают минимизировать задержки.

**Зачем нужен `preload` и чем он отличается от `prefetch`?**
`preload` — высокий приоритет, для ресурсов текущей страницы (критичный шрифт, hero image). `prefetch` — низкий приоритет, для ресурсов будущей навигации. `preload` без использования — warning в консоли.

**Как CSP защищает от XSS?**
CSP ограничивает, откуда могут загружаться ресурсы. `script-src 'self'` запрещает inline-скрипты и скрипты с чужих доменов. `nonce`-based CSP разрешает только скрипты с совпадающим одноразовым токеном.

**Зачем нужен `<meta name="viewport">`?**
Без него мобильный браузер рендерит страницу в ~980px ширины и масштабирует. `width=device-width` устанавливает ширину viewport равной ширине устройства. `initial-scale=1.0` задаёт начальный масштаб.
