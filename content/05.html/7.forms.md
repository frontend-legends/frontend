# Формы

Формы — один из самых интерактивных элементов HTML. На senior-уровне важно знать не только `<input>` типы, но и нативную валидацию (Constraint Validation API), `FormData` API и разницу между событиями `input` и `change`.

---

## `<form>`

```html
<form action="/api/submit" method="POST" enctype="multipart/form-data" novalidate>
  <!-- поля формы -->
</form>
```

| Атрибут | Описание | Значения |
|---|---|---|
| `action` | URL, куда отправляются данные | `/api/submit`, пустая строка = текущий URL |
| `method` | HTTP-метод | `GET` (по умолчанию), `POST` |
| `enctype` | Тип кодирования | `application/x-www-form-urlencoded` (default), `multipart/form-data` (для файлов), `text/plain` |
| `novalidate` | Отключает нативную HTML5-валидацию | Булевый |
| `autocomplete` | Автозаполнение браузером | `on` / `off` |

<b class="text-on-semi-light">Когда `GET`, когда `POST`:</b> `GET` — для поиска и фильтров (данные в URL, можно закладку). `POST` — для создания/изменения данных (данные в body, не кэшируется, нет ограничения длины).

<span class="text-gray">`enctype="multipart/form-data"` обязателен для загрузки файлов. Без него `<input type="file">` отправит только имя файла, а не содержимое.</span>

## Input types

HTML5 добавил множество специализированных типов, которые дают нативную валидацию и правильную клавиатуру на мобильных.

| Тип | Назначение | Мобильная клавиатура | Нативная валидация |
|---|---|---|---|
| `text` | Общий текст | Стандартная | — |
| `password` | Пароль (скрытый) | Стандартная | — |
| `email` | Email | С `@` и `.` | Формат email |
| `number` | Число | Цифровая | min, max, step |
| `tel` | Телефон | Цифровая (без +) | — (только формат клавиатуры) |
| `url` | URL | С `/` и `.com` | Формат URL |
| `search` | Поиск | С кнопкой очистки | — |
| `date` | Дата | Нативный datepicker | Валидная дата |
| `time` | Время | Нативный timepicker | Валидное время |
| `datetime-local` | Дата + время | Нативный picker | — |
| `range` | Слайдер | — | min, max, step |
| `color` | Выбор цвета | Color picker | — |
| `file` | Загрузка файла | Файловый менеджер | accept |
| `checkbox` | Флажок | — | required |
| `radio` | Радиокнопка | — | required (на группу) |
| `hidden` | Скрытое поле | — | — |

```html
<input type="email" required placeholder="user@example.com">
<input type="number" min="1" max="100" step="1">
<input type="date" min="2020-01-01" max="2030-12-31">
<input type="file" accept=".pdf,.doc" multiple>
<input type="range" min="0" max="100" value="50">
```

## Другие элементы форм

### `<label>`

Связывает текст с полем формы. **Обязателен для доступности** — без label скринридер не знает, что за поле.

```html
<!-- Явная связь через for/id -->
<label for="email">Email:</label>
<input id="email" type="email">

<!-- Неявная связь через вложенность -->
<label>
  Email:
  <input type="email">
</label>
```

### `<textarea>`

```html
<textarea rows="4" cols="50" maxlength="500" placeholder="Комментарий..."></textarea>
```

### `<select>`, `<option>`, `<optgroup>`

```html
<select name="city">
  <option value="" disabled selected>Выберите город</option>
  <optgroup label="Россия">
    <option value="msk">Москва</option>
    <option value="spb">Санкт-Петербург</option>
  </optgroup>
  <optgroup label="Казахстан">
    <option value="alm">Алматы</option>
  </optgroup>
</select>
```

### `<button>`

```html
<button type="submit">Отправить</button>  <!-- default type в <form> -->
<button type="button">Действие</button>   <!-- не сабмитит форму -->
<button type="reset">Сбросить</button>    <!-- сброс полей формы -->
```

<b class="text-on-semi-light">Важно:</b> `<button>` без `type` внутри `<form>` по умолчанию имеет `type="submit"`. Если кнопка не должна отправлять форму — явно указывай `type="button"`.

### `<fieldset>` и `<legend>`

Группировка связанных полей. Критично для a11y — скринридер объявляет `<legend>` как контекст группы.

```html
<fieldset>
  <legend>Способ оплаты</legend>
  <label><input type="radio" name="pay" value="card"> Карта</label>
  <label><input type="radio" name="pay" value="cash"> Наличные</label>
</fieldset>
```

### `<datalist>`

Автодополнение для `<input>` с предустановленными вариантами.

```html
<input list="browsers" name="browser">
<datalist id="browsers">
  <option value="Chrome">
  <option value="Firefox">
  <option value="Safari">
</datalist>
```

### `<output>`, `<progress>`, `<meter>`

```html
<!-- Результат вычисления -->
<output name="result">42</output>

<!-- Прогресс выполнения -->
<progress value="70" max="100">70%</progress>

<!-- Скалярное значение в известном диапазоне -->
<meter value="0.7" min="0" max="1" low="0.3" high="0.8" optimum="0.9">70%</meter>
```

## HTML5 Constraint Validation

HTML5 предоставляет **нативную валидацию без JavaScript**. Браузер показывает всплывающие подсказки и блокирует отправку формы.

### Атрибуты валидации

| Атрибут | Назначение | Пример |
|---|---|---|
| `required` | Обязательное поле | `<input required>` |
| `pattern` | Регулярное выражение | `<input pattern="[A-Za-z]{3,}">` |
| `min` / `max` | Мин/макс значение | `<input type="number" min="1" max="100">` |
| `minlength` / `maxlength` | Мин/макс длина | `<input minlength="3" maxlength="50">` |
| `step` | Шаг значения | `<input type="number" step="0.01">` |
| `type` | Формат (email, url, number) | `<input type="email">` — валидирует формат |

### CSS-псевдоклассы

```css
input:valid { border-color: green; }
input:invalid { border-color: red; }
input:required { border-left: 3px solid blue; }
input:placeholder-shown { opacity: 0.7; }
input:user-invalid { border-color: red; }  /* после взаимодействия, не сразу */
```

<span class="text-gray">`:invalid` применяется сразу при загрузке (пустое required поле = invalid). `:user-invalid` (новый) — только после того, как пользователь взаимодействовал с полем. Это удобнее для UX.</span>

### Constraint Validation API

JavaScript API для программной валидации.

```js
const input = document.querySelector('input');

// Проверка валидности
input.checkValidity();    // true/false, вызывает 'invalid' event при false
input.reportValidity();   // true/false + показывает нативную подсказку

// Кастомное сообщение об ошибке
input.setCustomValidity('Пароли не совпадают');
input.reportValidity();   // покажет "Пароли не совпадают"
input.setCustomValidity(''); // сброс (пустая строка = валидно)

// Объект validity — детальная информация
input.validity.valueMissing;   // required но пусто
input.validity.typeMismatch;   // type="email" но не email
input.validity.patternMismatch; // не соответствует pattern
input.validity.tooShort;       // короче minlength
input.validity.tooLong;        // длиннее maxlength
input.validity.rangeUnderflow; // меньше min
input.validity.rangeOverflow;  // больше max
input.validity.stepMismatch;   // не кратно step
input.validity.valid;          // всё ок
```

## FormData API

`FormData` — удобный API для сборки и отправки данных формы, включая файлы.

```js
const form = document.querySelector('form');

// Из существующей формы
const formData = new FormData(form);

// Чтение
formData.get('email');        // значение поля name="email"
formData.getAll('files');     // все файлы (для multiple)
formData.has('name');         // true/false

// Модификация
formData.set('email', 'new@mail.com');  // заменить
formData.append('tag', 'urgent');        // добавить (может быть несколько)
formData.delete('temp');                 // удалить

// Итерация
for (const [key, value] of formData) {
  console.log(key, value);
}

// Отправка через fetch
await fetch('/api/submit', {
  method: 'POST',
  body: formData, // Content-Type: multipart/form-data выставится автоматически
});
```

<b class="text-on-semi-light">Важно:</b> Не устанавливай `Content-Type` вручную при отправке `FormData` через fetch — браузер сам добавит `multipart/form-data` с правильным boundary. Ручная установка сломает отправку.

## События форм

### `submit`

```js
form.addEventListener('submit', (e) => {
  e.preventDefault(); // предотвращаем перезагрузку страницы
  const data = new FormData(form);
  // обработка...
});
```

### `input` vs `change`

| Событие | Когда срабатывает | Пример |
|---|---|---|
| `input` | **При каждом изменении** значения (каждый символ) | Живой поиск, подсчёт символов |
| `change` | **После завершения** изменения (blur для текста, выбор для select/checkbox) | Сохранение после редактирования |

```js
// input: "H" → "He" → "Hel" → "Hell" → "Hello" — 5 событий
input.addEventListener('input', (e) => {
  console.log(e.target.value); // каждый символ
});

// change: одно событие, когда поле потеряет фокус (или Enter)
input.addEventListener('change', (e) => {
  console.log(e.target.value); // финальное значение
});
```

<b class="text-on-semi-light">Для `<select>`, `<input type="checkbox/radio">`</b> — `change` срабатывает сразу при выборе (нет понятия «потеря фокуса»). `input` тоже срабатывает, но `change` — более распространённый выбор.

### Другие события

| Событие | Когда | Применение |
|---|---|---|
| `focus` / `blur` | Получение / потеря фокуса | Показ подсказок, валидация при blur |
| `focusin` / `focusout` | Как focus/blur, но **всплывают** | Делегирование через родительский элемент |
| `invalid` | Нативная валидация провалена | Кастомное оформление ошибок |
| `reset` | Нажата кнопка reset | Сброс кастомного состояния |

## На фронте: controlled vs uncontrolled

В React и Vue формы делятся на два подхода:

<b class="text-on-semi-light">Controlled (управляемые):</b> Состояние поля хранится в JS (state/ref). Каждое изменение проходит через обработчик.

```tsx
// React
const [email, setEmail] = useState('');
<input value={email} onChange={e => setEmail(e.target.value)} />
```

```vue
<!-- Vue -->
<input v-model="email" />
```

<b class="text-on-semi-light">Uncontrolled (неуправляемые):</b> Состояние в DOM. Значение читается при сабмите через `FormData` или `ref`.

```tsx
// React
const formRef = useRef<HTMLFormElement>(null);
const handleSubmit = () => {
  const data = new FormData(formRef.current!);
  console.log(data.get('email'));
};
<form ref={formRef}><input name="email" /></form>
```

<span class="text-gray">Для простых форм (контактная форма, подписка) uncontrolled + FormData часто проще. Controlled нужен, когда значение поля влияет на другие элементы (conditional rendering, live validation, derived state).</span>

## Частые вопросы на собесе

**В чём разница между `input` и `change`?**
`input` срабатывает при каждом изменении значения (каждый символ). `change` — после завершения (blur для текстовых, сразу для select/checkbox). Для live search — `input`, для save-on-leave — `change`.

**Как валидировать форму без JavaScript?**
Атрибутами: `required`, `pattern`, `min`/`max`, `minlength`/`maxlength`, `type="email/url"`. Браузер покажет нативные подсказки и заблокирует submit. Для кастомных сообщений — `setCustomValidity()`.

**Как отправить файл через fetch?**
`FormData` + fetch без ручной установки `Content-Type`. `new FormData(form)` автоматически собирает все поля, включая файлы. `enctype="multipart/form-data"` на `<form>` нужен только для нативной отправки.

**Зачем `type="button"` на `<button>`?**
Без `type` кнопка внутри `<form>` по умолчанию имеет `type="submit"` и отправит форму при клике. `type="button"` предотвращает это.

**Controlled vs uncontrolled — когда что?**
Controlled — когда значение поля влияет на UI в реальном времени (live validation, conditional fields, derived state). Uncontrolled — для простых форм, где значение нужно только при submit.
