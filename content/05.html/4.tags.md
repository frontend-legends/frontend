# Семантика и теги

Семантический HTML — это не про «красивый код». Это про **смысл разметки**: когда браузер, скринридер и поисковый бот одинаково правильно понимают структуру страницы. На senior-собесе спрашивают не «перечисли теги», а **«когда и почему использовать `<section>` вместо `<div>`»**.

---

## Зачем нужна семантика

| Кому выгодно | Как именно |
|---|---|
| **Доступность (a11y)** | Скринридеры используют семантику для навигации: landmarks, heading tree, lists |
| **SEO** | Поисковики понимают структуру: `<article>` = контент, `<nav>` = навигация, `<h1>` = главный заголовок |
| **Поддерживаемость** | Разработчик сразу видит структуру: `<header>` говорит больше, чем `<div class="header">` |
| **Парсеры и боты** | Reader mode, RSS, AI-агенты извлекают контент, опираясь на семантику |

> Семантический HTML — это **первый уровень доступности**. Без него ARIA превращается в костыли, а не в усиление.

---

## Семантические элементы HTML5

### Структурные (Sectioning)

Определяют **области** документа. Каждый создаёт свой implicit ARIA landmark.

| Элемент | Назначение | ARIA-роль | Правила |
|---|---|---|---|
| `<header>` | Шапка страницы или секции | `banner` (если потомок `<body>`) | Может быть несколько: один для страницы, другие — для `<article>`, `<section>` |
| `<footer>` | Подвал страницы или секции | `contentinfo` (если потомок `<body>`) | Аналогично `<header>` — может быть внутри секций |
| `<main>` | Основное содержимое | `main` | **Только один** на страницу. Не вкладывается в `<article>`, `<aside>`, `<header>`, `<footer>`, `<nav>` |
| `<nav>` | Блок навигации | `navigation` | Для основных навигационных ссылок. Не для каждого набора ссылок |
| `<section>` | Тематическая группировка | `region` (если есть `aria-label`) | Должен иметь заголовок. Без заголовка — вероятно, нужен `<div>` |
| `<article>` | Самостоятельный контент | `article` | Контент, который имеет смысл вне контекста: пост, комментарий, карточка |
| `<aside>` | Побочный контент | `complementary` | Не основной контент: sidebar, реклама, related links |

### Текстовые и контентные

| Элемент | Назначение | Пример |
|---|---|---|
| `<figure>` + `<figcaption>` | Иллюстрация с подписью | Изображение, диаграмма, код с описанием |
| `<details>` + `<summary>` | Раскрываемый блок (нативный accordion) | FAQ, дополнительная информация |
| `<time datetime="...">` | Дата/время в машиночитаемом формате | `<time datetime="2026-03-25">25 марта</time>` |
| `<mark>` | Выделение (highlight) | Результат поиска |
| `<dialog>` | Нативное модальное/немодальное окно | `dialog.showModal()` — с backdrop и focus trap |
| `<address>` | Контактная информация автора | Email, ссылки — не физический адрес |
| `<blockquote>` + `<cite>` | Цитата с источником | `<blockquote cite="url">...<cite>Автор</cite></blockquote>` |

## `<section>` vs `<article>` vs `<div>`

Один из самых частых вопросов. Простое правило:

```text
Контент самодостаточен? (пост, комментарий, виджет)
  ├── Да  → <article>
  └── Нет → Тематическая группировка с заголовком?
              ├── Да  → <section>
              └── Нет → <div> (нет семантического значения)
```

<b class="text-on-semi-light">Ключевые отличия:</b>

- **`<article>`** — контент, который имеет смысл **вне контекста**: статья в блоге, комментарий, карточка товара. Можно вынести в RSS — и он не потеряет смысла.
- **`<section>`** — тематическая группировка **внутри** страницы: «О нас», «Контакты», «Преимущества». Без заголовка `<section>` не имеет семантического смысла.
- **`<div>`** — **нет семантики**. Используй для стилизации и layout, когда ни `<article>`, ни `<section>` не подходят.

```html
<!-- article: самостоятельный контент -->
<article>
  <h2>Как работает CRP</h2>
  <p>Critical Rendering Path — это...</p>
</article>

<!-- section: тематическая группировка -->
<section>
  <h2>Преимущества</h2>
  <ul>...</ul>
</section>

<!-- div: layout-обёртка без семантики -->
<div class="grid-container">
  <article>...</article>
  <article>...</article>
</div>
```

## Block vs Inline

В HTML5 деление на block/inline **больше не существует** на уровне спецификации — его заменили content models. Но в CSS `display: block` и `display: inline` по-прежнему определяют поведение элементов.

### Дефолтное поведение

| Тип | CSS `display` | Занимает всю ширину | Перенос строки | Примеры |
|---|---|---|---|---|
| **Block** | `block` | Да | До и после | `<div>`, `<p>`, `<h1>`–`<h6>`, `<section>`, `<form>` |
| **Inline** | `inline` | Нет | Нет | `<span>`, `<a>`, `<strong>`, `<em>`, `<code>` |
| **Inline-block** | `inline-block` | Нет | Нет | `<img>`, `<input>`, `<button>` (replaced elements) |

<b class="text-on-semi-light">Replaced elements:</b> `<img>`, `<video>`, `<input>`, `<textarea>` — формально inline, но ведут себя как inline-block: к ним применимы `width`, `height`, `margin`/`padding` со всех сторон. Это **replaced elements** — их содержимое определяется не HTML, а внешним источником.

## Void elements (пустые элементы)

Элементы, которые **не имеют закрывающего тега** и не могут содержать потомков.

```html
<br>     <!-- перенос строки -->
<hr>     <!-- тематический разделитель -->
<img>    <!-- изображение -->
<input>  <!-- поле ввода -->
<meta>   <!-- метаинформация -->
<link>   <!-- ресурсная ссылка -->
<source> <!-- источник медиа -->
<area>   <!-- область карты -->
<col>    <!-- колонка таблицы -->
<embed>  <!-- встраиваемый объект -->
<wbr>    <!-- возможный перенос слова -->
<track>  <!-- субтитры для видео -->
```

<span class="text-gray">В HTML5 самозакрывающий синтаксис `<br />` допустим, но необязателен. `<br>` и `<br />` идентичны. В JSX (React) `<br />` обязателен, потому что JSX ближе к XML.</span>

## Document Outline (иерархия заголовков)

Заголовки `<h1>`–`<h6>` формируют **outline (структуру)** документа, которую используют скринридеры и поисковики.

### Правила

- **Один `<h1>` на страницу** — главный заголовок страницы (SEO best practice)
- Не пропускай уровни: `<h1>` → `<h2>` → `<h3>`, не `<h1>` → `<h3>`
- Каждый `<section>` / `<article>` может начинаться с `<h2>` и далее

**Плохо:**

```html
<h1>Главная</h1>
<h3>О нас</h3>          <!-- Пропущен h2 -->
<h4>Команда</h4>
<h2>Контакты</h2>       <!-- Нарушена иерархия -->
```

**Хорошо:**

```html
<h1>Frontend Legends</h1>
  <h2>Computer Science</h2>
    <h3>Введение</h3>
    <h3>ООП</h3>
  <h2>HTML</h2>
    <h3>Семантика</h3>
    <h3>Формы</h3>
```

<span class="text-gray">Спецификация HTML5 предлагала «алгоритм outline», позволяющий начинать с `<h1>` в каждой `<section>`. Но ни один браузер и скринридер его не реализовал. На практике используй плоскую иерархию `<h1>`–`<h6>`.</span>

## Устаревшие теги

| Устаревший тег | Что делал | Чем заменить |
|---|---|---|
| `<center>` | Центрирование | CSS: `text-align: center` или `margin: 0 auto` |
| `<font>` | Шрифт и цвет | CSS: `font-family`, `color`, `font-size` |
| `<marquee>` | Бегущая строка | CSS animations или JS |
| `<b>` | Жирный текст | `<strong>` (семантическое выделение) или CSS `font-weight` |
| `<i>` | Курсив | `<em>` (семантическое ударение) или CSS `font-style` |
| `<strike>` / `<s>` | Зачёркнутый | `<del>` (удалённый контент) или CSS `text-decoration` |
| `<frame>` / `<frameset>` | Фреймы | `<iframe>` или SPA-навигация |

<span class="text-gray">Технически `<b>` и `<i>` не deprecated в HTML5 — они переопределены: `<b>` = привлечение внимания без важности, `<i>` = альтернативный голос (технические термины, мысли). Но на практике `<strong>` и `<em>` предпочтительнее из-за семантической ясности.</span>

## Плохо / Хорошо: `<div>` soup vs семантика

**Плохо — «div soup»:**

```html
<div class="header">
  <div class="nav">
    <div class="nav-item"><a href="/">Главная</a></div>
    <div class="nav-item"><a href="/about">О нас</a></div>
  </div>
</div>
<div class="main">
  <div class="article">
    <div class="title">Заголовок</div>
    <div class="text">Текст статьи...</div>
  </div>
</div>
<div class="footer">
  <div class="copyright">© 2026</div>
</div>
```

**Хорошо — семантическая разметка:**

```html
<header>
  <nav aria-label="Основная навигация">
    <ul>
      <li><a href="/">Главная</a></li>
      <li><a href="/about">О нас</a></li>
    </ul>
  </nav>
</header>
<main>
  <article>
    <h1>Заголовок</h1>
    <p>Текст статьи...</p>
  </article>
</main>
<footer>
  <p><small>© 2026</small></p>
</footer>
```

<b class="text-on-semi-light">Разница:</b> Скринридер в первом случае видит «группу div'ов». Во втором — landmarks: header, nav, main, article, footer. Пользователь может перейти к `<main>` одной клавишей.

## Частые вопросы на собесе

**В чём разница между `<section>` и `<article>`?**
`<article>` — самостоятельный контент (пост, комментарий), имеет смысл вне контекста. `<section>` — тематическая группировка внутри страницы, требует заголовка. `<article>` может содержать `<section>`, и `<section>` может содержать `<article>`.

**Зачем семантические теги, если можно всё на `<div>` сделать?**
Доступность: скринридеры используют landmarks для навигации. SEO: поисковики лучше понимают структуру. Поддерживаемость: `<nav>` говорит больше, чем `<div class="nav">`. Reader mode и парсеры зависят от семантики.

**Block vs inline — это HTML или CSS?**
В HTML5 спецификации — content models (flow, phrasing и т.д.). Block/inline — это CSS display. Но дефолтный display элементов соответствует их исторической категории: `<div>` → block, `<span>` → inline.

**Почему один `<h1>` на страницу?**
SEO best practice: `<h1>` = главный заголовок страницы. Множественные `<h1>` (по одному в каждой `<section>`) предполагались outline-алгоритмом HTML5, но он не реализован ни в одном браузере. На практике — один `<h1>`, далее `<h2>`–`<h6>`.

**Что такое replaced element?**
Элемент, содержимое которого определяется внешним ресурсом, а не HTML: `<img>`, `<video>`, `<input>`, `<textarea>`, `<canvas>`. Они ведут себя как inline-block и принимают `width`/`height`, хотя формально inline.
