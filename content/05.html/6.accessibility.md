# Доступность (a11y)

Веб-доступность (accessibility, сокращённо **a11y**) — это практика создания сайтов, которыми могут пользоваться **все люди**, включая пользователей скринридеров, клавиатурной навигации и ассистивных технологий. На senior-уровне знание a11y — это не «приятный бонус», а требование: крупные компании (Google, Microsoft, Apple) проверяют его на собеседованиях, а в ряде стран доступность — юридическое обязательство.

---

## Зачем это нужно

| Причина | Пояснение |
|---|---|
| **Юридическая** | ADA (США), EN 301 549 (ЕС), ФЗ №419 (РФ) требуют доступности для госсайтов и публичных сервисов |
| **Бизнес** | ~15% населения — люди с инвалидностью. Это огромная аудитория |
| **SEO** | Семантический HTML и alt-тексты улучшают индексацию |
| **UX для всех** | Субтитры помогают в шумном метро, высокий контраст — на солнце, клавиатурная навигация — power users |
| **Этическая** | Веб создан для всех. Исключение людей — проблема проектирования, не пользователя |

---

## WCAG 2.1

**Web Content Accessibility Guidelines (WCAG)** — стандарт W3C, определяющий требования к доступности. Основан на четырёх принципах **POUR**:

| Принцип | Английский | Что значит | Примеры |
|---|---|---|---|
| **Воспринимаемость** | Perceivable | Контент должен быть доступен через органы чувств | alt для изображений, субтитры для видео, контраст текста |
| **Управляемость** | Operable | Интерфейс управляем всеми способами | Клавиатурная навигация, нет ловушек фокуса, достаточно времени |
| **Понятность** | Understandable | Контент и интерфейс понятны | Понятные лейблы, предсказуемое поведение, помощь при ошибках |
| **Надёжность** | Robust | Работает с разными user agents | Валидный HTML, ARIA, совместимость с ассистивными технологиями |

### Уровни соответствия

| Уровень | Требование | Пример |
|---|---|---|
| **A** | Минимальный | Все изображения имеют alt, контент доступен с клавиатуры |
| **AA** | Стандартный (целевой для большинства) | Контраст 4.5:1, resize до 200%, видимый фокус |
| **AAA** | Расширенный | Контраст 7:1, язык жестов для видео, упрощённый текст |

<b class="text-on-semi-light">На практике:</b> Большинство юридических требований и корпоративных стандартов ориентируются на **WCAG 2.1 Level AA**.

## Семантический HTML как основа

Первое правило доступности: **используй нативные HTML-элементы**. Они уже имеют встроенную доступность: роли, состояния, клавиатурное взаимодействие.

**Плохо:**

```html
<!-- div не фокусируется, не имеет роли, не реагирует на Enter/Space -->
<div class="button" onclick="handleClick()">Отправить</div>
```

**Хорошо:**

```html
<!-- button: фокусируется, role="button", реагирует на Enter/Space -->
<button onclick="handleClick()">Отправить</button>
```

Чтобы сделать `<div>` доступным как кнопку, нужно добавить: `role="button"`, `tabindex="0"`, обработчики `keydown` для Enter и Space, стили фокуса. **Нативный `<button>` делает всё это бесплатно.**

| Задача | Нативный элемент | Не нужно вручную |
|---|---|---|
| Кнопка | `<button>` | role, tabindex, keydown |
| Ссылка | `<a href>` | role, tabindex, cursor |
| Чекбокс | `<input type="checkbox">` | role, aria-checked, keydown |
| Выпадающий список | `<select>` | role, aria-expanded, навигация стрелками |
| Модалка | `<dialog>` | role, aria-modal, focus trap, Escape |

## ARIA

**Accessible Rich Internet Applications (ARIA)** — набор атрибутов, дополняющих семантику HTML для скринридеров и других ассистивных технологий.

### Первое правило ARIA

> **Не используй ARIA, если нативный HTML-элемент решает задачу.** ARIA не добавляет поведение — только метаинформацию. `role="button"` не делает div кликабельным.

### Roles (роли)

Определяют **тип** элемента для ассистивных технологий.

```html
<div role="alert">Ошибка сохранения!</div>
<div role="tablist">
  <button role="tab" aria-selected="true">Таб 1</button>
  <button role="tab" aria-selected="false">Таб 2</button>
</div>
<div role="tabpanel">Содержимое таба</div>
```

| Роль | Назначение | Нативный аналог |
|---|---|---|
| `button` | Кнопка | `<button>` |
| `link` | Ссылка | `<a href>` |
| `alert` | Уведомление (объявляется сразу) | — |
| `dialog` | Модальное окно | `<dialog>` |
| `tablist`, `tab`, `tabpanel` | Табы | — |
| `navigation` | Навигация | `<nav>` |
| `banner` | Шапка страницы | `<header>` (прямой потомок body) |

### States & Properties (атрибуты)

| Атрибут | Что делает | Пример |
|---|---|---|
| `aria-label` | Текстовая метка (для скринридера) | `<button aria-label="Закрыть">×</button>` |
| `aria-labelledby` | Ссылка на элемент с текстом метки | `<dialog aria-labelledby="title-id">` |
| `aria-describedby` | Дополнительное описание | `<input aria-describedby="hint-id">` |
| `aria-hidden="true"` | Скрывает от скринридера (но видимо!) | Декоративные иконки |
| `aria-live` | Объявляет обновления контента | `<div aria-live="polite">` — новости, чат |
| `aria-expanded` | Состояние раскрытия | `<button aria-expanded="false">Меню</button>` |
| `aria-controls` | Ссылка на контролируемый элемент | `<button aria-controls="menu-id">` |
| `aria-current` | Текущий элемент в наборе | `<a aria-current="page">Главная</a>` |
| `aria-required` | Обязательное поле | `<input aria-required="true">` |
| `aria-invalid` | Поле с ошибкой | `<input aria-invalid="true">` |

### `aria-label` vs `aria-labelledby`

```html
<!-- aria-label: текст прямо в атрибуте -->
<button aria-label="Закрыть модалку">×</button>

<!-- aria-labelledby: ссылка на существующий элемент -->
<h2 id="dialog-title">Подтверждение</h2>
<dialog aria-labelledby="dialog-title">
  <p>Вы уверены?</p>
</dialog>
```

<b class="text-on-semi-light">Приоритет:</b> `aria-labelledby` > `aria-label` > нативная метка (`<label>`, текст кнопки). Если есть видимый заголовок — используй `aria-labelledby`. Если нет — `aria-label`.

## Навигация с клавиатуры

### `tabindex`

| Значение | Поведение |
|---|---|
| `tabindex="0"` | Элемент попадает в порядок Tab (в порядке DOM) |
| `tabindex="-1"` | Фокусируется программно (`element.focus()`), но не через Tab |
| `tabindex="1+"` | **Антипаттерн!** Изменяет естественный порядок фокуса — путает пользователей |

```html
<!-- Нефокусируемый элемент сделан фокусируемым -->
<div tabindex="0" role="button" onclick="handle()">Кнопка</div>

<!-- Элемент можно сфокусировать программно, но Tab его пропустит -->
<div tabindex="-1" id="notification">Уведомление</div>
<script>
  document.getElementById('notification').focus(); // OK
</script>
```

### Focus trap (ловушка фокуса)

В модальных окнах фокус должен **оставаться внутри** модалки. Tab на последнем элементе → фокус на первом элементе модалки (не на элементе за ней).

```js
// Упрощённая реализация focus trap
dialog.addEventListener('keydown', (e) => {
  if (e.key !== 'Tab') return;

  const focusable = dialog.querySelectorAll(
    'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
  );
  const first = focusable[0];
  const last = focusable[focusable.length - 1];

  if (e.shiftKey && document.activeElement === first) {
    e.preventDefault();
    last.focus();
  } else if (!e.shiftKey && document.activeElement === last) {
    e.preventDefault();
    first.focus();
  }
});
```

<span class="text-gray">Нативный `<dialog>` через `showModal()` реализует focus trap автоматически. `inert` на остальном контенте усиливает изоляцию. Подробнее → [Атрибуты](/content/html/attributes).</span>

### Skip link

Позволяет пользователям клавиатуры перепрыгнуть через навигацию к основному контенту.

```html
<a href="#main" class="skip-link">Перейти к содержимому</a>
<nav>...</nav>
<main id="main">...</main>
```

```css
.skip-link {
  position: absolute;
  top: -100%;
  left: 0;
}
.skip-link:focus {
  top: 0;
  z-index: 1000;
}
```

## Скринридеры

Скринридеры (VoiceOver, NVDA, JAWS, TalkBack) **не видят** страницу визуально. Они строят модель из:

1. **Accessibility Tree** — дерево, построенное из DOM + ARIA. Отличается от DOM: `aria-hidden` элементы удалены, `role` переопределяет семантику.
2. **Landmarks** — навигационные точки: `<main>`, `<nav>`, `<header>`, `<footer>`, `<aside>`. Пользователь перемещается между ними горячими клавишами.
3. **Heading tree** — дерево заголовков `<h1>`–`<h6>`. Основной способ навигации по длинным страницам.
4. **Form labels** — скринридер объявляет имя поля через `<label>`, `aria-label` или `aria-labelledby`.

### `aria-live` регионы

Для контента, который обновляется динамически (уведомления, чат, результаты поиска):

```html
<!-- polite: объявит, когда скринридер закончит текущее -->
<div aria-live="polite" role="status">
  Найдено 42 результата
</div>

<!-- assertive: прерывает и объявляет немедленно -->
<div aria-live="assertive" role="alert">
  Ошибка: не удалось сохранить
</div>
```

## Распространённые ошибки

| Ошибка | Проблема | Решение |
|---|---|---|
| `<img>` без `alt` | Скринридер читает имя файла | Добавь `alt`. Декоративные: `alt=""` |
| `<div onclick>` вместо `<button>` | Не фокусируется, не объявляется | Используй `<button>` |
| `<input>` без `<label>` | Скринридер не знает, что за поле | `<label for="id">` или `aria-label` |
| Информация только цветом | Дальтонизм — ~8% мужчин | Добавь текст, иконку или паттерн |
| Нет `lang` на `<html>` | Скринридер использует неправильное произношение | `<html lang="ru">` |
| Скрытие через `display: none` от скринридера | `display: none` скрывает от всех | `aria-hidden="true"` для визуально видимого, но a11y-скрытого |
| `tabindex > 0` | Ломает естественный порядок навигации | Используй порядок DOM, `tabindex="0"` или `-1` |
| Автовоспроизведение видео со звуком | Дезориентирует, мешает скринридеру | `muted` по умолчанию или не `autoplay` |

## a11y-чеклист для компонентов

### Модальное окно (Dialog)

- Используй `<dialog>` + `showModal()` (нативный focus trap, Escape, backdrop)
- `aria-labelledby` → ссылка на заголовок модалки
- При открытии фокус на первый интерактивный элемент
- Escape закрывает модалку
- `inert` на контенте за модалкой
- Возврат фокуса на trigger после закрытия

### Выпадающее меню (Dropdown)

- Trigger: `aria-expanded="true/false"`, `aria-controls="menu-id"`
- Menu: `role="menu"`, items: `role="menuitem"`
- Стрелки вверх/вниз перемещают фокус между пунктами
- Escape закрывает меню, фокус на trigger

### Табы (Tabs)

- Container: `role="tablist"`
- Tabs: `role="tab"`, `aria-selected="true/false"`, `aria-controls="panel-id"`
- Panels: `role="tabpanel"`, `aria-labelledby="tab-id"`
- Стрелки влево/вправо переключают табы
- Неактивные panels: `hidden` или `tabindex="-1"`

### Уведомления (Toast / Alert)

- `role="alert"` или `role="status"` + `aria-live`
- `assertive` для ошибок, `polite` для информационных
- Автоматическое скрытие — не для критичных сообщений (пользователь может не успеть)

## Частые вопросы на собесе

**В чём разница между `aria-label` и `aria-labelledby`?**
`aria-label` — текст метки прямо в атрибуте. `aria-labelledby` — ссылка (по id) на элемент, чей текст станет меткой. `aria-labelledby` приоритетнее и удобнее, когда метка уже есть на странице (заголовок модалки).

**Какие значения `tabindex` бывают и когда использовать?**
`0` — элемент в потоке Tab (порядок DOM). `-1` — фокусируется только программно. Положительные значения — антипаттерн, ломают естественный порядок.

**Как сделать модалку доступной?**
Нативный `<dialog>` + `showModal()` — самый простой путь. Он даёт focus trap, Escape-закрытие, backdrop, role="dialog". Добавь `aria-labelledby` на заголовок и `inert` на контент за модалкой.

**Что такое accessibility tree?**
Упрощённое представление DOM, которое видят ассистивные технологии. Строится из DOM + ARIA: `aria-hidden` элементы удалены, `role` и `aria-*` переопределяют семантику. Можно инспектировать в DevTools (Accessibility tab).

**Зачем пустой `alt=""`?**
Для декоративных изображений. Скринридер полностью пропустит `<img alt="">`. Без `alt` скринридер прочитает имя файла (`IMG_20240301.jpg`), что бесполезно и раздражает.
