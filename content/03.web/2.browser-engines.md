# Архитектура браузера и движки

Браузер — это сложное многопроцессное приложение. Понимание его архитектуры критически важно для senior-интервью: вопрос «как работает браузер?» — один из самых частых и показательных. Разберём всё: от процессов до garbage collection.

---

## Многопроцессная архитектура

<b class="text-on-semi-light">Chromium как эталонная архитектура</b>

<span class="text-gray">Современные браузеры (Chrome, Edge, Opera, Brave, Arc) основаны на Chromium и используют многопроцессную модель:</span>

```
┌─────────────────────────────────────────────────────────┐
│                    Browser Process                       │
│         (UI, навигация, координация всех процессов)      │
├──────────┬──────────┬──────────┬────────────────────────┤
│ Renderer │ Renderer │ Renderer │   ...per site/tab      │
│ Process  │ Process  │ Process  │                        │
│ (tab A)  │ (tab B)  │ (iframe) │                        │
├──────────┴──────────┴──────────┴────────────────────────┤
│  GPU Process         │  Network Process                  │
│  (compositing,       │  (HTTP, DNS, TLS,                 │
│   WebGL, WebGPU)     │   resource loading)                │
├──────────────────────┴───────────────────────────────────┤
│  Utility Processes                                       │
│  (audio decoding, video decoding, file system, etc.)     │
└─────────────────────────────────────────────────────────┘
```

<b class="text-on-semi-light">Почему многопроцессность?</b>

| Причина | Объяснение |
|---------|------------|
| **Security** | Каждый renderer работает в песочнице (sandbox). Даже при эксплойте в рендерере — атакующий ограничен процессом без доступа к FS/сети |
| **Stability** | Падение одной вкладки не убивает весь браузер. Crash isolation — каждый renderer может упасть независимо |
| **Performance** | Параллельная работа на разных ядрах CPU. Heavy JS в одной вкладке не блокирует другие |

<b class="text-on-semi-light">Site Isolation</b>

<span class="text-gray">Начиная с Chrome 67, каждый cross-origin iframe получает отдельный renderer process. Это фундаментальная защита от атак Spectre/Meltdown: даже если злоумышленник эксплуатирует спекулятивное выполнение CPU, он не может прочитать память другого origin — она в другом процессе.</span>

Связанные HTTP-заголовки:
- **COOP** (`Cross-Origin-Opener-Policy`) — изолирует `window.opener` между origin'ами
- **COEP** (`Cross-Origin-Embedder-Policy`) — требует явное разрешение для cross-origin ресурсов
- Вместе они активируют **cross-origin isolated** контекст

```
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
```

<span class="text-gray">В cross-origin isolated контексте доступны:</span>
- `SharedArrayBuffer` — разделяемая память между потоками (Workers)
- `performance.now()` с полной точностью (без rounded до 100μs)
- `performance.measureUserAgentSpecificMemory()` — измерение потребления памяти

<span class="text-gray">Проверка в коде:</span>

```js
if (crossOriginIsolated) {
  // SharedArrayBuffer доступен
  const sab = new SharedArrayBuffer(1024);
}
```

## Компоненты рендерер-процесса

<b class="text-on-semi-light">Внутренняя структура</b>

```
Renderer Process
├── HTML Parser → DOM Tree
├── CSS Parser → CSSOM Tree
├── JavaScript Engine (V8 / SpiderMonkey / JSC)
├── Layout Engine
├── Paint Engine
└── Compositor Thread
```

<span class="text-gray">**HTML Parser** — парсит поток байтов в DOM-дерево. Работает инкрементально, может быть приостановлен скриптами (без `async`/`defer`).</span>

<span class="text-gray">**CSS Parser** — строит CSSOM (CSS Object Model). Блокирует рендеринг: браузер не начнёт paint без полного CSSOM.</span>

<span class="text-gray">**JavaScript Engine** — компилирует и исполняет JS. V8 в Chromium, SpiderMonkey в Firefox, JavaScriptCore в Safari.</span>

<span class="text-gray">**Layout Engine** — вычисляет геометрию каждого элемента (position, size). Результат — Layout Tree.</span>

<span class="text-gray">**Paint Engine** — генерирует display lists (инструкции рисования) для каждого layout-объекта.</span>

<span class="text-gray">**Compositor Thread** — работает отдельно от main thread. Разбивает страницу на layers, растеризует тайлы, собирает финальный кадр. Именно поэтому CSS-анимации на `transform`/`opacity` работают плавно даже при загруженном main thread.</span>

> Подробнее о rendering pipeline — см. [Rendering](/content/web/rendering)

## Движки браузеров

<b class="text-on-semi-light">Сравнительная таблица</b>

| Браузер | Rendering Engine | JS Engine | Доля рынка | Особенности |
|---------|-----------------|-----------|-----------|-------------|
| Chrome, Edge, Opera, Brave, Arc | **Blink** | **V8** | ~70% | Fork WebKit (2013). Самый быстрый feature adoption |
| Firefox | **Gecko** | **SpiderMonkey** | ~3% | Полностью независимый. Stylo — CSS-движок на Rust. Фокус на приватность |
| Safari | **WebKit** | **JavaScriptCore (JSC)** | ~18% | Старейший из выживших. Консервативное принятие фич. Обязателен на iOS (до EU DMA). Энергоэффективен |

> **Риск монокультуры.** Доминирование Blink (~70%) напоминает эру IE6. Когда один движок определяет стандарты де-факто, страдает открытость веба. Поддержка Firefox и Safari-specific тестирование — это не только QA, но и ответственность перед экосистемой.

<b class="text-on-semi-light">Blink vs Gecko vs WebKit — ключевые архитектурные отличия</b>

| Аспект | Blink | Gecko | WebKit |
|--------|-------|-------|--------|
| Процессная модель | Site Isolation (per-origin) | Fission (аналог Site Isolation, с 2021) | Per-tab (iOS: per-WebView) |
| CSS-движок | Собственный (C++) | Stylo (Rust, параллельный) | Собственный (C++) |
| Rendering pipeline | LayoutNG → Paint → Compositing | WebRender (GPU-first, Rust) | Layer-based compositing |
| JS compilation | Ignition → Sparkplug → Maglev → TurboFan | Baseline → Warp (IonMonkey successor) | LLInt → Baseline → DFG → FTL (B3/Air) |

## V8 в деталях (senior-critical)

<b class="text-on-semi-light">Compilation Pipeline</b>

<span class="text-gray">V8 использует многоуровневую компиляцию (tiered compilation). Каждый уровень даёт более оптимизированный, но более дорогой в генерации код:</span>

```
Source Code (UTF-16)
  │
  ▼
Scanner (tokenization)
  │
  ▼
Parser → AST (Abstract Syntax Tree)
  │
  ▼
Ignition (interpreter → bytecode)
  │  ← Быстрый старт, без оптимизаций
  ▼
Sparkplug (baseline compiler)
  │  ← Быстрая компиляция, прямая трансляция bytecode → machine code
  ▼
Maglev (mid-tier compiler)
  │  ← SSA-based IR, умеренные оптимизации
  ▼
TurboFan (optimizing compiler)
     ← Спекулятивная оптимизация на основе type feedback
     ← Inlining, escape analysis, loop unrolling
     ← Результат: near-native performance
```

<span class="text-gray">Sparkplug появился в 2021 как прослойка между Ignition и TurboFan. Maglev — в 2023, для заполнения разрыва между Sparkplug и TurboFan. Оба уровня значительно снизили overhead «холодного старта».</span>

<b class="text-on-semi-light">Представление чисел в V8</b>

<span class="text-gray">V8 использует тегированные указатели (tagged pointers) для оптимизации работы с числами:</span>

| Тип | Описание | Диапазон |
|-----|----------|----------|
| **Smi** (Small Integer) | Целое число, закодированное прямо в указателе (без аллокации) | -2³¹ до 2³¹-1 (64-bit) |
| **HeapNumber** | Число в куче (double, IEEE 754) | Любое число с плавающей точкой |
| **Mutable HeapNumber** | Оптимизация для полей объектов — число обновляется in-place | Любое число с плавающей точкой |

```js
// Smi — без аллокации, максимальная производительность
const count = 42;

// HeapNumber — аллокация в куче
const pi = 3.14159;
const big = 2 ** 32;

// Практический совет: используйте целые числа где возможно
// для массивов — V8 выбирает оптимальный internal representation:
const packed_smi = [1, 2, 3];         // PACKED_SMI_ELEMENTS
const packed_double = [1.1, 2.2];     // PACKED_DOUBLE_ELEMENTS
const packed_obj = [1, "two", {}];    // PACKED_ELEMENTS (generic)
const holey = [1, , 3];              // HOLEY_SMI_ELEMENTS (дыра!)

// Lattice деградации (необратимо!):
// PACKED_SMI → PACKED_DOUBLE → PACKED → HOLEY_SMI → HOLEY_DOUBLE → HOLEY
```

> **Для интервью:** массивы в V8 имеют внутренние типы. Переход «вниз» по lattice необратим. Никогда не создавайте holey arrays (массивы с дырами) без необходимости — они медленнее.

### Hidden Classes (Maps) и Inline Caching

<b class="text-on-semi-light">Концепция Hidden Classes</b>

<span class="text-gray">V8 создаёт внутреннюю структуру — **Map** (hidden class) — для каждой уникальной «формы» объекта. Объекты с одинаковым порядком свойств разделяют одну Map, что позволяет V8 использовать inline caching для быстрого доступа к свойствам.</span>

```js
// Хорошо: все объекты одной формы (monomorphic)
class Point {
  constructor(x, y) {
    this.x = x; // Map transition: {} → {x}
    this.y = y; // Map transition: {x} → {x, y}
  }
}
const p1 = new Point(1, 2); // same Map
const p2 = new Point(3, 4); // same Map → inline cache hit

// Плохо: разные формы (polymorphic → megamorphic)
const a = { x: 1, y: 2 };
const b = { y: 2, x: 1 }; // different property order → different Map!

// Плохо: добавление свойств после конструктора
const obj = {};
obj.a = 1; // Map transition
obj.b = 2; // Map transition
if (condition) obj.c = 3; // conditional → branch в Maps
```

<b class="text-on-semi-light">Уровни полиморфизма Inline Cache</b>

| Состояние | Описание | Производительность |
|-----------|----------|-------------------|
| **Monomorphic** | Одна Map на callsite | Максимальная — прямой offset lookup |
| **Polymorphic** | 2–4 Maps | Линейный поиск по массиву Maps |
| **Megamorphic** | >4 Maps | Fallback на hash lookup, кэш бесполезен |

> **Правило для интервью:** инициализируйте все свойства объекта в конструкторе, в одинаковом порядке. Не добавляйте свойства условно. Это обеспечит monomorphic inline caching.

### Деоптимизация

<b class="text-on-semi-light">Что вызывает bailout из TurboFan обратно в Ignition</b>

<span class="text-gray">TurboFan компилирует код на основе **предположений** (speculative optimization). Когда предположение нарушается — происходит **деоптимизация**: скомпилированный код выбрасывается, и выполнение откатывается к интерпретатору Ignition.</span>

Типичные причины деоптимизации:
- **Смена типа в горячей функции:** функция принимала только `number`, вдруг пришёл `string`
- **Изменение формы объекта** после JIT-компиляции
- **Выход за границы массива** (out-of-bounds access заставляет V8 проверять prototype chain)
- **Нетривиальное использование `arguments`** — деструктуризация, передача в другие функции
- **`delete` на свойствах** — переводит объект в slow mode (dictionary mode)

```js
// Пример деоптимизации по смене типа
function add(a, b) {
  return a + b;
}

// V8 оптимизирует под числа (Smi/HeapNumber)
for (let i = 0; i < 100_000; i++) {
  add(i, i + 1); // hot loop → TurboFan compiles with number speculation
}

add("hello", "world"); // DEOPT! Тип не совпал → bailout → recompile
```

<span class="text-gray">Деоптимизация — дорогая операция: скомпилированный machine code выбрасывается, создаётся новый bytecode frame, execution возвращается в Ignition. Повторная оптимизация может произойти, но с менее агрессивными предположениями.</span>

## Garbage Collection в V8 (senior-critical)

<b class="text-on-semi-light">Generational Hypothesis</b>

<span class="text-gray">Большинство объектов «умирают молодыми» — создаются и становятся недоступными почти сразу. V8 использует эту статистику, разделяя heap на поколения:</span>

```
V8 Heap
├── Young Generation (1–8 MB)
│   ├── From-Space (Nursery)
│   └── To-Space
└── Old Generation (до сотен MB)
    ├── Old Object Space
    ├── Large Object Space (>kMaxRegularHeapObjectSize)
    ├── Code Space (JIT-compiled code)
    └── Map Space (hidden classes)
```

<b class="text-on-semi-light">Young Generation — Scavenger (Minor GC)</b>

<span class="text-gray">Алгоритм: **semi-space copying** (Cheney's algorithm).</span>

1. Новые объекты аллоцируются в **from-space** (bump pointer allocation — O(1))
2. Когда from-space заполняется — запускается Scavenge
3. Живые объекты копируются в **to-space**, мёртвые — остаются и уничтожаются
4. From-space и to-space меняются ролями
5. Объекты, пережившие **2 scavenge**, промотируются в Old Generation

<span class="text-gray">Характеристики: ~1–2ms паузы, происходит часто (каждые несколько сотен миллисекунд под нагрузкой).</span>

<b class="text-on-semi-light">Old Generation — Mark-Sweep-Compact (Major GC)</b>

| Фаза | Описание |
|------|----------|
| **Mark** | Обход от корней (stack, global, handles) — пометка всех достижимых объектов |
| **Sweep** | Освобождение памяти непомеченных объектов. Возвращение в free lists |
| **Compact** | Дефрагментация: перемещение объектов для уменьшения фрагментации. Самая дорогая фаза |

<b class="text-on-semi-light">Orinoco — инфраструктура GC в V8</b>

<span class="text-gray">Orinoco — набор техник для минимизации stop-the-world пауз:</span>

- **Parallel** — несколько GC-потоков работают одновременно (параллельный scavenge/marking)
- **Incremental marking** — чередование marking и выполнения JS (write barriers отслеживают изменения)
- **Concurrent sweeping/compacting** — выполняется параллельно с JS на отдельных потоках
- **Lazy sweeping** — sweeping происходит по мере необходимости, а не сразу

> **Результат:** GC-паузы практически незаметны — <1ms для Young Gen, <10ms для большинства Major GC. V8 нацелен на 60fps без видимых GC-пауз.

### Что вызывает GC pressure

<b class="text-on-semi-light">Антипаттерны создания мусора</b>

```js
// Плохо: массовое создание короткоживущих объектов
function processItems(items) {
  return items.map(item => ({  // новый объект на каждый элемент
    ...item,                   // spread = ещё аллокации
    processed: true,
    timestamp: new Date(),     // ещё объект
  }));
}

// Лучше: мутация если допустимо, или переиспользование
function processItems(items) {
  for (const item of items) {
    item.processed = true;
  }
  return items;
}
```

<b class="text-on-semi-light">Другие источники утечек и давления на GC</b>

- **Замыкания, захватывающие большие scope** — весь scope chain остаётся в памяти, даже если используется одна переменная
- **Забытые таймеры** — `setInterval` без `clearInterval` удерживает callback и всё, на что он ссылается
- **Event listeners без очистки** — особенно на DOM-элементах, которые удаляются из дерева
- **Растущие кэши без eviction** — `Map`/`Object`, который только растёт; используйте `WeakMap` или LRU

```js
// Утечка: забытый таймер
class Dashboard {
  start() {
    this.intervalId = setInterval(() => {
      this.updateMetrics(); // ссылка на this → Dashboard живёт вечно
    }, 1000);
  }

  // Без этого метода — утечка!
  destroy() {
    clearInterval(this.intervalId);
  }
}

// Утечка: замыкание захватывает больше, чем нужно
function createHandler() {
  const hugeData = loadHugeDataset(); // 100MB
  const smallPart = hugeData.summary;

  return () => {
    // Используем только smallPart, но V8 может удерживать весь scope
    console.log(smallPart);
  };
  // Fix: присвоить hugeData = null перед return
}
```

> **Совет:** используйте Chrome DevTools → Memory tab → Allocation Timeline для обнаружения GC pressure. Heap Snapshot comparison покажет утечки.

## Что спрашивают на интервью

<b class="text-on-semi-light">Типичные вопросы и ожидаемая глубина ответов</b>

| Вопрос | Уровень | Что ожидают услышать |
|--------|---------|---------------------|
| «Как работает браузер?» | Middle+ | Многопроцессная архитектура, rendering pipeline (DOM → CSSOM → Layout → Paint → Composite) |
| «Что такое V8?» | Middle | JS-движок Chrome, tiered compilation (Ignition → TurboFan), GC |
| «Как V8 оптимизирует код?» | Senior | Hidden classes, inline caching, speculative optimization, deopt triggers |
| «Расскажите про GC в JS» | Senior | Generational GC, Scavenger vs Mark-Sweep-Compact, Orinoco, GC pressure anti-patterns |
| «Почему Site Isolation?» | Senior | Spectre/Meltdown, process-per-origin, COOP/COEP |
| «Как избежать memory leaks?» | Middle+ | Closures, timers, event listeners, WeakMap/WeakRef, DevTools profiling |

<b class="text-on-semi-light">Красные флаги в ответах кандидата</b>

<span class="text-gray">Интервьюеры обращают внимание на типичные заблуждения:</span>

- «JS — интерпретируемый язык» — нет, V8 компилирует JS в machine code
- «GC решает все проблемы с памятью» — нет, утечки всё ещё возможны (detached DOM, closures)
- «Все браузеры работают одинаково» — разные движки, разные оптимизации, разное поведение
- «`delete` — это просто удаление свойства» — `delete` переводит объект в dictionary mode, убивая inline caching
- Незнание разницы между renderer process и browser process

> **Совет:** на интервью демонстрируйте не только теоретическое знание, но и практический опыт. Упоминайте конкретные кейсы: «мы профилировали GC pressure через DevTools и обнаружили, что spread-оператор в hot path создавал миллионы временных объектов».
