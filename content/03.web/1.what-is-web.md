# Что такое веб

Веб (World Wide Web) — это распределённая система гипертекстовых документов, работающая поверх интернета. **Интернет** — это физическая сеть (кабели, роутеры, протоколы IP/TCP). **Веб** — это приложение поверх этой сети, построенное на трёх столпах: **URL** (адресация), **HTTP** (передача) и **HTML** (представление).

Для senior-разработчика важно понимать не просто «что такое веб», а как устроена платформа, на которой работает твой код, и какие архитектурные решения (SPA, SSR, SSG) вытекают из её ограничений и возможностей.

## Стандарты и организации

Веб — это не один продукт, а набор открытых стандартов, разрабатываемых несколькими организациями.

| Организация | Что стандартизирует | Ключевые спецификации |
|---|---|---|
| **WHATWG** | HTML, DOM, Fetch, URL, Streams | HTML Living Standard |
| **W3C** | CSS, Accessibility, WebRTC, Web APIs | CSS Modules, WCAG, Web Components |
| **TC39** | ECMAScript (JavaScript) | ES2015–ES2025+ (stage 0–4 process) |
| **IETF** | Сетевые протоколы | HTTP/1.1, HTTP/2, HTTP/3 (QUIC), TLS |

<b class="text-on-semi-light">Как фича попадает в браузер</b>

<span class="text-gray">Proposal (идея) → Spec draft (черновик спецификации) → Browser intent (Intent to Ship/Experiment) → Origin Trial (эксперимент на реальных сайтах) → Shipping (включение по умолчанию). Для JavaScript: TC39 Stage 0 (strawman) → 1 (proposal) → 2 (draft) → 3 (candidate) → 4 (finished, включён в стандарт). Отслеживать статус можно на **Chrome Platform Status**, **Can I Use** и **MDN**.</span>

## Путь от URL до пикселей

Самый классический вопрос на senior-интервью: **«Что происходит, когда ты вводишь URL в адресную строку?»**

```text
1. Пользователь вводит URL
   │
2. DNS-резолвинг: домен → IP-адрес
   │                                    ─┐
3. TCP-соединение (3-way handshake)      │ Подробнее:
   │                                     ├ 04. Network & API
4. TLS handshake (если HTTPS)            │
   │                                     │
5. HTTP-запрос → сервер → HTTP-ответ    ─┘
   │
6. Парсинг HTML → DOM Tree             ─┐
   │                                     │
7. Парсинг CSS → CSSOM Tree             │ Подробнее:
   │                                     ├ 03. Web → Rendering
8. JavaScript: скачивание, парсинг,      │
   │  компиляция, выполнение             │
   │                                     │
9. Render Tree (DOM + CSSOM)             │
   │                                     │
10. Layout → Paint → Composite          ─┘
   │
11. Пиксели на экране
```

> Каждый из этих шагов — отдельная тема для глубокого обсуждения. Сетевую часть (шаги 2–5) разбираем в [Network & API](/content/network-api). Рендеринг (шаги 6–11) — в [Rendering](/content/web/rendering).

## Архитектуры рендеринга

Один из ключевых вопросов для senior-разработчика: **как доставить контент пользователю** и какие tradeoff'ы у каждого подхода.

<b class="text-on-semi-light">MPA (Multi-Page Application)</b>

<span class="text-gray">Классическая модель: каждый переход — полная загрузка страницы с сервера. Сервер отдаёт готовый HTML. Навигация = новый HTTP-запрос. Примеры: традиционные сайты на Rails, Django, PHP.</span>

<b class="text-on-semi-light">SPA (Single-Page Application)</b>

<span class="text-gray">Браузер загружает один HTML-файл и JS-бандл. Все переходы обрабатываются на клиенте через History API. Контент загружается через API (fetch). Примеры: React SPA, Vue SPA.</span>

<b class="text-on-semi-light">SSR (Server-Side Rendering)</b>

<span class="text-gray">Сервер рендерит HTML для каждого запроса (как MPA), но после загрузки страница «гидратируется» и работает как SPA. Лучший TTFB + SEO, но нагрузка на сервер. Примеры: Next.js, Nuxt.</span>

<b class="text-on-semi-light">SSG (Static Site Generation)</b>

<span class="text-gray">HTML генерируется на этапе сборки (build time). Статические файлы раздаются через CDN. Самый быстрый TTFB, но контент фиксирован до следующей сборки. Примеры: Astro, Next.js static export, Nuxt generate.</span>

<b class="text-on-semi-light">ISR (Incremental Static Regeneration)</b>

<span class="text-gray">Гибрид SSG + SSR: страницы генерируются статически, но могут обновляться по запросу с заданным интервалом (revalidate). Примеры: Next.js ISR, Nuxt с routeRules.</span>

<b class="text-on-semi-light">Streaming SSR</b>

<span class="text-gray">Сервер начинает отправлять HTML **до завершения** рендеринга. Браузер получает shell (header, навигацию) мгновенно, а контент «подтягивается» по мере готовности. Уменьшает TTFB и FCP. Примеры: React Server Components + Suspense, Next.js App Router.</span>

| Архитектура | TTFB | FCP | TTI | SEO | Нагрузка на сервер | Сложность |
|---|---|---|---|---|---|---|
| **MPA** | Средний | Быстрый | Быстрый | Отличный | Средняя | Низкая |
| **SPA** | Быстрый | Медленный | Медленный | Плохой* | Нет | Средняя |
| **SSR** | Средний | Быстрый | Средний | Отличный | Высокая | Высокая |
| **SSG** | Мгновенный (CDN) | Мгновенный | Быстрый | Отличный | Нет | Низкая |
| **ISR** | Мгновенный (CDN) | Мгновенный | Быстрый | Отличный | Низкая | Средняя |
| **Streaming SSR** | Быстрый | Быстрый | Средний | Отличный | Высокая | Высокая |

<span class="text-gray">*SPA SEO можно улучшить через pre-rendering или SSR-гидратацию.</span>

> На senior-интервью не достаточно знать, что такое SSR — нужно уметь объяснить, **когда** какая архитектура подходит и **почему**. E-commerce = SSR/SSG (SEO критичен). Дашборд = SPA (SEO не нужен). Блог = SSG + ISR (контент меняется редко).

## Веб как среда выполнения

Браузер — это **sandboxed runtime**: он исполняет произвольный код из интернета, но ограничивает его возможности ради безопасности.

<b class="text-on-semi-light">Ограничения</b>

- **Один main thread** — JavaScript, DOM, layout, paint делят один поток. Долгий JS = зависший UI. Подробнее: [Потоки и параллелизм](/content/computer-science/threads).
- **Same-Origin Policy** — скрипт с одного origin не может читать данные другого origin без явного разрешения (CORS).
- **Нет доступа к FS** — браузерный JS не может читать файлы пользователя напрямую (только через File API с разрешения).
- **Ограниченный доступ к устройствам** — камера, микрофон, геолокация — только с permission prompt.

<b class="text-on-semi-light">Возможности</b>

- **DOM API** — манипуляция документом, события, анимации.
- **Web APIs** — fetch, WebSocket, IntersectionObserver, Web Crypto, Canvas, WebGL/WebGPU.
- **Storage** — cookies, localStorage, sessionStorage, IndexedDB, Cache API.
- **Workers** — Web Workers, Service Workers, Worklets — выход за пределы main thread.

| Характеристика | Веб-платформа | Нативное приложение |
|---|---|---|
| Распространение | URL, мгновенный доступ | App Store, установка |
| Обновления | Автоматические (при загрузке) | Требуют скачивания |
| Доступ к устройству | Ограниченный (с permissions) | Полный |
| Производительность | Ниже (sandbox, JIT) | Выше (AOT, нативный код) |
| Кросс-платформенность | По умолчанию | Требует портирования |
| Безопасность | Sandbox, изоляция origin | Полный доступ к системе |
