# Рендеринг в браузере

Rendering pipeline превращает HTML/CSS/JS в пиксели на экране. Понимание каждого этапа позволяет диагностировать и исправлять проблемы производительности. Это **самая часто спрашиваемая тема** на собеседованиях для senior frontend-разработчиков.

> Подробнее о том, как устроены движки браузеров — [Browser Engines](/content/web/browser-engines).

---

## Critical Rendering Path (CRP)

<b class="text-on-semi-light">Путь от байтов до пикселей</b>

```text
HTML ──> Парсинг ──> DOM Tree
                            ↘
CSS  ──> Парсинг ──> CSSOM   → Render Tree → Layout → Paint → Composite → Экран
                            ↗
JS   ──> V8 ──> DOM/CSSOM modifications
```

<span class="text-gray">CRP — это последовательность шагов, которую браузер выполняет для первого рендеринга страницы. Каждый шаг зависит от предыдущего, поэтому блокировка на любом этапе задерживает отображение.</span>

<b class="text-on-semi-light">Pixel pipeline — жизненный цикл кадра</b>

```text
JavaScript → Style → Layout → Paint → Composite
```

<span class="text-gray">Не каждый кадр проходит все стадии. Composite-only изменения (`transform`, `opacity`) пропускают Layout и Paint — это самый быстрый путь обновления.</span>

> Оптимизация CRP — это главный способ ускорить First Paint и Time to Interactive.

## Парсинг HTML и построение DOM

<b class="text-on-semi-light">Tokenization и Tree Construction</b>

```text
Bytes → Characters → Tokens → Nodes → DOM Tree
```

<span class="text-gray">Парсинг инкрементальный: браузер начинает строить DOM по мере поступления HTML (streaming). Не нужно ждать загрузки всего документа.</span>

<b class="text-on-semi-light">Speculative parsing (preload scanner)</b>

<span class="text-gray">Пока основной парсер заблокирован `<script>`, вторичный парсер (preload scanner) сканирует HTML вперёд и ставит в очередь загрузку ресурсов — изображений, стилей, скриптов. Это значительно сокращает время загрузки.</span>

## Блокирующие ресурсы — стратегии загрузки скриптов

| Атрибут | Загрузка | Выполнение | Блокирует парсинг | Порядок |
|---|---|---|---|---|
| `<script>` (нет) | Синхронно | Сразу | Да | В порядке документа |
| `async` | Параллельно | Как загрузится | Да (на момент выполнения) | Непредсказуемый |
| `defer` | Параллельно | После парсинга, до DOMContentLoaded | Нет | В порядке документа |
| `type="module"` | Параллельно | После парсинга | Нет (defer по умолчанию) | В порядке документа |

<b class="text-on-semi-light">Визуальная диаграмма загрузки</b>

```text
HTML parsing:  ████████████████████████████████████

<script>:      ████──download──██exec██────████████
               blocked                    resumed

async:         ████████████──dl──██exec██──████████
               parsing continues  interrupts briefly

defer:         ████████████████████████████──exec──
               parsing not blocked    after DOMContentLoaded

module:        ████████████████████████████──exec──
               same as defer, + import resolution
```

<span class="text-gray">`defer` — предпочтительный вариант для большинства скриптов. `async` — для независимых скриптов вроде аналитики, где порядок выполнения не важен.</span>

## Resource Hints

| Hint | Что делает | Когда использовать |
|---|---|---|
| `<link rel="preload">` | Загружает ресурс с высоким приоритетом | LCP image, critical font, above-the-fold CSS |
| `<link rel="preconnect">` | DNS + TCP + TLS к домену заранее | API домен, CDN, шрифты |
| `<link rel="prefetch">` | Загружает ресурс для будущей навигации (низкий приоритет) | Следующая страница |
| `<link rel="dns-prefetch">` | Только DNS резолвинг | Домены третьих сторон |

```html
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preload" href="/hero.webp" as="image" fetchpriority="high">
<link rel="prefetch" href="/about.html">
<script defer src="/app.js"></script>
```

<span class="text-gray">`preload` — для ресурсов текущей страницы, которые браузер обнаружил бы слишком поздно. `prefetch` — для ресурсов следующей навигации, загружается в idle time.</span>

## Построение CSSOM

<span class="text-gray">CSS является **render-blocking**: браузер не может отрендерить ничего, пока CSSOM не построен — потому что любое CSS-правило может повлиять на любой элемент (каскадирование). CSS также **script-blocking**: выполнение JS ожидает загрузки предшествующего CSS, потому что JS может запрашивать computed styles.</span>

<b class="text-on-semi-light">Оптимизация загрузки CSS</b>

- Inline critical CSS (above-the-fold) в `<style>` тег
- Загружать non-critical CSS асинхронно
- Использовать `media` атрибут для print/screen-specific стилей

```html
<!-- Critical CSS inline -->
<style>
  body { margin: 0; font-family: system-ui; }
  .hero { height: 100vh; display: flex; align-items: center; }
</style>

<!-- Non-critical CSS: async load -->
<link rel="preload" href="/styles.css" as="style" onload="this.rel='stylesheet'">
<noscript><link rel="stylesheet" href="/styles.css"></noscript>
```

<span class="text-gray">Critical CSS — это минимальный набор стилей для контента «выше сгиба» (above the fold). Всё остальное можно загрузить асинхронно, чтобы не блокировать первый рендер.</span>

## Render Tree

<b class="text-on-semi-light">Объединение DOM + CSSOM</b>

<span class="text-gray">Render Tree содержит только видимые элементы с вычисленными стилями. Элементы, которые не влияют на визуальный вывод, исключаются.</span>

- **Включены**: элементы с computed styles, которые видимы
- **Исключены**: `<head>`, `<meta>`, `<script>`, элементы с `display: none`

<b class="text-on-semi-light">Ключевые отличия свойств скрытия</b>

| Свойство | В Render Tree? | Занимает место? | Видим? | Composited слой? |
|---|---|---|---|---|
| Обычный элемент | Да | Да | Да | Нет |
| `display: none` | **Нет** | Нет | Нет | Нет |
| `visibility: hidden` | Да | **Да** | **Нет** | Нет |
| `opacity: 0` | Да | Да | Нет | **Да** (composited) |
| `content-visibility: auto` (offscreen) | Частично | Да (estimated) | Нет | — |

<span class="text-gray">`display: none` полностью убирает элемент из render tree — он не занимает места и не рендерится. `visibility: hidden` оставляет элемент в дереве — он занимает место, но невидим. `opacity: 0` создаёт composited layer, что может быть полезно для анимаций.</span>

## Layout (Reflow)

<span class="text-gray">Браузер вычисляет точную позицию и размер каждого элемента в render tree. Layout — один из самых дорогих этапов, особенно для глубоких DOM-деревьев.</span>

<b class="text-on-semi-light">Что вызывает Layout</b>

Чтение layout-свойств заставляет браузер синхронно вычислять layout:

- `offsetWidth`, `offsetHeight`, `offsetTop`, `offsetLeft`
- `clientWidth`, `clientHeight`
- `scrollWidth`, `scrollHeight`, `scrollTop`
- `getBoundingClientRect()`
- `getComputedStyle()` (если зависит от layout)
- `window.innerWidth`, `window.innerHeight`

CSS-свойства, вызывающие layout при изменении:

`width`, `height`, `padding`, `margin`, `border`, `top`/`left`/`right`/`bottom`, `font-size`, `line-height`, `display`, `position`, `float`, `overflow`

<b class="text-on-semi-light">Layout Thrashing</b>

<span class="text-gray">Чтение, затем запись DOM в цикле заставляет браузер пересчитывать layout на КАЖДОМ чтении — это называется layout thrashing (принудительная синхронная перекомпоновка).</span>

```js
// ❌ Layout thrashing: N принудительных layout
for (const el of elements) {
  const height = el.offsetHeight;          // READ → forces layout
  el.style.height = height * 2 + 'px';    // WRITE → invalidates layout
}

// ✅ Оптимизированный вариант: batch reads, then batch writes
const heights = Array.from(elements, el => el.offsetHeight); // все READ
elements.forEach((el, i) => {
  el.style.height = heights[i] * 2 + 'px'; // все WRITE
});

// ✅ Ещё лучше: requestAnimationFrame для записей
const heights2 = Array.from(elements, el => el.offsetHeight);
requestAnimationFrame(() => {
  elements.forEach((el, i) => {
    el.style.height = heights2[i] * 2 + 'px';
  });
});
```

<b class="text-on-semi-light">CSS Containment</b>

<span class="text-gray">Свойство `contain` сообщает браузеру, что внутренние изменения элемента не повлияют на внешний layout. Это позволяет браузеру оптимизировать reflow, пересчитывая только изолированное поддерево.</span>

| Значение | Что изолирует |
|---|---|
| `contain: layout` | Layout-изменения внутри не вызывают reflow родителя |
| `contain: paint` | Ничего внутри не рисуется за пределами границ элемента |
| `contain: size` | Размер элемента не зависит от потомков |
| `contain: content` | Сокращение для `layout` + `paint` (без `size`) |
| `contain: strict` | Все вышеперечисленные |

## Paint

<span class="text-gray">Преобразование узлов render tree в реальные пиксели. Чем сложнее визуальные свойства — тем больше работы на этапе paint.</span>

<b class="text-on-semi-light">Paint-expensive свойства</b>

`box-shadow`, `border-radius`, `filter`, `background-gradient`, `clip-path`, `mask`

<b class="text-on-semi-light">Reflow vs Repaint vs Composite-only</b>

| Действие | Layout (Reflow) | Paint (Repaint) | Composite only |
|---|---|---|---|
| Изменение `width`/`height`/`margin`/`padding` | ✅ | ✅ | — |
| Изменение `color`/`background`/`visibility` | — | ✅ | — |
| Изменение `transform` | — | — | ✅ |
| Изменение `opacity` | — | — | ✅ |
| Добавление/удаление DOM-элемента | ✅ | ✅ | — |
| Изменение `font-size`/`line-height` | ✅ | ✅ | — |
| Скролл (если composited) | — | — | ✅ |

> `transform` и `opacity` — единственные свойства, которые гарантированно не вызывают layout и paint. Используй их для анимаций.

## Compositing (Композитинг)

<span class="text-gray">Браузер разбивает страницу на слои (layers), рисует каждый слой независимо, затем компонует их вместе на GPU. Это позволяет обновлять отдельные слои без перерисовки всей страницы.</span>

<b class="text-on-semi-light">Что создаёт новый composited слой</b>

- `will-change: transform` / `will-change: opacity`
- `transform: translateZ(0)` / `translate3d(0,0,0)` (hack, лучше использовать `will-change`)
- `position: fixed` / `position: sticky`
- `<video>`, `<canvas>`, `<iframe>`
- `opacity` < 1 (в некоторых случаях)
- CSS `filter`, `backdrop-filter`
- `overflow: scroll` (для scrollable layers)

<b class="text-on-semi-light">GPU-ускоренные анимации</b>

```css
/* ✅ Composite-only: 60fps guaranteed */
.card {
  will-change: transform;
  transition: transform 0.3s ease, opacity 0.3s ease;
}
.card:hover {
  transform: translateY(-8px) scale(1.02);
  opacity: 0.95;
}

/* ❌ НЕ делай так: triggers layout on every frame */
.card-bad:hover {
  top: -8px;          /* layout trigger */
  width: 102%;        /* layout trigger */
}
```

<b class="text-on-semi-light">Подводные камни will-change</b>

<span class="text-gray">Каждый composited слой потребляет GPU-память. Слишком много слоёв = взрыв потребления памяти. Используй `will-change` точечно, только для элементов, которые действительно будут анимироваться.</span>

```css
/* ❌ Плохо: promote everything */
* { will-change: transform; }

/* ✅ Хорошо: promote only when needed */
.card:hover { will-change: transform; }
/* или через JS перед анимацией, убрать после */
```

```js
// JS подход: добавить will-change перед анимацией
el.addEventListener('mouseenter', () => {
  el.style.willChange = 'transform';
});
el.addEventListener('transitionend', () => {
  el.style.willChange = 'auto';
});
```

## content-visibility: auto

<span class="text-gray">Пропускает рендеринг offscreen-контента полностью. Огромный выигрыш производительности для длинных страниц — браузер пропускает layout, paint и compositing для элементов за пределами viewport.</span>

```css
.section {
  content-visibility: auto;
  contain-intrinsic-size: auto 500px; /* estimated height for scrollbar */
}
```

<span class="text-gray">Когда пользователь прокручивает к элементу — браузер рендерит его on demand. `contain-intrinsic-size` задаёт предполагаемый размер, чтобы скроллбар не прыгал.</span>

## requestAnimationFrame и requestIdleCallback

<b class="text-on-semi-light">requestAnimationFrame (rAF)</b>

<span class="text-gray">Выполняет callback перед следующим repaint. Браузер гарантирует, что callback вызовется в оптимальное время — обычно 60 раз в секунду (или по refresh rate монитора).</span>

```js
// ✅ Плавная анимация через rAF
function animate(timestamp) {
  // Обновить позицию элемента
  element.style.transform = `translateX(${pos}px)`;
  pos += 2;

  if (pos < 500) {
    requestAnimationFrame(animate); // запланировать следующий кадр
  }
}
requestAnimationFrame(animate);

// ❌ НЕ используй setInterval для анимаций
setInterval(() => {
  element.style.transform = `translateX(${pos}px)`;
  pos += 2;
}, 16); // не синхронизирован с рендерингом, может дёргаться
```

<span class="text-gray">`rAF` автоматически паузится, когда вкладка неактивна — экономит CPU и батарею. `setInterval` продолжает работать в фоне.</span>

<b class="text-on-semi-light">requestIdleCallback (rIC)</b>

<span class="text-gray">Выполняет callback, когда у браузера есть свободное время в конце кадра или когда пользователь неактивен. Подходит для некритичных задач.</span>

```js
// Отложить некритичную работу до idle time
requestIdleCallback((deadline) => {
  // deadline.timeRemaining() — сколько мс осталось до конца idle period
  while (deadline.timeRemaining() > 0 && tasks.length > 0) {
    performTask(tasks.shift());
  }

  // Если задачи остались — запланировать на следующий idle
  if (tasks.length > 0) {
    requestIdleCallback(processRemainingTasks);
  }
}, { timeout: 2000 }); // fallback: выполнить не позже чем через 2с
```

| API | Когда вызывается | Для чего использовать |
|---|---|---|
| `requestAnimationFrame` | Перед следующим repaint (~16ms) | Анимации, визуальные обновления |
| `requestIdleCallback` | В свободное время браузера | Аналитика, prefetch, некритичные вычисления |
| `setTimeout(fn, 0)` | После текущего task, из macro queue | Legacy, не оптимально для анимаций |
| `queueMicrotask` | После текущей задачи, до рендеринга | Согласование состояния, promise-like scheduling |

## Виртуальный DOM и реальный рендеринг

<b class="text-on-semi-light">Зачем нужен Virtual DOM</b>

<span class="text-gray">Прямые манипуляции с DOM дорогие — каждое изменение может вызвать reflow/repaint. Virtual DOM — это lightweight in-memory представление реального DOM. Фреймворк сравнивает предыдущее и новое виртуальное дерево (diffing), находит минимальный набор изменений и применяет их к реальному DOM (patching) за один batch.</span>

```text
State change → New VDOM → Diff(old VDOM, new VDOM) → Minimal DOM patches → Browser renders
```

<b class="text-on-semi-light">VDOM vs Direct DOM vs Signals</b>

| Подход | Фреймворки | Как обновляет DOM | Overhead |
|---|---|---|---|
| Virtual DOM + Diffing | React, Vue 2/3 | Batch diff + patch | Средний (diffing cost) |
| Прямые DOM-манипуляции | jQuery, Vanilla JS | Немедленно | Нет overhead, но нет batching |
| Signals / Fine-grained reactivity | Solid, Svelte, Angular Signals | Точечное обновление конкретных DOM-нод | Минимальный |

<span class="text-gray">Virtual DOM — не быстрее ручных DOM-операций. Его ценность в том, что он позволяет писать декларативный код с предсказуемой производительностью. Signals-подход (Solid, Svelte) может быть быстрее, потому что обновляет DOM без diffing.</span>

> Virtual DOM — это не оптимизация производительности, а оптимизация developer experience. Он делает DOM-манипуляции предсказуемыми, а не быстрыми.

## DevTools: диагностика проблем рендеринга

<b class="text-on-semi-light">Performance panel (Chrome DevTools)</b>

<span class="text-gray">Основной инструмент для диагностики rendering performance. Записывает timeline всех этапов pixel pipeline.</span>

```text
Performance tab → Record → Interact with page → Stop

Timeline показывает:
┌─ Frames ──────────────────────────────────┐
│ Зелёные полоски = кадры, красные = jank   │
├─ Main thread ─────────────────────────────┤
│ JS tasks, Style, Layout, Paint            │
├─ Compositor thread ───────────────────────┤
│ Composite, Raster                         │
├─ GPU ─────────────────────────────────────┤
│ GPU-задачи                                │
└───────────────────────────────────────────┘
```

<b class="text-on-semi-light">Ключевые индикаторы проблем</b>

| Что видишь | Проблема | Как исправить |
|---|---|---|
| Длинные жёлтые блоки (JS) | Тяжёлый JS блокирует main thread | Code splitting, Web Worker, debounce |
| Фиолетовые блоки (Layout) | Частые reflow | Batch DOM reads/writes, `contain` |
| Зелёные блоки (Paint) | Перерисовка больших областей | `will-change`, уменьшить paint area |
| Красные треугольники на Frames | Dropped frames (jank) | Анимировать через `transform`/`opacity` |
| "Forced reflow" warning | Layout thrashing | Разделить чтение и запись DOM |

<b class="text-on-semi-light">Rendering tab</b>

<span class="text-gray">Включи Paint flashing (зелёные прямоугольники показывают перерисовываемые области) и Layer borders (оранжевые границы composited layers) для визуальной диагностики.</span>

```text
DevTools → More tools → Rendering:
☑ Paint flashing        — подсвечивает repaint-области
☑ Layout shift regions  — подсвечивает layout shifts (CLS)
☑ Layer borders         — показывает границы composited layers
☑ Frame rendering stats — FPS meter + GPU memory
```

## Часто задаваемые вопросы на собеседовании

<b class="text-on-semi-light">1. Что происходит, когда браузер получает HTML?</b>

<span class="text-gray">Bytes → tokenization → DOM tree. Параллельно CSS → CSSOM. Объединение → Render Tree → Layout (позиции/размеры) → Paint (пиксели) → Composite (слои на GPU) → экран. JS может модифицировать DOM/CSSOM на любом этапе.</span>

<b class="text-on-semi-light">2. Чем отличается reflow от repaint?</b>

<span class="text-gray">**Reflow** (layout) — пересчёт геометрии: позиция, размер. Затрагивает потомков и иногда соседей. **Repaint** — перерисовка пикселей без изменения геометрии (цвет, фон). Reflow всегда вызывает repaint, но не наоборот.</span>

<b class="text-on-semi-light">3. Почему `transform` лучше `top`/`left` для анимаций?</b>

<span class="text-gray">`transform` обрабатывается на compositor thread и не вызывает layout/paint — элемент просто перемещается как текстура на GPU. `top`/`left` меняют layout, вызывая reflow + repaint на каждом кадре, что блокирует main thread.</span>

<b class="text-on-semi-light">4. Как оптимизировать рендеринг длинного списка?</b>

<span class="text-gray">Виртуализация (windowing): рендерить только видимые элементы + небольшой буфер. Библиотеки: `react-window`, `vue-virtual-scroller`, `@tanstack/virtual`. Также: `content-visibility: auto`, `contain: strict`, уменьшить сложность каждого элемента.</span>

<b class="text-on-semi-light">5. Что такое layout thrashing и как его избежать?</b>

<span class="text-gray">Чередование чтения и записи DOM в цикле: каждое чтение (`offsetHeight`) после записи (`style.height = ...`) вызывает принудительный synchronous layout. Решение: сначала все чтения, потом все записи, или использовать `requestAnimationFrame` для записей.</span>

<b class="text-on-semi-light">6. В чём разница между `defer` и `async`?</b>

<span class="text-gray">`defer`: загрузка параллельно парсингу, выполнение после парсинга, порядок сохраняется. `async`: загрузка параллельно, выполнение сразу после загрузки (прерывает парсинг), порядок не гарантирован.</span>

## Оптимизация рендеринга — сводка

<b class="text-on-semi-light">Frame budget при 60fps</b>

```text
16.67ms per frame:
┌─────────┬───────┬────────┬───────┬───────────┬──────┐
│   JS    │ Style │ Layout │ Paint │ Composite │ Idle │
│  ~6ms   │ ~1ms  │ ~2ms   │ ~2ms  │   ~1ms    │ ~4ms │
└─────────┴───────┴────────┴───────┴───────────┴──────┘
```

<b class="text-on-semi-light">Практический чеклист оптимизации</b>

1. Уменьшить parser-blocking ресурсы (`defer`/`async`, `type="module"`).
2. Inline critical CSS, загружать остальное асинхронно.
3. Preload LCP-ресурс (`<link rel="preload">` + `fetchpriority="high"`).
4. Избегать layout thrashing (batch reads/writes, `requestAnimationFrame`).
5. Анимировать только `transform` и `opacity` (composite-only).
6. Использовать `will-change` точечно, не глобально.
7. Применять `content-visibility: auto` для длинных страниц.
8. Использовать `contain` для изоляции layout.
9. Минимизировать глубину DOM (shallow trees = faster layout).
10. Lazy-load изображения ниже fold (`loading="lazy"`), но **не** LCP-изображение.

> 60fps = 16.67ms на кадр. Если JS занимает больше 10ms — анимации будут дёргаться. Оптимизируй или выноси в Web Worker.

<b class="text-on-semi-light">Итоговая таблица: какие изменения что вызывают</b>

| Что меняешь | Layout | Paint | Composite | Стоимость |
|---|---|---|---|---|
| `width`, `height`, `margin`, `padding` | ✅ | ✅ | ✅ | Высокая |
| `color`, `background`, `box-shadow` | — | ✅ | ✅ | Средняя |
| `transform`, `opacity` | — | — | ✅ | Низкая |
