# Будущее веба

Веб-платформа развивается быстрее, чем когда-либо. Новые API, стандарты и возможности расширяют границы того, что может браузер. Для senior-разработчика важно не просто знать текущие инструменты, а понимать направление платформы — это показывает стратегическое мышление на интервью.

> Этот раздел охватывает технологии, которые уже меняют веб или изменят его в ближайшие годы. Каждая секция содержит практические примеры и оценку production-readiness.

## WebAssembly (WASM)

<b class="text-on-semi-light">Бинарный формат инструкций для near-native скорости</b>

<span class="text-gray">WebAssembly — это низкоуровневый бинарный формат, который выполняется в браузере рядом с JavaScript. Код на C, C++, Rust, Go компилируется в `.wasm` и работает со скоростью, близкой к нативной. WASM не заменяет JS — он дополняет его для CPU-intensive задач.</span>

### Реальные примеры использования

- **Figma** — весь rendering engine на C++, компилируется в WASM
- **Adobe Photoshop** (web) — обработка изображений на WASM
- **Google Earth** — 3D-рендеринг в браузере
- **Squoosh** — сжатие изображений (codecs на WASM)
- **AutoCAD Web** — CAD-движок на WASM
- **FFmpeg.wasm** — обработка видео прямо в браузере

### Загрузка и использование WASM из JS

```js
// Загрузка и инициализация WASM-модуля
const response = await fetch('/math.wasm');
const buffer = await response.arrayBuffer();
const { instance } = await WebAssembly.instantiate(buffer, {
  env: {
    // Импорт функций из JS в WASM
    log: (value) => console.log('WASM says:', value),
  },
});

// Вызов экспортированных WASM-функций
const result = instance.exports.fibonacci(42);
console.log(result); // вычислено на near-native скорости

// Оптимизированный подход: streaming compilation
const streamedModule = await WebAssembly.instantiateStreaming(
  fetch('/math.wasm'),
  { env: { log: console.log } }
);
```

<span class="text-gray">`instantiateStreaming` — предпочтительный метод: начинает компиляцию во время загрузки (streaming), что значительно быстрее, чем ожидание полной загрузки + `instantiate`.</span>

### WASI — WebAssembly System Interface

<span class="text-gray">WASI — стандартизированный интерфейс для запуска WASM вне браузера. Он предоставляет доступ к файловой системе, сети, часам — но в sandboxed-виде. Это позволяет использовать WASM как универсальный runtime (альтернатива Docker для edge-функций).</span>

### Component Model

<span class="text-gray">Component Model — следующая эволюция WASM: позволяет модулям, написанным на разных языках, взаимодействовать через стандартизированные интерфейсы (WIT — WASM Interface Types). Rust-модуль может вызывать Go-модуль без JS-прослойки.</span>

> WASM не заменяет JavaScript — он дополняет его. JS остаётся для UI, DOM, orchestration. WASM — для тяжёлых вычислений: кодеки, криптография, физика, ML-инференс.

## Web Components

<b class="text-on-semi-light">Нативные компоненты без фреймворка</b>

<span class="text-gray">Web Components — набор браузерных стандартов для создания переиспользуемых UI-компонентов с инкапсуляцией. Три основных технологии: Custom Elements, Shadow DOM, HTML Templates.</span>

### Custom Elements

```js
class UserCard extends HTMLElement {
  // Наблюдаемые атрибуты — при их изменении вызывается attributeChangedCallback
  static observedAttributes = ['name', 'avatar'];

  constructor() {
    super();
    this.attachShadow({ mode: 'open' });
  }

  // Lifecycle callbacks
  connectedCallback() {
    // Элемент добавлен в DOM
    this.render();
  }

  disconnectedCallback() {
    // Элемент удалён из DOM — cleanup
  }

  attributeChangedCallback(name, oldValue, newValue) {
    // Атрибут изменился
    if (oldValue !== newValue) this.render();
  }

  render() {
    this.shadowRoot.innerHTML = `
      <style>
        :host {
          display: flex;
          align-items: center;
          gap: 12px;
          padding: 16px;
          border: 1px solid #e0e0e0;
          border-radius: 8px;
        }
        img { width: 48px; height: 48px; border-radius: 50%; }
        ::slotted(p) { color: #666; margin: 0; }
      </style>
      <img src="${this.getAttribute('avatar') || '/default.png'}" alt="">
      <div>
        <strong>${this.getAttribute('name') || 'Unknown'}</strong>
        <slot></slot>
      </div>
    `;
  }
}

customElements.define('user-card', UserCard);
```

```html
<!-- Использование -->
<user-card name="Alice" avatar="/alice.jpg">
  <p>Frontend Developer</p>
</user-card>
```

### Shadow DOM — инкапсуляция

<span class="text-gray">Shadow DOM создаёт изолированное DOM-дерево: стили не утекают наружу и не проникают внутрь. Это решает главную проблему CSS — глобальную область видимости. `:host` стилизует сам custom element, `::slotted()` — проецируемый контент.</span>

### Declarative Shadow DOM — совместимость с SSR

```html
<!-- Работает без JS — браузер создаёт Shadow DOM при парсинге HTML -->
<user-card>
  <template shadowrootmode="open">
    <style>
      :host { display: block; padding: 16px; }
    </style>
    <slot></slot>
  </template>
  <p>SSR-compatible content</p>
</user-card>
```

<span class="text-gray">Declarative Shadow DOM решает главную проблему Web Components — необходимость JS для создания Shadow DOM. Теперь SSR-фреймворки могут отдавать Web Components с сервера.</span>

### Когда Web Components vs фреймворк-компоненты

| Сценарий | Web Components | React/Vue/Svelte |
|---|---|---|
| Design system (shared across teams) | Отличный выбор | Привязка к фреймворку |
| Cross-framework компоненты | Идеально | Не подходит |
| Micro-frontends | Хорошая изоляция | Возможно, но сложнее |
| Сложный SPA | Слишком низкоуровнево | Идеально |
| Виджеты для встраивания (embed) | Лучший выбор | Тянет runtime |

## Новые Web APIs

### View Transitions API

<b class="text-on-semi-light">Плавные переходы между страницами и состояниями</b>

<span class="text-gray">View Transitions API позволяет создавать анимированные переходы между DOM-состояниями — как в нативных приложениях. Работает и для SPA (same-document), и для MPA (cross-document).</span>

```js
// SPA: плавный переход при обновлении контента
async function navigateTo(url) {
  const data = await fetch(url).then((r) => r.json());

  // Браузер делает скриншот текущего состояния,
  // затем анимирует переход к новому
  document.startViewTransition(() => {
    updateDOM(data);
  });
}
```

```css
/* Кастомизация анимации перехода */
::view-transition-old(root) {
  animation: fade-out 0.3s ease;
}
::view-transition-new(root) {
  animation: fade-in 0.3s ease;
}

/* Разные анимации для разных элементов */
.hero-image {
  view-transition-name: hero;
}
::view-transition-group(hero) {
  animation-duration: 0.5s;
}
```

<span class="text-gray">Для MPA (cross-document) используется `@view-transition` at-rule в CSS — переходы между отдельными HTML-страницами без JS. Работает в Chrome с `navigation` типом `same-origin`.</span>

### Popover API

<b class="text-on-semi-light">Нативные поповеры без JavaScript</b>

```html
<!-- Нативный popover — никакого JS не нужно -->
<button popovertarget="my-popup">Открыть</button>

<div id="my-popup" popover>
  <p>Это нативный popover!</p>
  <p>Закрывается при клике вне, Escape, или программно.</p>
</div>
```

<span class="text-gray">Popover API предоставляет: top-layer rendering (поверх всего, как `<dialog>`), light dismiss (клик вне закрывает), управление фокусом, встроенная доступность (accessibility). Два режима: `popover="auto"` (закрывается при клике вне) и `popover="manual"` (только программное закрытие).</span>

```js
// Программное управление
const popup = document.querySelector('#my-popup');
popup.showPopover();
popup.hidePopover();
popup.togglePopover();

// События
popup.addEventListener('toggle', (e) => {
  console.log(e.newState); // 'open' или 'closed'
});
```

### Navigation API

<b class="text-on-semi-light">Современная замена History API</b>

<span class="text-gray">Navigation API упрощает SPA-навигацию: единый `navigate` event вместо `popstate` + `pushState` + `hashchange`. Поддерживает перехват навигации, отмену, скролл-менеджмент.</span>

```js
navigation.addEventListener('navigate', (event) => {
  const url = new URL(event.destination.url);

  if (url.pathname.startsWith('/app/')) {
    event.intercept({
      async handler() {
        const content = await fetchPage(url.pathname);
        document.querySelector('#content').innerHTML = content;
      },
    });
  }
});
```

### Speculation Rules API

<b class="text-on-semi-light">Prerender и prefetch следующих навигаций</b>

<span class="text-gray">Speculation Rules позволяют браузеру заранее загрузить или полностью отрендерить следующую страницу в фоне. Переход происходит мгновенно.</span>

```html
<script type="speculationrules">
{
  "prerender": [
    {
      "where": { "href_matches": "/products/*" },
      "eagerness": "moderate"
    }
  ],
  "prefetch": [
    {
      "urls": ["/about", "/contact"],
      "eagerness": "eager"
    }
  ]
}
</script>
```

<span class="text-gray">`eagerness` контролирует агрессивность: `eager` (сразу), `moderate` (при hover), `conservative` (при клике). `prerender` рендерит страницу полностью в скрытом tab — переход будет мгновенным. `prefetch` только загружает ресурсы.</span>

### Scheduler API

<b class="text-on-semi-light">Приоритетное планирование задач</b>

```js
// User-blocking: реакция на ввод пользователя (высший приоритет)
await scheduler.postTask(() => {
  updateSearchResults(query);
}, { priority: 'user-blocking' });

// User-visible: видимые обновления (средний приоритет)
scheduler.postTask(() => {
  updateDashboardWidgets();
}, { priority: 'user-visible' });

// Background: фоновые задачи (низший приоритет)
scheduler.postTask(() => {
  sendAnalytics();
  prefetchNextPage();
}, { priority: 'background' });

// scheduler.yield() — отдать контроль браузеру, сохранив приоритет
async function processLargeList(items) {
  for (const item of items) {
    processItem(item);
    await scheduler.yield(); // позволяет браузеру обработать input/рендер
  }
}
```

<span class="text-gray">`scheduler.yield()` — замена паттерна `setTimeout(0)` для разбивки длинных задач. В отличие от `setTimeout`, `scheduler.yield()` сохраняет приоритет задачи и не отправляет её в конец очереди.</span>

## WebGPU

<b class="text-on-semi-light">Современный GPU API — замена WebGL</b>

<span class="text-gray">WebGPU предоставляет низкоуровневый доступ к GPU, основанный на архитектуре Vulkan/Metal/Direct3D 12. Помимо графики, поддерживает compute shaders — общие вычисления на GPU (ML-инференс, физика, обработка данных).</span>

| Характеристика | WebGL / WebGL2 | WebGPU |
|---|---|---|
| API-стиль | OpenGL ES (stateful) | Vulkan-like (stateless, explicit) |
| Compute shaders | Нет | Да |
| Многопоточность | Нет | Command buffers из Workers |
| Язык шейдеров | GLSL | WGSL (WebGPU Shading Language) |
| Error handling | Implicit (silent errors) | Explicit (validation) |
| Производительность | Ограничена driver overhead | Меньше overhead, ближе к металлу |
| ML inference | Невозможно | Да (через compute) |

<span class="text-gray">WebGPU уже доступен в Chrome, Edge, Firefox (nightly). Основные библиотеки: Three.js (WebGPU renderer), Babylon.js (WebGPU support), ONNX Runtime Web (ML на WebGPU).</span>

## AI в браузере

<b class="text-on-semi-light">Нейросети прямо на устройстве пользователя</b>

<span class="text-gray">Тренд на on-device AI: модели запускаются локально, данные не покидают устройство. Это даёт приватность, работу offline и отсутствие серверных затрат.</span>

### Текущие API и подходы

- **WebNN** (Web Neural Network API) — стандартизированный API для аппаратного ускорения inference (GPU, NPU, CPU). Пока в ранней стадии
- **WebGPU for ML** — compute shaders для ML-операций. Используется TensorFlow.js, ONNX Runtime Web, Transformers.js
- **Chrome Built-in AI** (`window.ai`) — эксперимент Chrome с встроенными LLM (Gemini Nano). API: `ai.languageModel.create()`, `ai.summarizer`, `ai.translator`

### Практические сценарии

- **Автокомплит / автозамена** — работает offline
- **Классификация изображений** — на камере устройства, без отправки на сервер
- **Суммаризация текста** — локально через встроенную модель
- **Перевод** — offline-перевод в браузере
- **Smart compose** — подсказки при наборе текста

<span class="text-gray">On-device AI — одно из самых перспективных направлений. NPU (Neural Processing Unit) появляется в новых процессорах (Apple M-series, Snapdragon, Intel Meteor Lake), и WebNN даст веб-приложениям доступ к этому железу.</span>

## Baseline & Interop

<b class="text-on-semi-light">Как понять, что API можно использовать в production</b>

<span class="text-gray">**Baseline** — инициатива Google, Mozilla, Apple, Microsoft для прозрачности browser support. Два уровня:</span>

| Уровень | Что значит | Пример (2024) |
|---|---|---|
| **Newly available** | Поддерживается последними версиями всех основных браузеров | View Transitions (SPA), Popover API |
| **Widely available** | Поддерживается 30 месяцев+ (безопасно для production) | Container Queries, `:has()`, CSS Nesting |

<b class="text-on-semi-light">Interop Project</b>

<span class="text-gray">Interop — ежегодный проект, где Apple, Google, Mozilla, Microsoft совместно выбирают приоритетные области для улучшения совместимости. Благодаря Interop 2023-2024 были значительно улучшены Container Queries, `:has()`, Subgrid, Scroll-driven animations, Popover.</span>

> Проверяйте Baseline статус на [web.dev/baseline](https://web.dev/baseline) и [caniuse.com](https://caniuse.com). Если API имеет статус «Widely available» — можно использовать без полифиллов.

## Что учить сейчас, а что наблюдать

<b class="text-on-semi-light">Практическая таблица для планирования обучения</b>

| Категория | Технологии | Рекомендация |
|---|---|---|
| **Production-ready** | View Transitions (SPA), Container Queries, `:has()`, CSS Nesting, Popover API, `dialog`, Declarative Shadow DOM | Учить и использовать сейчас |
| **Experimental** | Speculation Rules, Scheduler API, Navigation API, View Transitions (MPA), Scroll-driven Animations | Изучать, пробовать в non-critical фичах |
| **Emerging** | WebGPU, WebNN, WASM Component Model, Chrome Built-in AI, CSS Anchor Positioning | Следить за развитием, экспериментировать |

<b class="text-on-semi-light">Стратегия для senior-разработчика</b>

- **Production-ready** — эти технологии уже в Baseline Widely Available. Используйте их в production. На собеседованиях от вас ожидают их знание
- **Experimental** — понимание этих API показывает, что вы следите за платформой. Упоминайте их на собеседованиях как «next steps» и «architectural considerations»
- **Emerging** — знание на концептуальном уровне. Показывает стратегическое мышление и видение будущего фронтенда

<span class="text-gray">Веб-платформа конвергирует с нативными возможностями. View Transitions заменяют custom-анимации переходов, Popover убирает необходимость в tooltip-библиотеках, Speculation Rules делает навигацию мгновенной. Тренд: меньше JS-библиотек, больше нативных API.</span>

> Лучшая стратегия — не гнаться за каждым новым API, а понимать тренды: нативные возможности браузера заменяют JS-библиотеки, on-device AI расширяет offline-возможности, WASM открывает веб для новых категорий приложений.
