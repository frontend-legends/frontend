# Browser APIs

Современные браузеры предоставляют десятки API помимо DOM. На собеседованиях уровня senior от вас ожидают знание IntersectionObserver, Performance API, requestAnimationFrame/requestIdleCallback, паттернов AbortController и Streams API. Именно эти API отделяют код senior-разработчика от junior-кода — они показывают, что вы понимаете, **как браузер работает под капотом**, и умеете писать производительные, отзывчивые приложения.

> В этом разделе рассматриваются API, которые чаще всего встречаются на техническом интервью и критически важны для production-приложений. Каждый раздел содержит примеры кода, готовые к использованию.

## IntersectionObserver

<b class="text-on-semi-light">Асинхронное наблюдение за видимостью элементов</b>

<span class="text-gray">IntersectionObserver позволяет асинхронно отслеживать пересечение элемента с viewport или родительским контейнером. В отличие от слушателя `scroll` + `getBoundingClientRect()`, он не блокирует main thread и работает через внутренний механизм браузера.</span>

Типичные сценарии использования:
- **Lazy loading** изображений и компонентов
- **Infinite scroll** (подгрузка данных при скролле)
- **Impression tracking** (отслеживание просмотров рекламы/контента)
- **Animate on scroll** (запуск анимаций при появлении элемента)

### Основные параметры

| Параметр | Тип | Описание |
|---|---|---|
| `threshold` | `number \| number[]` | Массив порогов видимости (0 — появился, 1 — полностью виден) |
| `rootMargin` | `string` | Расширение/сужение области наблюдения (`'200px 0px'`) |
| `root` | `Element \| null` | Элемент-контейнер (по умолчанию — viewport) |

<b class="text-on-semi-light">Пример: lazy loading изображений с cleanup</b>

```js
const observer = new IntersectionObserver((entries) => {
  for (const entry of entries) {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src;
      observer.unobserve(img); // прекращаем наблюдение после загрузки
    }
  }
}, { rootMargin: '200px' }); // начинаем загрузку за 200px до viewport

document.querySelectorAll('img[data-src]').forEach(img => observer.observe(img));

// Cleanup (например, при unmount компонента)
// observer.disconnect();
```

<span class="text-gray">Ключевой момент: вызов `unobserve()` после загрузки изображения предотвращает утечки памяти. В React/Vue обязательно вызывайте `disconnect()` в cleanup-функции (`useEffect` return / `onUnmounted`).</span>

## ResizeObserver

<b class="text-on-semi-light">Наблюдение за изменением размеров элемента</b>

<span class="text-gray">ResizeObserver отслеживает изменение размеров конкретных элементов — не только окна (`window.resize`), а любого DOM-элемента. Это критично для компонентов, которые должны адаптироваться к своему контейнеру.</span>

Типичные сценарии:
- Responsive-компоненты без media queries
- Пересчёт размеров графиков (charts)
- Container-aware layouts

### Доступные размеры

| Свойство | Описание |
|---|---|
| `contentRect` | Размер содержимого (без padding/border) |
| `contentBoxSize` | Размер content box (массив `ResizeObserverSize`) |
| `borderBoxSize` | Размер border box |
| `devicePixelContentBoxSize` | Размер в физических пикселях |

```js
const observer = new ResizeObserver((entries) => {
  for (const entry of entries) {
    const { width } = entry.contentRect;
    entry.target.classList.toggle('compact', width < 400);
  }
});
observer.observe(element);

// Cleanup
// observer.disconnect();
```

<span class="text-gray">Замечание: CSS Container Queries (`@container`) часто являются лучшим решением для стилизации в зависимости от контейнера. Однако ResizeObserver по-прежнему необходим для JS-логики — пересчёт canvas, обновление данных графика и т.д.</span>

## Performance API

<b class="text-on-semi-light">Измерение производительности — критически важно для senior-уровня</b>

<span class="text-gray">Performance API позволяет точно измерять время выполнения операций, отслеживать загрузку ресурсов и выявлять узкие места. На собеседованиях ожидают, что вы умеете профилировать приложение не только через DevTools, но и программно.</span>

### User Timing

<b class="text-on-semi-light">Свои метки и замеры</b>

```js
performance.mark('fetch-start');
const data = await fetch('/api/data').then(r => r.json());
performance.mark('fetch-end');

performance.measure('api-fetch', 'fetch-start', 'fetch-end');
const measure = performance.getEntriesByName('api-fetch')[0];
console.log(`API took ${measure.duration}ms`);
```

### PerformanceObserver

<b class="text-on-semi-light">Реактивное наблюдение за метриками</b>

```js
// Наблюдение за Long Tasks (>50ms)
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.log(`Long task: ${entry.duration}ms`, entry);
    // Отправляем в аналитику
  }
});
observer.observe({ type: 'longtask', buffered: true });
```

<span class="text-gray">`buffered: true` — получить также записи, созданные до подписки. Это важно, если вы подключаете PerformanceObserver не сразу при загрузке страницы.</span>

```js
// Наблюдение за Resource Timing
const resObserver = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.duration > 1000) {
      console.warn(`Slow resource: ${entry.name} took ${entry.duration}ms`);
    }
  }
});
resObserver.observe({ type: 'resource', buffered: true });
```

### Navigation Timing

<b class="text-on-semi-light">Метрики навигации и загрузки страницы</b>

```js
// Время от начала навигации до DOM interactive
const nav = performance.getEntriesByType('navigation')[0];
console.log('DOM Interactive:', nav.domInteractive - nav.startTime);
console.log('DOM Complete:', nav.domComplete - nav.startTime);
console.log('TTFB:', nav.responseStart - nav.startTime);
```

| Метрика | Что измеряет |
|---|---|
| `TTFB` (Time to First Byte) | Время до первого байта ответа |
| `domInteractive` | DOM готов к взаимодействию |
| `domComplete` | Все ресурсы загружены |
| `loadEventEnd` | Событие `load` завершилось |

## requestAnimationFrame

<b class="text-on-semi-light">Синхронизация с частотой обновления дисплея</b>

<span class="text-gray">`requestAnimationFrame` (rAF) вызывает callback перед следующей отрисовкой кадра. Это единственный правильный способ обновлять визуальное состояние из JavaScript — он синхронизирован с refresh rate дисплея.</span>

- На 60Hz мониторе: ~60 вызовов/сек, бюджет кадра ~16.67ms
- На 120Hz мониторе: ~120 вызовов/сек, бюджет кадра ~8.33ms
- Всегда используйте для визуальных обновлений/анимаций
- Отменяйте через `cancelAnimationFrame(id)`

```js
let animationId;
function animate(timestamp) {
  // Обновляем позиции, трансформы и т.д.
  element.style.transform = `translateX(${position}px)`;
  animationId = requestAnimationFrame(animate);
}
animationId = requestAnimationFrame(animate);

// Cleanup
cancelAnimationFrame(animationId);
```

<b class="text-on-semi-light">Когда использовать rAF, а когда CSS-анимации?</b>

| Критерий | requestAnimationFrame | CSS Transitions/Animations |
|---|---|---|
| Контроль | Полный (JS логика) | Декларативный |
| Производительность | Зависит от кода | Оптимизирован браузером |
| Сложные анимации | Да (physics, game loop) | Ограничен keyframes |
| Простые переходы | Overkill | Идеально |
| Compositor thread | Нет (main thread) | Да (`transform`, `opacity`) |

<span class="text-gray">Правило: если анимацию можно описать через `transform` или `opacity` — используйте CSS. Если нужна физика, интерактивность или зависимость от данных — используйте rAF.</span>

## requestIdleCallback

<b class="text-on-semi-light">Выполнение работы в свободное время браузера</b>

<span class="text-gray">`requestIdleCallback` вызывает callback, когда браузер простаивает — нет pending-задач, rAF-вызовов или рендеринга. Идеально для некритичной работы: аналитика, предзагрузка, логирование.</span>

- `deadline.timeRemaining()` — сколько миллисекунд осталось в idle-периоде
- **НЕ подходит** для time-critical задач (может не вызваться, если страница постоянно занята)
- Параметр `timeout` гарантирует вызов через указанное время

```js
function processAnalytics(deadline) {
  while (analyticsQueue.length > 0 && deadline.timeRemaining() > 5) {
    const event = analyticsQueue.shift();
    sendToAnalytics(event);
  }

  if (analyticsQueue.length > 0) {
    requestIdleCallback(processAnalytics);
  }
}

requestIdleCallback(processAnalytics, { timeout: 2000 }); // максимальное ожидание 2с
```

<b class="text-on-semi-light">Современная альтернатива: scheduler.postTask()</b>

<span class="text-gray">`scheduler.postTask()` — новый API с уровнями приоритета, который предлагает более предсказуемое поведение, чем `requestIdleCallback`.</span>

```js
// Приоритет ниже, чем user-blocking, но гарантированный запуск
await scheduler.postTask(() => {
  processAnalytics();
}, { priority: 'background' });
```

| Приоритет | Описание | Аналог |
|---|---|---|
| `'user-blocking'` | Реакция на действия пользователя | Высокий |
| `'user-visible'` | Видимые обновления (по умолчанию) | Средний |
| `'background'` | Фоновые задачи | `requestIdleCallback` |

## AbortController

<b class="text-on-semi-light">Управление отменой асинхронных операций</b>

<span class="text-gray">AbortController — универсальный механизм отмены операций. Работает с `fetch`, event listeners, и любым кодом, который принимает `AbortSignal`. Это один из самых частых вопросов на собеседованиях по cleanup и memory leaks.</span>

```js
const controller = new AbortController();

// Отменяемый fetch
fetch('/api/data', { signal: controller.signal })
  .then(r => r.json())
  .then(data => render(data))
  .catch(err => {
    if (err.name === 'AbortError') {
      console.log('Запрос отменён');
    }
  });

// Отмена по таймауту
setTimeout(() => controller.abort(), 5000);
```

<b class="text-on-semi-light">AbortSignal.timeout() — встроенный таймаут</b>

```js
// Автоматическая отмена через 5 секунд (без ручного AbortController)
const response = await fetch('/api/data', {
  signal: AbortSignal.timeout(5000),
});
```

<b class="text-on-semi-light">Отмена event listener</b>

```js
const controller = new AbortController();

element.addEventListener('click', handleClick, {
  signal: controller.signal,
});
element.addEventListener('mousemove', handleMove, {
  signal: controller.signal,
});

// Один вызов убирает ВСЕ слушатели, привязанные к этому signal
controller.abort();
```

<span class="text-gray">Паттерн с AbortController особенно полезен в React `useEffect`, где нужно отменять запросы при размонтировании компонента, и в Vue `onUnmounted`.</span>

## Page Visibility API

<b class="text-on-semi-light">Определение состояния вкладки</b>

<span class="text-gray">Page Visibility API позволяет узнать, видит ли пользователь вашу вкладку. Это важно для экономии ресурсов — зачем гонять анимацию или polling, если вкладка в фоне?</span>

```js
document.addEventListener('visibilitychange', () => {
  if (document.hidden) {
    pauseVideo();
    clearInterval(pollingInterval);
  } else {
    resumeVideo();
    pollingInterval = setInterval(poll, 5000);
  }
});
```

<span class="text-gray">`document.visibilityState` возвращает `'visible'` или `'hidden'`. Свойство `document.hidden` — boolean shortcut. Событие `visibilitychange` срабатывает при переключении вкладок, сворачивании окна и блокировке экрана.</span>

## Broadcast Channel

<b class="text-on-semi-light">Простая коммуникация между вкладками (same origin)</b>

<span class="text-gray">BroadcastChannel позволяет вкладкам одного origin обмениваться сообщениями без сервера. Альтернативы: `localStorage` event, `SharedWorker`, но BroadcastChannel — самый простой вариант.</span>

```js
const channel = new BroadcastChannel('auth');

// Вкладка 1: пользователь разлогинился
channel.postMessage({ type: 'logout' });

// Вкладка 2: получаем сообщение
channel.onmessage = (e) => {
  if (e.data.type === 'logout') {
    window.location.href = '/login';
  }
};

channel.close(); // cleanup
```

Типичные сценарии:
- Синхронизация авторизации между вкладками
- Уведомления о новых данных
- Синхронизация темы оформления (dark/light mode)
- Координация WebSocket-подключений (одно на все вкладки)

## Streams API

<b class="text-on-semi-light">Потоковая обработка данных</b>

<span class="text-gray">Streams API позволяет обрабатывать данные **инкрементально**, по мере поступления, вместо ожидания полной загрузки. Три основных примитива: `ReadableStream`, `WritableStream`, `TransformStream`.</span>

<b class="text-on-semi-light">Streaming fetch — построчная обработка большого ответа</b>

```js
const response = await fetch('/api/large-data');
const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  const text = decoder.decode(value, { stream: true });
  processChunk(text);
}
```

<b class="text-on-semi-light">TransformStream — конвейерная обработка</b>

```js
const transform = new TransformStream({
  transform(chunk, controller) {
    const upper = chunk.toUpperCase();
    controller.enqueue(upper);
  }
});

const readable = new ReadableStream({ /* ... */ });
const writable = new WritableStream({ /* ... */ });

readable.pipeThrough(transform).pipeTo(writable);
```

<span class="text-gray">Streams особенно полезны для SSR-стриминга (React 18 `renderToPipeableStream`), обработки больших файлов, и реализации streaming-ответов от LLM (ChatGPT-like интерфейсы).</span>

## Типичные вопросы на собеседовании

> **В чём разница между IntersectionObserver и scroll event + getBoundingClientRect()?**
> IntersectionObserver асинхронный и не блокирует main thread. Scroll listener с `getBoundingClientRect()` вызывает forced reflow при каждом событии scroll, что критически влияет на производительность.

> **Как отменить fetch-запрос?**
> Через `AbortController`. Создаёте контроллер, передаёте `signal` в `fetch`, вызываете `controller.abort()`. Обрабатывайте `AbortError` в catch.

> **Зачем нужен requestAnimationFrame, если есть setInterval?**
> `setInterval` не синхронизирован с refresh rate, может вызывать jank (пропуск кадров) и продолжает работать в фоновых вкладках. rAF автоматически паузится в скрытых вкладках и гарантирует один вызов на кадр.

> **Как измерить Long Tasks на production?**
> `PerformanceObserver` с `type: 'longtask'` и `buffered: true`. Собирайте данные и отправляйте в аналитику (RUM — Real User Monitoring).
