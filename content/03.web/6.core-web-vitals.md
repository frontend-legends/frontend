# Core Web Vitals и метрики производительности

<span class="text-gray">Core Web Vitals (CWV) — это набор метрик от Google, измеряющих реальный пользовательский опыт: скорость загрузки, отзывчивость и визуальную стабильность. CWV влияют на SEO-ранжирование с 2021 года. На senior-интервью вас спросят не только «что такое LCP», но и «как вы диагностировали и исправили плохой LCP на проекте».</span>

---

## Зачем нужны метрики производительности

<b class="text-on-semi-light">Объективные данные vs «кажется быстро»</b>

<span class="text-gray">Субъективная оценка скорости ненадёжна. Разработчик на MacBook Pro с гигабитным интернетом не заметит проблем, которые критичны для пользователя на бюджетном Android с 3G. Метрики дают объективные числа.</span>

- **SEO ranking signal** — Google использует CWV как фактор ранжирования с 2021 года
- **RUM vs Lab** — field data (реальные пользователи) vs synthetic (контролируемая среда)
- **Бизнес-метрики** — каждые 100ms задержки загрузки снижают конверсию на 1-2%

> На собеседовании: «Как вы узнаёте, что ваш сайт быстрый?» — Ответ через конкретные метрики и инструменты, а не «я открыл и всё быстро загрузилось».

## LCP (Largest Contentful Paint) — глубокое погружение

<b class="text-on-semi-light">Что измеряет</b>

<span class="text-gray">Время до рендеринга самого большого видимого элемента контента во viewport. Это показатель воспринимаемой скорости загрузки — момент, когда пользователь видит основное содержимое страницы.</span>

<b class="text-on-semi-light">LCP-кандидаты</b>

- `<img>` — изображения
- `<video>` — poster-кадр видео
- Элемент с `background-image` (загруженным через CSS)
- Блочные текстовые элементы (`<h1>`, `<p>`, `<div>` с текстом)

| Оценка | Значение |
|---|---|
| Хорошо (Good) | ≤ 2.5s |
| Нужно улучшить | ≤ 4.0s |
| Плохо (Poor) | > 4.0s |

---

<b class="text-on-semi-light">Декомпозиция LCP на под-части</b>

```text
LCP = TTFB + Resource Load Delay + Resource Load Time + Element Render Delay
       │            │                     │                    │
       │            │                     │                    └─ JS/CSS блокирует рендер
       │            │                     └─ Скачивание ресурса (изображение, шрифт)
       │            └─ Задержка до начала загрузки (discoverable? приоритет?)
       └─ Ответ сервера
```

<span class="text-gray">Каждая под-часть — отдельная точка оптимизации. Сначала определите bottleneck, потом оптимизируйте.</span>

---

<b class="text-on-semi-light">Что ухудшает LCP и как исправить</b>

| Проблема | Решение |
|---|---|
| Медленный TTFB | CDN, edge caching, оптимизация серверной логики |
| Render-blocking CSS/JS | `defer`, `async`, critical CSS inline |
| Медленная загрузка ресурса | `preload`, сжатие, WebP/AVIF, `fetchpriority="high"` |
| CSR-задержка (hydration) | SSR/SSG, streaming HTML, partial hydration |
| Lazy-loading LCP-изображения | **Никогда** не ставьте `loading="lazy"` на LCP-элемент! |

```html
<!-- ✅ Preload LCP-изображения с высоким приоритетом -->
<link rel="preload" as="image" href="/hero.webp" fetchpriority="high">
<link rel="preconnect" href="https://cdn.example.com">

<!-- ✅ LCP image: eager loading, высокий приоритет -->
<img src="/hero.webp" alt="Hero" fetchpriority="high"
     width="1200" height="600" decoding="async">

<!-- ⚠️ Частая ошибка: lazy-loading для LCP-элемента -->
<img src="/hero.webp" loading="lazy"> <!-- НЕПРАВИЛЬНО для LCP-элемента! -->
```

```html
<!-- ✅ Подсказки браузеру для ускорения TTFB -->
<link rel="dns-prefetch" href="https://api.example.com">
<link rel="preconnect" href="https://cdn.example.com" crossorigin>
```

> На собеседовании: «Вы видите LCP 4.2s. Какие шаги предпримете?» — Определите LCP-элемент через DevTools/web-vitals attribution, разбейте на под-части и найдите bottleneck.

## INP (Interaction to Next Paint) — глубокое погружение

<b class="text-on-semi-light">Что измеряет</b>

<span class="text-gray">INP заменил FID в марте 2024 года. Измеряет отзывчивость на ВСЕ взаимодействия (клик, тап, нажатие клавиши), а не только первое. Берётся p98 — практически наихудшее взаимодействие за время жизни страницы.</span>

| Оценка | Значение |
|---|---|
| Хорошо (Good) | ≤ 200ms |
| Нужно улучшить | ≤ 500ms |
| Плохо (Poor) | > 500ms |

---

<b class="text-on-semi-light">Три фазы INP</b>

```text
User Click / Tap / Keypress
  │
  ├── Input Delay ──────── (main thread занят другими задачами)
  │
  ├── Processing Time ───── (выполнение event handler'ов)
  │
  └── Presentation Delay ── (style recalc → layout → paint → composite)
  │
Next Visual Update
```

<span class="text-gray">Все три фазы суммируются. Input delay — убирайте long tasks; processing — оптимизируйте обработчики; presentation — уменьшайте DOM и сложность стилей.</span>

---

<b class="text-on-semi-light">Что ухудшает INP</b>

| Проблема | Фаза | Решение |
|---|---|---|
| Long tasks (>50ms) | Input Delay | Code splitting, `scheduler.yield()`, web workers |
| Тяжёлые event handlers | Processing Time | Debounce, виртуализация, оптимизация логики |
| Большой DOM (>1500 элементов) | Presentation Delay | Виртуализация списков, `content-visibility` |
| Дорогие layout/paint | Presentation Delay | CSS containment, composite-only анимации |

<b class="text-on-semi-light">Код: оптимизация INP</b>

```js
// ❌ Long task блокирует main thread
button.addEventListener('click', () => {
  processHeavyData(); // 200ms синхронной работы
  updateUI();
});

// ✅ Разбиваем через scheduler.yield()
button.addEventListener('click', async () => {
  updateUI(); // визуальная обратная связь СРАЗУ
  await scheduler.yield(); // уступаем браузеру для рендеринга
  processHeavyData(); // тяжёлая работа после paint
});

// ✅ Альтернатива: requestIdleCallback для некритичной работы
button.addEventListener('click', () => {
  updateUI();
  requestIdleCallback(() => processHeavyData());
});
```

<b class="text-on-semi-light">Yield-паттерн для обработки большого массива</b>

```js
async function processInChunks(items, chunkSize = 50) {
  for (let i = 0; i < items.length; i += chunkSize) {
    const chunk = items.slice(i, i + chunkSize);
    chunk.forEach(processItem);
    if (i + chunkSize < items.length) {
      await scheduler.yield(); // уступаем main thread после каждого chunk
    }
  }
}
```

> INP отличает «просто работающий» сайт от действительно отзывчивого. FID измерял только первое взаимодействие и игнорировал все последующие проблемы.

## CLS (Cumulative Layout Shift) — глубокое погружение

<b class="text-on-semi-light">Что измеряет</b>

<span class="text-gray">Визуальная стабильность — насколько сильно элементы «прыгают» на странице. Каждый незапланированный сдвиг оценивается и суммируется через «session windows».</span>

```text
Layout Shift Score = Impact Fraction × Distance Fraction

Impact Fraction — доля viewport, затронутая сдвигом
Distance Fraction — максимальное расстояние сдвига / высота viewport
```

| Оценка | Значение |
|---|---|
| Хорошо (Good) | ≤ 0.1 |
| Нужно улучшить | ≤ 0.25 |
| Плохо (Poor) | > 0.25 |

---

<b class="text-on-semi-light">Частые причины CLS</b>

| Причина | Пример |
|---|---|
| Изображения/видео без размеров | `<img>` без `width`/`height` |
| Рекламные блоки без зарезервированного места | Ad slot подгружается и сдвигает контент |
| Динамически вставленный контент | Баннер вверху страницы после загрузки |
| Веб-шрифты (FOIT/FOUT) | Шрифт загружается, текст перерисовывается |

<b class="text-on-semi-light">Код: устранение CLS</b>

```html
<!-- ✅ Всегда указывайте размеры изображений -->
<img src="/photo.webp" width="800" height="600" alt="Photo"
     style="aspect-ratio: 800/600; width: 100%; height: auto;">
```

```css
/* ✅ Резервируем место для рекламного блока */
.ad-slot {
  min-height: 250px;
  contain: layout;
}

/* ✅ Стратегия загрузки шрифта без CLS */
@font-face {
  font-family: 'CustomFont';
  src: url('/font.woff2') format('woff2');
  font-display: optional; /* не вызывает layout shift */
  size-adjust: 105%;       /* подгоняем метрики fallback-шрифта */
}

/* ✅ content-visibility для off-screen контента */
.below-fold-section {
  content-visibility: auto;
  contain-intrinsic-size: 0 500px;
}
```

<span class="text-gray">Сдвиги в течение 500ms после user input (клик, нажатие клавиши) **не учитываются** в CLS — пользователь ожидает изменения после своего действия.</span>

## Сводная таблица Core Web Vitals

| Метрика | Что измеряет | Хорошо | Допустимо | Плохо | Тип |
|---|---|---|---|---|---|
| **LCP** | Скорость загрузки основного контента | ≤ 2.5s | ≤ 4s | > 4s | Loading |
| **INP** | Отзывчивость на все взаимодействия | ≤ 200ms | ≤ 500ms | > 500ms | Interactivity |
| **CLS** | Визуальная стабильность | ≤ 0.1 | ≤ 0.25 | > 0.25 | Visual Stability |

<span class="text-gray">Эти три метрики формируют Core Web Vitals с 2024 года (INP заменил FID). Google использует 75-й перцентиль (p75) field data для оценки — 75% посещений должны быть «зелёными».</span>

## Дополнительные метрики производительности

<b class="text-on-semi-light">TTFB (Time to First Byte)</b>

<span class="text-gray">Время от запроса до получения первого байта ответа сервера. Напрямую влияет на LCP. Хорошо: ≤ 800ms.</span>

<b class="text-on-semi-light">FCP (First Contentful Paint)</b>

<span class="text-gray">Момент появления первого контента (текст, изображение, SVG) на экране. Хорошо: ≤ 1.8s.</span>

<b class="text-on-semi-light">TBT (Total Blocking Time)</b>

<span class="text-gray">Lab-метрика: сумма «блокирующего времени» всех long tasks между FCP и TTI. Коррелирует с INP.</span>

```text
Task 1: 30ms  → blocking time: 0ms   (< 50ms threshold)
Task 2: 80ms  → blocking time: 30ms  (80 - 50)
Task 3: 120ms → blocking time: 70ms  (120 - 50)
                                ─────
TBT = 0 + 30 + 70 = 100ms
```

<b class="text-on-semi-light">TTI (Time to Interactive)</b>

<span class="text-gray">Момент, когда страница полностью интерактивна (deprecated в Lighthouse v11, но спрашивают на собеседованиях). Определяется как момент после FCP, когда нет long tasks в течение 5 секунд.</span>

<b class="text-on-semi-light">Speed Index</b>

<span class="text-gray">Как быстро контент визуально заполняет viewport. Рассчитывается по видеозаписи загрузки.</span>

<b class="text-on-semi-light">Long Animation Frames (LoAF)</b>

<span class="text-gray">Новый API, более детальный чем Long Tasks. Захватывает конкретные скрипты, время рендеринга, блокировку main thread. Помогает диагностировать причины плохого INP.</span>

```js
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.log('LoAF duration:', entry.duration);
    console.log('Blocking duration:', entry.blockingDuration);
    for (const script of entry.scripts) {
      console.log('Script:', script.sourceURL, script.duration);
    }
  }
});
observer.observe({ type: 'long-animation-frame', buffered: true });
```

## RUM vs Synthetic — senior-тема

<b class="text-on-semi-light">Два подхода к измерению производительности</b>

| Аспект | RUM (Field Data) | Synthetic (Lab Data) |
|---|---|---|
| **Источник** | Реальные пользователи | Контролируемая среда |
| **Вариативность** | Высокая (разные устройства, сети) | Низкая (воспроизводимо) |
| **Метрики** | LCP, INP, CLS, TTFB, FCP | LCP, CLS, TBT, FCP, SI |
| **Инструменты** | CrUX, web-vitals + аналитика | Lighthouse, WebPageTest |
| **Когда использовать** | Оценка реального опыта | Debugging, CI/CD gates |

<span class="text-gray">Почему расходятся: lab-тесты на мощном железе с быстрым интернетом. Реальные пользователи — на 3G с бюджетными телефонами. **Всегда** принимайте решения на основе field data.</span>

---

<b class="text-on-semi-light">Attribution builds — диагностика причин</b>

<span class="text-gray">Библиотека `web-vitals` имеет attribution-режим, объясняющий **почему** метрика плохая: какой элемент, какой ресурс, какая фаза.</span>

```js
import { onLCP } from 'web-vitals/attribution';

onLCP((metric) => {
  console.log('LCP element:', metric.attribution.element);
  console.log('LCP resource:', metric.attribution.url);
  console.log('TTFB:', metric.attribution.timeToFirstByte);
  console.log('Resource load delay:', metric.attribution.resourceLoadDelay);
  console.log('Element render delay:', metric.attribution.elementRenderDelay);
});
```

> Attribution builds увеличивают размер бандла — используйте для диагностики, а не в продакшене. При обнаружении проблемы включайте attribution для процента пользователей.

<b class="text-on-semi-light">Обзор инструментов</b>

| Инструмент | Тип | Метрики | Использование |
|---|---|---|---|
| **Lighthouse** | Lab | LCP, CLS, TBT, FCP, SI | Audit, CI/CD |
| **PageSpeed Insights** | Lab + Field | CWV + диагностика | Быстрая проверка |
| **Chrome DevTools → Performance** | Lab | Flame chart, long tasks | Глубокий debugging |
| **CrUX** (Chrome UX Report) | Field | CWV по реальным пользователям | BigQuery, API |
| **web-vitals** (JS library) | Field | LCP, INP, CLS, FCP, TTFB | RUM интеграция |
| **WebPageTest** | Lab | Waterfall, filmstrip | Детальный анализ |

---

<b class="text-on-semi-light">Интеграция web-vitals в проект</b>

```js
import { onLCP, onINP, onCLS, onFCP, onTTFB } from 'web-vitals';

function sendToAnalytics(metric) {
  const body = JSON.stringify({
    name: metric.name,
    value: metric.value,
    rating: metric.rating, // 'good' | 'needs-improvement' | 'poor'
    delta: metric.delta,
    id: metric.id,
    navigationType: metric.navigationType,
  });

  // navigator.sendBeacon для надёжной отправки при закрытии страницы
  if (navigator.sendBeacon) {
    navigator.sendBeacon('/analytics', body);
  } else {
    fetch('/analytics', { body, method: 'POST', keepalive: true });
  }
}

onLCP(sendToAnalytics);
onINP(sendToAnalytics);
onCLS(sendToAnalytics);
onFCP(sendToAnalytics);
onTTFB(sendToAnalytics);
```

## Оптимизация на практике — чек-лист

<b class="text-on-semi-light">LCP</b>

- [ ] `fetchpriority="high"` и `<link rel="preload">` для LCP-ресурса
- [ ] Никогда `loading="lazy"` на LCP-элементе
- [ ] Critical CSS inline в `<head>`, остальное — `defer`/`async`
- [ ] Оптимизация TTFB (CDN, кэширование)
- [ ] Современные форматы изображений (WebP, AVIF)

<b class="text-on-semi-light">INP</b>

- [ ] Разбивать long tasks через `scheduler.yield()` или `setTimeout`
- [ ] Визуальный feedback первым, тяжёлая работа — потом
- [ ] Виртуализация больших списков, `content-visibility: auto`
- [ ] Перенос тяжёлых вычислений в Web Workers

<b class="text-on-semi-light">CLS</b>

- [ ] `width` и `height` на всех `<img>` и `<video>`
- [ ] Зарезервированное место для рекламы и embed
- [ ] `font-display: optional` или `swap` + `size-adjust`
- [ ] Не вставлять контент выше текущего viewport

## Типичные вопросы на собеседовании

<b class="text-on-semi-light">Q: Чем INP отличается от FID?</b>

<span class="text-gray">FID измерял только **первое** взаимодействие (input delay). INP учитывает **все** взаимодействия, включает все три фазы (input delay + processing + presentation delay) и берёт p98 — почти наихудшее. INP гораздо строже и полнее.</span>

<b class="text-on-semi-light">Q: Как вы диагностируете плохой LCP на проекте?</b>

<span class="text-gray">1) Смотрю CrUX/RUM данные для определения масштаба. 2) PageSpeed Insights — вижу LCP-элемент и под-части. 3) web-vitals с attribution build для field-диагностики. 4) DevTools Performance — записываю загрузку, ищу bottleneck. 5) Исправляю и отслеживаю метрику в RUM.</span>

<b class="text-on-semi-light">Q: Можно ли доверять только Lighthouse-оценке?</b>

<span class="text-gray">Нет. Lighthouse — lab-метрика на мощном железе. Реальные пользователи могут иметь совсем другие результаты. Lighthouse подходит для debugging и CI/CD gates, но решения принимайте на основе field data (CrUX, RUM).</span>

<b class="text-on-semi-light">Q: Что такое Long Tasks и как они влияют на INP?</b>

<span class="text-gray">Long Task — задача на main thread > 50ms. Браузер не может обработать пользовательский ввод, что увеличивает input delay (первую фазу INP). Решение: разбивать задачи через yielding, использовать Web Workers.</span>
