# Progressive Web Apps

PWA — это веб-приложения, использующие современные API (Service Workers, Web App Manifest, Cache API) для обеспечения нативного опыта: работа offline, установка на устройство, push-уведомления. Для senior-разработчика ключевые темы — жизненный цикл Service Worker, стратегии кэширования и их tradeoff'ы.

> PWA не является конкретным фреймворком или библиотекой — это набор паттернов и API, которые делают веб-приложение надёжным, быстрым и «устанавливаемым». На собеседованиях ожидают глубокое понимание Service Worker lifecycle и осознанного выбора caching strategy.

---

## Что такое PWA

<b class="text-on-semi-light">Критерии установки (Installability)</b>

<span class="text-gray">Чтобы браузер предложил установку (Install prompt), приложение должно соответствовать трём условиям:</span>

1. **HTTPS** — Service Worker работает только по защищённому протоколу (или `localhost`)
2. **Web App Manifest** — валидный `manifest.json` с обязательными полями (`name`, `icons`, `start_url`, `display`)
3. **Service Worker** — зарегистрированный SW с обработчиком `fetch`

<b class="text-on-semi-light">PWA vs Native vs Web — capabilities vs reach</b>

| Характеристика | Web (обычный сайт) | PWA | Native App |
|---|---|---|---|
| Reach (охват) | Максимальный (URL) | Высокий (URL + install) | Низкий (app store) |
| Offline | Нет | Да (Service Worker) | Да |
| Push-уведомления | Нет | Да (Web Push) | Да |
| Установка на устройство | Нет | Да (Add to Home Screen) | Да (app store) |
| Доступ к hardware | Ограничен | Расширяется (Project Fugu) | Полный |
| Обновления | Мгновенные | Мгновенные (SW update) | Через app store |
| Размер | 0 (streaming) | Маленький (кэш) | Большой (bundle) |
| Discoverability | SEO, ссылки | SEO, ссылки, store | Только store |

<span class="text-gray">PWA занимает «золотую середину» — высокий reach веба + capabilities, приближающиеся к нативным. Google, Microsoft, Starbucks, Twitter Lite, Pinterest — все используют PWA в production.</span>

## Service Worker Lifecycle

<b class="text-on-semi-light">Жизненный цикл — критически важная тема для senior</b>

<span class="text-gray">Service Worker — это JavaScript-файл, работающий в отдельном потоке (не в main thread). Он перехватывает сетевые запросы, управляет кэшем и обрабатывает push-уведомления. Понимание его lifecycle — ключевое для собеседования.</span>

```text
                    ┌──────────────────────────────────────┐
                    │          Service Worker Lifecycle      │
                    └──────────────────────────────────────┘

  register()          install           activate           fetch
     │                  │                  │                  │
     ▼                  ▼                  ▼                  ▼
┌─────────┐     ┌──────────────┐   ┌──────────────┐   ┌──────────┐
│ Parsed  │────▶│  Installing  │──▶│  Activating  │──▶│  Active  │
└─────────┘     └──────────────┘   └──────────────┘   └──────────┘
                     │    │              │                   │
                     │    ▼              │                   ▼
                     │  Error ──▶ Redundant           Handles fetch,
                     │                                push, sync
                     ▼
                  Waiting
               (если старый SW
                ещё активен)
```

### Этапы жизненного цикла

| Этап | Событие | Что происходит |
|---|---|---|
| **Registration** | — | Браузер скачивает и парсит SW-файл |
| **Installing** | `install` | Предзаполнение кэша (precaching) |
| **Waiting** | — | Новый SW ждёт, пока все вкладки со старым SW закроются |
| **Activating** | `activate` | Очистка старых кэшей, подготовка к работе |
| **Active** | `fetch`, `push`, `sync` | SW перехватывает запросы и события |
| **Redundant** | — | SW заменён новым или произошла ошибка при install |

### Регистрация Service Worker

```js
// main.js — регистрация SW
if ('serviceWorker' in navigator) {
  window.addEventListener('load', async () => {
    try {
      const registration = await navigator.serviceWorker.register('/sw.js', {
        scope: '/', // область действия SW
      });
      console.log('SW registered:', registration.scope);
    } catch (error) {
      console.error('SW registration failed:', error);
    }
  });
}
```

<span class="text-gray">Регистрацию SW рекомендуется делать после `load` event, чтобы не конкурировать с загрузкой критических ресурсов страницы.</span>

### install event — precaching

```js
// sw.js
const CACHE_NAME = 'app-v1';
const PRECACHE_URLS = [
  '/',
  '/index.html',
  '/styles.css',
  '/app.js',
  '/offline.html',
];

self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then((cache) => {
      return cache.addAll(PRECACHE_URLS);
    })
  );
});
```

### activate event — очистка старых кэшей

```js
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then((cacheNames) => {
      return Promise.all(
        cacheNames
          .filter((name) => name !== CACHE_NAME)
          .map((name) => caches.delete(name))
      );
    })
  );
});
```

### skipWaiting() и clients.claim()

<b class="text-on-semi-light">Опасные, но необходимые инструменты</b>

<span class="text-gray">`skipWaiting()` заставляет новый SW пропустить фазу waiting и сразу перейти в active. `clients.claim()` позволяет активному SW взять под контроль все открытые вкладки, не дожидаясь перезагрузки.</span>

```js
// В install event
self.addEventListener('install', (event) => {
  self.skipWaiting(); // пропустить waiting
  event.waitUntil(/* precaching */);
});

// В activate event
self.addEventListener('activate', (event) => {
  event.waitUntil(
    clients.claim() // взять контроль над всеми вкладками
  );
});
```

> **Подводные камни:** `skipWaiting()` + `clients.claim()` означает, что страница, загруженная старым SW, внезапно переключится на новый. Если новый SW меняет формат кэша или API-ответов — возможны баги. Используйте осознанно, с промежуточной проверкой совместимости.

### Обновление Service Worker

<span class="text-gray">Браузер проверяет обновления SW при каждой навигации (или вручную через `registration.update()`). Если файл SW отличается хотя бы на один байт (byte-diff) — запускается новый lifecycle. Даже если вы установили `Cache-Control` на SW-файл, браузер не будет кэшировать его дольше 24 часов (24h max cache для SW).</span>

## Стратегии кэширования

<b class="text-on-semi-light">Senior-critical: выбор стратегии — один из самых частых вопросов</b>

| Стратегия | Когда использовать | Плюсы | Минусы |
|---|---|---|---|
| **Cache First** | Статика (CSS, JS, шрифты, изображения) | Мгновенный ответ, работает offline | Нужна версионность для обновлений |
| **Network First** | API-данные, динамический контент | Всегда свежие данные | Медленнее при плохой сети |
| **Stale-While-Revalidate** | Баланс (контент, аватары, non-critical API) | Быстрый + обновляется в фоне | Пользователь видит stale данные |
| **Network Only** | Аналитика, POST-запросы, real-time данные | Никакого кэша | Не работает offline |
| **Cache Only** | Precached ресурсы, которые никогда не меняются | Предсказуемо | Нет fallback на сеть |

### Cache First

```js
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then((cached) => {
      return cached || fetch(event.request).then((response) => {
        const clone = response.clone();
        caches.open(CACHE_NAME).then((cache) => cache.put(event.request, clone));
        return response;
      });
    })
  );
});
```

### Network First

```js
self.addEventListener('fetch', (event) => {
  event.respondWith(
    fetch(event.request)
      .then((response) => {
        const clone = response.clone();
        caches.open(CACHE_NAME).then((cache) => cache.put(event.request, clone));
        return response;
      })
      .catch(() => caches.match(event.request))
  );
});
```

### Stale-While-Revalidate

```js
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.open(CACHE_NAME).then((cache) => {
      return cache.match(event.request).then((cached) => {
        const fetchPromise = fetch(event.request).then((response) => {
          cache.put(event.request, response.clone());
          return response;
        });
        return cached || fetchPromise;
      });
    })
  );
});
```

<span class="text-gray">Stale-While-Revalidate — пользователь мгновенно получает кэшированную версию, а в фоне SW обновляет кэш. При следующем запросе данные уже будут свежими. Это лучшая стратегия для «почти статического» контента.</span>

### Cache Invalidation

<b class="text-on-semi-light">Инвалидация кэша — «одна из двух сложных проблем CS»</b>

- **Versioned URLs** (`app.a1b2c3.js`) — лучший подход, кэш навсегда, новая версия = новый URL
- **Cache-Control interaction** — SW-кэш и HTTP-кэш — это разные уровни; SW fetch сначала проходит через HTTP cache
- **Versioned cache names** (`cache-v2`) — при обновлении SW в `activate` удаляем старые кэши
- **Runtime caching с TTL** — проверять `Date` header и очищать старые записи

## Offline

<b class="text-on-semi-light">Offline fallback — базовый паттерн</b>

```js
// Показать offline-страницу, если нет сети и нет кэша
self.addEventListener('fetch', (event) => {
  if (event.request.mode === 'navigate') {
    event.respondWith(
      fetch(event.request).catch(() => {
        return caches.match('/offline.html');
      })
    );
  }
});
```

<b class="text-on-semi-light">Background Sync API</b>

<span class="text-gray">Позволяет отложить действие (отправку формы, like, комментарий) до восстановления сети. Браузер гарантирует выполнение sync event, даже если пользователь закрыл вкладку.</span>

```js
// В main.js — регистрация sync
async function sendMessage(data) {
  // Сохранить данные в IndexedDB
  await saveToIndexedDB('outbox', data);

  // Запросить background sync
  const registration = await navigator.serviceWorker.ready;
  await registration.sync.register('send-messages');
}

// В sw.js — обработка sync event
self.addEventListener('sync', (event) => {
  if (event.tag === 'send-messages') {
    event.waitUntil(
      getFromIndexedDB('outbox').then((messages) => {
        return Promise.all(
          messages.map((msg) =>
            fetch('/api/messages', {
              method: 'POST',
              body: JSON.stringify(msg),
            }).then(() => removeFromIndexedDB('outbox', msg.id))
          )
        );
      })
    );
  }
});
```

<b class="text-on-semi-light">Periodic Background Sync</b>

<span class="text-gray">Позволяет периодически обновлять данные в фоне (новости, статьи). Доступен только для установленных PWA с достаточным engagement score. Минимальный интервал — обычно 12 часов.</span>

```js
// Регистрация периодической синхронизации
const registration = await navigator.serviceWorker.ready;
await registration.periodicSync.register('update-news', {
  minInterval: 12 * 60 * 60 * 1000, // 12 часов
});

// В sw.js
self.addEventListener('periodicsync', (event) => {
  if (event.tag === 'update-news') {
    event.waitUntil(fetchAndCacheLatestNews());
  }
});
```

<b class="text-on-semi-light">Display modes</b>

## Push Notifications

<b class="text-on-semi-light">Поток push-уведомлений</b>

```text
1. Запрос разрешения           → Notification.requestPermission()
2. Подписка на push            → PushManager.subscribe() → получить endpoint
3. Отправка endpoint на сервер → POST /api/subscribe
4. Сервер отправляет push      → web-push library → push service
5. SW получает push event      → self.addEventListener('push', ...)
6. SW показывает notification  → self.registration.showNotification()
7. Пользователь кликает        → notificationclick event
```

```js
// main.js — подписка
async function subscribeToPush() {
  const permission = await Notification.requestPermission();
  if (permission !== 'granted') return;

  const registration = await navigator.serviceWorker.ready;
  const subscription = await registration.pushManager.subscribe({
    userVisibleOnly: true, // обязательно — пуш должен показать уведомление
    applicationServerKey: urlBase64ToUint8Array(VAPID_PUBLIC_KEY),
  });

  // Отправить subscription на сервер
  await fetch('/api/push/subscribe', {
    method: 'POST',
    body: JSON.stringify(subscription),
    headers: { 'Content-Type': 'application/json' },
  });
}

// sw.js — обработка push
self.addEventListener('push', (event) => {
  const data = event.data?.json() ?? { title: 'Notification', body: '' };

  event.waitUntil(
    self.registration.showNotification(data.title, {
      body: data.body,
      icon: '/icons/icon-192.png',
      badge: '/icons/badge-72.png',
      actions: [
        { action: 'open', title: 'Открыть' },
        { action: 'dismiss', title: 'Закрыть' },
      ],
    })
  );
});

// sw.js — клик по уведомлению
self.addEventListener('notificationclick', (event) => {
  event.notification.close();

  if (event.action === 'open') {
    event.waitUntil(clients.openWindow('/'));
  }
});
```

<span class="text-gray">**VAPID keys** (Voluntary Application Server Identification) — пара ключей (public + private), которая идентифицирует ваш сервер перед push-сервисом. Public key передаётся при подписке, private key — используется при отправке push с сервера. Это предотвращает несанкционированную отправку push от чужих серверов.</span>

## Navigation Preload

<b class="text-on-semi-light">Устранение задержки запуска SW при навигации</b>

<span class="text-gray">Когда пользователь переходит по ссылке, браузер сначала запускает SW, затем SW обрабатывает fetch event. Запуск SW может занять 50-500ms. Navigation Preload позволяет начать сетевой запрос параллельно с запуском SW, устраняя эту задержку.</span>

```js
// В activate event
self.addEventListener('activate', (event) => {
  event.waitUntil(
    (async () => {
      if (self.registration.navigationPreload) {
        await self.registration.navigationPreload.enable();
      }
    })()
  );
});

// В fetch event — использовать preloadResponse
self.addEventListener('fetch', (event) => {
  if (event.request.mode === 'navigate') {
    event.respondWith(
      (async () => {
        // Попробовать preloaded response
        const preloadResponse = await event.preloadResponse;
        if (preloadResponse) return preloadResponse;

        // Fallback на сеть
        try {
          return await fetch(event.request);
        } catch {
          return caches.match('/offline.html');
        }
      })()
    );
  }
});
```

## Ограничения PWA

<b class="text-on-semi-light">iOS — главная головная боль PWA-разработчиков</b>

| Ограничение | iOS (Safari / WebKit) | Android (Chrome) |
|---|---|---|
| Push-уведомления | Только с iOS 16.4+ (и только installed PWA) | Полная поддержка |
| Background Sync | Не поддерживается | Поддерживается |
| Periodic Background Sync | Не поддерживается | Ограничено (engagement) |
| Install prompt (`beforeinstallprompt`) | Нет — только ручной Add to Home Screen | Да |
| Storage persistence | Агрессивное удаление (7 дней без визита) | Persistent Storage API |
| Badge API | Не поддерживается | Поддерживается |
| Web Bluetooth, NFC, USB | Нет | Частично |

<span class="text-gray">Safari/WebKit на iOS — единственный допустимый движок (до EU DMA). Все браузеры на iOS используют WebKit под капотом, поэтому ограничения Safari = ограничения всех iOS-браузеров. С iOS 17.4 в EU появилась возможность альтернативных движков, но adoption пока минимальный.</span>

<b class="text-on-semi-light">Storage quotas</b>

- Chrome: до 80% доступного дискового пространства (на origin)
- Safari: ~1GB, агрессивная очистка после 7 дней без использования
- `navigator.storage.estimate()` — проверить доступный объём
- `navigator.storage.persist()` — запросить persistent storage (не удалять при нехватке места)

## Когда PWA, а когда нет

| Тип приложения | Рекомендация | Почему |
|---|---|---|
| Контентный сайт (блог, документация) | SSG / SSR | Контент статический, SEO важнее offline |
| Dashboard (аналитика, admin) | SPA | Всегда online, не нужен offline |
| E-commerce | PWA (рассмотреть) | Offline-каталог, push для акций, быстрый повторный визит |
| Field workers (инспекторы, курьеры) | PWA (сильная рекомендация) | Работа в зонах без сети, offline forms, background sync |
| Социальная сеть | PWA | Push, offline лента, быстрый запуск |
| Игра / мультимедиа | Native или PWA + WASM | Нужен доступ к GPU, сенсорам |
| Банковское приложение | Native | Требования безопасности, биометрия, app store distribution |

> PWA — не серебряная пуля. Выбор архитектуры зависит от целевой аудитории (iOS vs Android ratio), требований offline и уровня интеграции с ОС. На собеседовании важно показать, что вы понимаете tradeoff'ы, а не просто «за» PWA.
