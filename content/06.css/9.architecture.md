# CSS-архитектура

На senior-собесе спрашивают не только «как написать стили», но и **как организовать CSS** для масштабируемого проекта. BEM, CSS Modules, CSS-in-JS, utility-first — у каждого подхода свои trade-offs. Понимание этих подходов и обоснованный выбор — то, что показывает senior-мышление.

---

## Проблема масштабирования CSS

CSS по умолчанию **глобальный**: любое правило может повлиять на любой элемент. При росте проекта это приводит к:

- **Naming collisions** — два разработчика создают `.button` с разными стилями
- **Specificity wars** — каскад `!important` и длинных селекторов
- **Dead code** — неиспользуемый CSS накапливается, никто не решается удалить
- **Side effects** — изменение одного правила ломает другой компонент
- **Непредсказуемость** — порядок импорта влияет на результат

> Все подходы ниже решают одну проблему: **как ограничить scope CSS** и сделать стили предсказуемыми.

## Методологии именования

### BEM (Block Element Modifier)

Конвенция именования: `.block__element--modifier`

- **Block** — самостоятельный компонент: `.card`, `.nav`, `.form`
- **Element** — часть блока, не имеет смысла вне его: `.card__title`, `.card__image`
- **Modifier** — вариация блока или элемента: `.card--featured`, `.card__title--large`

```css
/* Block */
.card { padding: 16px; border-radius: 8px; }

/* Elements */
.card__title { font-size: 1.25rem; font-weight: bold; }
.card__description { color: gray; }
.card__image { width: 100%; border-radius: 8px 8px 0 0; }

/* Modifiers */
.card--featured { border: 2px solid var(--primary); }
.card--compact { padding: 8px; }
.card__title--large { font-size: 1.5rem; }
```

**Плохо — неконсистентное именование:**

```css
.cardTitle { ... }
.card-title { ... }
.card .title { ... }  /* зависимость от DOM-структуры */
#card .title { ... }  /* высокая специфичность */
```

**Хорошо — BEM:**

```css
.card__title { ... }       /* плоская специфичность (0,1,0) */
.card__title--large { ... } /* легко переопределить */
```

<b class="text-on-semi-light">Преимущества BEM:</b>

- **Плоская специфичность** — все селекторы (0,1,0), нет войн
- **Самодокументирующийся** — `.card__title--large` сразу понятен
- **Изоляция** — стили блока не влияют на другие блоки

<b class="text-on-semi-light">Недостатки:</b>

- Verbose: длинные имена классов
- Не предотвращает naming collisions технически (только по конвенции)
- Ручное обеспечение уникальности

### Другие методологии

| Методология | Идея | Особенность |
|---|---|---|
| **SMACSS** | Категории: base, layout, module, state, theme | Разделение по назначению |
| **OOCSS** | Разделение structure и skin | Максимальное переиспользование |
| **ITCSS** | Inverted Triangle: от generic к specific | Контроль каскада через порядок |
| **Atomic CSS** | Один класс = одно свойство | Прообраз utility-first |

<span class="text-gray">На практике BEM — самая распространённая методология. ITCSS хорошо сочетается с BEM. SMACSS/OOCSS встречаются реже. Atomic CSS эволюционировал в Tailwind.</span>

## CSS Modules

**CSS Modules** — build-time решение. Каждый CSS-файл скоупится к компоненту через уникальные hash-имена классов.

```css
/* Card.module.css */
.title { font-size: 1.25rem; }
.featured { border: 2px solid var(--primary); }
```

```tsx
// React component
import styles from './Card.module.css';

function Card({ featured }) {
  return (
    <div className={featured ? styles.featured : ''}>
      <h2 className={styles.title}>Заголовок</h2>
    </div>
  );
}
```

```html
<!-- Результат в DOM -->
<h2 class="Card_title_a1b2c3">Заголовок</h2>
```

### `composes` — композиция

```css
/* shared.module.css */
.base { padding: 16px; border-radius: 8px; }

/* Card.module.css */
.card { composes: base from './shared.module.css'; background: white; }
```

| Преимущества | Недостатки |
|---|---|
| Автоматический scoping (hash) | Нет dynamic styles (нужен className toggle) |
| Zero runtime cost | Build-time dependency (bundler) |
| Работает с обычным CSS | TypeScript type safety требует setup |
| Нет naming collisions | Отладка: классы = хеши |

<b class="text-on-semi-light">На фронте:</b> CSS Modules — стандарт в Next.js, Vite, CRA. Nuxt использует `<style scoped>` (аналогичный принцип с hash-атрибутами вместо hash-классов).

## CSS-in-JS

Стили пишутся **в JavaScript**, генерируются runtime или build-time.

### Runtime (styled-components, emotion)

```tsx
import styled from 'styled-components';

const Card = styled.div<{ $featured?: boolean }>`
  padding: 16px;
  border-radius: 8px;
  border: ${p => p.$featured ? '2px solid var(--primary)' : 'none'};
`;

const Title = styled.h2`
  font-size: 1.25rem;
  color: ${p => p.theme.text};
`;
```

### Zero-runtime (vanilla-extract, Panda CSS)

```ts
// Card.css.ts (vanilla-extract)
import { style } from '@vanilla-extract/css';

export const card = style({
  padding: 16,
  borderRadius: 8,
});

export const featured = style({
  border: '2px solid var(--primary)',
});
```

### Trade-offs

| Подход | Scoping | Runtime cost | Dynamic styles | SSR | DX |
|---|---|---|---|---|---|
| **Runtime** (styled-components) | Авто | Да (inject `<style>`) | Полная (props) | Требует setup | Отличный |
| **Zero-runtime** (vanilla-extract) | Авто | Нет | Через variants | Встроенный | Хороший |

<span class="text-gray">Тренд: уход от runtime CSS-in-JS в сторону zero-runtime (vanilla-extract, Panda CSS) и utility-first (Tailwind). Runtime CSS-in-JS (styled-components) добавляет ~12–15KB к bundle и замедляет SSR из-за style injection.</span>

## Utility-first CSS

Философия: вместо написания CSS-классов для компонентов — **комбинируем утилитарные классы** прямо в разметке.

### Tailwind CSS / UnoCSS

```html
<!-- Utility-first -->
<div class="p-4 rounded-lg border-2 border-primary bg-white shadow-md hover:shadow-lg transition-shadow">
  <h2 class="text-xl font-bold text-gray-900">Заголовок</h2>
  <p class="mt-2 text-gray-600">Описание карточки</p>
</div>
```

### Extraction через `@apply` (Tailwind)

```css
/* Если утилиты повторяются — извлекаем в компонентный класс */
.card {
  @apply p-4 rounded-lg border-2 bg-white shadow-md hover:shadow-lg transition-shadow;
}
```

### Преимущества и недостатки

| Преимущества | Недостатки |
|---|---|
| Нет naming — не нужно придумывать имена | Verbose HTML |
| Consistency — значения из design system | Кривая обучения (запоминание классов) |
| Маленький bundle (purge неиспользуемых) | Сложнее дебажить (много классов) |
| Нет dead CSS (только используемое) | Стили привязаны к HTML |
| Быстрая разработка | Custom значения требуют config |

<b class="text-on-semi-light">На фронте:</b> Tailwind CSS — самый популярный utility-first framework. UnoCSS — быстрее (build-time), используется в этом проекте (Frontend Legends). Оба совместимы с React, Vue, Svelte, Angular.

## Design Tokens

**Design tokens** — абстрактные значения дизайн-системы (цвета, отступы, типографика), не привязанные к конкретной технологии.

```json
{
  "color": {
    "primary": { "value": "#e94560" },
    "background": { "value": "#ffffff" }
  },
  "spacing": {
    "sm": { "value": "8px" },
    "md": { "value": "16px" },
    "lg": { "value": "24px" }
  }
}
```

Трансформируются в CSS custom properties, SASS variables, JS constants, Tailwind config:

```css
:root {
  --color-primary: #e94560;
  --spacing-sm: 8px;
  --spacing-md: 16px;
}
```

<span class="text-gray">Инструменты: **Style Dictionary** (Amazon), **Tokens Studio** (Figma plugin). Sync: дизайнер определяет токены в Figma → экспорт в JSON → трансформация в CSS/JS → CI автоматически обновляет код.</span>

## Сравнение подходов

| Подход | Scoping | Runtime | Bundle | DX | Лучше для |
|---|---|---|---|---|---|
| **BEM** | Конвенция | 0 | Средний | Средний | Проекты без bundler, legacy |
| **CSS Modules** | Hash (build) | 0 | Средний | Хороший | React, Next.js, Vite |
| **CSS-in-JS (runtime)** | JS runtime | Да | Больше | Отличный | Dynamic theming, props-based |
| **CSS-in-JS (zero-rt)** | Build-time | 0 | Средний | Хороший | Type-safe, SSR-friendly |
| **Utility-first** | Purge (build) | 0 | **Маленький** | Быстрый | Rapid development, consistency |
| **`@layer` + vars** | Cascade | 0 | Минимальный | Нативный | Без build tools, простота |

## Архитектурные паттерны

### Layer structure

Структура стилей, которая предотвращает конфликты:

```css
@layer reset, tokens, base, components, utilities;

@layer reset { *, *::before, *::after { margin: 0; box-sizing: border-box; } }
@layer tokens { :root { --primary: #e94560; --radius: 8px; } }
@layer base { body { font-family: 'Inter', sans-serif; color: var(--text); } }
@layer components { .card { padding: 16px; border-radius: var(--radius); } }
@layer utilities { .hidden { display: none !important; } }
```

### Component-scoped

Каждый компонент владеет своими стилями: CSS Module, scoped style, или styled-component. Нет глобальных стилей кроме reset + tokens.

### Token-driven

Все значения берутся из custom properties / design tokens. Ни одного «магического числа» в коде:

```css
/* Плохо */
.card { padding: 16px; border-radius: 8px; color: #333; }

/* Хорошо */
.card { padding: var(--spacing-md); border-radius: var(--radius); color: var(--text); }
```

## На фронте: что выбрать

| Фреймворк | Рекомендуемые подходы |
|---|---|
| **React** | CSS Modules, Tailwind, styled-components, vanilla-extract |
| **Vue** | `<style scoped>`, UnoCSS, CSS Modules |
| **Angular** | Component styles (ViewEncapsulation), Tailwind |
| **Next.js** | CSS Modules (default), Tailwind |
| **Nuxt** | `<style scoped>`, UnoCSS (этот проект) |

<b class="text-on-semi-light">Нет «лучшего» подхода</b> — есть подходящий для контекста. Маленький проект? CSS Modules или utility-first. Дизайн-система для 5 команд? Design tokens + component-scoped. Legacy без bundler? BEM + @layer.

## Частые вопросы на собесе

**BEM в 2026 — всё ещё актуален?**
Да, для проектов без build tools и для legacy. В проектах с bundler'ом CSS Modules или utility-first удобнее — они обеспечивают scoping технически, а не по конвенции. BEM-мышление (block, element, modifier) полезно в любом подходе.

**CSS Modules vs CSS-in-JS?**
CSS Modules — zero runtime, build-time scoping, обычный CSS. CSS-in-JS (runtime) — dynamic styles от props, JS API, но runtime cost. Zero-runtime CSS-in-JS (vanilla-extract) — лучшее из обоих миров: type safety + zero runtime.

**Зачем utility-first, если это «стили в HTML»?**
Consistency (значения из design system), нет naming problem, нет dead CSS (purge), быстрая разработка. Verbose HTML — trade-off, решаемый extraction (`@apply`) и компонентными абстракциями.

**Как организовать global styles?**
`@layer` structure: reset → tokens → base → components → utilities. Reset и tokens — глобальные. Components — scoped (CSS Modules / scoped styles). Utilities — глобальные с `!important`.

**Что такое design tokens?**
Абстрактные значения дизайн-системы: цвета, отступы, шрифты. Определяются в JSON, трансформируются в CSS variables, SASS, JS. Single source of truth между дизайном (Figma) и кодом.
